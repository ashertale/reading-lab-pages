Exponential Backoff and Jitter

重試不是越快越可靠；指數退避降低壓力，抖動避免所有 client 在同一秒再次撞上去。

Orientation

重試也會成為負載

Exponential backoff 讓每次失敗後等待時間逐步增加；jitter 則把等待時間隨機化，避免群體同步。

Core Tension

單一 client 快速重試看起來合理，但上萬個 client 同時這樣做，故障中的服務會被重試洪峰壓得更久。

01 先分錯誤類型

永久錯誤不該重試；暫時錯誤才適合 backoff。

02 再增加等待

連續失敗代表環境可能仍未恢復，下一次嘗試應更謹慎。

03 最後加入 jitter

隨機化能讓重試分散，減少 synchronized retry storm。

Setup

一批裝置同時醒來搶連線

停電後網路恢復，幾千台設備同時重新連線；若每台都用固定間隔重試，AP 和後端會看到週期性尖峰。

最小場景

第一次連線失敗後，所有裝置都在 1 秒、2 秒、4 秒重試，尖峰仍然對齊。

真正難點

每台裝置都遵守同一個看似溫和的策略，整體卻形成同步攻擊般的負載形狀。

Key Point

好的 retry policy 要保護被呼叫端，也要保護 client 自己的電量、排程與狀態機。

Lenses

重試策略要同時管時間與副作用

退避只是其中一個旋鈕；還要看請求是否安全、是否冪等、以及失敗訊號是否可信。

錯誤分類

timeout、429、bus busy、auth failed 代表的處理方式不同。

退避上限

等待不能無限長；需要 max interval、max attempts 或 deadline。

Jitter 形狀

full jitter、equal jitter、decorrelated jitter 會產生不同重試分布。

冪等性

非冪等操作重試前要有 request id 或去重機制，否則可能重複扣款或重複寫入。

錯誤適合重試嗎策略
暫時 timeout 通常適合 退避、jitter、deadline
權限錯誤 通常不適合 刷新憑證或回報配置問題
server 429/503 看回應提示 尊重 Retry-After 並加入 jitter

Psychology

快點再試很像負責任

重試讓人感覺系統沒有放棄，但它也可能把局部等待變成全域塞車。

行動偏誤

工程師不喜歡失敗後什麼都不做，於是寫出太積極的 retry loop。

單機視角

在一台設備上測試很正常，上量後才出現同步尖峰。

成功案例遮蔽

短期重試成功會讓人忽略它在事故時放大了負載。

Retry review 要問的是整群 client 的時間分布，而不是單次 request 的勇敢程度。

Applications

從雲端 API 到韌體重連

重試策略跨越軟體層與硬體層；只要共享資源會短暫不可用，就需要管制重試節奏。

OTA download

下載失敗後分散重試，避免 CDN 或 server 在版本發布後承受同步尖峰。

Wi-Fi/BLE reconnect

掉線後加入 jitter，降低同區域設備同時掃描或連線造成的碰撞。

I2C/SPI 周邊忙碌

短退避可處理暫時 busy，但要設 deadline，避免狀態機卡死。

  • 把 retry budget 當成規格：最多幾次、多久放棄、如何回報給上層。
  • 在壓測中模擬同時失敗，而不是只測單一 client 的 happy retry。

Misreadings

退避不是把錯誤藏起來

重試可以吸收暫時性失敗，但不能修復永久配置、資料不一致或非冪等副作用。

失敗就一直 retry 校正

無限重試會消耗電力、頻寬與 server capacity，還會延後真正的錯誤處理。

固定 delay 夠簡單 校正

固定 delay 容易讓大量 client 對齊，形成週期性尖峰。

所有請求都能重試 校正

寫入、付款、狀態轉移需要 idempotency key 或去重設計。
