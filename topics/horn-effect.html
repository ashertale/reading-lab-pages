Horn Effect

一個明顯缺點會像污點一樣滲到其他欄位，讓我們把尚未檢查的能力、動機與可靠度一起扣分。

Orientation

壞印象最麻煩的地方，是它會替還沒被審查的欄位先寫答案

角效應不是叫你忽略缺點，而是提醒你：一個缺點可以是真實的，同時它的推論半徑也可能太大。

Core Tension

負面線索常有保護價值，所以我們不該把它全部消音；但只要它變成總評，後面的證據就會被迫替第一印象服務。

01 把缺點留在原欄位

先說清楚被扣分的是準時、品質、溝通還是風險控管，避免一句「不可靠」吃掉所有維度。

02 要求外溢提出證據

若要從一次失誤推到長期能力，必須補上時間、情境與重複模式，不讓直覺直接升級成結論。

03 設計恢復門檻

沒有可回到中性的條件，負面標籤就會變成永久狀態，而不是可被新證據修正的判斷。

Setup

一次事故之後，團隊開始審查的是人，還是下一個 patch

最小場景不是「有人犯錯」而已，而是錯誤被記住後，接下來每個訊號都被拿來證明那個人本來就有問題。

最小劇情

某位工程師曾在 release 中漏掉一個 migration case。三個月後他提出 rollback plan，討論還沒看細節，就有人先說「他上次不是也沒想周全」。

真正難點

壞紀錄不能刪掉，但也不能讓它取代這次設計本身的證據；難的是在會議壓力下守住這條線。

記住一次失誤
→
把模糊訊號讀成佐證
→
讓新證據失去翻案能力
Key Point

角效應最該被檢查的時刻，通常不是你很討厭某人時，而是你覺得自己「只是合理小心」時。

Lenses

三個問題可以把壞印象拆回可討論的證據

不要急著問「這個人到底好不好」，先問負面線索在哪裡成立、推到哪裡停止、什麼新證據可以改變判斷。

證據欄位

把扣分寫成具體欄位：例如測試覆蓋不足、估時過度樂觀或 escalation 太晚。欄位越清楚，外溢越難假裝成事實。

推論距離

從「這次失手」到「這人不可靠」中間至少隔著頻率、情境相似度與補救行動。跳過這些橋，判斷就已經越界。

翻案條件

若一個人無論做什麼都只能被視為補償而不是改善，評價系統其實已經拒絕學習新資料。

Psychology

負面外溢有保護感，所以它不像錯誤那麼容易被放下

人會優先保存壞印象，因為它看起來能降低未來損失；問題是保護感常常比證據跑得更快。

風險記憶比較黏

一次糟糕經驗會留下明確故事，後續中性表現卻很難留下同樣鮮明的反證，於是記憶本身就開始失衡。

一致性讓人省力

把一個人看成「本來就粗心」比每次重看情境更省力；這種一致敘事會讓矛盾證據變得不順眼。

群體也會接力貼標

團隊裡的耳語會把個人印象轉成共享背景，後來加入討論的人可能已經承接了結論，卻沒看過原始證據。

角效應需要的不是善意濾鏡，而是一套能讓壞印象被記錄、被限制、也能被新事實更新的評價紀律。

Applications

在工程現場，角效應最常混進人員評價與事故敘事

只要評價會影響權限、review 強度或升遷，負面外溢就不只是心理偏誤，而會改變後續工作的機會結構。

Code review

先遮住作者名字或至少先列 patch 風險，再談作者歷史。這能避免「他寫的我不放心」直接蓋過實際 diff。

Incident review

復盤要把行為、決策環境與系統護欄分開寫。若只留下「某人判斷差」，下一次仍然看不到流程怎麼讓錯誤變大。

Vendor selection

供應商一次回覆失誤可以扣分，但要明確對應 SLA、修復能力或溝通品質，不能順手推成所有技術能力都不足。

Misreadings

別把校正角效應誤會成替缺點開脫

這個概念要保護的是判斷比例，不是保護某個人免於被批評。

誤讀成「不要記仇」 校正

壞紀錄應該留下；需要改的是它能不能自動代表所有能力。

把警覺當成結論 校正

警覺可以提高檢查強度，但不能替代檢查結果。兩者混在一起，偏誤就變得很像負責任。

以為一次修正就夠 校正

角效應會在後續互動裡反覆回來，因此需要評價表、復盤紀錄與決策會議共同約束。

Questions

把壞印象放回可稽核的位置

扣分到底落在哪一欄

我能不能把這個負面判斷寫成具體行為，而不是一句人格或能力總評？

什麼證據能讓分數回升

如果沒有任何可回升條件，我是在評估現況，還是在維持一個已經結案的標籤？

延伸路線

Sources

核心概念

Horn Effect
相鄰概念
