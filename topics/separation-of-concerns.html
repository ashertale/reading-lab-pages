關注點分離

Separation of Concerns

關注點分離不是把檔案切小，而是讓不同變動理由不互相綁架。

Read For

理解責任邊界、變更理由、資料流和接口契約如何讓模組能獨立演進。

Orientation

真正要分開的是變動理由，不只是程式碼位置

關注點分離要求不同責任有清楚邊界，讓某一種改變不必牽動無關的判斷、資料或流程。

Core Tension

把東西放在一起能快速完成，但責任混雜後，任何小改都可能跨過多個概念，讓維護變成猜測。

01 用變更理由切邊界

同一模組裡的東西應該因相同原因改變。

02 讓接口承載語意

邊界不是隱藏所有東西，而是把必要的資料和責任用清楚契約傳遞。

03 避免雙向知道太多

如果兩邊都需要理解對方內部，邊界只是檔案分隔。

Setup

UI 按鈕直接改硬體暫存器，讓產品文案變更牽動 driver 風險

最小場景是展示、業務規則和底層操作混在同一條流程裡。

最小劇情

設定頁切換某個模式時，前端 handler 直接組 protocol bytes 並寫入裝置。後來 UI 文案和模式分類改名，竟然影響底層 command mapping。

真正難點

程式能跑不代表責任清楚；當變更理由不同的東西混在一起，測試和 review 都會擴散。

不同責任混在同處
→
小改跨過不相關層
→
維護成本和錯誤面增加
Key Point

關注點分離的成果，是讓某類變更有明確落點和有限影響範圍。

Lenses

用變更路徑檢查邊界是否有效

好的邊界會讓產品文案、業務規則、通訊協定和硬體控制各自有負責位置。

責任聚合

這段 code 同時處理展示、策略、持久化、通訊和硬體細節嗎？

變更半徑

改一個文案、規則或底層 protocol 時，會牽動哪些無關檔案？

接口方向

高層是否知道太多低層細節，低層是否反過來依賴產品語意？

Psychology

快速把東西串起來很有成就感，直到第二次變更到來

初次實作時混合責任最省路；真正成本通常在後續變更、測試和新人理解時出現。

短期速度有回報

把所有邏輯寫在同處能很快看到結果，讓邊界設計看起來像多餘。

抽象焦慮

團隊怕過度抽象，於是延後分離，最後責任已糾纏到難以拆開。

檔案分層假象

目錄看起來分層，不代表資料流和依賴方向真的分離。

檢查關注點分離時，請追一個變更需求會穿過哪些責任，而不是只看資料夾名稱。

Applications

讓不同責任有不同修改入口

前端狀態、韌體 driver 和服務架構都需要用變更理由設計邊界。

前端狀態

把展示狀態、server cache 和業務判斷分開，避免 UI component 變成所有邏輯入口。

韌體模組

讓 driver 處理硬體細節，policy layer 處理模式決策，避免產品語意滲入 register 操作。

服務架構

把驗證、授權、資料存取和外部 side effect 切出明確接口，讓測試可針對責任。

Misreadings

關注點分離不是越碎越好

切分要服務變更理由；任意拆小只會增加跳轉和接口噪音。

把檔案拆小當完成 校正

如果改同一需求仍要碰十個無關責任，拆檔沒有解決問題。

把所有耦合都視為壞 校正

必要資料流和明確契約是正常耦合，問題是隱性責任交纏。

一開始就大抽象 校正

可以從最常一起變或最常互相牽連的地方開始分離。

Questions

用下一次變更測試邊界

下面兩問適合用在模組 review 和重構前。

這段 code 有幾個變更理由？

列出文案、規則、資料、協定、硬體或操作流程各自為何會改。

改一個責任會碰到哪些無關責任？

若常跨層修改，就需要重新設計接口或責任分配。
