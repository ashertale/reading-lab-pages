垃圾桶模型

Garbage Can Model

垃圾桶模型描寫的是某些組織決策不是先有問題再找解法，而是問題、解法、參與者和機會剛好在某個時刻碰上。

Read For

理解組織混亂、注意力流動和解法等待問題時，決策如何被偶然組合塑形。

Orientation

有些決策不是解答問題，而是多條流在會議室裡相撞

垃圾桶模型把組織看成一個鬆散耦合的場域：問題、解法、人和決策窗口各自流動，碰在一起時就形成決策。

Core Tension

我們喜歡把決策寫成理性流程，但實際上常是某個方案早已存在，只是在等待能掛上的問題。

01 分辨四條流

問題、解法、參與者和選擇機會不一定同步出現。

02 檢查方案先行

若解法比問題更早被推銷，要小心問題被改寫來配合它。

03 管理注意力窗口

誰出席、何時開會、議程怎麼排，都會影響哪個組合被採納。

Setup

年度預算會上，早就想買的工具突然變成所有問題的答案

最小場景是一個解法在多個鬆散問題間尋找落點。

最小劇情

公司開年度效率會議。有人早已想導入新的專案管理系統；會中提到交付延遲、文件散落和責任不清，該工具很快被包裝成共同解法。

真正難點

這些問題可能真的存在，但它們是否都由同一工具解決，並沒有被逐一驗證。

Key Point

垃圾桶模型提醒你檢查決策的組合過程，而不是只看事後寫得很整齊的理由。

Lenses

把會議裡漂浮的東西分流

先分流，才能看出哪些連結是必要，哪些只是同時在場。

問題流

目前被說出口的痛點是否被明確定義和排序？

解法流

哪些方案在會議前就已經有人想推？

參與者流

誰剛好在場，誰缺席，誰握有決策權？

Psychology

手上有方案時，人會更快看見適合它的問題

組織注意力有限，現成方案提供了一種令人安心的封口方式。

解法舒適感

有可採購、可排程的方案，比承認問題模糊更輕鬆。

會議動能

一旦大家想結束討論，能形成共識的方案會被高估。

參與者偶然性

某些決策只是因為當天關鍵反對者不在。

當一個方案突然解決太多問題，請暫停，把每個問題和假設效果拆開。

Applications

把決策流程改成先定義問題，再允許方案競爭

採購、組織改革和技術選型都容易出現解法先行。

工具採購

要求每個候選工具對應明確痛點和成功指標。

組織改組

先說哪個決策延遲被解決，再談新架構。

技術選型

避免因團隊熟悉某框架，就把所有問題描述成適合它。

Misreadings

垃圾桶模型不是說組織決策全都荒謬

它描述的是高不確定、目標模糊和參與流動時常見的決策形態。

否定所有會議 校正

會議仍可有效，只是需要清楚分流。

把偶然當陰謀 校正

很多組合是注意力與時機造成，不一定是操弄。

忽略事後合理化 校正

紀錄很漂亮不代表當下流程真的那麼線性。

Questions

問方案是怎麼和問題配對的

下面兩問適合用在採購、改革和策略會議後檢查。

這個問題在方案出現前是否已被獨立定義？

避免用解法回頭重寫痛點。

今天缺席的人會改變哪個配對？

檢查參與者流對決策的影響。

延伸路線

可接著讀 organizational decision-making，理解不確定組織如何形成選擇。
