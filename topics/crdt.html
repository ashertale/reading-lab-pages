Conflict-free Replicated Data Type (CRDT)

CRDT 讓副本可以各自接受更新，但收斂不是靠運氣，而是靠資料型別本身限定哪些操作能被安全合併。

Orientation

讓合併規則成為資料型別的一部分

CRDT 的設計目標是讓副本在沒有同步協調的情況下各自更新，之後仍能用確定規則收斂到同一狀態。

Core Tension

離線寫入提高可用性，卻把衝突推到合併階段；若資料型別沒有明確語意，收斂只會變成任意覆蓋。

01 先限制操作語意

不是所有業務動作都能無損並發；可合併性要在型別裡設計。

02 再保留因果資訊

副本需要足夠 metadata 辨認已看過的更新與尚未合併的分岔。

03 最後驗證不變量

有些 invariant 例如餘額不能為負，單靠無協調 merge 未必保得住。

Setup

一份檢查清單在兩台筆電上分岔

A 離線新增「量測 boot current」，B 離線把「驗證 rollback」標成完成；同步時兩個副本都不能被粗暴覆蓋。

最小場景

新增項目可用 grow-only set 合併；完成狀態可能需要 observed-remove set 或更細的操作記錄。

真正難點

使用者想要的是語意正確，不只是兩邊 bytes 最後一樣。

Key Point

CRDT 把「怎麼合」從事後例外處理，搬進資料模型本身。

Lenses

看它靠狀態合，還是靠操作合

CRDT 家族有多種設計；選哪種會影響 metadata、網路傳輸、刪除語意與除錯方式。

State-based

副本交換完整或增量狀態，merge 通常像 join semilattice。

Operation-based

傳播可交換操作，要求可靠廣播與去重語意。

Tombstone cost

刪除常需要留下痕跡，否則舊新增可能在合併時復活。

Invariant boundary

可收斂不等於可保所有商業限制；需要 escrow、協調或補償策略。

資料型別容易合併的部分危險邊界
Counter 遞增可按副本分量相加或取最大 遞減與下限約束需要額外設計
Set 新增可用 union 刪除需處理 add/remove 並發
Text 插入位置要穩定識別 metadata 成長與使用者意圖可能分離

Psychology

離線可用會讓合併成本晚一點才出現

Local-first 的手感很好，因為每個人都能立刻動；真正的設計壓力在幾個副本重新見面時才浮出來。

Demo 偏向順路案例

兩人改不同欄位很漂亮，同一欄位或刪除新增競賽才暴露語意。

把衝突當 UI 問題

合併不是只做提示框；資料型別若錯，UI 只能顯示錯誤結果。

忽略 metadata 生命週期

向量、tombstone、actor id 會跟著資料長大，需要壓縮策略。

評估 CRDT 時，請用最尷尬的並發操作測試，而不是只跑新增與同步成功案例。

Applications

無協調不是免費，適合用在能接受收斂語意的地方

CRDT 很適合多端局部自治，但不適合把所有強一致問題都包成 eventually consistent。

協作筆記

多端插入、刪除與格式變更需要穩定 position id 與明確衝突語意。

Edge 狀態

邊緣節點可先接收 local updates，回雲端時用 CRDT 合併非關鍵狀態。

現場檢查表

離線填寫、補照片、加註記通常比唯一數值覆蓋更適合 CRDT 化。

  • 先列出不可違反的 invariant；若無協調無法保住，就不要硬包成 CRDT。
  • 把 actor identity、clock、tombstone 清理與資料遷移納入設計文件。

Misreadings

CRDT 不會替產品做價值判斷

它能讓特定操作安全收斂，但不會告訴你兩個人同時改價格時哪個才是商業上正確。

所有衝突都消失 校正

衝突只是被資料型別語意吸收；吸收方式若不合產品，結果仍然錯。

等同 last-write-wins 校正

LWW 是一種簡化策略，常會丟掉並發更新，不是 CRDT 的核心精神。

不用測試合併 校正

merge law、重放、亂序、重複傳送與 schema upgrade 都需要測。

Questions
