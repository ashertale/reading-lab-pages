Canary Release

金絲雀釋出不是慢慢推而已；它把新版本先交給小範圍真流量，讓監控和回退在全量前發揮作用。

Read For

適合 OTA、server rollout、driver update、feature flag 與任何不能一次把風險推給全部使用者的變更。

Orientation

先讓小範圍真實世界替你暴露風險

金絲雀釋出把新版本先導向一小部分使用者、設備、區域或服務實例，並用監控決定是否擴大。

Core Tension

測試環境能抓很多錯，但抓不到所有資料分布、硬體批次、網路條件與使用者行為；一次全量釋出把未知直接放大。

01 先選暴露切片

切片要小到可承受，又要足以代表關鍵風險。

02 再設 guardrails

crash、latency、error rate、rollback rate 與 business metrics 要有停止門檻。

03 最後確保回退

沒有可用 rollback 的 canary，只是慢一點的冒險。

Setup

OTA 先推 1% 裝置

新 firmware 通過 lab 測試，但 field 裝置有不同電池老化、網路品質、sensor revision 與使用模式。

最小場景

第一批只選 1% 裝置，觀察 boot success、crash loop、功耗與特定錯誤碼。

真正難點

小批次若沒有明確觀測與停止條件，只是把全量風險分批延後。

Key Point

Canary 的關鍵不是比例小，而是每一步都有能否繼續的證據。

Lenses

四個讓 canary 有牙齒的條件

漸進釋出要能真的降低風險，必須把分群、觀測與回退一起設計。

切片策略

可以按內部使用者、地區、硬體 revision、客戶群或 service instance 分批。

Guardrail metrics

指標要接近使用者傷害，不只看 server 還活著。

相容性窗口

前後端、資料庫、protocol 與 mobile/firmware 版本需要能共存。

Rollback path

回退是否會遇到資料格式降級、設定不相容或設備不可遠端恢復。

Canary 訊號可能代表動作
error rate 上升 新版本行為回歸 暫停擴大並切回舊版
功耗上升 field 條件下 loop 或重試異常 縮小範圍並抓取診斷資料
rollback 成功率低 恢復路徑不可靠 停止 rollout，先修 recovery

Psychology

通過測試後最容易過度自信

release pressure 會讓團隊把 staging green 誤認成 production safe。

綠燈偏誤

CI 全綠是必要條件，不是 field 風險已消失的證明。

小比例安慰

1% 聽起來小，但若使用者基數很大，影響仍可能很可觀。

不想按停止鍵

指標稍微變差時，團隊常希望再等等；停損規則應在釋出前定義。

Canary review 要把停止 rollout 視為成功防護，而不是釋出失敗。

Applications

OTA、後端與功能旗標都需要分階段證據

任何能被遠端開啟或更新的功能，都可以把全量風險拆成多個可觀測步驟。

Firmware OTA

按硬體 revision、地區或電池狀態分批，避免同一種 field 條件一次全中。

Backend rollout

先導一部分 traffic 到新版本，觀察 latency、error budget 與 downstream 壓力。

Feature flags

先對內部或小群開啟，並確保 flag 可以快速關閉且不留下資料不相容。

  • 把 rollout plan 寫成階段表：比例、觀察時間、指標門檻、負責人與回退指令。
  • 定期演練 rollback；沒有演練過的 rollback 在事故中常只是願望。

Misreadings

Canary 不是產品實驗的同義詞

它可以共享技術機制，但目標和 A/B test 不同：一個看安全，一個看假設。

等同 A/B test 校正

Canary 關心 regression 與營運風險，A/B test 關心使用者行為差異。

比例小就安全 校正

如果第一批選錯或監控不足，小比例也抓不到真正風險。

回退一定簡單 校正

資料 migration、protocol change 或 firmware downgrade 都可能讓 rollback 變難。

Questions

誰先承受風險，誰決定停止

如果第一批出事

影響範圍是否可承受，診斷資料是否足夠，回退是否能在目標時間內完成？

如果指標有小幅惡化

哪個門檻會自動暫停 rollout，誰有權覆寫，理由如何留下紀錄？
