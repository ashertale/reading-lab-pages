碰到 工具定律，最容易出錯的不是沒聽過它，而是把 工具熟悉度、問題框架 與 方法偏食 混成一句好聽卻無法執行的話。

Core Tension

當你手上特別熟悉某種工具或方法時，世界裡越來越多問題就會被你看成適合那個工具的樣子。

01 先確認 工具熟悉度 沒被偷渡進來

如果 工具熟悉度 沒先寫出來，後面看起來像分歧，實際上只是各自補腦。

02 接著看 問題框架 被誰省略

把 問題框架 講清楚後，你才看得出這條原則是在保護判準，還是在轉移代價。

03 最後把 方法偏食 變成驗收

少了 方法偏食，工具定律 很容易停在口頭正確，卻不知道怎麼在 除錯框架 或 組織流程選型 裡執行。

Setup

先別問你支不支持，先問 工具熟悉度 和 問題框架 誰該先落筆

先把場景壓回 除錯框架 或 組織流程選型 這種真的要決定責任的時刻，你會更清楚 工具熟悉度、問題框架 和 方法偏食 哪一個不能事後才補。

最小劇情

你熟悉某套分析框架、語言或流程，久了之後，連原本不該這樣切的問題也會被你硬切成它能處理的形狀。

Key Point

這類原則一旦失手，通常不是理論錯，而是 工具熟悉度 沒先寫、問題框架 沒算進去、方法偏食 也沒被翻成行動。

Lenses

先把 工具先行、問題被重塑 與 替代方案失明 分開看

同一個原則落到現場後，往往會分裂成 工具先行、問題被重塑 和 替代方案失明 三種完全不同的操作方式。

工具先行

把焦點放在 工具先行 時，你其實是在說：工具熟悉度 應該先被寫清楚，否則後面的結論都站不穩。

問題被重塑

問題被重塑 這條線在乎的，不是誰比較有氣勢，而是 問題框架 一旦沒被交代，現場會怎麼開始失真。

替代方案失明

順著 替代方案失明 讀，你會發現爭點不在口號，而在 方法偏食 是否真的被變成判準。

Psychology

哪種省事直覺最常讓 工具定律 失焦

一到現場，人最容易先抓住能讓自己省事的那一段，於是 工具定律 很快就從判準退化成表態。

熟悉就是安全感

熟悉就是安全感 會讓人覺得只要把話說得夠完整，就不必再補 工具熟悉度 的證據或代價。

快解決比先定義問題舒服

快解決比先定義問題舒服 常不是純粹錯誤，而是一種討論捷徑：先把自己放在合理的一邊，再晚點處理 問題框架。

成功經驗自我複製

碰到 成功經驗自我複製 時，最危險的是大家以為原則已經被引用，實際上 方法偏食 還沒被具體說明。

只要討論還停在『誰比較像站在對的一邊』，工具定律 就還沒真正開始工作。

Applications

回到 除錯框架、組織流程選型 與 技術選擇

把 工具定律 放回 除錯框架、組織流程選型 這些場景時，它的作用不是增加說服力，而是強迫大家先把責任、代價和驗收順序排好。

除錯框架

除錯框架 最需要 工具定律 的時候，通常就是大家都急著往前走，卻還沒把 工具熟悉度 寫成共識。

組織流程選型

只要到了 組織流程選型，你就能看出 問題框架 究竟是被真的承擔，還是只是被口頭提過。

技術選擇

在 技術選擇 這種場景，工具定律 幫你的不是贏辯論，而是把 方法偏食 先釘在桌面上，避免事後才補。

Misreadings

三個常見誤讀

原則類概念一旦變得順耳，就很容易失手；工具定律 最常就是在 除錯框架 或 組織流程選型 這些情境裡，被縮成一句不用交代代價的口號。

工具定律 只是好聽口號 校正

真正難的是把 工具定律 翻成一組可檢查的判準，而不是把它當成結論本身。

只要背了 工具定律 就能直接套用 校正

原則若不落回 工具熟悉度、問題框架 和代價分配，通常會在現場變成誤用。

工具定律 只適合高層決策 校正

其實從 code review 到 方法與問題框架 的日常判斷，都會碰到同樣的壓力。
