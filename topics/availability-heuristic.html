Availability Heuristic

Orientation

哪些記憶最先跳出來，就先改寫你的判斷

可得性捷思讓大腦用回想速度當作風險估計。它很快，也常常有用，但在樣本偏斜時會把注意力帶歪。

Core Tension

最近、鮮明、情緒強的案例比較容易浮上心頭；可是容易想起來不等於基準率高，也不等於最值得投入資源。

01 先承認它是捷徑

快速回想能幫你先抓方向，尤其在 incident response 初期很有價值。

02 再補上基準率

用歷史資料、fleet telemetry、缺陷分布或 field return rate 校正直覺。

03 最後看決策代價

如果錯估風險會造成安全、量產或客戶事故，就不能只靠最鮮明的案例。

Setup

把問題縮到最小

最小情境是：一個很戲劇化的 bug 剛發生，所有人都覺得它一定是最大風險。

最小劇情

上週某客戶遇到 boot failure，會議裡每個人都記得那張失敗截圖，於是大家自然把 boot path 放到最高優先。

真正難點

boot failure 也許很嚴重，但你還需要知道它的發生率、影響範圍、可重現性，以及是否只是單一治具或單一版本造成。

鮮明案例進入記憶

Key Point

可得性捷思提醒你：優先權不應只由哪個故事最容易被講出來決定，而要由頻率、嚴重度、可偵測性與修復成本一起決定。

Lenses

四種讓事件變得可得的因素

一件事容易被想起來，可能是因為它真的常見，也可能只是因為它最近、戲劇化或被重複提起。

近期性

剛發生的事比較容易支配判斷，讓團隊誤以為它代表長期趨勢。

鮮明性

有截圖、有客訴、有大場面事故的 case，比安靜累積的小錯誤更容易被記住。

重複曝光

同一個問題在多場會議被提起，會讓人感覺它比實際分布更普遍。

個人經驗

自己踩過的坑會特別有重量，但個人經驗常不是整體樣本。

Psychology

為什麼腦中最先浮現的案例會搶走權重

人腦不會每次都從統計資料重算世界，常用記憶搜尋速度來估計可能性。

記憶搜尋很省力

如果例子立刻冒出來，大腦會把這種流暢感誤讀成高頻率或高重要性。

情緒放大記憶

恐慌、尷尬或高壓事故會讓記憶更鮮明，也讓後續判斷更容易被它帶動。

團隊會放大同一記憶

同一個 incident 在 postmortem、週會和客戶會議裡反覆出現，會形成集體可得性。

重點是設計風險判斷的校正機制，而不是否定工程師的現場直覺。

Applications

把印象強度拉回 incident 判斷

可得性捷思最容易影響 priority、風險排序、測試覆蓋與 postmortem 後的改善方向。

Incident response

剛爆掉的問題需要快速處理，但長期防線要回到趨勢資料，不要只修最有戲劇性的 case。

測試策略

若測試只追逐最近失敗的路徑，可能會忽略低調但高頻的 regression。

產品決策

聲音最大的客戶需求不一定代表最大市場；需要把 anecdote 和 segment data 分開。

  • 風險排序時至少列出 frequency、severity、detectability、blast radius，避免只用印象排序。
  • 每次 postmortem 後問一句：這是系統性高頻問題，還是單一高鮮明度事件？

Misreadings

三個常見誤讀

可得性捷思不是說直覺一定錯，而是提醒直覺需要基準率校正。

容易想到就一定重要 校正

容易想到可能只是因為近期、鮮明或被重複提起；重要性還要看資料。

只要看 dashboard 就夠 校正

dashboard 也可能只呈現容易被量測的項目，沒有解決樣本偏斜。

這只是媒體或大眾問題 校正

工程團隊也會被最近的客訴、最痛的 incident 或最熟的模組牽著走。

Questions

把它變成你的風險校正

下面兩個問題適合用在 priority review 或 postmortem action item 排序。

這件事為什麼會被我想到

是因為它高頻、高嚴重度，還是只是最近、戲劇化、被高層提過？

如果看完整資料，排序會不會改變

把最近案例拿掉，只看三個月資料、客戶分布和 field rate，決策是否仍一樣？

延伸路線

延伸閱讀可看 confirmation bias、base rate neglect、availability cascade；也可以把它接到 incident management 的優先權模型。
