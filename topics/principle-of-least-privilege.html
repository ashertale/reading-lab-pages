最小權限原則

Principle of Least Privilege

最小權限原則要求每個人、服務或程式只拿完成任務所需的權限，讓失誤和入侵的爆炸半徑變小。

Read For

Orientation

方便的全權限，常把小錯誤變成大事故

最小權限不是不信任人，而是承認憑證會外洩、程式會有 bug、人會按錯；權限越小，錯誤影響越小。

Core Tension

廣權限讓工作順暢，窄權限讓事故變小；好的設計要讓日常工作可做，同時不把所有門都交給同一把鑰匙。

01 按任務授權

權限應對應具體操作，而不是對應職稱或方便程度。

02 限制時間與範圍

臨時高權限要有到期時間、目標資源和審計紀錄。

03 分離高風險能力

讀、寫、刪除、部署、簽章和金流操作不應無條件集中。

Setup

背景同步服務只需要讀設定，卻拿了整個資料庫寫入權

最小場景是服務為了省配置使用高權限帳號，實際任務只需要很小一部分。

最小劇情

一個 telemetry collector 只需讀取 device metadata，卻使用共用 admin database credential。程式漏洞被利用後，攻擊者能刪除使用者資料。

真正難點

過大權限在平常像便利，事故時才顯示它把所有後果綁在一起。

任務需求很小
→
配置給予廣權限
→
漏洞或誤操作放大傷害
Key Point

最小權限的價值，是讓單一憑證或流程失守時只失去它本該能碰的範圍。

Lenses

把權限拆成操作、資源與時間

權限不是一個 yes/no，而是誰、能對哪個資源、做哪個操作、多久。

操作範圍

服務需要 read、write、delete、deploy、sign 還是只是查詢？

資源範圍

權限是否限制到 tenant、table、bucket、device group 或環境？

時間範圍

臨時權限是否自動過期，長期 token 是否可輪替和撤銷？

Psychology

過大權限通常不是惡意，而是為了少被流程卡住

當最小權限太難申請或常阻塞工作，團隊會自然偏向共用高權限捷徑。

便利立即可見

全權限讓開發和維運少掉摩擦，風險則通常晚到事故才可見。

權限清單會腐爛

人員轉組、服務改責任、臨時例外若不回收，權限會越堆越寬。

共用帳號模糊責任

多人共用同一高權限帳號時，審計和回收都變困難。

最小權限能長期成立，必須讓正確權限容易取得，讓過大權限容易被看見。

Applications

把高權限改成可申請、可到期、可追蹤

服務帳號、韌體 debug 和營運後台都需要把權限縮到任務邊界。

服務角色

為每個服務建立專用 role，只允許它需要的 API、table 和環境。

韌體 debug

量產裝置的 debug capability 應有簽章、期限和裝置範圍，不保留永久後門。

營運後台

高風險操作使用 just-in-time elevation、雙人確認和完整 audit log。

Misreadings

最小權限不是讓工作做不了

它要讓必要工作有剛好的權限；設計太卡會逼人繞過系統。

把最小權限當禁止權限 校正

目標是必要且足夠，不是盲目收緊。

只做初始設定 校正

權限需要隨任務變更定期審查和回收。

忽略緊急情境 校正

緊急高權限可以存在，但要短期、可追蹤、可回收。

Questions

問這把鑰匙能開哪些不必要的門

這個任務真正需要哪些操作？

把 read、write、delete、deploy、sign、admin 分開列。

權限失守時最大傷害是什麼？

用資源範圍、時間範圍和審計能力估算爆炸半徑。

延伸路線

可接著讀 privilege separation，理解如何把高風險能力拆到不同邊界。
