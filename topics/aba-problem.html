ABA Problem

值從 A 變成 B 又變回 A，看起來沒變；但對併發演算法來說，歷史已經改變了。

Orientation

看起來一樣，不代表從未變過

ABA 問題出現在併發場景：某個位置原本是 A，其他執行緒改成 B 又改回 A，第一個執行緒只比較值，誤以為世界沒變。

Core Tension

CAS 的成功條件可能只證明目前值相同，卻不能證明中間沒有發生 pop、free、reuse 或語意變更。

01 先分值與歷史

相同位元模式不代表同一個物件生命週期，也不代表資料結構拓撲沒變。

02 再加版本資訊

tagged pointer、counter 或 stamp 能讓第二個 A 與第一個 A 可區分。

03 最後處理回收

memory reclamation、hazard pointer 或 epoch scheme 常是 ABA 修補的一部分。

Setup

Lock-free stack 的舊 head 回來了

想像一個 thread 讀到 stack head 指向節點 A，準備 CAS 把 head 改到 A.next。

最小場景

另 一個 thread 先 pop A，再 pop B，之後把 A 重新 push 回來；第一個 thread 看到 head 仍是 A，CAS 成功，但 A.next 的意義已經不是原本那個結構。

真正難點

單純比較指標值會漏掉節點生命週期與 next pointer 的歷史變化；資料結構可能被悄悄破壞。

→

Key Point

ABA 問題的核心是：相等比較缺少時間資訊。

Lenses

三個地方最容易藏 ABA

ABA 不只發生在教科書 stack；任何把狀態值拿來代表完整歷史的 lock-free 設計都要小心。

指標重用

allocator 釋放又重用同一地址，讓 pointer value 看起來回到原點。

狀態編碼

有限狀態機在 A/B 間切換，觀察者只看到現在值，漏掉中間 transition。

free list

節點快速回收與重掛載時，CAS 成功可能建立在過期的 next 上。

版本 wraparound

加 tag 也要考慮 counter 何時回捲，以及回捲是否足以在系統壽命內發生。

防法 能解什麼 注意事項
Tagged pointer 區分第一個 A 與第二個 A 需要位元空間或 double-width CAS。
Hazard pointer 避免仍被使用的節點回收 實作與掃描成本較高。
鎖或 critical section 簡化歷史一致性 犧牲 lock-free 的進度性質。

Psychology

相等比較讓人放心太快

人看到 expected value 仍然相等，很容易把它解讀成沒有競爭；ABA 正好利用這個直覺漏洞。

現在值遮住歷史

Debug print 通常只顯示當前 pointer 或 enum，沒有顯示它來回變過幾次。

低機率讓測試沉默

ABA 需要特定 interleaving，壓測不一定容易重現；一旦發生又很難留下完整證據。

效能誘惑

Lock-free code 追求快速路徑，容易把版本與回收的複雜度推遲到出事後。

面對 ABA，問『它現在是不是 A』不夠，還要問『它是不是同一個 A』。

Applications

用在 ISR、atomic 與 lock-free queue

Firmware 裡雖然不一定大量使用無鎖資料結構，但 ISR/main loop、DMA descriptor 與多核心共享 queue 都可能碰到相似問題。

ISR 與 main loop

共享狀態若可能在兩次讀取間來回變化，單一 flag 可能不足以表示事件歷史。

Descriptor ring

slot index 重用時，consumer 需要 generation 或 sequence，避免把舊 descriptor 當新資料。

Free list allocator

中斷或多核心回收節點時，要確認 pointer value 不會獨自代表節點生命週期。

  • 對 lock-free 結構做 interleaving review：值回到原點時，哪些隱含欄位已經不同？
  • 若資源會重用，給 slot 或 pointer 加 generation，比只看地址更安全。

Misreadings

ABA 不只是 compare-and-swap 的小缺點

它揭露的是 value equality 與 object identity 被混用的問題。

只要 CAS 成功就安全 校正

CAS 只證明當前值符合 expected；它不證明中間沒有被改過。

加 counter 永遠解決 校正

counter 若會回捲，或沒有跟 pointer 一起原子更新，仍可能留下洞。

只有高階 OS 會遇到 校正

任何共享 slot、ring index 或 generation 重用的 embedded 設計都可能有同類風險。

Questions
