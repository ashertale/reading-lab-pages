訊噪比

Signal-to-Noise Ratio

訊號是否能被可靠判讀，不只看訊號有多大，也看背景雜訊如何吃掉你的判斷餘裕。

Orientation

有訊號不等於能判斷

訊噪比描述的是有用訊號相對於背景噪聲的強度；它決定你能不能穩定分辨狀態、事件或趨勢。

Core Tension

訊號可以存在，但若它只比噪聲高一點，門檻、濾波與重複量測就會開始主宰結論，而不是物理現象本身。

01 先量噪聲底

沒有 noise floor，就無法知道 observed delta 是否有意義。

02 再看動態範圍

ADC resolution、front-end gain 與 saturation 會決定小訊號能不能被保留下來。

03 最後設計判定餘裕

threshold 應該離噪聲有足夠 margin，並考慮溫度、供電與製程差異。

Setup

一個看似跳動的 sensor 值

把問題縮到電池電壓或電流感測：數值在門檻附近上下跳，狀態機因此頻繁進出保護模式。

最小場景

平均值看起來接近門檻，但量測噪聲、ground bounce 與 ADC 量化誤差讓單次讀值不可靠；狀態機沒有 hysteresis，也沒有濾波策略。

真正難點

如果只調門檻，會在誤觸發與漏保護之間來回；真正要先估的是訊號變化相對於噪聲的距離。

Key Point

SNR 低時，問題不只是讀值不漂亮，而是整個判定流程缺少足夠的決策餘裕。

Lenses

從量測鏈找噪聲從哪裡進來

SNR 不是單點公式而已；它會穿過 sensor、analog front-end、ADC、佈線、韌體濾波與判定邏輯。

物理噪聲

熱噪聲、供電紋波、EMI 與機械震動會把真訊號淹在背景裡。

量化噪聲

ADC 位元數、參考電壓與取樣策略會限制可分辨的最小變化。

處理噪聲

平均、濾波與 decimation 能改善可讀性，也可能延遲事件或抹掉尖峰。

判定噪聲

沒有 hysteresis 或 debounce 的狀態機，會把小幅抖動放大成頻繁狀態切換。

現象 可能來源 先查什麼
門檻附近來回跳 噪聲幅度接近判定餘裕 noise floor、hysteresis、濾波延遲。
不同板子差很多 layout、零件公差或接地路徑 跨板分布與參考量測。
平均後仍不穩 低頻漂移或真事件混入 頻譜、溫度與負載條件。

Psychology

人會被單次讀值說服

Debug 時看到一個數字超過門檻，很容易立刻下結論；SNR 逼你把數字放回雜訊分布裡。

門檻讓判斷看似二元

一旦有 pass/fail，團隊容易忘記門檻附近其實是一段不確定區。

漂亮平均掩蓋尖峰

平均值穩不代表瞬態安全；尖峰也可能才是保護邏輯真正需要看到的訊號。

調參比找噪聲容易

改 threshold 很快，找 ground path、coupling 或 ADC 設定比較痛，但後者才會改善根因。

SNR 的思考方式能降低單點讀值帶來的過度自信。

Applications

用在 sensor、ADC 與狀態機設計

Firmware 常要把連續且帶噪的世界轉成離散決策；SNR 決定這個轉換是否穩。

ADC 讀值

選擇 oversampling、reference、gain 與 filter 時，先定義需要分辨的最小實際變化。

保護門檻

過溫、過流、欠壓保護應有 hysteresis、debounce 與故障前後快照，避免噪聲驅動狀態。

通訊品質

RSSI 或 link metric 不該單獨使用；要同時看 noise、重試率、封包錯誤與環境變化。

  • 每個 threshold 旁邊最好有 noise margin 與跨條件測試資料。
  • 若 SNR 不足，先改善量測鏈，再討論演算法門檻。

Misreadings

別把噪聲當成可以最後處理的小事

SNR 低時，後端邏輯會把量測問題放大成產品問題。

多平均幾次就好 校正

平均可壓低某些隨機噪聲，但會增加延遲，也對漂移與尖峰不一定有效。

訊號大就安全 校正

若噪聲也一起變大，絕對幅度沒有保證；要看相對距離。

軟體濾波能補所有硬體問題 校正

ADC 前端、layout 與 reference 的問題常會限制軟體能救的範圍。

Questions

判斷前先估噪聲餘裕

如果讀值剛好跨過門檻

這個跨越比 noise floor 大多少？在溫度、負載與不同板子上仍然成立嗎？

如果要加濾波

你願意用多少反應延遲換多少噪聲下降？哪些 fault 會被濾波拖慢或抹掉？
