規劃謬誤

Planning Fallacy

人即使知道過去常拖延，仍會在下一次估時時相信這次例外地會順利，於是時間、成本與風險被系統性低估。

Read For

這頁要幫你辨認一個很熟悉的幻覺：明明知道上次也拖了，這次排期時卻還是覺得自己終於算準了。

Orientation

計畫之所以樂觀，是因為腦中只排了順路版本

規劃謬誤抓到的是一種很頑固的樂觀：人會把未來計畫看成一條乾淨路徑，而不是一個會被中斷、返工與依賴纏住的系統。

Core Tension

你越熟悉任務內容，越容易陷入 inside view，只看這次要做的步驟，卻忽略過去類似任務實際怎麼收場。

01 先看參考類別

估時前先找相似任務的實際完工分布，而不是只看理想步驟表。

02 再列隱性依賴

跨團隊等待、硬體回板、bug 返修與 review 往往比核心實作更拖時間。

03 最後留風險緩衝

沒有緩衝的排程通常不是高效率，而是把不確定性偷偷轉成延誤。

Setup

把問題縮到最小

這個偏誤可怕的地方在於，它常發生在很有經驗的人身上，因為經驗會讓人更相信自己對流程的內部模擬。

最小劇情

團隊預估新板 bring-up 兩週完成，因為主流程很清楚；但實際上驅動整合、量測治具、供應商回覆與例外修補把時程拉到五週。

真正難點

計畫中的步驟會被想得比現實更連續，沒有發生的阻礙則幾乎不會自動進入你的估時模型。

腦中模擬順利路徑
→
忽略類似任務分布
→
時程與風險被低估
Key Point

規劃謬誤不是沒有經驗，而是經驗沒有被轉成外部參考資料，仍然只存在於『這次應該會順』的感覺裡。

Lenses

三個修正估時的方法

把 inside view 拉回現實，通常要靠參考類別、阻塞清單與機率式預估，而不是靠更努力想像一次。

參考類別

找相似任務過去實際花多久，而不是只問這次理想流程要做哪些事。

阻塞清單

把等待、返工、同步、依賴與外部回應列成估時一等公民。

機率估時

不要只給單一日期，改給範圍與信心，才比較接近真實不確定性。

看法 看起來像 真正風險
只看實作步驟 排程很漂亮 等待與返工在表上消失，最後都變延誤。
看相似任務分布 估時變保守 較不討喜，但能防止一再重演同樣錯誤。
給機率區間 承諾不夠俐落 對管理比較難受，但對真實風險更誠實。

Psychology

為什麼 inside view 總讓時程看起來太乾淨

人會自然偏愛自己可以控制的那一段流程，因此把其他干擾視為例外，而不是計畫的一部分。

想像中的未來會被美化

當下的自己容易在腦中跑一條乾淨的順利路徑，低估雜訊、等待與反覆修補。

過去失敗很少真正入模

我們記得上次拖延，卻不會在這次估時時把那個分布嚴格算進來。

重點在於把『知道過去會拖』轉成可操作的估時流程，而不是停留在自我提醒。

Applications

把參考類別帶回估時與整合排程

Firmware、硬體整合、跨團隊專案與 migration 工作都特別容易出現這個偏誤，因為等待與返工常比主流程更重。

Bring-up 與整合排程

板子、治具、驅動與量測彼此卡住時，inside view 會嚴重低估交互依賴成本。

版本發布

最後一哩常被認為只是打包與驗證，實際上容易被回歸問題、風險評估與協作延遲吃掉。

  • 估時時先拿過去三個類似任務的實際耗時做底線，比再想一次理想流程更有效。
  • 若不能給機率區間，至少明確分出『理想值』『較可能值』『保守值』三種日期。

Misreadings

別把修正方法簡化成「全部乘二」

修正規劃謬誤，不代表永遠用最悲觀排程；重點是讓樂觀不再沒有成本。

所以只要把時間全部乘二 校正

有時能救急，但那只是粗暴緩衝，不能取代參考類別與依賴分析。

有經驗的人就不會犯 校正

經驗若沒被結構化成外部分布，一樣可能在每次新計畫裡重新樂觀一次。
