希克定律

Hick's Law

當可選項越多、越難比較時，人做出決定所需的時間通常也會增加，因為真正負擔的是判斷，不是點擊。

Orientation

人卡住的常不是點哪個，而是先得判斷哪個值得點

希克定律之所以對產品設計有穿透力，是因為它把『資訊多』翻譯成『決策成本高』。

Core Tension

更多選項可以提高彈性與覆蓋度，但也會拉高辨識、比較與猶豫成本；真正好的設計不是一味刪，而是安排決策節點。

01 先數選項，再看可比性

十個高度相似的選項，比十個功能分明的選項更容易讓人停住。

02 再利用預設

好的預設值能把高頻決策從每次都重選，變成少數需要偏離時再處理。

03 最後分層揭露

把低頻複雜選項藏到下一層，通常比一開始全展開更有用。

Setup

從一個塞滿設定的畫面開始

這個定律最常出現在設計者覺得『都放給使用者選很自由』的時候。

最小劇情

韌體工具頁面一次塞了十多個 mode、logging 等級與 retry 策略，結果新使用者在真正要操作時反而不敢下手。

真正難點

問題不一定是選項太多本身，而是使用者沒有足夠線索知道哪些差異真的重要。

選項暴露過多
→
比較成本升高
→
行動被延遲或隨便選
Key Point

希克定律最有價值的地方，是提醒你決策不是免費的，尤其在高壓與不熟悉情境裡。

Lenses

四個看選項負擔的角度

不只是選項數量，下面這幾件事也會大幅改變決策成本。

相似度

越像的選項越難比較，因為每個都需要額外解碼。

任務緊迫度

時間越緊、風險越高，額外決策越容易讓人停滯。

預設品質

好的預設能把常見路徑變成低摩擦，不好的預設則會放大猶豫。

揭露節奏

把所有選項一次攤開，與按情境逐步揭露，對認知負荷差很多。

設計做法 看起來像 實際決策成本
一次展開所有選項 很完整很自由 新手要先理解一堆差異才敢動
提供強預設 限制比較多 多數高頻情境的決策成本大幅下降
分層進階設定 需要多點一步 把低頻複雜度留給真的需要的人
相似命名堆滿 專業感很強 使用者在名詞差異上耗掉大量心力

Psychology

為什麼更多自由有時反而讓人不動

因為選擇不是免費禮物，它同時帶來判斷負擔與做錯的焦慮。

選項多會放大出錯感

人越覺得後果重要，越怕自己在一堆相似選項裡選錯。

沒有明確預設時，大腦要自建排序

這會讓本來簡單的操作變成小型分析任務。

過多選擇會拖慢起手

很多人不是沒能力完成，而是卡在開始前的判斷。

希克定律不是反選擇，而是反對把決策成本隱藏在看似完整的選單裡。

Applications

把決策成本帶回介面與工具設計

從設定頁、安裝流程到 CLI 參數面，這個概念都在提醒你：易用性不只看功能有沒有，而要看人能不能順利做出選擇。

安裝流程

先給推薦路徑，再讓進階使用者展開細節，通常比一開始全開更少流失。

CLI 參數

若常用情境需要記一長串 flag，很多使用者會寧願跳回手動甚至放棄。

裝置設定

把高頻與低頻設定分層，能讓現場操作在壓力下更不容易出錯。

  • 減少決策成本的核心不是盲刪功能，而是把高頻路徑做成明確預設與清楚階層。
  • 每增加一個選項，都問一次：它替誰創造價值，又替多少人增加了判斷負擔？

Misreadings

三種容易把它用太死的方式

希克定律不是叫你把世界壓成單一按鈕，而是叫你設計決策節奏。

所以選項越少越好 校正

不是。太少可能會犧牲必要控制，重點是選項是否在對的地方出現。

這只是 UI 規則 校正

任何需要做選擇的介面、流程與命令列，都有同樣的決策成本。

只要做得漂亮就沒問題 校正

視覺乾淨有幫助，但若資訊結構沒整理，猶豫仍然存在。

Questions

這裡在賣自由，還是在賣猶豫

使用者現在真的需要做這個決策嗎

還是其實可以先提供安全預設，等真的偏離時再要求判斷？

這些選項彼此的差異，是否對目標使用者一眼可懂

如果不行，複雜度就還沒被真正設計掉。
