Orientation

如果一套說法能解釋所有結果，為什麼這反而可能代表它沒有真正說明任何事情

可反駁性 不像一句立場聲明，它更像討論時的排序器：先把 失敗條件 放回桌面，再決定 預先承諾 和 可檢驗性 要怎麼分配。

Core Tension

一個主張若永遠能在事後把任何結果解釋回來，它就很難真的接受檢驗；可反駁性要求你先說清楚什麼會讓自己輸。

01 先把 失敗條件 拉回起點

很多爭論表面在談結論，其實一開始就把 失敗條件 當成默認前提。

02 再問 預先承諾 是誰在付

現場最常漏掉的不是立場，而是 預先承諾 到底由誰承擔、何時結帳。

03 最後確認 可檢驗性 誰能檢查

真正把事情做完的，往往不是表態，而是把 可檢驗性 翻成可被檢查的動作。

Setup

哪個前提一旦不寫，可反駁性 就會被講歪

把討論縮到一次具體決策，失敗條件 是否先被交代、預先承諾 是否被偷省、可檢驗性 是否有驗收方式，就會一眼看出來。

最小劇情

某理論聲稱自己永遠能對；一旦資料不合，就再補一個例外、條件或新詮釋，最後再也沒有真正會讓它失敗的時刻。

真正難點

如果一套說法能解釋所有結果，為什麼這反而可能代表它沒有真正說明任何事情

有人先把 失敗條件 當既成前提
→
真正要付的 預先承諾 被留到後面
→
收尾時才發現 可檢驗性 根本沒有操作方式
Key Point

把 可反駁性 用好，重點不在記住一句格言，而在於知道 失敗條件 要先說、預先承諾 要有人扛、可檢驗性 要能驗收。

Lenses

先把 失敗條件先寫、不怕輸才算能驗 與 事後修補要有代價 分開看

如果不把 失敗條件先寫、不怕輸才算能驗 與 事後修補要有代價 拆開，你很容易以為大家在討論同一條原則，實際上是在搶不同的判準。

失敗條件先寫

順著 失敗條件先寫 讀，你會發現爭點不在口號，而在 失敗條件 是否真的被變成判準。

不怕輸才算能驗

把焦點放在 不怕輸才算能驗 時，你其實是在說：預先承諾 應該先被寫清楚，否則後面的結論都站不穩。

事後修補要有代價

事後修補要有代價 這條線在乎的，不是誰比較有氣勢，而是 可檢驗性 一旦沒被交代，現場會怎麼開始失真。

Psychology

為什麼人一忙起來就最容易把 可反駁性 用歪

這類原則常被用歪，不是因為內容太難，而是因為把它說成一句漂亮話，比真的交代 失敗條件、預先承諾 和 可檢驗性 容易太多。

永遠有話可以說看起來很強

碰到 永遠有話可以說看起來很強 時，最危險的是大家以為原則已經被引用，實際上 失敗條件 還沒被具體說明。

反例太刺眼不想收

反例太刺眼不想收 會讓人覺得只要把話說得夠完整，就不必再補 預先承諾 的證據或代價。

預先承諾比事後解釋難

預先承諾比事後解釋難 常不是純粹錯誤，而是一種討論捷徑：先把自己放在合理的一邊，再晚點處理 可檢驗性。

一條原則會變空話，通常不是因為字太抽象，而是因為 可檢驗性 從來沒有被翻成現場動作。

Applications

回到 實驗設計、KPI 定義 與 根因假說管理

真正有差的，不是會議上有沒有人引用 可反駁性，而是 實驗設計 和 KPI 定義 裡的決策流程有沒有因此變得更可檢查。

實驗設計

在 實驗設計 這種場景，可反駁性 幫你的不是贏辯論，而是把 失敗條件 先釘在桌面上，避免事後才補。

KPI 定義

KPI 定義 最需要 可反駁性 的時候，通常就是大家都急著往前走，卻還沒把 預先承諾 寫成共識。

根因假說管理

只要到了 根因假說管理，你就能看出 可檢驗性 究竟是被真的承擔，還是只是被口頭提過。

Misreadings

三個常見誤讀

這類頁面最常被讀歪的地方，不是概念太抽象，而是大家太快把 失敗條件 當成價值宣示，忘了它還得連著 預先承諾 和 可檢驗性 一起成立。

可反駁性 只是好聽口號 校正

真正難的是把 可反駁性 翻成一組可檢查的判準，而不是把它當成結論本身。

只要背了 可反駁性 就能直接套用 校正

原則若不落回 失敗條件、預先承諾 和代價分配，通常會在現場變成誤用。

可反駁性 只適合高層決策 校正

其實從 code review 到 方法與驗證 的日常判斷，都會碰到同樣的壓力。

Questions

把 失敗條件、預先承諾 與 可檢驗性 變成會議上的追問

下面兩個問題不是要你再複述一次原則，而是逼 實驗設計、KPI 定義 這類場景裡的人把空白前提補完。

如果 實驗設計 現在就要拍板

誰要先把 失敗條件 說清楚？預先承諾 由誰承擔？若 可檢驗性 無法驗收，你還願意讓決定往前走嗎？

如果團隊說『可反駁性 不是大家都同意嗎』

怎樣才算真的用了 可反駁性，而不是只借它替既有結論加上道德光澤？
