布隆過濾器

Bloom Filter

布隆過濾器用極少記憶體回答集合成員問題；它允許 false positive，但不允許 false negative。

Read For

適合查 cache miss、封包去重、儲存索引與任何只需要快速排除不存在項目的場景。

Orientation

它最有價值的回答是「一定不在」

布隆過濾器用多個 hash function 把元素映射到位元陣列；查詢時只要有任一位元為 0，就能確定元素沒被加入過。

Core Tension

它用少量空間換來一種不對稱錯誤：可能把不存在的元素誤判為存在，但不會把已加入元素判成不存在。

01 先決定可錯方向

只有 false positive 可接受時，Bloom filter 才是好交易。

02 再估容量

元素數超過設計量後，位元陣列變滿，誤判率會快速上升。

03 最後保留真資料

Filter 只是一個前置判斷；真正需要資料時仍要查 backing store。

Setup

用幾 KB 記住最近看過的 ID

一個小型 gateway 要避免重複處理 event ID，但 RAM 只夠放位元陣列，無法保存完整近期集合。

最小場景

收到 ID 後寫入 k 個 hash 對應位元；下次查詢若有任一位元為 0，就知道它沒出現過。

真正難點

若所有位元都是 1，只能說可能出現過；系統必須能承受偶爾多做一次查證或跳過某些低價值工作。

Key Point

Bloom filter 的核心不是更快的 hash set，而是用概率承諾壓低記憶體。

Lenses

四個會決定誤判率的旋鈕

布隆過濾器看似簡單，實際設計要同時估元素數、位元數與 hash 數量。

位元陣列大小

越大越能降低碰撞，但佔用 RAM 或 cache 空間也越多。

Hash 個數

太少會分散不足，太多會快速填滿位元並增加 CPU 成本。

元素上限

超過設計容量後，所有查詢都越來越像 positive。

刪除策略

標準 Bloom filter 不能安全刪除；需要 counting variant 才能處理移除。

需求適合不適合
快速排除不存在 查 backing store 前先擋掉 miss 需要精確列舉集合
RAM 很少 embedded cache、邊緣設備 每次誤判都會造成安全問題
資料只增不刪 批次索引、短期去重 頻繁刪除且不能留殘影

Psychology

「可能存在」很容易被聽成「存在」

概率型資料結構最大的風險，常不是演算法，而是介面命名讓使用者誤會承諾。

API 語意太強

函式若叫 contains，呼叫者可能忘記它只代表 probably contains。

誤判被低估

1% false positive 在每天數百萬次查詢時並不小。

容量超載難察覺

Filter 飽和通常不會爆炸，只是默默變得越來越沒鑑別力。

好的 Bloom filter 介面應把可能性寫進命名、文件與監控，而不是埋在 implementation detail 裡。

Applications

用在先篩掉不可能的查詢

Bloom filter 適合當第一道低成本檢查，尤其在 backing store 很慢或記憶體有限時。

Cache miss avoidance

查磁碟或網路前先判斷 key 是否必定不存在，減少昂貴查詢。

封包或事件去重

短時間內避免重複處理；誤判成本需要被流程吸收。

儲存索引

LSM tree、資料庫與分散式儲存常用 filter 降低不必要讀取。

  • 在監控裡記錄 estimated fill ratio 與 false positive sample，而不是只看查詢速度。
  • 把回傳值命名成 maybe_present / definitely_absent，降低語意誤用。

Misreadings

它不是安全名單，也不是可列舉集合

Bloom filter 的錯誤方向讓它很有用，也讓某些用途直接不合格。

可以用來擋惡意 ID 校正

false positive 可能放錯或擋錯；安全決策不能只靠它。

可以取代 hash set 校正

它不存原始元素，不能列舉、取值或回答確定存在。

誤判率固定不變 校正

加入元素越多，誤判率會跟著改變。

Questions
