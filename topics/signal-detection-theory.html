Orientation

同一個量測，可以切出不同錯誤

訊號偵測理論把判斷問題拆成兩層：資料本身能不能把 signal 和 noise 分開，以及決策者把 threshold 放在哪裡。

Core Tension

降低門檻會抓到更多真問題，也會增加誤報；提高門檻會讓報告乾淨，卻可能把真正故障放走。

01 先分 sensitivity

若 signal 與 noise 分布重疊很大，光調門檻無法變出可靠偵測。

02 再定 criterion

門檻應該跟誤報與漏報成本相連，而不是只追求報表好看。

03 最後看 ROC

不同門檻會形成取捨曲線；選點要回到現場成本與後續處理能力。

Setup

一個過熱告警要多早叫

想像裝置用溫度、電流與負載預測即將 thermal throttling；模型輸出一個 risk score，門檻由 firmware 決定。

最小場景

門檻太低，使用者常看到不必要的降頻；門檻太高，真正過熱時又太晚保護，造成效能突降或硬體壽命風險。

真正難點

團隊容易爭論模型好不好，卻沒有把誤報、漏報、使用者體感與硬體保護成本放在同一張表上。

Key Point

如果不先說清楚哪種錯誤比較貴，threshold 調校就會變成各方憑感覺拉扯。

Lenses

把偵測拆成能力與取捨

同一條曲線可以用在測試站、告警系統與 ML classifier；每次都要分清楚資料分離能力與門檻選擇。

命中與漏報

hit rate 高不一定好，若它靠低門檻換來大量 false alarm，後端處理會被淹沒。

誤報成本

誤報可能造成退貨、重測、使用者不信任或工程人員 alert fatigue。

漏報成本

漏掉真正故障可能導致 field failure、安全風險或更昂貴的召回。

基準率

若真故障很稀有，即使偵測器看似準確，也可能產生大量陽性裡的假警報。

門檻調整 好處 代價
降低門檻 更早抓到真問題 誤報、重測與干擾增加。
提高門檻 報告更乾淨 漏掉低強度但重要的早期訊號。
改善量測 整條 ROC 變好 需要更好 sensor、特徵或實驗設計。

Psychology

人會把錯誤種類混成一句不準

『不準』常同時包含太敏感、太遲鈍、資料太 noisy 與成本定義不清；拆不開就很難修。

誤報比較吵

false positive 會立即打擾團隊，所以常被高估；false negative 常到事故後才被看見。

單一準確率很誘人

accuracy 把不同錯誤折成一個數字，容易遮住低基準率下的真正問題。

門檻像政治決策

不同角色承擔的成本不同：測試、客服、硬體可靠性與產品體驗會偏好不同門檻。

這個模型的實用性在於讓爭論從『太吵』變成『我們願意用多少誤報換多少早期保護』。

Applications

用在測試、診斷與監控門檻

Firmware 判定經常是 threshold design：ADC 值、error counter、watchdog timing、factory limit 與 field alert 都在同一個框架裡。

Factory test limit

限值太寬會放走 marginal unit；太窄會增加誤殺與重工。限值應接上失效分布與成本。

Watchdog 與 health check

timeout 太短會把短暫抖動當死鎖；太長會讓真 hung state 延遲恢復。

Anomaly alert

告警系統要同時定義 triage 能量；抓得越多不代表處理得越好。

  • 調門檻前先分三件事：量測品質、真故障基準率、誤報/漏報成本。
  • 若門檻怎麼調都不滿意，問題可能不在 criterion，而在 signal 與 noise 根本分不開。

Misreadings

不是每個錯都叫模型不好

訊號偵測理論最有用的地方，是把偵測器能力與門檻偏好拆開。

誤報多代表系統差 校正

不一定。可能是門檻刻意偏保守，也可能是基準率太低。

準確率高就安全 校正

若故障稀有，全部判正常也可能有高 accuracy；要看漏報與陽性可信度。

只要調 threshold 校正

若分布重疊太大，需要改善 sensor、特徵、資料標註或測試設計。
