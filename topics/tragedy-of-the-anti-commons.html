反公地悲劇

Tragedy of the Anti-Commons

當一件事需要太多人各自點頭、各自授權、各自放行時，資源不一定被濫用，反而可能因過度否決權而長期閒置。

Read For

這一頁要把『多一層控管更安全』和『多到最後誰都動不了』拆開，因為很多流程不是被濫用害死，而是被過度管制凍死。

Orientation

問題有時不在搶太兇，而在每個人都能擋，最後誰也推不動

反公地悲劇像是公地悲劇的鏡像：前者的病是過少限制，後者的病是限制太碎太多。

Core Tension

多層授權原本是為了保護品質與風險，但一旦每一層都能單獨卡住整體，資源利用率就可能被磨到近乎停擺。

01 先數清楚否決點

不是每個審核都一樣，真正麻煩的是彼此獨立、彼此不對齊的多重 gate。

02 再看閒置成本

如果資源一直因權限摩擦而用不出去，保護機制本身就正在創造浪費。

03 最後重組權限結構

把審核集中、分層或預設授權，常比再加一層控制更有幫助。

Setup

從一件理論上大家都同意有價值、實際卻一直動不了的事開始

反公地悲劇最常出現在跨部門專案、法務授權、平台權限與共享資源治理。

最小劇情

一個新功能需要資料、法務、資安、平台與營運各自批准；每方都有合理顧慮，但沒有一方對整體停滯負責。

真正難點

每個 gate 單看都合理，可一旦組合起來，系統就從保護變成癱瘓。

資源被切成多個獨立控制點
→
任何一點都可單獨阻擋
→
整體利用率因協調摩擦而下滑
Key Point

反公地悲劇最值得警覺的是：沒人濫用，不代表系統健康；資源也可能因過度否決而慢慢荒廢。

Lenses

四個看過度控管的角度

真正的問題不只是流程長，而是控制權碎到沒有任何人能對整體效率負責。

權限碎片化

重要資源是否被拆成太多彼此獨立的授權片段？

責任歸屬

誰對『整體卡住』負責，而不只是對自己那一關負責？

預設路徑

現在的預設是允許後追蹤，還是預設卡住直到全部點頭？

摩擦可見性

團隊是否真正看得見每一道 gate 累積起來的時間與機會成本？

場景 表面上在保護什麼 實際可能造成什麼
跨部門核准 降低單點風險 專案長期卡住、責任稀釋
共享資料權限 避免誤用與外洩 資料完全用不起來，價值閒置
平台發版流程 確保每層都審過 緊急修復速度慢到失去時效

Psychology

為什麼多一層批准總讓人感覺更安全

因為單關審核的收益很具體，整體停滯的成本卻是分散而延後的。

單關視角很容易成立

每一關都能說出自己存在的理由。

停滯成本沒有人單獨承受

大家只看到自己避免了一個風險，看不到整體損失了多少機會。

責任容易向上疊加

多加一層批准在政治上常比刪掉一層批准更容易被接受。

反公地悲劇不是反控制，而是在提醒你控制也會有超量點。

Applications

把權限摩擦帶回流程治理與平台設計

對工程團隊來說，反公地悲劇常出現在發版權限、資料存取、資安流程與跨部門決策接口。

發版與 hotfix

每個團隊都要同意才能發時，緊急修正常在風險最低的時間點也發不出去。

資料平台

所有人都怕資料被誤用，最後連最合理的分析需求都拿不到。

共享元件維護

多人共同擁有但無人能單獨拍板時，重要重構容易長期擱置。

  • 流程設計時不只算風險攔截率，也要算被你一起凍住的價值與時間。
  • 若每一關都合理卻整體很卡，問題常不是態度，而是權限結構本身。

Misreadings

三個常見誤讀

反公地悲劇不是主張去監管，而是反對把控制權切得過碎。

所以越少審核越好 校正

不是。重點是把責任與決策權設計得可運作，而不是完全無 gate。

只要流程快就表示健康 校正

有些高風險領域仍需要審查，只是審查不該碎成彼此平行卡點。

這只是政府或法規問題 校正

企業內部的權限系統、平台治理與跨部門流程同樣常見。

Questions

現在是風險被控制住，還是價值被卡死了

如果把所有 gate 攤開來看，有沒有哪幾關其實在做高度重複的判斷

那通常就是最值得整併的地方。

資源閒置或專案延遲的成本，現在有沒有被任何人看見與負責

若沒有，流程只會越疊越厚。
