腦裂

Split-Brain

腦裂發生時，多個分區各自以為自己是主節點；真正可怕的是它們同時接受互相衝突的真實。

Orientation

最危險的不是沒有人當主，而是兩邊都相信自己是主

腦裂描述的是分區後多方同時取得主控身份，讓寫入、控制或決策出現不可合併的衝突。

Core Tension

高可用追求在故障中繼續服務，但如果每個分區都為了可用性接受寫入，一致性和安全就會付出代價。

01 用 quorum 決定主權

只有取得足夠多投票或租約的一方能提供寫入。

02 隔離舊主

Failover 不只選新主，還要 fence 掉可能仍在動的舊主。

03 明確分區策略

網路不通時，是停止寫入、只讀、降級，還是允許衝突後合併？

Setup

兩個資料中心斷線後，各自選出主庫接受寫入

最小場景是雙方都失去和對方的連線，卻都認為自己該繼續提供主服務。

最小劇情

A、B 兩個資料中心之間網路分區。A 看不到 B，B 也看不到 A；兩邊的自動 failover 都把本地節點提升為 primary，接著接受同一帳戶的不同寫入。

真正難點

每一邊都在提高本地可用性，合起來卻破壞了全局唯一主控假設。

網路分區切斷心跳
→
多方各自判定對方失效
→
雙主寫入產生衝突真實
Key Point

處理腦裂要先保證主權唯一，再談如何恢復服務。

Lenses

檢查主權如何被授予與撤銷

高可用系統要能回答誰有權寫、誰必須停、以及如何證明舊主已被隔離。

仲裁來源

主節點是否需要 quorum、外部仲裁或 lease，還是單靠心跳超時？

Fencing

新主上任前，舊主是否被切斷電源、儲存、網路或寫入權？

衝突語意

若仍允許多方寫入，衝突如何合併，哪些操作不能合併？

Psychology

可用性壓力會讓人想讓每一邊都繼續服務

分區時停止一邊看起來像保守甚至浪費，但雙主造成的資料修復成本可能更高。

本地健康感誤導

每個分區內部都看起來正常，容易低估全局一致性已經失效。

自動化信心太強

Failover 自動化若沒有 fencing，只是自動製造雙主風險。

短期可用壓過長期修復

讓兩邊都寫入能暫時服務更多人，後續資料衝突卻可能不可逆。

腦裂設計的第一問不是誰接手最快，而是誰有不可爭辯的寫入權。

Applications

把分區時的寫入權說清楚

資料庫叢集、分散式 lock 和控制系統都需要避免雙主。

資料庫 HA

Failover 使用 quorum 和 fencing，避免舊 primary 在隔離網段繼續寫入。

分散式 lock

Lock lease 要考慮時鐘漂移和網路延遲，避免兩方同時持有。

控制系統

主控切換前要確保舊 controller 無法繼續輸出命令。

Misreadings

腦裂不是普通服務中斷

服務中斷可能只是不可用；腦裂是多方同時可用但產生互相衝突的權威。

只看心跳超時 校正

心跳失敗不等於對方死了，也可能只是網路分區。

以為雙寫可事後合併 校正

有些操作可合併，餘額、權限和控制命令常不能簡單合併。

忽略舊主隔離 校正

選出新主不夠，舊主若仍能寫，雙主仍存在。

Questions

問誰有唯一寫入權

分區時哪一方可以接受寫入？

答案要依 quorum、lease 或仲裁規則，而不是依各自本地判斷。

新主上任前舊主如何被 fence？

確認電源、儲存、網路、憑證或寫入權能被確實撤銷。
