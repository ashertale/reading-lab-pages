Hanlon's Razor

當一個糟糕結果同時可以被疏忽、誤解或能力邊界解釋時，先不要急著把它讀成惡意。

Orientation

先別把難看的結果直接翻譯成壞心

韓隆剃刀不是要你天真，而是先把無能、忙亂、誤解與設計缺陷放進解釋清單。

Core Tension

把錯誤立刻讀成惡意很省力，但它會讓你跳過真正可修的能力與制度問題；反過來，若永遠拒絕承認惡意，風險也會失焦。

01 先拆機制

先看資訊是怎麼遺失、交接怎麼斷掉、介面怎麼誤導，再討論人品。

02 再看能力邊界

很多看似荒謬的行為，其實是人在錯誤心智模型下做出的正常反應。

03 最後才談敵意

如果要判斷是惡意，就應補上更直接的反覆模式、利益關係或掩飾行為。

Setup

先把衝突縮成一個小場景

這個原則最適合用在你很想生氣、很想問責、很想立刻貼標籤的時刻。

最小劇情

release note 沒寫清楚某個 register 預設值已改，整合端沿用舊假設，結果把整片裝置刷成錯誤配置。

真正難點

出事之後每個人都覺得對方一定很不負責，但真正造成損害的，也許是文件、命名與 review 邊界一起失手。

Key Point

韓隆剃刀要你先把最便宜且最可驗證的解釋排一遍，而不是讓情緒直接決定故事版本。

Lenses

四個把歸因拉回地面的透鏡

同樣一個糟糕結果，可能來自完全不同的來源；把來源分清，修法才不會只剩指責。

資訊落差

對方拿到的需求、版本、時序或限制，可能和你以為的一直不一樣。

能力邊界

有些錯不是態度差，而是對複雜系統只理解到一半時，很自然會做出的錯判。

制度摩擦

如果流程鼓勵趕進度、弱化回饋或把責任切碎，錯誤就會被系統性放大。

直接證據

若你懷疑惡意，就要找重複模式、掩飾、利益衝突或故意迴避澄清的痕跡。

現象 韓隆剃刀先問 若要判成惡意還缺什麼
需求被做反 規格是否本來就含混、命名是否誤導 是否存在明知故犯與刻意迴避確認
對方一直沒回信 是否卡在權限、時區、資訊來源或優先序 是否有證據顯示故意拖延來交換利益
同樣 bug 又出現 修法是否只堵表面、知識是否沒有傳遞 是否有人反覆隱匿已知風險
會議語氣很差 是壓力與誤會在升高，還是有意羞辱與操控 需補更多上下文與歷史互動模式

Psychology

為什麼大腦會偏好把人讀成壞人

惡意敘事有一種特別強的情緒完成感，因為它讓混亂立刻變得有臉孔。

責怪比診斷省力

把複雜事故縮成一個壞人，比追一整串脆弱機制容易得多。

自我保護會推高歸因

當結果讓你受損時，大腦傾向找一個帶責任的人來承接情緒。

道德判決有敘事快感

一旦故事能用善惡切開，很多灰色條件就會被你自動忽略。

這個原則不是替糟糕行為開脫，而是避免在證據不足時把診斷偷換成宣判。

Applications

把歸因校正帶回工程現場

對 firmware 與跨團隊整合來說，最昂貴的不是單一失誤，而是失誤後還沿著錯的故事修三輪。

Incident review

先分開 proximate cause、contributing factors 與責任判斷，避免第一版情緒敘事綁死後續調查。

Code review

看到奇怪寫法時，先問是需求壓力、歷史相容性還是作者真的漏掉基本原則。

跨部門協作

若測試、韌體與硬體各自只看本地上下文，很容易把彼此的盲點誤讀成故意擺爛。

  • 每次想說「他們就是故意不配合」之前，先列三個非惡意但可驗證的替代解釋。
  • 如果最後真的要談責任，也要先把可修的流程與介面缺陷寫清楚，避免下次重演。

Misreadings

三個常見走偏點

韓隆剃刀常被誤聽成善意口號，但它真正要求的是更好的證據排序。

所以不要究責 校正

不是。它只是要求你把究責建立在更扎實的證據上。

惡意幾乎不存在 校正

也不是。惡意存在，只是不能因為結果很糟就直接跳到那個結論。

只要出於善意就沒問題 校正

善意不會自動讓系統安全；可修的能力與流程問題仍然要修。

Questions

什麼時候該先收情緒，什麼時候該提高警覺

我現在手上的證據，是在說明機制，還是在滿足情緒

如果把所有描述人品的字刪掉，剩下的事實還支不支撐我的判斷？

若要把這件事判成惡意，我還需要看到什麼

是重複模式、掩飾行為、利益衝突，還是明知後果仍持續推進的記錄？
