Shannon Entropy

一串資料真正佔多少資訊，不看它表面有多長，而看接收者在看見前有多少不確定性被消掉。

Orientation

資訊量來自不確定性被消除

香農熵衡量的是一個訊息來源的平均不確定性；越難猜、越平均分散，看到結果後消除的疑問就越多。

Core Tension

工程現場常把資料量當資訊量，但一百萬筆幾乎相同的 log 可能比一小段罕見 fault trace 含有更少決策資訊。

01 先看分布

若某個值 99.9% 都相同，它每次出現帶來的新資訊很低；罕見分支才可能改變理解。

02 再談編碼

好的壓縮能利用可預測性；接近隨機的資料則很難再被無損壓縮。

03 最後連到通道

有限 bandwidth 下，應優先送能降低不確定性的欄位，而不是最容易取得的欄位。

Setup

兩份同樣大的 log，價值可能差很多

把情境縮到 field telemetry：裝置每秒回傳 200 個欄位，其中大部分長期不變，真正能定位問題的是少數狀態轉移。

最小場景

一份 50 MB log 幾乎全是正常 heartbeat；另一份 5 MB log 包含 power mode、error counter 與 reset reason 的罕見變化。

真正難點

如果只用容量衡量資料價值，團隊會花成本傳輸與保存大量可預測訊號，卻漏掉真正降低 debug 不確定性的欄位。

Key Point

香農熵讓你把資料設計從「多收一點」改成「哪些欄位最能減少下一次判斷的不確定」。

Lenses

從資訊量看資料，而不是只看容量

同一筆資料可以在儲存、通訊、觀測與決策上有不同價值；熵提供一個共同語言來討論可預測性。

平均驚訝

越意外的事件帶來越多資訊；高機率事件帶來的新增資訊較少。

冗餘

重複欄位、固定格式與高度相關訊號能幫助壓縮，也表示原始資料量高估了資訊量。

編碼下限

無損壓縮不可能永遠突破來源熵；若資料近似隨機，檔案縮不下來是正常結果。

決策資訊

對工程判斷有用的不是最大熵，而是能區分狀態、故障與假設的資訊。

資料現象 熵的讀法 工程動作
大量相同 heartbeat 可預測性高，資訊密度低 改成事件式或摘要式回報。
少見錯誤碼 出現機率低但診斷價值高 保留上下文與前後狀態。
壓縮效果差 來源可能接近隨機或已壓縮 檢查是否仍值得逐筆傳輸。

Psychology

資料越多越安心的錯覺

人面對不確定時，很容易用收集更多資料取代設計更好的觀測；香農熵把這個習慣拉回資訊密度。

容量有可見重量

檔案越大、欄位越多，越像做了完整監控；但它們可能只是重複同一個已知狀態。

罕見事件被低估

少見 fault trace 平常看不到，容易被產品化 telemetry 排除，直到事故後才發現缺它。

壓縮被誤讀成魔法

壓縮率不是工具好壞的唯一指標，它也反映來源本身的可預測性。

熵不是要你追求資料越亂越好，而是提醒你用不確定性下降來評估觀測價值。

Applications

改寫 log、trace 與 telemetry 設計

Firmware 常受限於 flash、RAM、上傳頻寬與隱私；香農熵能幫你挑出最該留下的訊號。

事件式 logging

固定週期 heartbeat 可轉成狀態變化、counter delta 與 anomaly snapshot，讓低熵資料少佔通道。

壓縮策略

若欄位高度相關，delta encoding、run-length 或 dictionary 可能比通用壓縮更可控。

debug trace

trace buffer 應保留能區分假設的狀態，例如 transition、timestamp、error context，而不是只堆重複訊息。

  • 設計 telemetry 時，替每個欄位寫一句：它會消除哪個 debug 不確定性？
  • 資料保留政策不要只看大小，也要看罕見事件是否被低頻摘要抹掉。

Misreadings

別把熵直接等同混亂

香農熵的語境是訊息來源與機率分布，不是日常口語裡的凌亂程度。

熵高代表資料品質高 校正

不一定。純隨機 noise 熵高，卻未必對診斷有用。

檔案越大資訊越多 校正

檔案大小包含冗餘、格式與重複；資訊量要看不確定性被消除多少。

壓縮率就是唯一目標 校正

高壓縮率可能表示資料重複，也可能表示你收了太多沒有決策價值的訊號。

Questions

把資料量換成資訊量來問

如果只能保留 10% telemetry

哪些欄位最能分辨正常、退化與故障？哪些欄位只是重複已知狀態？

如果壓縮率突然變差

是來源變得更隨機、格式改變、加密已經發生，還是系統進入更複雜的狀態？
