Orientation

資料會限制解釋，但不一定會替你選完解釋

證據不足決定不是說證據沒用，而是說有限證據常需要搭配簡潔性、範圍、可測性和背景知識來選擇理論。

Core Tension

我們想讓資料說話，但資料通常透過模型、測量、假設和背景理論被解讀；不同理論可能都能解釋目前結果。

01 列出候選解釋

至少保留兩個能解釋資料的理論，避免把第一個故事當事實。

02 暴露輔助假設

測量方式、環境條件、取樣偏差和工具誤差都可能幫某理論撐場。

03 設計區分測試

下一步實驗應該讓候選理論產生不同預測。

Setup

同一串錯誤 log 可以指向網路、序列化或版本不一致

最小場景是觀察資料不足以區分多個根因。

最小劇情

裝置上報資料間歇失敗。log 顯示 timeout 和 checksum error；可能是網路抖動、韌體序列化 bug，也可能是 server schema 變更後兼容不足。

真正難點

每個解釋都能套上現有證據，真正需要的是下一個能拉開差異的觀測。

Key Point

證據不足決定要求你把『符合資料』和『最佳解釋』分開。

Lenses

用下一個觀測區分理論

當前資料若無法決定，好的問題是：哪個新證據會讓候選解釋分出勝負。

預測差異

不同理論對下一次實驗、特定環境或邊界案例有什麼不同預測？

解釋成本

哪個理論需要更多額外假設才能保住？

適用範圍

哪個理論能解釋更多案例，同時保留可被推翻的風險？

Psychology

人會把最順的故事誤當成證據已經決定

當某個解釋能把零散資料串起來，心理上會很舒服；但舒服不等於唯一。

故事完整感很有說服力

能串起資料的故事容易被記成真相。

第一個可行解釋會定錨

debug 時第一個 plausible root cause 會吸住後續觀察。

不確定讓人想收斂

團隊壓力越大，越想把多重解釋快速壓成單一結論。

把每個根因假設寫成會被什麼觀測推翻，才能讓資料繼續工作。

Applications

在資料不足時管理候選解釋

Debug、A/B test 和模型選擇都常遇到多個理論同時符合資料。

Incident debug

根因會議保留候選清單，要求每個候選提出下一個區分觀測。

A/B test

效果差異可能來自功能、流量品質、季節性或實驗污染，不只是一個敘事。

模型選擇

多個模型同樣 fit 現有資料時，要用外部驗證、簡潔性和失敗模式比較。

Misreadings

不是所有理論都一樣好

證據不足決定不是相對主義；它要求補上理論選擇判準和新測試。

以為證據沒有力量 校正

證據會排除很多理論，只是不一定排到只剩一個。

把偏好當判準 校正

選擇理論需要可說明的簡潔性、範圍和預測差異。

停在多元解釋 校正

列候選只是開始，下一步要設計區分測試。

Questions

問哪些解釋同樣符合資料

下面兩問適合用在 debug、研究設計和模型比較。

還有哪些理論能解釋同一組證據？

至少列出兩個替代解釋和各自需要的假設。

下一個觀測如何區分它們？

設計能讓候選理論產生不同預測的測試。
