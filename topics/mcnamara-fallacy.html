McNamara Fallacy

當決策只承認可量化的東西，最後常不是世界變清楚，而是把最關鍵卻難量的部分一起從判斷裡刪掉。

Orientation

看得見的數字，常不是整個問題，而只是最容易被看到的切片

麥納瑪拉謬誤抓到的是一種管理誘惑：把可量當成可信，把難量當成次要，最後把判斷變成報表消費。

Core Tension

量化能減少模糊，但如果決策只肯相信量化結果，組織就會系統性低估文化、信任、風險傳導與真正的現場摩擦。

01 先問沒被量到的是什麼

真正難處往往在報表外，例如心智模型、協作邊界與制度信任。

02 再問量化是否偷換了本體

容易計數的東西，不一定就是最值得優先管理的東西。

03 最後補上質性判準

如果只能看儀表板，決策就會開始對儀表板負責，而不對現實負責。

Setup

從一張漂亮的 dashboard 開始

麥納瑪拉謬誤不是說數據沒用，而是說『只有數據才算數』這個信念本身會製造盲區。

最小劇情

團隊用 defect count、velocity 與 coverage 追蹤開發品質，結果 review 深度、設計清晰度與跨模組理解這些難量卻關鍵的東西逐漸退位。

真正難點

一旦管理只認表上的量，所有不在表上的訊號都會被視為主觀、雜訊或情緒。

Key Point

麥納瑪拉謬誤最危險的地方，不是它讓你不知道，而是它讓你誤以為自己已經知道夠多。

Lenses

四個看報表偏食的角度

不是所有數字都壞；真正要問的是，哪些重要東西在你現在的量測設計裡根本沒有說話權。

可量不等於重要

容易收集的資料往往只是低摩擦指標，不一定代表高價值判準。

質性訊號不等於不可信

很多早期風險其實先出現在抱怨、猶豫、繞路與不安，而不是等到統計量變壞。

指標會篩選注意力

一旦績效會議只看報表，大家自然把語言和努力都轉向報表欄位。

盲區常在跨層級介面

領導看到的是抽象數字，現場承受的是具體摩擦，兩者之間最容易失聯。

管理情境 只看數字時容易得到的結論 被漏掉的真問題
開發效率 velocity 夠快所以流程沒問題 設計債、跨模組依賴與知識集中可能正在堆高
產品滿意度 NPS 沒掉代表使用者沒事 高摩擦但高黏著使用者可能只是暫時忍耐
供應商品質 退貨率低表示製程穩 現場也許靠人工補救把風險吞進內部成本

Psychology

為什麼人會對可量的東西特別安心

因為數字提供了一種很強的秩序感，好像只要畫面夠整齊，世界就已經被管理住了。

可比較感會降低焦慮

數字讓模糊問題變得像能排序、能打分、能責任歸屬。

質性資訊處理成本高

對管理者來說，聽現場細節比看趨勢圖更慢、更吵，也更難統一。

報表會反過來塑造現實

當組織只獎勵數字敘事，細膩但關鍵的現場觀察就越來越不敢被提出。

這個謬誤不是反報表，而是提醒你『有表可看』和『問題已被看見』是兩件不同的事。

Applications

把難量但重要的東西重新放回工程判斷

對 firmware、硬體整合與產品團隊來說，很多最昂貴的失誤，正好都長在量測不完整的區域。

Review 品質

comment 數量和 merge 速度都很好看，不代表 reviewer 真的理解了時序、狀態機或風險邊界。

客訴追蹤

ticket 量下降不一定是體驗改善，也可能只是使用者學會繞過或放棄回報。

跨團隊合作

會議出席率和 SLA 都符合，不代表資訊真的被共享到可執行的程度。

  • 每次看報表時，強迫自己補一欄『現在最重要但沒有被量到的是什麼』。
  • 如果高層看不到現場的質性訊號，就要刻意設計能把異常故事帶上來的機制。

Misreadings

三個常見走偏點

麥納瑪拉謬誤不是叫你拒絕數據，而是叫你別把數據當作世界的邊界。

只要補更多欄位就行 校正

欄位再多也可能只是把可量部分堆得更滿，仍沒有碰到真正難量的核心。

質性資訊都很主觀 校正

主觀不代表無用，很多風險本來就只能先透過敘事與經驗浮現。

這只是歷史案例 校正

今天的 dashboard culture、A/B test culture 與績效看板同樣會遇到這種偏食。

Questions

我現在看的是現實，還是現實的方便切片

哪些最讓現場頭痛的摩擦，現在完全不在報表裡

如果回答不出來，通常代表決策界面太乾淨了。
