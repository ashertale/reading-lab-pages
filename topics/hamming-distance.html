漢明距離

Hamming Distance

兩個碼字差幾位，不只是相似度問題；它決定錯誤能不能被偵測、校正，或被誤認成另一個合法狀態。

Orientation

表示法本身也能提供保護

漢明距離計算兩個等長字串在多少位置不同；在二進位碼裡，它直接對應需要翻轉多少 bit 才能把一個碼字變成另一個。

Core Tension

若合法碼字彼此距離太近，少量 bit error 就可能把錯誤資料變成看似合理的另一個狀態；距離拉開後，錯誤才有機會被發現或修正。

01 先看最小距離

整組合法碼字中最近的一對，決定了系統能偵測或校正錯誤的下限。

02 再分偵測與校正

距離足夠大可以發現錯誤；要自動修正則需要更大的間隔，避免猜錯目標碼字。

03 最後設計非法區

不是所有 bit pattern 都該合法；保留無效碼可讓硬體或韌體更早發現資料毀損。

Setup

一個 bit flip 變成合法命令

想像 bootloader 的 command byte 只用連續整數表示：0x01 erase、0x02 write、0x03 verify。

最小場景

傳輸或 NVM 讀取時翻了一個 bit，原本非法或低風險的值可能剛好變成 erase；若沒有 CRC、magic pattern 或距離設計，錯誤會被當成正常命令。

真正難點

很多協定欄位被當成語意問題處理，卻忽略 bit-level 距離；錯誤不是只會產生亂碼，也可能產生有意義但危險的碼。

Key Point

漢明距離讓你在設計表示法時先留下錯誤緩衝區，而不是事後只靠例外處理。

Lenses

用距離檢查狀態與封包

同樣是幾個 bit，安排方式不同，錯誤偵測能力就會不同；表示法是可靠性設計的一部分。

最小碼距

若任何兩個合法值只差一位，單 bit error 就可能無聲變成另一個合法值。

校驗碼

parity、checksum、CRC 與 ECC 都是在資料旁邊加入能揭露或修正錯誤的結構。

保留碼

故意不用滿整個編碼空間，可以讓錯誤落入 invalid state，而不是落入危險指令。

語意距離

高風險命令不應與常見命令碼距太近；bit-level 距離應配合操作風險。

Psychology

人習慣把編號當成無害細節

狀態碼常在實作尾端被快速分配；漢明距離提醒你，編號方式會影響錯誤後的行為。

語意遮住位元

工程師討論 erase/write/verify 的語意，卻少看它們在 bit pattern 上是否貼得太近。

測試少碰資料毀損

正常流程測試很少隨機翻 bit，所以錯誤距離問題常到 field 才被看見。

校驗被當成額外成本

CRC 或 ECC 需要空間與時間，但它們買的是錯誤不被默默合理化的能力。

距離設計的價值通常表現在壞事發生時：錯誤變成可見失敗，而不是安靜地做錯事。

Applications

用在 NVM、協定與安全命令

Firmware 常處理 raw bytes；漢明距離能把資料表示、錯誤校驗與危險操作一起納入設計。

NVM metadata

magic number、version、length 與 CRC 應共同防止半寫入或 bit rot 被當成有效資料。

Bootloader command

erase、unlock、mass update 這類高風險命令可使用多 byte magic 或 challenge，避免單點翻轉觸發。

狀態機編碼

安全狀態與危險狀態不要只差一個 bit；必要時加入冗餘欄位與交叉檢查。

  • 替高風險碼字做 bit-flip audit：翻 1 到 2 位後會變成什麼？
  • 別把所有 pattern 都用滿；非法狀態是錯誤偵測的一部分。

Misreadings

它不是只給通訊理論用

漢明距離也能幫你設計狀態、命令與持久化資料，而不只是考試裡的編碼題。

只要有 checksum 就不用管碼距 校正

checksum 很重要，但核心命令與狀態碼本身仍應避免危險接近。

距離越大永遠越好 校正

距離需要空間與成本；要依錯誤模型與操作風險分配。

錯誤只會讓資料無效 校正

若合法碼字太密，錯誤可能產生另一個有效狀態，這比直接 fail 更危險。
