欠壓重置

Brown-Out Reset (BOR)

欠壓重置不是多一道保守開關，而是把電壓落在不可靠區間時的混亂執行，轉成可預期的重置或保持狀態。

Read For

適合看 MCU 上電、電池設備、馬達啟動、flash write、低功耗喚醒與 field 中偶發重開或狀態毀損問題。

主題類型
嵌入式可靠性 / 電源監控

Orientation

低電壓時最危險的是半醒著執行

Brown-out reset 偵測供電低於門檻後讓 MCU reset 或保持 reset，避免核心與外設在未保證區域繼續操作。

Core Tension

系統想撐過短暫電壓下陷，但若電壓已低到規格不保，繼續跑可能比停止更糟。

01 先查安全電壓

CPU frequency、flash programming、ADC 與 oscillator 各有不同最低條件。

02 再設 reset 門檻

BOR threshold 要高於真正失控區，並考慮容差與溫度。

03 最後控制釋放

電壓回升後仍需等待 clock、rail 與外設狀態穩定。

Setup

馬達一啟動，設定頁就偶發毀損

產品在 field 偶爾丟設定。log 看不到 crash，重現時發現馬達 inrush 讓 Vcc 短暫掉到 flash programming 規格以下。

最小場景

韌體剛寫入 calibration page，電壓下陷讓 flash controller 行為不再保證，CPU 卻還沒 reset。

真正難點

軟體看到的是下一次開機 checksum 錯；真正事件發生在前一次欠壓窗口。

負載拉低 Vcc
→
MCU 落入未保證區
→
BOR 介入或資料被寫壞
Key Point

BOR 是把模糊的低電壓行為切成明確的停與放行。

Lenses

欠壓保護要同時看類比門檻與重置語意

只知道有 BOR 不夠；門檻、遲滯、反應時間與 reset release 都會影響結果。

Threshold

門檻要對齊 MCU 速度、flash 寫入與外設最低操作電壓。

Hysteresis

進出門檻差距避免供電在邊界附近反覆 reset chatter。

Reset delay

電壓剛回升時，clock、regulator 與外部感測器可能尚未穩定。

Supervisor choice

內建 BOR 方便，外部 supervisor 可提供更精準門檻或多 rail 監控。

症狀可能電源線索檢查方法
偶發設定毀損 NVM 寫入遇到 Vcc sag 示波器觸發 flash write 與 Vcc
啟動循環 reset release 太早或遲滯不足 量 reset pin、rail ramp 與 clock ready
高負載才重開 inrush 或 regulator transient 加電子負載與 motor start profile

Psychology

Brown-out 很像軟體鬼影

欠壓常留下的是下一次開機才看見的狀態異常，因此 debug 直覺會先追 firmware flow。

Log 斷在錯地方

電壓掉下去時 log 也可能寫不完，留下錯誤時間線。

實驗室電源太乾淨

bench supply 不會重現電池內阻、線材、馬達 inrush 與溫度條件。

Reset 被當成功能

反覆重啟看似恢復，實際可能掩蓋資料一致性問題。

遇到 field-only 的隨機重啟或資料毀損時，把電源波形放進第一輪證據。

Applications

把電源安全條件寫進韌體狀態機

BOR 是硬體保護，但韌體仍要配合處理寫入、恢復與診斷。

NVM write gate

寫 flash/EEPROM 前確認 supply margin，寫入中避免進入高負載動作。

Battery products

低電量時降頻、停馬達或延後 radio burst，讓 BOR 不只是最後防線。

Field diagnostics

保存 reset reason、brown-out count 與最近一次 high-current action，幫助回推現場條件。

  • 用示波器同時量 Vcc、reset、clock ready 與 high-current enable，才看得到事件順序。
  • 若有多個 rail，確認 BOR 監的是哪一條，其他 rail 掉電是否也會破壞外設狀態。

Misreadings

BOR 不是電源設計的替身

它能避免未定義執行，卻不能補足太小的 bulk capacitor、差的 layout 或不合理的負載階躍。

有 BOR 就安全 校正

門檻錯、反應慢或 reset release 太早，一樣會留下不可靠窗口。

閾值越高越好 校正

太高會造成正常 transient 被誤判，讓產品頻繁重啟。

軟體 ADC 監控足夠 校正

當 CPU 或 ADC 自身已不可靠時，軟體偵測可能太晚。
