精確率與召回率

Precision and Recall

同一套偵測系統可以少誤報，也可以少漏報；精確率與召回率迫使你說清楚哪種錯比較貴。

Orientation

兩種好，分母不一樣

Precision 問的是被你抓出來的有多少是真的；recall 問的是所有真的案例裡你抓到了多少。

Core Tension

想少誤報，就會提高門檻；想少漏報，就會降低門檻。指標不是替你消除取捨，而是讓取捨被明白寫出來。

01 Precision 看被選中的品質

高 precision 代表工程師打開告警時，比較少遇到空包彈。

02 Recall 看真案例的覆蓋

高 recall 代表真問題比較不容易被系統漏掉。

03 門檻決定立場

同一個模型只要換 threshold，就可能從保守偵測變成激進篩選。

Setup

一條過熱告警規則該多敏感

假設 firmware 監控某電源 rail 的 transient，偵測器會把 log 片段標成 suspicious 或 normal。

最小場景

門檻設得太寬鬆，半夜告警大量湧入；門檻設得太嚴格，少數真異常滑過 release gate。

真正難點

團隊不是在追求一個抽象最高分，而是在決定哪種錯誤會傷害安全、吞吐、信任或工程時間。

Key Point

精確率與召回率讓你把「抓得準」和「抓得全」分開談。

Lenses

看分類器時先拆四件事

沒有錯誤成本的指標討論，很快會變成大家各自挑一個好看的分數。

混淆矩陣

TP、FP、TN、FN 是所有後續指標的底層帳本。

門檻移動

降低 threshold 通常提高 recall、犧牲 precision；提高 threshold 則反過來。

基準率

罕見事件下，即使 false positive rate 很低，也可能產生大量誤報。

人力回路

若每個 positive 都要工程師讀 log，precision 會直接影響 triage 成本。

目標偏好的指標例子
漏掉真失效很危險 偏 recall 安全相關 fault detection
誤報會耗盡人力 偏 precision 每日人工 triage queue
排序後人工看前幾筆 看 Precision@k 搜尋、缺陷推薦、相似 log

Psychology

人很容易愛上單一分數

單一分數帶來管理上的舒服感，但它也會把錯誤類型、資料分布與門檻選擇壓扁。

Accuracy 的誘惑

資料極度不平衡時，永遠猜 normal 也可能有漂亮 accuracy。

誤報比較吵

FP 會立刻打擾團隊，FN 則常等事故或客訴才被看見。

報表會塑造行為

如果 KPI 只獎勵 precision，大家可能把困難案例留在系統外。

好的評估會把錯誤代價寫進決策，而不是只把分數放進簡報。

Applications

從模型評估回到工程流程

不只 ML 需要這組概念；任何篩選、告警、搜尋與測試判讀都在做 precision/recall 取捨。

ATE failure triage

若 false fail 會卡住大量良品，precision 重要；若 false pass 會出貨風險，recall 不能低。

Log classifier

前幾筆要給人看的推薦系統，通常要把 Precision@k 設為主要觀察點。

Security alert

初期偵測可偏 recall，但必須設計分級與去重，否則人會被告警淹沒。

  • 每次調 threshold 都記錄 precision、recall、樣本數與人工處理成本。
  • 在 review 裡問清楚：這個系統寧可吵一點，還是寧可漏少一點？

Misreadings

F1 不是風險委員會

F1 把 precision 與 recall 合成一個數，但它不知道你的 false negative 是小麻煩還是大事故。

F1 最高就最好 校正

如果錯誤代價不對稱，F1 可能選到商業或安全上錯的門檻。

Precision 高代表模型可靠 校正

也可能只是門檻太高，幾乎什麼都不抓。

Recall 高代表系統安全 校正

抓很多不等於可用；沒有 triage 能力，高 recall 會變成噪音製造器。

Questions

先替錯誤標價，再調門檻

如果漏報一次

它會造成 field return、安全風險、客戶信任損失，還是只是一個可補的分析延遲？

如果誤報一百次

誰要讀、誰要決策、哪個流程會被卡住，系統能否自動降噪或分級？
