波斯托爾定律

Postel's Law / Robustness Principle

對外送出的東西要保守清楚，對外接收的東西要寬容穩健，但這種寬容若失控，也可能把錯誤與歧義養成永久相容性包袱。

Read For

這一頁要把『寬容地收』的短期好處和長期風險拆開，因為很多協定穩定性與相容性債都從這裡長出來。

Orientation

寬容能救相容，但也可能悄悄把歧義升格成事實標準

波斯托爾定律在網路協定時代非常有用，但到了大型 API 生態，它也常被拿來合理化過度容忍錯誤輸入。

Core Tension

如果接收端太嚴，互通性會下降；如果接收端太寬，錯誤行為會被默默吸收，久了反而變成大家都得相容的隱性規格。

01 輸出端先守契約

你送出去的東西越模糊，下游越難知道什麼才是正確用法。

02 接收端要分清容錯與縱容

短期容忍能提升兼容，但也可能讓錯誤格式被永久保留下來。

03 演化時要設計收斂路線

若現場已經養出歧義輸入，要用 deprecation、telemetry 與明確錯誤訊息把生態拉回來。

Setup

從一個看似善意的 parser 開始

很多工程團隊一開始只是想讓東西更能互通，最後卻變成每個人都在猜系統到底真正接受什麼。

最小劇情

某個裝置通訊欄位明明要求固定格式，但接收端為了容錯接受多種變體；多年後，各家工具都依賴不同變體，誰也不敢收緊。

真正難點

寬容輸入在短期看起來像穩健，長期卻可能把錯誤資料流與怪異依賴一起養大。

為了互通而放寬接收
→
錯誤輸入也被默默接受
→
歧義行為變成隱性契約
Key Point

波斯托爾定律最值得學的，不是機械地寬容，而是知道寬容會把什麼樣的未來一起買進來。

Lenses

四個看容錯邊界的透鏡

不是所有寬容都壞；真正要問的是，它在解決短期互通，還是在製造長期語意混亂。

契約清晰度

協定文件是否已足夠明確，還是大家其實在靠實作猜測真規則？

錯誤可見性

接收端若默默吞掉異常，發送端就永遠不會被迫修正。

生態規模

使用者越多，任何被接受的怪異行為越容易變成大家依賴的現實。

演化成本

現在的寬容是可回收的暫時橋接，還是未來每個版本都要背的相容性包袱？

做法 短期收益 長期代價
嚴格拒收非法格式 錯誤更早暴露 短期互通性與導入摩擦較高
默默接受多種變體 比較不會當場壞掉 錯誤輸入被養成隱性規格
接受但回警告 兼容與可見性可兼得 需要團隊真的追警告與安排收斂

Psychology

為什麼寬容輸入看起來總是比較善良

因為立即不壞掉比長期語意乾淨更能被感受到，團隊自然傾向買短期平順。

當下成功會壓過未來成本

資料一進來能跑通，會讓人低估之後維護每種怪異變體的代價。

錯誤被吞掉就不再被看見

一旦接收端代為修正，真正製造錯誤的人反而感受不到問題。

相容常被當成仁慈

團隊容易把『能跑』當成『設計合理』，忽略語意正在慢慢腐壞。

這個原則最難的地方不是背口訣，而是知道哪種寬容在幫系統，哪種寬容在養債。

Applications

把容錯邊界帶回韌體與 API 設計

對 firmware、協定解析器與 SDK 來說，波斯托爾定律幾乎每天都在出現，只是常被包裝成『先讓它 work』。

封包解析

容忍欄位順序或空白差異能提升互通，但若連非法值都靜默接受，錯誤設備會永遠留在生態裡。

CLI 與設定檔

過度自動修正雖然方便，卻會讓使用者更難知道自己到底寫對了沒有。

SDK 演化

若函式對模糊輸入過度寬容，下游程式會把那些偶然行為當成正式契約。

  • 若決定容忍異常輸入，最好同步留下 telemetry、warning 或 deprecation 路徑。
  • 對外輸出的格式越清楚穩定，接收端就越不需要靠寬容去補上前端的鬆散。

Misreadings

三個常見誤讀

波斯托爾定律常被簡化成『永遠寬容』，但那其實只學到一半。

接收端越寬越好 校正

不是。過度寬容會把錯誤輸入固定成新的相容性負債。

只要能互通就是成功 校正

互通若建立在語意模糊上，之後的修復成本可能更高。

這只是網路協定老觀念 校正

今天的 JSON API、裝置設定與資料交換格式同樣會遇到這題。

Questions

我是在做穩健設計，還是在替未來養更多例外

下面兩個問題很適合放進 parser 與 API 設計 review。

如果今天接受這個模糊輸入，兩年後我還能輕易撤回嗎

若不能，它就不是小容錯，而是正在擴張契約面。

發送端是否能清楚知道自己送錯了

如果接收端把錯誤全部吞掉，整個生態就失去自我校正能力。
