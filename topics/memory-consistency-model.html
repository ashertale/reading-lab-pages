記憶體一致性模型

Memory Consistency Model

多核心系統裡，程式寫下的順序不必然就是其他核心、DMA 或中斷看見的順序；一致性模型定義了這份契約。

Orientation

程式順序不是觀察順序的保證

記憶體一致性模型定義多個執行單元對讀寫順序與可見性的共同規則；它是硬體、編譯器與程式之間的契約。

Core Tension

為了效能，CPU、cache 與 compiler 可能重排或延遲可見性；為了正確，程式必須在共享邊界使用適當同步。

01 先分 coherence 與 consistency

同一地址的寫入順序是一回事，不同地址之間的可見順序又是另一回事。

02 再看同步原語

atomic acquire/release、fence、barrier 與 lock 定義哪些讀寫不能被跨越。

03 最後對上硬體代理

DMA、peripheral、ISR 與另一顆 core 都可能有不同的 cache 與 ordering 要求。

Setup

資料寫完了，flag 卻先被看見

一個 producer 先填 descriptor，再把 ready flag 設成 1；consumer 看到 flag 後讀 descriptor。

最小場景

在弱排序或 compiler optimization 下，flag 的可見性可能早於 descriptor 欄位，或 consumer 的讀取被提早；結果 consumer 看到 ready，卻讀到舊資料。

真正難點

程式碼文字順序給了人安全感，但硬體與編譯器只遵守模型允許的順序，不遵守你的直覺。

Key Point

共享邊界不是靠註解保證，而是靠明確同步原語把順序寫進契約。

Lenses

看記憶體模型要拆四層

同一個 bug 可能來自 compiler、CPU、cache 或 peripheral ordering；逐層拆才能找出需要哪種 barrier。

Compiler reordering

編譯器可以在不改變單執行緒語義下重排；共享資料需要 atomic 或特定約束。

CPU reordering

核心可能讓 store buffer、load speculation 或 cache 行為改變其他 agent 的觀察順序。

Cache visibility

DMA 或 peripheral 可能看不到 CPU cache 裡尚未 clean 的資料。

Device ordering

MMIO 存取常需要特定 barrier，避免 register write 被合併、延遲或跨越。

場景 常見誤解 需要檢查
flag + payload 先寫 payload 就一定先可見 release/acquire 或 explicit fence。
DMA descriptor memory write 後 DMA 自然看得到 cache clean/invalidate 與 device barrier。
ISR flag volatile 等於同步 volatile 只處理可見存取，不等於跨核心 ordering。

Psychology

原始碼順序很會騙人

人閱讀程式時自然按行號想像執行；記憶體模型讓你改看誰被允許觀察到什麼順序。

單核心經驗外溢

在簡單 MCU 上成立的直覺，移到 cache、多核心或 DMA 後可能失效。

volatile 被過度信任

volatile 常被當成同步萬用工具，但它無法替代 atomic ordering 與 cache maintenance。

偶發性讓 bug 像硬體問題

ordering bug 常隨負載、最佳化等級與中斷時序出現，容易被誤判成雜訊。

一旦共享資料跨越 core、ISR、DMA 或 peripheral，就該把同步契約寫出來。

Applications

用在 DMA、ISR 與多核心 firmware

Firmware 最容易踩到記憶體模型的地方，往往不是高階 lock-free，而是描述子、flag、cache 與 MMIO 的邊界。

DMA ring

CPU 更新 descriptor 後，要確保資料與 descriptor 對 DMA 可見，再啟動 doorbell。

ISR/main handoff

若 ISR 發布事件並填資料，main loop 讀取前需要明確的順序與臨界區策略。

多核心共享狀態

release/acquire 或平台 barrier 要與 cache policy 一起設計，不要只靠 shared struct。

  • 替每個 shared flag 寫出它保護哪些 payload，以及需要哪種 publish/consume ordering。
  • 把 cache maintenance、barrier 與 MMIO ordering 放進 driver API 合約，而不是散在呼叫端。

Misreadings

volatile 不是記憶體模型答案

volatile 可能防止某些編譯器最佳化，但不代表跨核心或 DMA 的同步已經正確。

行號就是順序 校正

原始碼順序只是 programmer view；其他 agent 看到的順序要看模型與 barrier。

加 delay 可以修 ordering 校正

delay 只能改時間，不提供正式可見性或順序保證。

所有平台都一樣 校正

x86、ARM、RISC-V、MCU bus 與 DMA 的規則不同；要讀平台手冊。

Questions

把共享邊界改寫成同步契約

如果 consumer 看到 ready flag

哪些資料被保證已經可見？這個保證來自 atomic ordering、barrier、lock，還是只是直覺？

如果 DMA 讀到舊資料

CPU cache 是否 clean？descriptor write 與 doorbell write 之間是否有 device ordering 保證？
