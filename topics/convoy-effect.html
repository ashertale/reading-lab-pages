護送隊效應

Convoy Effect

護送隊效應出現時，整個隊伍的速度被最慢工作拖住；快工作不是不能跑，而是排在慢工作後面。

Orientation

慢工作卡在前面時，後面再快也只能一起慢

護送隊效應不是平均負載太高，而是排隊順序和共享資源讓慢工作形成移動瓶頸。

Core Tension

先來先服務看似公平，但如果服務時間差異很大，公平順序會把短工作綁在長工作後面，讓系統看起來整體變慢。

01 看隊首而不是只看平均

平均 latency 可能還好，但 p95 或 p99 被少數慢任務拖長。

02 隔離長短工作

把 blocking I/O、CPU heavy 與短 request 分池或分級處理。

03 讓排程看見成本

如果 scheduler 不知道任務長短或資源需求，就容易把不相容工作排在一起。

Setup

一個慢 I/O 任務佔著 worker，短請求排隊等它釋放

最小場景是快工作和慢工作共用同一條隊伍與同一批 worker。

最小劇情

API thread pool 同時處理短查詢和外部檔案下載。幾個下載任務阻塞 worker 後，原本十毫秒可完成的查詢也開始排隊數秒。

真正難點

系統不是沒有算力，而是排程把不同服務時間的任務混在同一等待線裡。

慢任務佔住共享資源
→
短任務在後面等待
→
整體延遲被隊首放大
Key Point

護送隊效應的修正重點，是讓短工作不要被長工作當成同一種負載處理。

Lenses

從等待線找出誰在拖著誰

看 CPU 使用率不夠，要追蹤任務等待時間、服務時間和共享資源持有時間。

服務時間分布

短任務和長任務是否被混在同一隊列，導致短任務被無關長任務拖累？

資源持有

慢任務是否長時間持有 worker、mutex、socket 或 DMA channel？

排程策略

FIFO、公平排程或批次處理是否在特定負載下放大尾端延遲？

Psychology

團隊常先加 worker，卻沒有問工作是否該排在一起

看到排隊就擴容很直覺；但若隊列混雜或共享資源設計不當，擴容只會延後問題。

平均值遮住尾端

儀表板的平均延遲可能讓護送現象不明顯，直到使用者感受到尖峰。

公平感掩蓋效率損失

FIFO 容易被理解為公平，卻可能讓大量短工作為少數長工作付出代價。

局部慢被誤判成全局慢

慢外部依賴讓整個 worker pool 卡住時，團隊可能以為後端整體容量不足。

處理護送隊效應時，先把等待和服務拆開量測，再決定分流、限流或重新排程。

Applications

讓慢工作有自己的通道

Thread pool、RTOS 任務與資料處理 pipeline 都需要避免不同成本工作互相排隊。

Thread pool

把 blocking I/O 和短 request 分到不同 pool，並限制長工作佔用比例。

RTOS 排程

確認低速周邊操作不會長時間阻塞高頻控制任務的執行時槽。

資料 pipeline

把大檔案批次和小事件流分開處理，避免少數大工作拉長整批提交。

Misreadings

護送隊效應不是所有排隊問題

如果每個工作都一樣慢，那是容量或算法問題；護送隊效應特別關心慢工作拖住快工作。

只靠加 worker 校正

更多 worker 可能緩解，但如果共享鎖或外部 I/O 仍卡住，尾端延遲會回來。

把 FIFO 當永遠公平 校正

公平順序不等於公平結果；短工作被長工作拖住時，使用者體感會很差。

忽略資源持有時間 校正

真正的瓶頸可能不是任務執行，而是它持有的鎖或連線讓後續工作不能開始。

Questions

找出隊首正在拖住誰

下面兩問適合用在 latency debug 與 scheduler review。

哪些短工作排在長工作後面？

把等待時間和服務時間分開看，確認快工作是否被隊首拖慢。

慢工作持有哪些共享資源？

檢查 worker、mutex、socket 或硬體通道是否被長時間占住。

延伸路線

可接著讀 scheduling，理解排程策略如何在公平、吞吐與延遲之間取捨。
