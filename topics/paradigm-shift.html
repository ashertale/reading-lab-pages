Orientation

如果新理論只是比較準，為什麼歷史上的大轉變常像整個世界觀一起被換掉

典範轉移 不像一句立場聲明，它更像討論時的排序器：先把 框架更替 放回桌面，再決定 常態科學 和 危機累積 要怎麼分配。

Core Tension

有些改變不是在舊框架內修補，而是連『什麼問題值得問、什麼證據算有力』都一起改寫。

01 先把 框架更替 拉回起點

很多爭論表面在談結論，其實一開始就把 框架更替 當成默認前提。

02 再問 常態科學 是誰在付

現場最常漏掉的不是立場，而是 常態科學 到底由誰承擔、何時結帳。

03 最後確認 危機累積 誰能檢查

真正把事情做完的，往往不是表態，而是把 危機累積 翻成可被檢查的動作。

Setup

哪個前提一旦不寫，典範轉移 就會被講歪

把討論縮到一次具體決策，框架更替 是否先被交代、常態科學 是否被偷省、危機累積 是否有驗收方式，就會一眼看出來。

最小劇情

舊框架下累積太多異常後，新框架不是只補一個洞，而是連評價標準、問題語言與工具箱都跟著換了一套。

真正難點

如果新理論只是比較準，為什麼歷史上的大轉變常像整個世界觀一起被換掉

有人先把 框架更替 當既成前提
→
真正要付的 常態科學 被留到後面
→
收尾時才發現 危機累積 根本沒有操作方式
Key Point

把 典範轉移 用好，重點不在記住一句格言，而在於知道 框架更替 要先說、常態科學 要有人扛、危機累積 要能驗收。

Lenses

先把 問題本身改寫、標準一起換 與 新舊語言不完全可直譯 分開看

如果不把 問題本身改寫、標準一起換 與 新舊語言不完全可直譯 拆開，你很容易以為大家在討論同一條原則，實際上是在搶不同的判準。

問題本身改寫

順著 問題本身改寫 讀，你會發現爭點不在口號，而在 框架更替 是否真的被變成判準。

標準一起換

把焦點放在 標準一起換 時，你其實是在說：常態科學 應該先被寫清楚，否則後面的結論都站不穩。

新舊語言不完全可直譯

新舊語言不完全可直譯 這條線在乎的，不是誰比較有氣勢，而是 危機累積 一旦沒被交代，現場會怎麼開始失真。

Psychology

為什麼人一忙起來就最容易把 典範轉移 用歪

這類原則常被用歪，不是因為內容太難，而是因為把它說成一句漂亮話，比真的交代 框架更替、常態科學 和 危機累積 容易太多。

只把改變看成版本升級

碰到 只把改變看成版本升級 時，最危險的是大家以為原則已經被引用，實際上 框架更替 還沒被具體說明。

框架依戀

框架依戀 會讓人覺得只要把話說得夠完整，就不必再補 常態科學 的證據或代價。

熟悉成功不想放下

熟悉成功不想放下 常不是純粹錯誤，而是一種討論捷徑：先把自己放在合理的一邊，再晚點處理 危機累積。

一條原則會變空話，通常不是因為字太抽象，而是因為 危機累積 從來沒有被翻成現場動作。

Applications

回到 架構轉型、研究方向變更 與 團隊工作模型切換

真正有差的，不是會議上有沒有人引用 典範轉移，而是 架構轉型 和 研究方向變更 裡的決策流程有沒有因此變得更可檢查。

架構轉型

在 架構轉型 這種場景，典範轉移 幫你的不是贏辯論，而是把 框架更替 先釘在桌面上，避免事後才補。

研究方向變更

研究方向變更 最需要 典範轉移 的時候，通常就是大家都急著往前走，卻還沒把 常態科學 寫成共識。

團隊工作模型切換

只要到了 團隊工作模型切換，你就能看出 危機累積 究竟是被真的承擔，還是只是被口頭提過。

Misreadings

三個常見誤讀

這類頁面最常被讀歪的地方，不是概念太抽象，而是大家太快把 框架更替 當成價值宣示，忘了它還得連著 常態科學 和 危機累積 一起成立。

典範轉移 只是好聽口號 校正

真正難的是把 典範轉移 翻成一組可檢查的判準，而不是把它當成結論本身。

只要背了 典範轉移 就能直接套用 校正

原則若不落回 框架更替、常態科學 和代價分配，通常會在現場變成誤用。

典範轉移 只適合高層決策 校正

其實從 code review 到 框架與變革 的日常判斷，都會碰到同樣的壓力。

Questions

把 框架更替、常態科學 與 危機累積 變成會議上的追問

下面兩個問題不是要你再複述一次原則，而是逼 架構轉型、研究方向變更 這類場景裡的人把空白前提補完。

如果 架構轉型 現在就要拍板

誰要先把 框架更替 說清楚？常態科學 由誰承擔？若 危機累積 無法驗收，你還願意讓決定往前走嗎？

如果團隊說『典範轉移 不是大家都同意嗎』

怎樣才算真的用了 典範轉移，而不是只借它替既有結論加上道德光澤？
