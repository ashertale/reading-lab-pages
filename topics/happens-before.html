Happens-Before Relation

Happens-before 不是時間先後的直覺描述，而是語言與記憶體模型用來保證可見性與順序的偏序關係。

Orientation

先發生，要先由規則保證

Happens-before 定義事件間的偏序；若寫入 happens-before 讀取，語言模型才保證讀取能看見那個寫入或其後續一致結果。

Core Tension

你在 source code 裡先寫 data 再寫 flag，不代表另一個 thread 會照同樣順序看見；compiler、CPU 與 cache 都可能重排或延遲可見性。

01 先找同步邊

lock/unlock、release/acquire、thread start/join 等才會建立明確關係。

02 再用傳遞性

A happens-before B，B happens-before C，才能推 A 對 C 可見。

03 最後排除 data race

沒有同步的共享讀寫，常讓程式落入未定義或不可靠語意。

Setup

Ready flag 不等於資料已安全可見

Producer 寫入 buffer 後把 ready 設成 true；consumer 看到 ready 就讀 buffer。單次測試都對，不代表模型保證它永遠對。

最小場景

ready 不是 atomic，也沒有 memory barrier；consumer 可能看見 flag 卻讀到舊 buffer。

真正難點

人用牆上時間理解先後，CPU 和語言模型要求你用同步原語建立可見性。

writer 更新資料
→
同步原語發布狀態
→
reader acquire 後讀取資料
Key Point

Happens-before 是多執行緒程式可推理的地基。

Lenses

把順序拆成偏序，而不是時間線

並行程式裡有些事件有順序，有些事件彼此並發；happens-before 幫你分清楚哪一種。

Program order

同一 thread 內的語句順序通常提供基本排序，但不自動跨 thread。

Synchronizes-with

release/acquire、mutex 或 condition variable 可建立跨 thread 邊。

Transitivity

同步關係可以串起來，形成更長的可見性保證。

Race freedom

若衝突存取沒有被 happens-before 排序，就要檢查 data race。

做法是否建立關係原因
mutex unlock/lock 是 unlock synchronizes-with 後續 lock
sleep 10 ms 否 等待時間不是同步語意
atomic release/acquire flag 是 可發布並取得先前寫入

Psychology

多執行緒最會欺騙單執行緒直覺

人讀程式會自然想像一條時間線；memory model 要你證明可見性路徑。

測試通過的陷阱

race condition 可能只在特定 CPU、最佳化或負載下出現。

Delay 安慰感

等待讓 bug 機率下降，卻沒有建立語言保證。

volatile 語意混淆

不同語言的 volatile 承諾不同，不能把它當萬用同步。

Concurrency review 的好問題是：請指出哪一行建立 happens-before，而不是問跑起來有沒有錯。

Applications

從 ISR 到 C++ atomics 都需要可見性語意

嵌入式程式也會遇到記憶體模型問題，只是常包在 ISR、DMA、RTOS 與 compiler optimization 裡。

ISR/main loop

ISR 設 flag、main 讀資料時，要考慮 volatile、atomic、critical section 與 barrier。

C++ atomics

release store 搭配 acquire load 可以發布資料；relaxed 則不提供同樣排序。

RTOS primitives

queue、semaphore、mutex 通常同時提供同步與可見性，不只是排程。

  • 把 shared state 的 writer、reader 與同步原語畫成小圖，確認每條讀取路徑都有邊。
  • 對 lock-free code 特別保守；沒有註解說明 memory order 理由時，很難維護。

Misreadings

看起來先，不等於保證先

Happens-before 是規範語意，不是 log timestamp 的直覺排序。

加 delay 就安全 校正

Delay 改變時機，不建立同步關係。

volatile 就是 thread-safe 校正

volatile 不等於 atomic，也不一定提供跨 thread ordering。

x86 上沒壞就通用 校正

較強硬體順序不等於語言層保證；compiler 仍有優化空間。
