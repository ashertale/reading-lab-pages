線性化一致性

Linearizability

線性化一致性要求每個操作看起來都在呼叫與回應之間的某一瞬間生效，並尊重真實時間順序。

Read For

適合 review lock-free data structure、distributed KV store、leader-based service 與讀寫 API 的一致性承諾。

主題類型
並行正確性 / 強一致模型
核心問題
Concurrent operations appear to occur atomically in real-time order
主要判準
線性化點、real-time order、atomic object、history
讀完產出
能檢查一個並行物件或服務是否能被當成單一原子物件推理

Orientation

並行世界裡的原子瞬間

如果一個物件是 linearizable，每個操作都可以被放在呼叫與回應之間的某個點上，使整體結果符合合法的 sequential history。

Core Tension

線性化讓使用者推理簡單，代價是實作要處理同步、複製、延遲與故障；弱一致模型則把部分負擔交給 caller。

01 先找 linearization point

每個操作何時正式生效，必須能被指出。

02 再尊重 real-time order

一個操作完成後才開始的另一個操作，順序不能被顛倒。

03 最後看重疊操作

時間上重疊的操作可以用任一合法順序解釋。

Setup

寫入成功後，下一個讀取仍看到舊值

設定服務回覆 write OK；另一個 client 隨後讀取同一 key，讀到舊設定。這是 bug 還是允許行為，取決於服務承諾。

最小場景

A 的 write response 已返回，B 的 read invocation 之後才開始；linearizable store 應讓 B 看見 A 或其後更新。

真正難點

分散式系統常為延遲與可用性放鬆一致性；API 若沒說清楚，使用者會用強一致直覺寫錯流程。

Key Point

Linearizability 是使用者可把服務當成單一原子物件的強承諾。

Lenses

檢查 history，而不是只看最後狀態

同樣最終資料可能一致，但過程中的讀寫順序已經違反 API 承諾。

History

記錄每個 invocation、response、參數與回傳值，才能檢查一致性。

Linearization point

可能在 CAS 成功、lock 內寫入、leader commit 或 quorum commit 時發生。

Overlapping operations

重疊的 read/write 可有多種合法排序；非重疊操作要尊重真實時間。

Composability

多個 linearizable object 各自正確，不一定讓跨物件 transaction 自動正確。

模型使用者感受代價
Linearizable 像單一即時物件 延遲、同步與可用性成本高
Sequential consistency 有單一順序但不保證真實時間 較弱但仍可推理
Eventual consistency 最後可能收斂 caller 要處理暫時舊值

Psychology

人會自然假設讀到最新

大多數使用者不會在每次讀取前想 consistency model；他們用本機變數的直覺理解遠端服務。

API 沉默

文件沒寫一致性時，caller 會自行補上最方便的假設。

測試太短

單機或低延遲測試常看不見舊讀、重排與 failover 時的歷史。

成功回應的重量

write OK 讓人以為全世界都已看見，實際可能只代表某個節點接受。

強一致不是道德優勢；它是一個需要明確承諾與成本支付的 API 合約。

Applications

哪裡需要像單一物件一樣可靠

不是每個資料都需要 linearizability，但控制面、鎖、租約與設定常需要更強承諾。

Configuration service

若更新 feature flag 後仍讀到舊值，部署流程可能做出錯誤判斷。

Concurrent queue

enqueue/dequeue 必須能對應到合法順序，否則工作可能遺失或重複。

Distributed lock

鎖若不 linearizable，兩個 client 可能同時以為自己獲得排他權。

  • 在 API 文件中明確說明 read-after-write、monotonic read 與 failover 期間的承諾。
  • 使用 history checker 或 Jepsen 類測試思路檢查重疊操作，而不是只看最後資料。

Misreadings

最後一致，不等於當下可當真

不同一致性模型各有用途；混用名詞會讓 caller 寫出錯誤假設。

Eventual 就夠接近 校正

如果流程需要 write OK 後立即可見，eventual consistency 不提供這個保證。

單 leader 一定 linearizable 校正

還要看 commit、read path、lease、failover 與 clock assumptions。

Serializable 等於 linearizable 校正

transaction ordering 與 real-time ordering 是不同維度。
