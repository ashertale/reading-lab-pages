Orientation

組織邊界，如何悄悄長成系統邊界

康威定律指出，系統不是只由技術理想塑形，也會被組織溝通成本塑形。

Core Tension

你可以畫出理想模組邊界，但如果團隊邊界、會議節奏、ownership 和資訊流不支持它，實作最後往往會回到組織最容易溝通的形狀。

01 先看溝通圖

不要只看架構圖，也畫出哪些 team 需要頻繁同步、哪些同步成本最高。

02 再對齊 ownership

讓模組邊界與長期負責邊界盡量一致，減少跨 team handoff 成為日常。

03 最後設計反向調整

若想要某種架構，可能要調整團隊結構，這就是 inverse Conway maneuver 的精神。

Setup

把問題縮到最小

最小情境是：你想做清楚的模組化系統，但團隊被切成前端、後端、測試、平台，每個 feature 都要跨很多 handoff。

最小劇情

系統架構圖上功能 A 應該是一條完整 vertical slice，但組織上它被拆給三個 team。結果每次改動都要排同步、等對方、補文件。

真正難點

技術上你想要低耦合，組織上卻製造了高溝通成本。最後 code 會長出 bridge、adapter、workaround 和模糊 ownership。

團隊邊界決定日常溝通
→
溝通成本影響接口設計
→
系統邊界映出組織邊界
Key Point

康威定律不是宿命論，而是提醒你：若架構和組織長期互相打架，通常需要一起改。

Lenses

四種看康威定律的透鏡

它可以被當成診斷工具，也可以被當成設計工具；重點是不要只在技術層面看架構。

溝通成本

需要頻繁協調的東西若分散在高摩擦邊界上，系統會長出額外耦合或等待。

模組 ownership

沒有清楚 owner 的模組容易變成公地；太多 owner 的模組則容易變成政治邊界。

接口形狀

跨 team 邊界通常會被做成更正式的接口，這可能是好事，也可能造成過早抽象。

反向康威

如果你想要服務自治或產品能力導向架構，可能要先建立相對自治的長期團隊。

組織形狀 常見系統形狀 可能調整
按技術層分 team UI/API/DB 分層很厚，feature 交付需多次 handoff 改成 capability team 或明確 feature owner
平台 team 過度集中 所有變更都卡在平台 queue 下放安全擴充點與 clear contract
多地時區分散 模組被切成低頻同步的邊界 把高頻耦合工作放在同時區或建立強接口
責任邊界模糊 共用 code 反覆被局部修改 建立 owner、contract test 與變更流程

Psychology

為什麼組織邊界會滲進架構

人會降低溝通成本，系統也會沿著最低摩擦的協作路徑成長。

就近溝通

能直接問到的人，會影響你怎麼切接口、怎麼做假設、怎麼處理例外。

避免等待

若跨 team 等待成本高，工程師會傾向複製邏輯、繞路或做 local workaround。

責任安全感

清楚 ownership 讓人敢改；模糊 ownership 讓人傾向加 layer 保護自己。

這裡的心理與組織連結是教學性整理：重點是把架構討論放回真實協作成本，而不是把所有問題都歸咎於 team structure。

Applications

把它帶回 firmware 與產品架構

硬體、firmware、cloud、app、QA、製造常跨很多邊界；康威定律能幫你提前看見邊界成本。

Firmware / hardware 邊界

如果硬體與韌體 team 只在 milestone 才同步，接口假設會晚爆，最後用 workaround 補時序與行為差異。

平台與產品線

平台 team 若不理解產品線變化，平台 API 會偏向內部方便，而不是支援產品快速交付。

測試與開發分離

若 QA 只在最後接手，系統會缺少可測性設計；測試工具也會像外掛，而不是架構的一部分。

  • 架構 review 時同時畫 team interaction map，找出高頻溝通是否跨越高摩擦邊界。
  • 若某接口一直不穩，先問它是不是剛好位在組織權責不清的位置。

Misreadings

三個常見誤讀

康威定律常被當成嘲諷組織的口號，但它更有用的地方是幫你設計 team 與 architecture 的共同演化。

所以組織結構決定一切 校正

不是。技術選擇仍重要；康威定律只是提醒溝通結構會施加長期壓力。

只要改組就會有好架構 校正

不一定。改組需要搭配接口、ownership、工具與交付節奏，不然只是換名字。

這只適用軟體 校正

Conway 原本談的是廣義系統設計；firmware、硬體、流程與文件系統也會受到溝通結構影響。
