Simpson's Paradox

同一組資料在各子群裡都指向一個方向，合併後卻可能反過來，因為你把不同結構的樣本壓成了一個總數。

Orientation

總表不是上帝視角，它也可能是在說反話

辛普森悖論讓人不安，是因為它告訴你：同一份資料，光是換個聚合方式，方向都可能變掉。

Core Tension

每個子群看都像 A 比 B 好，總表卻可能顯示 B 贏；真正關鍵不在數學花招，而在你是不是把不同結構的樣本硬塞進同一桶。

01 先問有沒有分層

資料來自哪些不同群體、版本、難度或時間段，常比平均值更重要。

02 再看權重怎麼分配

若樣本量與難度分布不均，匯總結果很容易被大群體拖走。

03 最後回到因果問題

不是每個反轉都表示矛盾，很多時候只是你缺少關鍵條件。

Setup

從一張看似很正常的報表開始

辛普森悖論常出現在你最相信總表的時刻，因為總表看起來最乾淨、最像結論。

最小劇情

兩條產線各自看來新製程的不良率都比較低，但把兩條線的資料合併後，總不良率卻顯示新製程比較差。

真正難點

不是算錯，而是高難度批次大多落在新製程那邊，匯總時把難度差一起混進去了。

子群各自有趨勢
→
樣本權重與結構不同
→
總表方向被翻轉
Key Point

辛普森悖論提醒你：任何漂亮的總表，都值得被問一句『分開看還一樣嗎』。

Lenses

四個看反轉的角度

不是每次分組都會翻轉，但這四個點最常是關鍵。

混雜變項

有個沒被拉出來的條件，同時影響了分組與結果。

權重失衡

大群體的數量優勢，可能蓋過每個子群內真正的方向。

難度分配

不同方法若接到不同難度的樣本，總表就容易誤導。

因果偷換

聚合後的相關性，未必還能回答你原本想問的因果問題。

你看的層級 看起來像 可能漏掉什麼
總表 A 比 B 好 樣本難度與群體大小是否嚴重不均
依版本分組 每個版本都說 A 較好 某個版本是否承接最多高風險案例
依客群分組 高低端客群趨勢不同 市場分布是否讓總表被單一客群主導
依時間分組 某段時間特別差 是否剛好碰到外部條件改變

Psychology

為什麼總表特別容易被誤當真相

因為總表最簡潔、最像結論，也最適合進簡報和會議首頁。

平均值有權威感

人很容易把匯總數字誤認成『把雜訊洗掉後的真相』。

分層看起來比較麻煩

一旦要拆群體、拆難度、拆時間，故事就不再那麼俐落。

我們偏好單一答案

當報表能只給一個排名時，團隊通常比較安心，即使那個排名很脆弱。

這個悖論不是要你永遠拒絕匯總，而是要求你在匯總前先把結構問清楚。

Applications

把分層意識帶回實驗與品質報表

對工程團隊來說，辛普森悖論最危險的地方，是它會把錯的結論包裝成很會管理的 dashboard。

A/B test

不同流量來源與客群結構若不平衡，總體 conversion 可能說出和分群分析相反的話。

不良率分析

新舊治具、批次與型號混在一起看時，總表很容易把責任錯丟給製程或韌體。

支援績效

若艱難 ticket 多半分給資深工程師，單看平均處理時間會誤判誰效率低。

  • 看到總表方向明顯時，至少用版本、客群、難度或時間切一次，確認方向沒有翻。
  • 若你真正想回答的是因果問題，就不要只停在聚合相關性。

Misreadings

三個會把它用歪的地方

辛普森悖論不是叫你永遠不要合併資料，而是不要把合併視為無害。

所以任何總表都不能看 校正

不是。總表可以看，但要知道它回答的是哪個層級的問題。

只要分組就一定更真實 校正

分組也可能過度切碎；關鍵是分組是否對應真正重要的結構差異。

這只是統計冷知識 校正

實驗設計、品質監控與績效評估裡，這種反轉會直接改變決策方向。

Questions

這個總表結論，有沒有可能一拆就翻

下面兩個問題很適合放在任何報表解讀前。

如果按真正重要的條件分層，方向還一樣嗎

型號、客群、版本、難度或時間拆開後，原本的結論是否仍穩定？

這個匯總在回答哪個問題

你是在問整體表現，還是在問方法本身的好壞？這兩者不一定能用同一張表回答。
