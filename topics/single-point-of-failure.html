單點故障

Single Point of Failure

單點故障是一個元件、流程或人失效時，整個系統跟著失效的結構性脆弱點。

Read For

學會用依賴圖、替代路徑與演練檢查哪個點失效會讓全局中斷。

主題類型
可靠性 / 脆弱節點
核心問題
One component can drop the whole system
主要判準
依賴集中、替代路徑、故障域、恢復時間
讀完產出

Orientation

真正的脆弱點，是它壞掉時沒有人能繞過它

單點故障不一定是最大元件；它是依賴關係中沒有替代路徑、沒有降級方式或沒有接手人的關鍵點。

Core Tension

集中化能簡化管理與一致性，但若所有路徑都穿過同一點，簡單會換來整體不可用。

01 畫出依賴圖

找出所有請求、部署、資料與操作流程必經的節點。

02 定義故障域

問某個點壞掉會影響一個功能、一個區域，還是整個系統。

03 驗證替代路徑

備援、手動流程與降級策略要演練，否則只是文件上的安慰。

Setup

所有服務都有備份，卻都依賴同一個配置服務啟動

最小場景是表面上有多個副本，實際上存在共同必經點。

最小劇情

後端服務部署在多個 zone，也有多份資料副本；但所有實例啟動時都要連同一個配置中心。配置中心故障後，滾動重啟讓原本健康的服務也無法回來。

真正難點

冗餘通常畫在主資料路徑上，啟動、部署、認證、配置和人工作業路徑常被漏掉。

多條路徑依賴同一點
→
該點故障或不可達
→
全局服務失去替代路徑
Key Point

找單點故障要看實際依賴，不只看元件副本數。

Lenses

找出所有不可繞過的必經點

資料面、控制面和操作面都可能藏有單點，尤其是平常不在 request path 上的元件。

資料面

使用者請求是否都經過同一 load balancer、database writer 或 queue？

控制面

部署、配置、認證、DNS 或 feature flag 是否有共同必經點？

人員面

是否只有一個人知道操作、憑證或恢復流程？

Psychology

人容易在主要路徑看到冗餘，卻忘了輔助路徑也會決定存活

架構圖常強調多副本服務和資料庫，較少標出啟動、變更、憑證和手動修復依賴。

副本數帶來安全感

三個服務副本不代表沒有單點，因為它們可能共享同一個控制面。

低頻路徑被低估

部署、恢復和 rotate secret 不常發生，因此更容易在事故時才暴露。

人員單點被視為正常

組織常把唯一懂的人當效率來源，直到他不在場時才發現那是故障點。

單點故障審查要包含 request path、startup path、change path 和 recovery path。

Applications

把不可繞過的點改成可降級或可接手

系統架構、CI/CD 和操作流程都需要單點故障檢查。

配置服務

服務應能用最後有效配置啟動或降級，而不是配置中心短暫不可用就全局失敗。

CI/CD

部署平台故障時要保留緊急 rollback 或 freeze 能力。

知識交接

把關鍵操作 runbook、憑證流程與演練分散到多人。

Misreadings

單點故障不是只看有沒有備機

備機若共享同一電源、網路、憑證或操作人員，仍可能同時失效。

把副本當獨立 校正

副本共享控制面或故障域時，不能算真正獨立。

只看線上流量 校正

啟動、部署、恢復和管理面也會讓系統全局失效。

忽略恢復時間 校正

有替代路徑但切換要數小時，對使用者來說仍可能是嚴重單點。

Questions

問哪個點不能壞

下面兩問適合用在架構 review、DR 演練和操作流程盤點。

如果這個元件不可用，哪些路徑完全停止？

同時檢查請求、啟動、部署、認證、配置與恢復路徑。

替代路徑最近一次演練是什麼時候？

沒有演練的備援不能當成已驗證能力。

延伸路線

可接著讀 fault tolerance，理解系統如何在局部故障下繼續運作。
