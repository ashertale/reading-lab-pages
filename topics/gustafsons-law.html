Orientation

資源增加後，目標本身也會變大

古斯塔夫森定律反駁的是一種過窄的效能想像：假設問題永遠固定不變，只問同一件事能不能縮到更短。

Core Tension

阿姆達爾提醒串行段會封頂；古斯塔夫森則提醒，當可用算力增加，人常把工作規模放大，讓更多可並行部分進入原本的時間預算。

01 先固定時間預算

很多工程工作不是追求無限縮短，而是在一晚、一小時或一次 release gate 內做更多有價值的檢查。

02 再擴大可並行內容

新增 runner 可能讓你加入更多 seed、更多硬體組態或更長 soak，而不是只把舊測項跑快。

03 最後追問結果價值

問題變大必須帶來更好的覆蓋、信心或資料品質；放大本身不是目的。

Setup

同樣一晚，測更多世界

想像原本 nightly 只能跑一組 board、三個溫度點與短時間壓力測試；擴充後不只是縮短時間，而是把更多環境納入同一晚。

最小場景

團隊把測試機從 4 台加到 24 台，release gate 仍維持隔天早上出報告，但新增了更多 SKU、power mode 與隨機壓力組合。

真正難點

若只看舊測項完成時間，擴建好像沒有達到線性加速；但真正收益可能是以前被剪掉的風險終於被覆蓋。

Key Point

古斯塔夫森定律適合問：新增資源後，我們要縮短舊任務，還是把原本因成本太高而沒做的任務放回來？

Lenses

看 scaled speedup 的三條線

這個概念的重點不在反駁阿姆達爾，而在換一個問題設定：時間大致固定，問題規模隨可用資源而變。

固定時間

CI、batch job、模擬與資料分析常有外部時間窗；超過窗口就失去決策價值。

可放大工作

只有能產生額外信心或資訊的部分值得放大，例如更多資料、更多組態或更細粒度掃描。

剩餘串行段

即使工作量放大，初始化、彙整與人工 sign-off 仍會限制整體流程。

價值密度

放大的工作若只是重複低風險案例，scaled speedup 會在品質上失真。

問法 阿姆達爾視角 古斯塔夫森視角
同一批 test 加機器後能縮短多少？ 同一晚可加入多少新覆蓋？
資料處理 固定資料集能快幾倍？ 同樣時間能分析多大樣本？
模擬驗證 單一場景多久完成？ 時間窗內能掃過多少參數空間？

Psychology

人常把容量收益看成時間收益

效能改善最容易被報成時間縮短；但很多團隊真正缺的是覆蓋不足、樣本太少或不敢跑長測。

簡單 KPI 會遮住規模

把改善寫成 runtime reduction 很直覺，卻可能忽略新增案例帶來的風險下降。

固定任務讓討論省事

同一份 workload 容易比較，卻未必代表實際工作會在資源增加後保持不變。

放大需要產品判斷

要決定多測哪裡，必須理解 field risk、客戶組態與失效成本，不只是排程問題。

古斯塔夫森的心理提醒是：不要讓容易比較的舊工作量，替真正該放大的工作量做決策。

Applications

容量規劃不只是在追更短時間

對 firmware 團隊來說，scaled speedup 可以轉成覆蓋率、環境廣度、測試長度與資料品質，而不只是報表上的分鐘數。

長時間 soak

新增機台後，可以把過去只跑 30 分鐘的壓力測試擴成跨溫度、跨電源事件的長時間觀察。

參數掃描

校準、DSP filter 或 power policy 可把更多參數組合放進同一個決策窗口。

多 SKU 驗證

同一晚涵蓋更多 board revision 與 vendor component，比單純把一個 SKU 跑得更快更有用。

  • 寫擴建理由時，同時列出時間收益與新增覆蓋收益，避免只用舊 workload 判斷投資。
  • 如果規模放大後報告太慢，下一個瓶頸通常會轉到資料彙整、triage 與 ownership。

Misreadings

別把它當成樂觀版阿姆達爾

古斯塔夫森定律不是保證平行化一定划算；它只是說問題設定變了，效益也應該重估。

工作量放大就一定好 校正

錯。放大的內容若不能降低風險或增加知識，只是在填滿容量。

可以忽略串行瓶頸 校正

不行。結果彙整、資料上傳與人工判斷仍可能在規模放大後變成新瓶頸。

它否定阿姆達爾 校正

兩者回答不同問題：一個問固定工作量的上限，一個問固定時間內能做多大的事。

Questions

把容量變成有效工作量

如果 CI runner 明天多五倍

你會加入哪些以前因時間太貴而被砍掉的驗證？這些驗證對 release 風險有多大影響？

如果主管只問縮短幾分鐘

你能不能用新增覆蓋、更多樣本或更早發現 field issue 的方式說明收益？
