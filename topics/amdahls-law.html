Amdahl's Law

Orientation

加速不是平均分給每個核心

阿姆達爾定律把效能討論從「有幾個處理單元」拉回「工作裡有多少比例真的能同時做」。

Core Tension

只要有一小段工作必須按順序完成，無論後面堆多少核心，整體速度都會逐漸撞上那段串行工作的天花板。

01 先找不可拆的段落

初始化、鎖、單一匯流排、序列化協定與收尾合併，常比主迴圈更能決定上限。

02 再看同步吃掉多少

平行化本身會帶來分派、等待、cache miss 與合併成本，這些不會自動免費。

03 最後估理論之外的現場

量測要包含真實資料、I/O、interrupt 與排程抖動；漂亮公式只能告訴你該追哪裡。

Setup

一次優化會議裡的冷水

把問題縮到一個 firmware build 或測試流程：九成步驟能丟給多台機器，剩下一成必須等待同一份簽章、同一個治具或同一條硬體連線。

最小場景

團隊想把 nightly validation 從 60 分鐘降到 10 分鐘，於是加了 8 台 runner；實跑後只降到 24 分鐘，因為 flashing、license checkout 與結果彙整仍排隊。

真正難點

直覺會把 runner 數量當成加速倍率，但總時間裡最硬的那段不是 test case 本身，而是所有 job 都必須回到同一個關口。

Key Point

阿姆達爾的重點不是公式漂亮，而是讓你在買硬體之前先問：總時間裡哪一段仍只能一個一個來？

Lenses

四個判斷 speedup 的角度

同一份 workload 可以從比例、粒度、共享資源與觀測窗口來拆；任何一個角度漏掉，都會把效能計畫估得過度樂觀。

串行比例

若 5% 時間必須單線完成，無限多核心也不會讓整體快過 20 倍；這個比例越早量出來越好。

工作粒度

任務切得太細會讓排程與同步超過計算本體；切得太粗又會讓部分 worker 提早閒置。

共享瓶頸

flash programmer、global lock、memory bus、log sink 或 license server 都可能把平行工作重新壓成排隊。

固定問題大小

阿姆達爾通常假設問題規模不變；如果你因資源增加而處理更大資料集，就要同時看古斯塔夫森定律。

現場訊號 容易誤判 阿姆達爾提醒
CPU 使用率不滿 再加核心就會快 也許 worker 正在等鎖、I/O 或單一設備。
單一測項很快 全部測項可等比縮短 setup、teardown 與報告彙整可能沒有一起縮短。
平均時間下降 最佳化成功 尾端等待可能仍決定 release 何時能過關。

Psychology

為什麼人喜歡相信多加資源會線性變快

更多核心、更多人、更多機器給人立即可見的控制感；串行比例卻常藏在流程邊界與工具限制裡。

可見資源比隱形等待好算

硬體數量可以列在表上，等待鎖或等待治具的時間卻常散在 log 裡。

局部 benchmark 讓人高估

一段 kernel 或 algorithm 變快，不代表整條 pipeline 的固定成本也被移除。

買資源比改流程痛感低

拆掉單點關口常牽涉 ownership、工具鏈與驗證策略，比加 worker 難討論。

這個心理入口特別適合用在效能會議：先把不可並行比例公開，團隊才不會把希望全部押在擴機。

Applications

帶回 firmware 效能與測試排程

Firmware 團隊常同時碰到多核、DMA、CI runner、硬體治具與 release gate；阿姆達爾能把這些資源討論拉成一張上限表。

多核 RTOS 任務

把工作分到多核心前，先量 critical section、cache invalidation 與 IPC 等待；它們會變成真正的串行段。

硬體在迴路測試

HIL 測試通常不是 test script 慢，而是 fixture、power cycle、flash 與人工換線成了共享瓶頸。

資料處理 pipeline

log parsing 可以平行，最後排序、去重、簽章與 artifact 上傳仍可能讓整體卡在固定尾巴。

  • 做效能 proposal 時，同步列出可並行比例、不可並行比例與同步成本，而不是只列新增資源。
  • 如果 speedup 卡住，優先找共享關口與固定成本；它們比主程式迴圈更常限制整體改善。

Misreadings

容易把它讀得太悲觀

阿姆達爾定律不是叫你放棄平行化，而是要求先承認上限從哪裡來。

所以平行化不值得 校正

錯。它只說 speedup 有上限；若串行比例低，平行化仍可能非常值得。

只要算一次比例就夠 校正

比例會跟資料集、硬體、工具版本與 workload 改變；舊估算不能永久當真。

瓶頸只能靠硬體解 校正

很多串行段其實是流程或協定設計造成的，重排 ownership 可能比買設備有效。

Questions

估加速前先問這兩件事

如果新增 4 倍資源

總時間裡哪幾段完全不會受益？它們合計佔比多少？

如果 speedup 低於預期

你會先查 CPU 飽和、鎖等待、I/O 排隊、fixture 佔用，還是結果彙整？為什麼？
