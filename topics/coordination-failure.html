協調失靈

Coordination Failure

協調失靈不是大家不同意目標，而是每個人的最佳行動都取決於別人是否也一起行動。

Read For

理解多方互相等待、共同預期和協調點如何讓好結果卡在沒人先動。

Orientation

每個人都願意做，但只在相信別人也會做的時候

協調失靈出現在多個均衡之間：大家都做會更好，但單獨先做的人可能承擔損失。

Core Tension

合作收益需要同時性；如果沒有人能確信其他人跟上，理性的等待就會把團隊困在較差狀態。

01 找出互相依賴

誰的收益取決於誰也採用、遷移、投資或遵守？

02 建立可信承諾

用日期、合約、資源鎖定或公開承諾降低對他人行動的不確定。

03 提供過渡支架

讓先行者不必獨自承擔全部成本。

Setup

每個團隊都想換新 API，但都等別人先支援

最小場景是遷移只有在多方一起完成時才有價值。

最小劇情

平台提供新認證 API，產品團隊都同意舊 API 應退場。但產品端不想先改，因為平台還要支援舊版；平台也不敢關舊 API，因為產品端還沒改。

真正難點

卡住的不是價值判斷，而是缺少共同承諾和過渡節點。

大家偏好新狀態
→
單獨行動成本高
→
互相等待維持舊狀態
Key Point

協調失靈要靠共同預期被重新建立，而不是再說一次目標很好。

Lenses

找出缺少的共同預期

真正的問題常不是方案內容，而是誰相信誰會在何時行動。

參與者集合

哪些人必須同時改變，少了誰就會讓收益消失？

承諾可信度

口頭支持之外，有沒有排程、資源、owner 和失敗後果？

過渡均衡

是否能設計中間狀態，讓部分採用也有收益？

Psychology

等待別人先動可以很理性，也會讓壞均衡活很久

每個人都怕當唯一先行者，因此穩定停在大家都不喜歡的狀態。

先行者風險

單獨投入可能得不到回報，甚至被舊系統拖累。

承諾不可信

大家都說會改，但沒有成本的承諾很難讓人放心。

舊狀態有慣性

即使較差，只要大家都知道如何在舊狀態運作，它就能持續。

處理協調失靈時，請把『誰先動』改成『怎樣讓大家相信會一起動』。

Applications

創造可相信的共同起跑點

標準、遷移和冷啟動都需要把共同預期設計出來。

API 遷移

公布凍結日期、遷移儀表板和產品線 owner，讓各方看到進度。

標準採用

提供 adapter 或兼容層，降低早期採用者成本。

產品冷啟動

先聚焦小群體密度，而不是讓所有人等待所有人加入。

Misreadings

協調失靈不是大家懶得合作

它常發生在大家都想要好結果，但無法相信同步行動時。

只責怪某一方拖延 校正

拖延可能是對他人不確定的合理反應。

只重申願景 校正

願景不能替代可信承諾和遷移支架。

忽略中間狀態 校正

若只有全改或全不改，協調門檻會很高。

Questions

問誰需要相信誰

下面兩問適合用在跨團隊計畫和平台遷移。

每個參與者在等誰先動？

把等待關係畫出來，而不是只列任務。

哪個承諾會讓等待變得不理性？

日期、資源、契約、adapter 或公開進度都可能成為協調點。

延伸路線

可接著讀 coordination game，理解多方如何在共同預期下選擇均衡。
