這頁真正要處理的，不是你喜不喜歡 貝氏定理，而是遇到 故障診斷 或 風險告警 時，哪些前提若沒先寫出來，討論就會失焦。

Core Tension

面對新證據時，理性的重點不只是看那個證據有多醒目，還要看它在不同前提下有多可能出現，以及原本的基線有多高。

01 第一步先把 先驗機率 寫清楚

讀 貝氏定理 的第一步，不是贊成或反對，而是確認誰把 先驗機率 先當真。

02 再把 證據力 的成本攤開

證據力 一旦只留在抽象層，這條原則就很容易被拿去替某一方省成本。

03 最後問 更新 怎麼落地

更新 不是附註；它決定這條原則是可操作的判準，還是說完就散的口號。

Setup

把 貝氏定理 放進一次真實判斷，看誰先該交代

這類原則一離開場景就會變口號；放回 不確定性與證據 裡的實際判斷後，誰要先說明、誰要先承擔、誰要先驗收會立刻分開。

最小劇情

某測試準確率很高，但目標事件本來就罕見；若忽略基線，只看測試命中，結論就會被拉得過頭。

Key Point

貝氏定理 真正幫你做的，不是替某個立場加分，而是把 先驗機率、證據力 和 更新 排成可檢查的順序。

Lenses

先把 先驗不能丟、證據是乘法不是魔法 與 更新不是重開局 分開看

先驗不能丟、證據是乘法不是魔法 與 更新不是重開局 看起來都在談 貝氏定理，其實各自是在回答不同的追問：誰先說明、誰先承擔、誰來收尾。

先驗不能丟

先驗不能丟 這條線在乎的，不是誰比較有氣勢，而是 先驗機率 一旦沒被交代，現場會怎麼開始失真。

證據是乘法不是魔法

順著 證據是乘法不是魔法 讀，你會發現爭點不在口號，而在 證據力 是否真的被變成判準。

更新不是重開局

把焦點放在 更新不是重開局 時，你其實是在說：更新 應該先被寫清楚，否則後面的結論都站不穩。

Psychology

現場通常怎麼把 貝氏定理 讀成順手口號

真正的困難通常不在反對者，而在支持者也常不願意把 證據力 的代價和 更新 的驗收一併寫出來。

新證據太有畫面

新證據太有畫面 常不是純粹錯誤，而是一種討論捷徑：先把自己放在合理的一邊，再晚點處理 先驗機率。

基線抽象又無聊

碰到 基線抽象又無聊 時，最危險的是大家以為原則已經被引用，實際上 證據力 還沒被具體說明。

人喜歡直接跳結論

人喜歡直接跳結論 會讓人覺得只要把話說得夠完整，就不必再補 更新 的證據或代價。

真正要防的不是有人沒背過 貝氏定理，而是整個討論都在引用它，卻沒人把 先驗機率 和 證據力 寫成可檢查條件。

Applications

回到 故障診斷、風險告警 與 實驗結果判讀

這類原則離開 故障診斷 或 風險告警 很容易變格言；一回到實際流程，你就得回答誰先說明、誰先買單、誰來驗證。

故障診斷

只要到了 故障診斷，你就能看出 先驗機率 究竟是被真的承擔，還是只是被口頭提過。

風險告警

在 風險告警 這種場景，貝氏定理 幫你的不是贏辯論，而是把 證據力 先釘在桌面上，避免事後才補。

實驗結果判讀

實驗結果判讀 最需要 貝氏定理 的時候，通常就是大家都急著往前走，卻還沒把 更新 寫成共識。

Misreadings

三個常見誤讀

最麻煩的不是有人反對 貝氏定理，而是很多人自以為同意，卻同意的是不同版本。

貝氏定理 只是好聽口號 校正

真正難的是把 貝氏定理 翻成一組可檢查的判準，而不是把它當成結論本身。

只要背了 貝氏定理 就能直接套用 校正

原則若不落回 先驗機率、證據力 和代價分配，通常會在現場變成誤用。

貝氏定理 只適合高層決策 校正

其實從 code review 到 不確定性與證據 的日常判斷，都會碰到同樣的壓力。
