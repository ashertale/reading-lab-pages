縱深防禦

Defense in Depth

縱深防禦不是把同一把鎖裝很多次，而是讓不同層各自承擔不同失敗假設。

Read For

分辨邊界防護、身份、最小權限、偵測、隔離和恢復如何形成互補層。

Orientation

每一層都要假設上一層可能失手

縱深防禦把安全看成多層限制：外層擋不住時，內層仍能降低權限、暴露面、橫向移動和恢復成本。

Core Tension

多層防線能吸收錯誤，但層數增加也會提高複雜度；有效縱深要看層與層是否真的獨立且互補。

01 給每層不同任務

防止、偵測、限制、記錄和恢復不是同一件事。

02 檢查共同失效

如果多層依賴同一秘密、同一身份或同一配置，攻破一處可能全破。

03 讓內層最小化傷害

即使入口被攻破，權限、網段、資料和操作也要被分區限制。

Setup

API gateway 驗證失效時，內部服務不能假設所有請求都可信

最小場景是外層防線失手，內層若完全信任外層就一起失守。

最小劇情

Gateway 原本負責 JWT 驗證；某次配置錯誤讓未驗證請求進入內網。內部服務沒有再檢查 scope，也用高權限 service account 存取資料庫。

真正難點

外層驗證存在不代表內層可以放棄身份、授權和操作限制。

外層防線被繞過
→
內層仍假設請求可信
→
單一失誤擴大成資料暴露
Key Point

縱深防禦要求每層都能在上一層失效時限制爆炸半徑。

Lenses

看每層限制的是哪一種失敗

有效防線不是重複檢查同一條件，而是覆蓋不同攻擊階段。

入口層

身份、輸入驗證、rate limit 和基本邊界是否存在？

內部層

服務間授權、最小權限、網路分段和資料範圍是否能限制橫向移動？

偵測恢復層

異常是否被記錄、告警，且能快速撤銷 token、隔離服務或回復狀態？

Psychology

人容易把第一層設計得很厚，然後讓內層過度信任

強邊界帶來安全感，但真正事故常來自邊界配置錯誤、憑證外洩或內部假設過度寬鬆。

邊界思維太舒服

把世界分成內外很直覺，卻會讓內部服務預設安全。

重複控制被誤當縱深

多個相同密碼或相同 ACL 不是多層防線，因為它們可能同時失效。

偵測被晚放

團隊重視阻擋，卻低估阻擋失敗後需要快速看見和收斂。

設計縱深時，逐層問：如果前一層壞了，這一層能限制什麼。

Applications

讓安全控制分散在入口、內部與恢復路徑

API、韌體更新和雲端資料存取都需要互補防線。

API 安全

Gateway 驗證之外，服務內仍檢查 scope、租戶和操作權限。

韌體更新

下載通道安全之外，裝置端仍驗簽、版本檢查和 rollback protection。

資料存取

即使 app server 被攻破，資料庫帳號也只能存取必要表與必要操作。

Misreadings

縱深防禦不是安全控制越多越好

沒有不同失敗假設的層數只是複雜度，甚至會讓人誤以為安全。

把重複當縱深 校正

同一種檢查複製三次，不代表能抵抗不同失效模式。

忽略可用性成本 校正

過度層層阻擋會讓維運繞路，反而製造陰影流程。

只看預防 校正

偵測、記錄、隔離和恢復也是縱深的一部分。

Questions

逐層假設上一層已失手

下面兩問適合用在 threat modeling 和安全架構 review。

如果入口驗證失效，內層還能限制什麼？

檢查服務授權、資料範圍、網段和操作權限。

哪些防線會被同一個秘密或設定同時打穿？

找出共享憑證、共享角色和共同配置來源。

延伸路線

可接著讀 security 與 least privilege，理解權限限制如何成為內層防線。
