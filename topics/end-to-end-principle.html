端到端原則

End-to-End Principle

有些正確性只有端點知道；中間層可以幫忙，但不該假裝自己能完整替端點完成語意判斷。

Orientation

中間層能協助，端點才知道完成

端到端原則指出，某些功能若要完全正確，必須由通訊端點依自身語意檢查；底層最多提供不完整但有用的支援。

Core Tension

把可靠性全塞到底層會增加成本與複雜度，但完全不在底層防護又會讓端點承擔過多重試、偵錯與性能損失。

01 先找語意歸屬

只有應用端知道檔案是否完整、firmware image 是否可開機、交易是否真的完成。

02 再決定底層幫忙

link retry、CRC、flow control 可以降低常見錯誤，但不能取代端點的最終驗證。

03 最後避免重複錯位

同一功能在多層重複實作時，要說清楚每層負責哪種失效。

Setup

傳輸成功不等於更新成功

把問題縮到 OTA：傳輸層說封包都收到了，flash driver 也寫入成功，但裝置能不能 boot 到新版本是另一件事。

最小場景

中間層做了重試與 CRC，卻沒有端點層的 image manifest、簽章、版本與 boot validation；最後一次 power loss 讓資料看似完整但語意上不可用。

真正難點

每一層都可能說自己成功，卻沒有任何一層負責應用真正要的成功定義。

Key Point

端到端原則不是否定底層可靠性，而是防止底層成功被誤認成整體語意成功。

Lenses

功能應該放在哪一層

分層設計的難題不是把功能往上或往下推，而是看哪一層具備足夠資訊與合適成本來處理它。

語意可見性

若某層看不到完整物件或使用者意圖，它就不能做最終正確性判斷。

性能與局部修復

底層重試能快速修復常見短暫錯誤，避免每次都讓端點付出昂貴代價。

失效隔離

底層保護應限制局部錯誤擴散；端點保護則確認全局結果。

複雜度分配

功能下放會讓所有使用者共同付成本；上放則讓需要者按語意自行承擔。

功能 底層可做 端點仍要做
資料傳輸 CRC、重試、順序控制 檔案 hash、版本與完整語意檢查。
韌體更新 flash write verify image signature、rollback 與 boot 成功判定。
通訊可靠性 link-level retransmission request idempotency 與交易結果確認。

Psychology

人喜歡把成功切在最近的一層

工程分工會讓每層都想定義自己的 done；端到端原則提醒你，局部 done 不一定等於使用者目標完成。

局部指標比較好量

封包成功率、write success 或 checksum pass 都容易量，端點語意成功反而跨模組。

責任邊界會漂移

當事故發生，底層、協定層與應用層可能各自說自己沒錯，因為沒人定義端到端成功。

過度通用化的誘惑

底層想替所有情境做好，但越通用越難掌握各應用真正需要的語意。

這個原則最能幫你問清楚 ownership：誰能看見真正的完成條件？

Applications

放進協定、driver 與更新流程

Firmware 系統有很多中間層成功：DMA complete、flash write ok、packet ack、driver init done。端到端原則要求再問一次使用者語意。

OTA 與 boot chain

傳輸層和 flash 層可以保護片段，bootloader 與 application 仍要驗簽、切換與回滾。

Driver API

driver 可確認 register write 完成，但上層仍要確認 peripheral 進入預期模式並可提供服務。

資料同步

ack 只代表對方收到了某層訊息；業務語意可能還需要 commit record 或 idempotent token。

  • 替每個流程寫出低層成功、中層成功與端點成功三種定義。
  • 底層保護用來降低常見錯誤成本，端點驗證用來確認語意真的成立。

Misreadings

不是所有功能都只能放端點

端到端原則常被過度簡化成『底層不要做事』，這會錯過局部保護的價值。

底層可靠性都沒用 校正

錯。底層重試、CRC 與流控能降低錯誤率，只是不能取代端點語意檢查。

端點檢查可以省略底層保護 校正

也不對。沒有底層協助，端點可能付出太高延遲、頻寬或能源成本。

原則給出唯一答案 校正

它給的是追問順序：哪一層看得到語意，哪一層能便宜處理常見錯誤。

Questions

替每一層寫清楚成功定義

如果底層回報 success

端點真正需要的物件、狀態或交易是否已被驗證？誰負責最後確認？

如果要把功能下放

所有上層都需要這個功能嗎？下放後增加的成本與語意限制是否值得？
