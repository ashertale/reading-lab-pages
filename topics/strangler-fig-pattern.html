絞殺榕模式

Strangler Fig Pattern

絞殺榕模式不主張一次重寫；它在舊系統周圍建立新路徑，逐步移走行為，直到舊核心可以安全退休。

Read For

適合 legacy service、舊 driver、bootloader protocol、工廠工具與任何一次重寫風險太大的系統。

Orientation

大重寫最危險的部分，是以為需求已經懂了

絞殺榕模式在舊系統外建立新入口或新服務，把特定行為逐步導向新實作，而不是一次關掉舊系統。

Core Tension

舊系統的行為常藏著例外、客戶依賴與歷史 workaround；一次重寫會把未知集中在同一天爆開。

01 先找 seam

可攔截、可路由、可比較結果的邊界是第一個替換點。

02 再平行運行

新舊路徑可短期共存，用觀測資料確認行為差異。

03 最後刪掉舊路徑

每次遷移都應縮小 legacy，而不是只增加一層轉接。

Setup

舊工具不能停，新工具也不能等兩年才上線

產線工具有大量舊流程與客戶例外；團隊想換 UI、協定與資料模型，但停線重寫不可接受。

最小場景

新工具先接管查詢與 log 匯出，燒錄與特殊 recovery 仍導向舊工具。

真正難點

轉接層要能觀察誰還在用舊路徑，否則新系統只是在舊系統旁邊多一套。

建立可攔截邊界
→
逐步導流到新實作
→
移除已被取代的舊行為
Key Point

漸進替換的勝利條件是 legacy surface 逐步縮小，而不是新舊永遠共存。

Lenses

四個讓遷移真的前進的判準

遷移計畫要避免只新增 adapter；每個 slice 都要有觀測、比對與退休節點。

Seam

找到能攔截呼叫、資料或 UI flow 的位置，讓單一行為可被替換。

Routing

用 feature flag、proxy、command dispatcher 或 protocol gateway 控制流量走向。

Behavior comparison

在切換前後比對結果、錯誤率與使用情境，降低 hidden behavior 風險。

Retirement criteria

當使用量、覆蓋率與例外處理達標，舊路徑要被刪除或封存。

Slice適合先做不適合先做
低風險高頻查詢 容易比對、價值早出現 資料語意仍不清楚
高風險寫入 要等冪等與回退設計完成 直接作為第一刀
少數客戶例外 可先記錄與隔離 不理解就硬搬進新模型

Psychology

新系統在紙上總是比較乾淨

重寫計畫常低估舊系統裡那些沒人喜歡、卻仍被現場依賴的行為。

乾淨模型誘惑

設計新模型時很容易把例外當髒資料，但客戶流程可能靠它活著。

遷移疲勞

新舊並存需要操作成本，若沒有退休節奏，團隊會失去耐心。

成功片段被忽略

漸進替換的價值是每一段都能上線學習，而不是等最後一天一次證明。

這個模式要求技術與產品一起決定：哪個行為先搬、怎麼驗證、何時宣布舊路徑死亡。

Applications

Legacy driver、服務與工具都可以切片替換

只要能找到穩定邊界，就能把重寫改成一系列小型可驗證遷移。

Backend service

用 proxy 把單一路由導到新服務，其餘仍由 monolith 處理。

Firmware protocol

先讓新 command set 支援少數操作，gateway 負責新舊格式轉換與 telemetry。

Factory tool

先替換 log viewer、查詢、報表等低風險高頻功能，累積使用資料。

  • 為每個 slice 定義 done：流量比例、錯誤率、使用量、舊路徑刪除日期。
  • 轉接層要有 owner；沒有 owner 的 adapter 很容易變成下一代 legacy。

Misreadings

漸進不是無期限共存

這個模式降低大重寫風險，但也會製造過渡架構；若不移除，它會變成新負擔。

先包一層就好了 校正

包覆只是開始；真正工作是逐步移走行為。

不需要理解舊系統 校正

恰好相反，切 slice 需要理解邊界、例外與使用流量。

永遠保持雙軌最安全 校正

雙軌越久，測試、資料同步與操作複雜度越高。

Questions

第一個 slice 要小到能驗證，也大到有價值

如果要選第一刀

哪個功能邊界最清楚、使用量高、錯誤代價低，且能用指標比較新舊行為？

如果轉接層上線

它的退休條件、owner、觀測指標與移除日期是否已經寫進計畫？
