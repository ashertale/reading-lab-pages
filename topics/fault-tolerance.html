容錯

Fault Tolerance

容錯不是保證零故障，而是承認元件會壞，並讓壞掉的部分不立刻拖垮整體服務。

Read For

分辨 fault、error、failure，並把偵測、隔離、冗餘與降級設計成一條路徑。

Orientation

可靠系統不是不會壞，而是壞了還知道怎麼縮小傷害

容錯把故障視為正常輸入：硬體、網路、依賴、資料和人都可能出錯，設計要負責吸收其中一部分。

Core Tension

冗餘、重試和 failover 能提高存活率，也會增加複雜度；沒有明確故障模型時，容錯機制可能變成新的故障來源。

01 先寫故障模型

明確說明要容忍單機、網路、資料、電源還是人為操作故障。

02 讓偵測早於擴散

故障要被快速發現、隔離和標記，避免錯誤狀態繼續服務。

03 驗證恢復路徑

Failover、restore 和降級不能只存在架構圖，要定期演練。

Setup

一台節點壞掉不可怕，可怕的是其他節點不知道它壞了

最小場景是單一故障被系統當成正常回覆繼續使用。

最小劇情

感測節點回傳固定舊值，監控只檢查連線是否存在，因此控制系統把卡住的讀值當成真實環境，持續做錯誤調整。

真正難點

容錯不只是備援硬體，也包含判斷哪個輸出已不可信、何時隔離、由誰接手。

元件發生 fault
→
錯誤輸出未被辨認
→
整體服務出現 failure
Key Point

容錯的起點是故障假設；不知道要容忍什麼，就無法評估冗餘是否有用。

Lenses

把故障從發生到恢復逐段檢查

容錯鏈條至少包含偵測、隔離、替代路徑、狀態修復和事後學習。

偵測

系統如何知道某個輸出、節點或依賴已經不可信？

隔離

錯誤是否會被限制在局部，還是繼續污染資料與決策？

恢復

替代路徑啟動後，原狀態如何修復、回補或重新同步？

Psychology

人容易把冗餘當成安心感，卻少檢查切換條件

看到雙機、備援或重試很容易覺得可靠性提高，但真正的風險常在偵測和切換細節。

架構圖太樂觀

圖上有兩條路徑不代表失敗時真的會切過去。

重試被誤當容錯

無限制重試可能放大流量，讓局部故障變成整體壓力。

演練成本被延後

Failover 演練會打擾正常運作，因此常被推到真正事故才第一次測。

容錯設計要問的不是『有沒有備援』，而是『故障如何被發現、隔離、接手與修復』。

Applications

把容錯拆成可演練的操作路徑

硬體節點、雲端服務與韌體裝置都需要把故障處理流程具體化。

韌體感測

對固定值、跳變值與過期 timestamp 加入 plausibility check，而不是只看連線。

服務 failover

把健康檢查、流量切換和 state sync 納入演練，避免災難時才發現不能切。

資料恢復

備份要定期 restore 驗證，否則只是心理安慰。

Misreadings

容錯不是堆更多備援

更多元件也帶來更多狀態和切換邏輯；沒有故障模型的冗餘會增加複雜度。

把高可用等同容錯 校正

高可用是結果目標，容錯是處理故障的設計能力。

只設計 happy failover 校正

切換失敗、資料落後與腦裂情境也要被考慮。

忽略人為操作 校正

部署、回滾和手動修復也是故障來源，需要 guardrail。

Questions

問故障如何被吸收

下面兩問適合用在 reliability review 和 FMEA 前。

我們假設哪個元件會壞？

請具體到節點、網路、資料、感測器、依賴或操作步驟。

壞掉後多久會被隔離？

偵測時間、錯誤擴散範圍和接手路徑都要能被驗證。

延伸路線

可接著讀 redundancy，理解備援如何提高可靠性，也如何增加切換複雜度。
