Orientation

工作痕跡也能成為協調訊號

Stigmergy 指的是個體透過改變環境留下痕跡，後續個體再依痕跡行動，形成間接協調。

Core Tension

直接溝通昂貴且容易遺失；環境痕跡能降低同步成本，但痕跡若模糊或過期，也會引導錯誤行動。

01 先留下狀態

工作到哪、誰負責、卡在哪裡，要在環境中可見。

02 再引導下一步

好的痕跡不只記錄過去，還提示後續可行動作。

03 最後清理殘影

過期標籤、壞 dashboard 與舊文件會變成反向協調。

Setup

下一個值班工程師不用先問一輪

一個 nightly build 失敗；CI artifact、owner label、bisect result 與 issue 狀態若完整，接手者能直接行動。

最小場景

前一位工程師留下失敗測試、可疑 commit、已排除原因與下一個檢查點。

真正難點

若所有脈絡只存在聊天訊息或腦中，交接就必須靠同步溝通，且容易漏資訊。

行動留下痕跡
→
他人讀取痕跡
→
後續行動改變環境
Key Point

跡化協作把部分 coordination 存進共享環境。

Lenses

痕跡要能被看見、理解與更新

不是所有紀錄都有協調價值；好的痕跡要在正確時間出現在正確地方。

可見性

狀態藏在私人筆記或聊天深處，就無法引導團隊行動。

語意清楚

標籤、狀態與欄位要有共同定義，否則每個人讀法不同。

回饋速度

CI、監控、review bot 的回饋越靠近工作點，越能引導下一步。

腐敗控制

過期痕跡需要被移除或標記，避免舊訊號誤導新行動。

環境痕跡協調作用壞掉時
CI 狀態 指出可合併或需修復 flaky 測試讓大家忽略紅燈
Issue label 顯示 owner、priority、blocked reason 標籤過期造成錯誤排程
Runbook 註記 保留事故處理路徑 未更新會導向舊系統

Psychology

我們常把協調想成開會

會議能解決不確定性，但若每次都要開會，代表環境沒有承載足夠狀態。

口頭脈絡依賴

知道的人一離線，工作就停住。

工具被當倉庫

如果 issue tracker 只是事後歸檔，它就不會引導當下行動。

訊號疲勞

過多 bot 訊息和狀態會讓真正重要痕跡被淹沒。

協作工具的設計問題常不是資訊太少，而是資訊沒有變成可行動的痕跡。

Applications

讓工作流自己提示下一步

工程團隊每天都在使用 stigmergic systems，只是常沒有用這個名字。

Issue tracker

狀態、owner、blocked reason 與 reproduction steps 讓任務能非同步接力。

Code review

inline comments、check runs 與 requested changes 在程式碼旁留下可處理痕跡。

Incident response

timeline、current hypothesis、mitigation status 讓下一班不用重建整個故事。

  • 把下一步放在工作物件旁，而不是只放在聊天訊息裡。
  • 定期刪除或標記 stale labels、flaky checks 與過期 runbook，避免環境說謊。

Misreadings

自組織不等於沒責任

環境痕跡能降低協調成本，但仍需要 owner、清理規則與例外處理。

不用溝通 校正

複雜衝突、價值判斷與模糊責任仍需要直接對話。

工具會自動協作 校正

工具只承載痕跡；痕跡品質取決於團隊行為。

越多狀態越好 校正

太多欄位和 bot 訊息會讓重要線索失去注意力。

Questions

下一個人能不能只看環境就接上

如果 owner 離線

issue、branch、CI artifact 和文件是否足以讓別人知道目前判斷與下一步？

如果某個標籤過期

誰會清理，多久清理一次，系統是否會提醒它已不可信？
