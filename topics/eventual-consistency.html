最終一致性

Eventual Consistency

最終一致性承認副本會暫時不同步；真正的設計問題是差異期間系統允許什麼、拒絕什麼、如何收斂。

Orientation

最終會一致，不代表中間可以隨便不一致

最終一致性允許副本在一段時間內不同，但必須能說清楚何時收斂、如何處理衝突，以及使用者在中間會看到什麼。

Core Tension

放寬即時一致性能提高可用性和延遲表現，但中間狀態會進入使用者決策、交易流程和後續系統。

01 定義中間語意

使用者看到舊資料、重複資料或缺資料時，系統要有明確承諾。

02 設計收斂路徑

同步、重試、補償和衝突解決要可觀測，不能只說稍後會好。

03 保護不可逆操作

付款、權限、庫存等不可逆決策可能需要更強一致性或最後確認。

Setup

A 區更新了權限，B 區還在用舊副本批准操作

最小場景是寫入已完成於某個副本，但讀取落在尚未同步的副本。

最小劇情

使用者在主區撤銷 API token，幾秒內另一區的邊緣節點仍讀到舊 token 狀態，繼續允許請求進入。

真正難點

短暫不一致可能對推薦列表無傷，對權限撤銷卻可能是安全缺口。

Key Point

最終一致性要按資料語意分級，不是把所有不一致都交給時間解決。

Lenses

把收斂前的世界也納入設計

系統在收斂前同樣在提供服務，因此那些讀寫結果也是產品與安全承諾的一部分。

陳舊讀取

使用者可能讀到多舊的資料，會造成顯示錯誤、交易錯誤還是安全錯誤？

衝突解決

同一資料多地更新時，用 last-write-wins、merge、CRDT 還是人工介入？

補償能力

如果中間狀態允許了錯誤操作，是否能撤銷、補償或通知？

Psychology

人容易把『最終』聽成保證，卻忘了中間狀態也會被使用

最終一致聽起來像穩定承諾，但事故常發生在尚未最終的那幾秒、幾分鐘或幾小時。

時間窗口被低估

幾秒不一致在報表上很短，在安全操作或金融流程裡可能足夠造成傷害。

收斂被當成自然現象

網路分區、重試失敗和 schema mismatch 都可能讓收斂停住。

產品語意被留白

工程說 eventual，使用者卻感受到資料忽隱忽現或操作被反悔。

採用最終一致性時，請把『最後會好』改寫成『中間允許什麼，最後如何證明已好』。

Applications

按資料風險選擇一致性承諾

同步資料、離線裝置和跨區服務都需要把不一致窗口產品化。

跨區讀寫

對低風險 profile 可接受短暫舊值，對權限撤銷採用強一致或全局黑名單。

離線裝置

允許本地暫存操作時，要設計同步衝突、版本提示和人工修復路徑。

事件驅動系統

下游 view 落後時要標示 freshness 或避免用它做不可逆決策。

Misreadings

最終一致性不是隨便一致

它需要明確的收斂保證、衝突規則和中間狀態承諾。

以為所有資料都適合 校正

不同資料的陳舊成本差異很大，不能共用同一策略。

只說 async 就結案 校正

非同步傳播不是一致性設計，還要處理失敗、重放和衝突。

忽略使用者回饋 校正

中間狀態若會被使用者看見，就要用文案、狀態或限制行為解釋。

Questions

問收斂前會發生什麼

下面兩問適合用在分散式資料設計和同步流程 review。

使用者在不一致窗口內能做什麼？

列出讀、寫、付款、撤權、刪除和回復等操作。

如何證明資料已經收斂？

需要版本、watermark、lag metric、reconciliation log 或補償紀錄。
