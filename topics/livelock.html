活結

Livelock

活結看起來很忙：每一方都在回應、退讓、重試，卻沒有任何一方真正完成工作。

Read For

分辨忙碌活動和有效進展，並用退避、隨機化與協調規則打破互相讓路。

主題類型
併發問題 / 忙碌不前
核心問題
Everyone keeps moving, nothing finishes
主要判準
持續活動、無進展、互相回應、破同步
讀完產出
能在併發 retry、協定協商與人工作業流程中辨認忙而不前

Orientation

系統一直動，不代表狀態正在前進

活結和死結相反：參與者沒有靜止等待，而是不斷改變狀態來避開彼此，結果仍無法完成。

Core Tension

自動重試和禮讓規則能提高恢復能力，但如果多方用同樣節奏互相反應，就會形成永遠同步的閃避舞步。

01 定義進展指標

不要只看 request rate 或 task activity，要看完成數、commit 數和狀態推進。

02 打破同步

使用隨機退避、jitter、leader 或 token，避免多方同時做同一反應。

03 限制無效重試

重試要有 backoff、budget 和觀測，否則活動量會掩蓋沒進展。

Setup

兩個服務都很有禮貌地退讓，結果誰也沒完成交易

最小場景是每一方都在回應對方，但回應規則讓雙方同步重複。

最小劇情

兩個 worker 搶同一筆工作，偵測到衝突後都釋放並立刻重試。它們總是在相同時間重新取得、再次衝突、再次釋放，log 顯示非常活躍卻沒有 commit。

真正難點

每個單次反應都合理，問題是反應規則在群體中形成週期。

→
雙方同時退讓
→

Key Point

活結的修正不是讓系統更忙，而是讓某一方能在規則內真正完成。

Lenses

把活動量和完成量分開看

活結最會騙過監控，因為它產生大量事件、重試和狀態變更。

完成指標

Request、retry 或 context switch 很高時，commit、dequeue 或狀態推進是否也增加？

同步來源

多方是否用相同 timeout、同樣 backoff 或同一觸發條件重試？

破局角色

是否有 leader、優先權、token 或隨機化讓其中一方能先完成？

Psychology

忙碌會讓人誤以為系統正在自我修復

看到大量 retry 和協商 log 時，團隊可能以為機制正在運作，卻沒有檢查是否真的完成工作。

活動感掩蓋停滯

人容易把有 log、有流量、有狀態變化當成有進展。

對稱規則看起來公平

所有人同樣退讓很合理，但對稱性正是週期重複的原因。

重試被當成韌性

重試沒有 backoff 和 budget 時，可能只是把無進展放大成噪音。

排查活結時，請先問完成率，而不是先問系統有沒有在動。

Applications

讓衝突後的下一步不再完全對稱

活結常出現在 retry protocol、分散式協商與人工作業互相等待中。

Retry protocol

加入 exponential backoff 和 jitter，避免所有 client 同時重試。

工作佇列

衝突後由單一 owner 或 lease 決定誰繼續，避免 worker 互相釋放。

人工作業

兩個團隊互相退回需求時，指定決策者和截止條件，避免反覆請對方先確認。

Misreadings

活結不是死結的另一個名字

死結是等待不動，活結是一直動但不前進；監控和修法都不同。

看到活動就排除卡住 校正

活動量高可能正是活結徵兆，要看完成量。

只增加重試次數 校正

更多重試可能讓同步衝突更嚴重。

忽略對稱性 校正

雙方規則完全一樣時，合理反應可能形成穩定循環。

Questions

問忙碌有沒有換成完成

下面兩問適合用在 retry storm 和流程卡住時。

哪些指標證明狀態真的前進？

用 commit、完成、移出佇列或狀態轉移，不要只看事件數。

多方是否在同一節奏退讓與重試？

若是，加入 jitter、leader、token 或明確優先權。

延伸路線

可接著讀 deadlock，對照等待不動和忙而不前的不同失敗型態。
