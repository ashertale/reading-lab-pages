Debounce

機械接點不會乾淨地從 0 跳到 1；debounce 的任務是把一串抖動訊號判成一次人類意圖。

Read For

適合處理按鍵、旋鈕、limit switch、GPIO interrupt 與任何 mechanical contact 進入數位邏輯的地方。

主題類型
訊號處理 / 接點雜訊
核心問題
Turn noisy transitions into one intended event
主要判準
取樣窗、穩定時間、延遲、誤觸
讀完產出
能設計不會把一次按壓變成多次事件的輸入處理

Orientation

數位事件背後常是一團類比抖動

機械開關接觸時會在短時間內反覆導通與斷開；若直接把每個 edge 當事件，軟體會看到不存在的多次按壓。

Core Tension

Debounce 越保守越能壓住抖動，但也增加輸入延遲；越靈敏越快，越容易把雜訊當意圖。

01 先量 bounce

用示波器或高頻取樣看實際接點行為，而不是猜。

02 再定事件語意

press、release、long press、repeat 都需要清楚狀態機。

03 最後保護中斷

GPIO interrupt 應喚醒判讀流程，不該直接觸發產品動作。

Setup

按一次，狀態機收到三次

Power key 接到 GPIO interrupt，韌體在 rising edge 切換狀態；使用者只按一次，裝置卻先開機又關機。

最小場景

接點 bounce 在 5 ms 內產生多個 edge，ISR 每次都送出 press event。

真正難點

人類意圖是一次按壓，但硬體訊號不是乾淨方波；軟體必須明確定義何時接受事件。

Key Point

Debounce 是把物理訊號翻譯成產品語意的邊界層。

Lenses

去彈跳可以在硬體或軟體做

沒有唯一正解；不同功耗、成本、延遲與安全要求會推向不同實作。

RC / Schmitt

硬體先濾掉高頻抖動，提高進入 MCU 前的訊號品質。

固定穩定窗

偵測變化後等待一段時間，確認狀態穩定再發事件。

積分取樣

連續多次取樣累積信心，比單純 sleep 更能處理雜訊。

狀態機

press、release、hold、repeat 和 cancel 應分開建模，避免補丁式 if。

方法優點代價
硬體濾波 減少 MCU 負擔與 interrupt storm 增加 BOM 與板級限制
固定 delay 實作簡單 延遲固定且可能不適合所有開關
取樣狀態機 可調、可測、可表達產品語意 需要 timer 與清楚規格

Psychology

偶發多按很容易被歸咎於手殘

按鍵 bug 的難處在於它常和操作手感、機構、公差與 firmware 時序混在一起。

不可重現

不同力道、角度、溫度與老化會改變 bounce，單次 lab 測試不代表量產。

ISR 崇拜

看到 edge 就立刻反應很有工程快感，但產品事件未必該在 ISR 裡決定。

延遲恐懼

為了手感快把 window 設太短，結果讓少數雜訊變成 field issue。

按鍵手感不是 debounce 的反面；好的狀態機會同時處理延遲與誤觸。

Applications

按鍵、旋鈕與限位開關都需要事件語意

Debounce 不只在人手按鍵；所有 mechanical contact 都可能讓數位輸入看到多次轉換。

Power key

需要區分短按、長按與卡鍵，並避免開關機狀態被 bounce 來回切換。

Rotary encoder

需要處理相位、抖動與轉速，否則旋轉方向或步數會錯。

Limit switch

機械震動會讓到位訊號抖動，控制系統要避免反覆進出狀態。

  • 把 debounce 參數放進可測試配置，並記錄它對最短按壓、最大連點速度與功耗的影響。
  • 用量產樣品與不同溫度/老化條件量測 bounce profile，再定 window。

Misreadings

Delay 不是設計，只是最粗的實作

固定延遲可能足夠，但它需要被量測與產品需求支持。

睡 20 ms 一定好 校正

不同開關 bounce 差異很大；固定值要靠量測證明。

只 debounce press 校正

release bounce 也會影響 long press、repeat 與狀態切換。

ISR 裡處理全部 校正

ISR 應短小；複雜事件判讀最好交給 timer/state machine。

Questions

先決定一個按壓何時算成立

如果使用者快速連點

debounce window 會不會吞掉合法第二次按壓，產品是否接受？

如果接點老化或震動

目前演算法會輸出多次事件、延遲變長，還是進入故障判斷？
