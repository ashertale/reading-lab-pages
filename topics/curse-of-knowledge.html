Curse of Knowledge

一旦你已經知道某件事，就很難真實地想像不知道它的人會卡在哪裡，於是說明、教學與介面都容易高估對方的起點。

這一頁會把『我已經講清楚了』和『對方真的能跟上』拆開，因為專家最常漏掉的是自己以為太基本而沒說的那一步。

Orientation

知道太多的人，最難感覺到哪一步其實沒被說出來

知識詛咒的難點不在惡意，而在於一旦你看見了全圖，就很難再用初學者視角重新走一次。

Core Tension

專家越熟，越容易壓縮解釋、跳過背景與默認關鍵前提；對新手來說，最痛的往往正是那些被省略掉的理所當然。

01 先盤點前提

你覺得不需要說的背景，往往正是別人進不去的門檻。

02 再測真正理解

點頭、安靜或照抄，不代表對方已經建立同樣的模型。

03 最後用例子補橋

抽象說明若沒有具體例子與反例，知識落差很難被跨過。

Setup

從一份你覺得很清楚的文件開始

知識詛咒最常出現在你真心覺得自己已經說得夠明白的時候。

最小劇情

你寫了一份 setup 指南，覺得步驟都列齊了；新人卻卡在第三步，因為你默認他知道哪個 toolchain 已安裝、哪個 env var 要先設。

真正難點

不是你故意講太快，而是你已經無法自然感受到哪些前提對新人其實完全不可見。

專家腦中前提變自動化
→
說明時把關鍵步驟壓縮
→
新手在看不見的缺口卡住
Key Point

知識詛咒最有價值的提醒是：清楚不等於完整，尤其當聽的人和你不在同一個地板高度上。

Lenses

四個看專家盲區的角度

若你懷疑溝通失敗不是對方不用心，而是你跳步了，可以先看這四件事。

隱含前提

有些背景知識你已經內化到忘記它其實不是常識。

語彙密度

術語堆疊太快時，新手常在第一個沒懂的詞就整段掉隊。

步驟壓縮

專家會把多步驟動作自動合併成一句話，聽者卻需要逐段展開。

驗證錯覺

對方說『好』，有時只是社交上的回應，不代表理解模型一致。

溝通情境 專家看起來像 新手實際可能卡在哪
安裝說明 步驟已經很清楚 隱含前提與環境差異完全沒說出來
Code review 建議 這很基本吧 其實對方還沒建立你在用的風險模型
UI 文案 字已經夠短 短到把必要解釋一起切掉
教學演示 我都一步一步講了 中間很多推理在你腦中是自動完成的

Psychology

為什麼一旦你懂了，就很難回到還不懂的世界

因為知識會改寫你看問題的方式，讓很多中間步驟直接消失在意識裡。

自動化遮蔽了細節

熟練者會把多個判斷壓成一個直覺，於是忘記自己曾經也需要那些階梯。

我們高估共同背景

只要對方和自己在同一團隊、同一專業，就容易誤以為起點差不多。

社交回饋會騙人

新手常不想一直打斷，所以表面配合會讓專家更以為自己講清楚了。

這個偏誤最適合拿來校正教學、文件與產品設計，不是拿來批評誰不會講話。

Applications

把前提可見化帶回文件與介面

對工程團隊來說，知識詛咒最昂貴的地方，是它會把原本可以被設計掉的 onboarding 與 UX 摩擦偽裝成『對方程度不夠』。

文件與 SOP

真正好用的文件會把前提、常見錯誤與驗證方式一起寫出來，而不是只寫操作名詞。

CLI 與工具 UX

專家覺得理所當然的參數組合，對新手可能像沒有地圖的黑盒。

Mentoring

帶人不是重複答案，而是幫對方看到你心裡那些自動化的判準。

  • 做說明時，先列出三個你本來不打算寫、但其實是前提的資訊。
  • 驗證理解時，請對方自己走一遍或用自己的話重述，通常比問『有沒有問題』更有效。

Misreadings

三個很容易掉進去的誤解

知識詛咒不是說專家不該簡潔，而是說簡潔不該建立在偷省前提。

所以所有說明都要寫超長 校正

不是。重點是把必要前提補齊，而不是把所有東西都灌進去。

對方沒懂就是他程度差 校正

有時真正缺的是橋接，不是智力。

這只發生在教學 校正

產品設計、文件、指令列與跨團隊溝通都常被這個偏誤影響。

Questions

我省掉的哪一步，正是對方最需要的那一步嗎

下面兩個問題很適合在寫文件、做 demo 或 review 後回頭檢查。

若我完全不知道這套東西，我會在哪一步第一次失去方向

把那一步補出來，通常比再精煉高層摘要更有幫助。

我現在依賴了哪些沒明說的共同背景

工具、術語、假設與慣例是否真的被共享了？
