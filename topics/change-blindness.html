變化盲視

Change Blindness

人不只會漏掉沒注意的東西，也會漏掉已經改變的東西；如果變化被遮斷、分散或間隔包住，眼前差異也可能消失。

Read For

練習把『我看過前後狀態』改成『我有比較機制』，避免變更在視覺與流程縫隙中溜走。

Orientation

變化不會自動被看見；它需要一個穩定的比較錨點

變化盲視提醒我們，前後都看過不等於知道哪裡不同；人需要提示、記憶或工具幫忙對齊差異。

Core Tension

快速掃描讓人能處理大量畫面，但只要比較錨點消失，重大變更也可能被當成同一個場景。

01 建立前後錨點

讓審查者能直接對照舊狀態與新狀態，而不是憑印象回想差異。

02 標出刪除與移動

新增通常醒目，移除和搬移比較容易被忽略，因此要有額外提示。

03 把比較交給工具

關鍵變更應由 diff、changelog 或 review checklist 承接，不該只靠人的記憶。

Setup

最容易漏掉的變更，常發生在你以為自己已經看過的地方

最小場景是 review 者看了新畫面，卻沒有可靠方式把它和舊畫面逐項對照。

最小劇情

設計稿更新後，主要按鈕位置與權限提示一起改了。評審者看過前版也看過新版，卻只注意到顏色變好，沒發現提示文字被拿掉。

真正難點

變化需要被比較才會成為資訊；沒有對齊前後狀態，人會用整體熟悉感代替差異偵測。

前後畫面被分開看
→
熟悉感蓋過差異
→
變更影響延後浮現
Key Point

變化盲視的解法不是叫人眼尖，而是提供可對照、可標記、可追蹤的變更線索。

Lenses

把變化偵測拆成前版、後版與差異提示

若流程只展示結果，不展示差異，審查就會依賴記憶，而記憶最不可靠。

前版錨點

保留舊狀態或自動 diff，讓比較不靠腦中印象。

差異提示

把移除、改名、位置變動與權限影響標出來，而不只讓人看新版。

檢查節奏

把掃描和比較分開；先理解新版，再逐項核對變更。

Psychology

熟悉感會讓人以為自己掌握全局

人看到熟悉場景時，會快速補全缺口；這種補全能力在變更偵測上會變成風險。

整體相似會掩蓋局部差異

畫面大致相同時，大腦傾向相信細節也差不多。

記憶不是精準快照

人常記得語意和大概位置，卻不記得每個文字、欄位與狀態。

注意轉移會切斷比較

一旦中間有遮蔽、跳轉或任務切換，差異就需要額外標記才容易被抓到。

校正變化盲視，就是把差異從記憶問題改成介面與流程問題。

Applications

所有變更都該有自己的可見性設計

工程與產品流程裡，許多風險不是沒被呈現，而是變更沒有被明確標成變更。

UI review

用 visual diff 和變更註記標出刪除、移動與文案差異，避免只靠評審者印象。

配置變更

把 default、權限與回退值逐項列出前後差異，讓隱性變更浮上來。

文件審查

重要文件不要只傳最新版，應提供 changelog 與關鍵段落 diff。

Misreadings

漏看不等於眼睛沒看到

這個概念提醒我們檢查比較條件，而不是用『明明前後都看過』結案。

把漏看變更當成不用心 校正

如果前後狀態沒有並排或標記差異，漏看很可能是流程問題。

以為熟悉系統就會看出差異 校正

越熟悉越容易用舊模型補全畫面；熟悉感反而可能遮住局部改動。

只強調新增內容 校正

刪除、移動和預設值變更常比新增更難被注意，卻可能影響更大。

Questions

讓前後狀態直接對齊

前後狀態如何被對齊？

審查者是否能同時看到舊版、新版與差異，還是只能靠記憶比較？

哪些刪除比新增更難看見？

移除的提示、欄位、權限或限制是否被單獨列出？

延伸路線

可接著讀 inattentional blindness，對照注意任務如何讓人漏掉未預期訊號。
