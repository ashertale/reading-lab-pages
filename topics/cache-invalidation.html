快取失效

Cache Invalidation

快取失效真正難的不是把資料存快，而是判斷哪一刻舊資料已經不能再代表真實狀態。

Read For

看懂 fresh、stale、TTL、事件通知與回寫路徑如何一起決定資料可信度。

Orientation

快不是唯一目標；舊資料何時變成錯資料才是問題

快取讓讀取變快，也讓資料多出一份副本；副本越有用，何時同步、何時丟棄、誰負責通知就越重要。

Core Tension

系統想降低延遲與後端負載，但只要快取留住過期狀態，使用者看到的就可能是已不存在的權限、庫存或設定。

01 定義陳舊窗口

先說明資料可以舊多久，而不是先選 TTL 數字。

02 標出寫入來源

任何能改變真實狀態的路徑，都要能觸發或等待相應失效。

03 讓錯誤可觀測

命中率漂亮不代表安全，還要能看 stale read、miss storm 與失效延遲。

Setup

一筆權限更新完成了，快取卻還在批准舊身份

最小場景是一份副本繼續提供答案，但原始資料已經改變。

最小劇情

管理員移除某帳號的存取權，database 已更新成功；API gateway 仍使用五分鐘前的 permission cache，讓舊 token 繼續通過敏感操作。

真正難點

TTL、手動 purge、事件通知和 read-through cache 都可能合理；錯誤在於沒先定義安全資料能容忍多少陳舊。

Key Point

快取失效的設計要從資料語意出發：有些資料可舊一分鐘，有些資料一秒也不能舊。

Lenses

把失效責任放回資料生命週期

快取不是單一元件，而是一條從寫入、通知、重建到讀取的生命週期。

資料語意

設定、權限、價格、庫存和報表可以容忍的陳舊程度不同。

失效觸發

寫入後是同步刪除、事件廣播、版本號比對，還是等待 TTL 自然過期？

重建行為

大量鍵同時過期時，是否會把後端打爆，或讓使用者看到不一致結果？

Psychology

團隊容易被命中率吸引，卻少問命中的是不是還能信

快取指標常讓人先看到速度收益，資料語意和失效責任反而被留到事故後補。

局部優化很誘人

降低 latency 和 DB load 很容易被展示，stale read 風險卻常要等到業務語意破掉才可見。

責任跨過邊界

寫入服務、讀取服務、消息系統和 CDN 都可能各自覺得失效不是自己完整責任。

測試樣本太乾淨

單元測試常只看命中與未命中，很少模擬寫入後讀取、事件延遲與重建尖峰。

快取策略要把速度收益和語意風險一起寫進設計，而不是讓 TTL 成為默認答案。

Applications

把 cache policy 寫成資料契約

權限、商品、設定和韌體發佈資料的快取策略不應該共用同一個答案。

權限檢查

使用短 TTL 之外還要加入版本號或撤銷清單，避免已撤權身份繼續通過。

商品與庫存

允許短暫陳舊時，要在下單前重新確認可售量，讓展示快和交易準分開。

韌體 rollout

裝置查詢更新資訊可快取，但撤回版本或封鎖批次需要可立即失效的通道。

Misreadings

快取失效不是只把 TTL 調短

TTL 只是其中一種近似方法；資料語意、寫入頻率和失效成本才決定策略。

以為短 TTL 就安全 校正

短 TTL 降低平均陳舊時間，但關鍵撤銷仍可能在窗口內出錯。

把快取當純效能層 校正

只要快取內容影響決策，它就是資料一致性設計的一部分。

忽略重建尖峰 校正

同時過期可能造成 cache stampede，讓修失效變成新的可用性問題。

Questions

先問資料可以舊多久

下面兩問適合放在 cache design review 和上線前檢查。

這份資料的最大可接受陳舊時間是多少？

請依資料語意回答，而不是先拿系統預設 TTL。

哪條寫入路徑會讓快取答案立刻不能用？

列出所有改變真實狀態的入口，確認它們能觸發失效或版本檢查。

延伸路線

可接著讀 caching 與 cache stampede，理解副本帶來的效能收益和重建風險。
