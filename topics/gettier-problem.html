葛梯爾問題

Gettier Problem

葛梯爾問題指出：一個信念即使合理且剛好為真，也可能只是靠運氣踩中真相。

Read For

分辨 justified true belief 為何仍不足以保證知識，並看出運氣在判斷中的位置。

Orientation

你可以有好理由，也可以答對，但仍然不是知道

葛梯爾反例挑戰的是知識三要件：信念、真實、理由。它讓我們看到，理由和真相之間若靠錯誤路徑相遇，知識感就會破裂。

Core Tension

我們需要用理由判斷知識，但好理由可能經過偶然、誤認或錯誤中介，最後碰巧得到真結論。

01 檢查理由鏈

結論為真不夠，要看支持結論的理由是否以正確方式連到事實。

02 排除幸運命中

若同樣推理在鄰近情境很容易錯，就不應太快稱為知道。

03 分開結果和方法

回顧成功時要問方法是否可靠，而不是只看結果答對。

Setup

你看見一個假證據，卻推到一個剛好真的結論

最小場景是推理路徑中有錯誤，但最後命中事實。

最小劇情

你看到同事桌上有一張假的得獎通知，於是相信他得獎了。剛好他真的在另一個比賽得獎，只是你看到的那張通知不是原因。

真正難點

結論是真的，你也有看似合理的理由，但理由不是讓結論為真的那個事實。

信念有理由
→
理由鏈含有誤導
→
結論剛好為真
Key Point

葛梯爾問題要求我們檢查真信念是否以非偶然方式連到事實。

Lenses

把答對拆成理由可靠性和事實連結

知識不只是結果正確，而是認知路徑能穩定避開鄰近錯誤。

理由來源

證據本身是否可靠，還是只是看起來像證據？

連結方式

理由是否真正解釋結論為何為真？

鄰近世界

稍微改變情境，這套推理是否很容易得出假結論？

Psychology

人很容易把結果正確回寫成方法可靠

成功會替推理鍍金；葛梯爾提醒我們，成功可能只是錯誤路徑碰巧落在真相上。

結果偏誤很強

答對後，人會低估中間推理的漏洞。

證據外觀會騙人

看起來像正式通知、測試通過或權威來源，就容易被當成可靠理由。

偶然性不容易被記錄

運氣通常不像錯誤一樣在 log 裡顯示，需要主動檢查。

好的回顧要獎勵可靠方法，不只獎勵剛好答對。

Applications

讓成功也接受證據鏈審查

測試、除錯和策略決策中，剛好正確常被誤認成真的知道。

測試判讀

測試通過若 mock 錯、assert 太弱或資料剛好，不能代表功能被理解。

Root cause

修一行 code 後症狀消失，不代表找到根因；要證明機制連結。

策略回顧

押對市場方向可能來自運氣，需檢查當時理由是否能穩定支持選擇。

Misreadings

葛梯爾問題不是說知識不可能

它要求補強知識條件，尤其是排除認知運氣。

把它當成懷疑一切 校正

問題不是所有信念都不可靠，而是三要件還不夠。

只看結論是否真 校正

真結論可能由錯誤理由碰巧推出。

只在哲學裡有用 校正
