向量時鐘

Vector Clock

向量時鐘不追求全域幾點幾分，而是記錄每個節點知道哪些事件已經發生，藉此區分因果先後與真正並發。

Orientation

時間不是一條線，而是一張知道關係

每個節點維護一個向量，記錄自己與其他節點的事件進度；比較兩個向量時，可以看出包含、落後或並發。

Core Tension

系統很想用 timestamp 選勝者，但 wall clock 漂移與離線更新會把真正並發的事件偽裝成先後。

01 先記本地進度

節點發生事件時增加自己的分量。

02 再交換知識

訊息攜帶向量；接收端逐分量取最大值並記錄新事件。

03 最後比較偏序

一方全分量不小於另一方才代表因果包含；否則可能是並發。

Setup

兩台設備離線後都改了設定

Device A 和 Device B 各自改同一個 threshold。重新連線時，只看最後修改時間會讓其中一份直接消失。

最小場景

A 的版本向量是 4/1，B 的版本向量是 3/2；A 知道較多 A 事件，B 知道較多 B 事件。

真正難點

兩個向量互相都不是完全大於對方，代表它們不是單純新舊，而是各自分岔。

Key Point

向量時鐘的價值在於告訴你何時不該安靜覆蓋。

Lenses

用四個動作讀懂版本關係

向量時鐘的運算規則簡單，困難在於把比較結果接到正確的產品語意。

Increment

本地寫入、狀態轉移或事件發生時，只增加自己的 slot。

Merge

收到遠端向量後逐項取最大值，表示已吸收對方知道的歷史。

Dominance

若 X 每項都大於等於 Y，且至少一項更大，X 因果上晚於 Y。

Concurrency

兩邊各有較大分量時，代表沒有包含關係，需要衝突策略。

比較結果語意同步動作
A dominates B A 已包含 B 的歷史 可用 A 覆蓋或讓 B 快進
B dominates A B 已包含 A 的歷史 可用 B 作為較新版本
不可比較 兩邊並發分岔 執行 merge rule 或保留衝突

Psychology

人會把鐘面順序誤認成因果

看到較晚 timestamp 很舒服，因為它給出一個單一答案；但分散式系統裡，較晚不一定代表知道較多。

最後寫入迷信

Last-write-wins 簡單，但會把離線期間的合法工作丟掉。

時鐘權威錯覺

NTP 或 RTC 讓時間看似共同，但網路延遲與裝置漂移仍會破壞判斷。

衝突羞恥感

團隊常把衝突視為系統失敗，其實有些衝突是世界真的同時分岔。

向量時鐘把不確定性留在資料裡，逼產品決定並發更新的處理方式。

Applications

把衝突偵測放進同步協議

只要系統允許多端離線寫入，版本欄位就不能只是一個自增整數或牆上時間。

Device configuration

工廠工具與現場設備各自修改設定，回傳後用向量判斷誰包含誰。

Distributed tracing

跨服務事件可用邏輯時間輔助判讀因果，而不是只看 log 排列。

Replicated document

協作文件的編輯若不可比較，系統需要 CRDT、OT 或人工合併策略。

  • 節點數固定且小時，完整向量很直觀；大型系統需考慮 version vector、dotted version vector 或壓縮表示。
  • 向量時鐘只偵測因果，不替業務決定哪個值正確。

Misreadings

它不是更準的時鐘

向量時鐘的名字容易誤導；它不量秒數，而是量每個節點看過的事件前綴。

可以取代所有 timestamp 校正

審計、UI 顯示與 timeout 仍需要實際時間；向量解的是因果比較。

能自動合併衝突 校正

它只標出並發，merge 規則仍要依資料型別設計。

節點越多越簡單 校正

向量大小跟參與者數量相關；動態 membership 需要額外工程處理。
