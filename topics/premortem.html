Orientation

如果大家都想成功，為什麼反而要先假設專案已經死了

讀 事前驗屍，先別急著表態自己站哪一邊；更重要的是看在 規劃與預先檢討 裡，預先失敗、逆向追因 和 風險具體化 誰應該先被說清楚。

Core Tension

與其在案子死掉後才做檢討，不如在開始前先假設它已經失敗，再倒推可能怎麼死。

01 先確認 預先失敗 沒被偷渡進來

如果 預先失敗 沒先寫出來，後面看起來像分歧，實際上只是各自補腦。

02 接著看 逆向追因 被誰省略

把 逆向追因 講清楚後，你才看得出這條原則是在保護判準，還是在轉移代價。

03 最後把 風險具體化 變成驗收

少了 風險具體化，事前驗屍 很容易停在口頭正確，卻不知道怎麼在 專案 kickoff 或 發版前檢查 裡執行。

Setup

問題不是立場，而是 風險具體化 有沒有被當成默認

先把場景壓回 專案 kickoff 或 發版前檢查 這種真的要決定責任的時刻，你會更清楚 預先失敗、逆向追因 和 風險具體化 哪一個不能事後才補。

最小劇情

團隊在立案或上線前，先一起想像半年後這案子失敗了，再逐條寫出最可能的死因與現在能補的缺口。

真正難點

Key Point

這類原則一旦失手，通常不是理論錯，而是 預先失敗 沒先寫、逆向追因 沒算進去、風險具體化 也沒被翻成行動。

Lenses

先把 先把失敗說出口、風險變得可討論 與 樂觀偏誤被拉回來 分開看

同一個原則落到現場後，往往會分裂成 先把失敗說出口、風險變得可討論 和 樂觀偏誤被拉回來 三種完全不同的操作方式。

先把失敗說出口

把焦點放在 先把失敗說出口 時，你其實是在說：預先失敗 應該先被寫清楚，否則後面的結論都站不穩。

風險變得可討論

風險變得可討論 這條線在乎的，不是誰比較有氣勢，而是 逆向追因 一旦沒被交代，現場會怎麼開始失真。

樂觀偏誤被拉回來

順著 樂觀偏誤被拉回來 讀，你會發現爭點不在口號，而在 風險具體化 是否真的被變成判準。

Psychology

哪種省事直覺最常讓 事前驗屍 失焦

一到現場，人最容易先抓住能讓自己省事的那一段，於是 事前驗屍 很快就從判準退化成表態。

一開始談失敗像唱衰

一開始談失敗像唱衰 會讓人覺得只要把話說得夠完整，就不必再補 預先失敗 的證據或代價。

順利想像比較舒服

順利想像比較舒服 常不是純粹錯誤，而是一種討論捷徑：先把自己放在合理的一邊，再晚點處理 逆向追因。

隱憂需要安全出口

碰到 隱憂需要安全出口 時，最危險的是大家以為原則已經被引用，實際上 風險具體化 還沒被具體說明。

只要討論還停在『誰比較像站在對的一邊』，事前驗屍 就還沒真正開始工作。

Applications

回到 專案 kickoff、發版前檢查 與 高風險決策會議

把 事前驗屍 放回 專案 kickoff、發版前檢查 這些場景時，它的作用不是增加說服力，而是強迫大家先把責任、代價和驗收順序排好。

專案 kickoff

專案 kickoff 最需要 事前驗屍 的時候，通常就是大家都急著往前走，卻還沒把 預先失敗 寫成共識。

發版前檢查

只要到了 發版前檢查，你就能看出 逆向追因 究竟是被真的承擔，還是只是被口頭提過。

高風險決策會議

在 高風險決策會議 這種場景，事前驗屍 幫你的不是贏辯論，而是把 風險具體化 先釘在桌面上，避免事後才補。

Misreadings

三個常見誤讀

原則類概念一旦變得順耳，就很容易失手；事前驗屍 最常就是在 專案 kickoff 或 發版前檢查 這些情境裡，被縮成一句不用交代代價的口號。

事前驗屍 只是好聽口號 校正

真正難的是把 事前驗屍 翻成一組可檢查的判準，而不是把它當成結論本身。

只要背了 事前驗屍 就能直接套用 校正

原則若不落回 預先失敗、逆向追因 和代價分配，通常會在現場變成誤用。

事前驗屍 只適合高層決策 校正

其實從 code review 到 規劃與預先檢討 的日常判斷，都會碰到同樣的壓力。
