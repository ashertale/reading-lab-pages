驚群問題

Thundering Herd Problem

驚群問題發生在大量等待者同時被喚醒，大家一起搶同一份有限資源，結果把恢復動作變成新一波壓力。

Read For

理解喚醒策略、重試節奏、cache 重建和限流如何避免同時起跑。

主題類型
效能問題 / 競爭喚醒
核心問題
Too many waiters waking at once
主要判準
等待者數量、喚醒範圍、共享資源、退避策略
讀完產出

Orientation

問題不是有人醒來，而是所有人同時醒來

驚群問題把等待變成同步競爭：一個事件釋放後，太多工作同時爭搶鎖、連線、cache 或任務。

Core Tension

喚醒等待者能恢復進展，但全量喚醒會讓有限資源瞬間被淹沒，讓系統在剛恢復時再次失穩。

01 只喚醒需要的人

能喚醒一個就不要喚醒全部，能分片就不要全域通知。

02 加入退避和抖動

重試、重連和 cache rebuild 要有 jitter，避免同一時間重新衝擊。

03 保護被搶資源

用 singleflight、lock、限流或 queue 讓同一份工作只被重建一次。

Setup

一個熱門 cache key 過期，所有請求同時打到資料庫

最小場景是等待者本來被擋住，某個條件解除後一起衝向同一資源。

最小劇情

首頁推薦資料的 cache key 過期後，數千個請求同時 miss。每個 request 都嘗試重建同一份資料，原本要保護資料庫的 cache 反而製造尖峰。

真正難點

單一 request 重建 cache 是合理的；大量 request 同步做同一件事才是災難。

大量等待者累積
→
同一事件觸發喚醒
→
有限資源被同步競爭打滿
Key Point

驚群問題的修正，是讓恢復動作分散、合併或排隊，而不是讓每個等待者各自衝刺。

Lenses

檢查喚醒是否比資源容量更寬

當喚醒範圍大於可服務容量，恢復點就會變成尖峰點。

喚醒粒度

事件會喚醒一個 worker、一組 shard，還是所有等待者？

重建合併

同一 key、同一連線或同一任務是否能由單一 owner 處理，其他人等待結果？

尖峰保護

是否有 rate limit、queue、backoff 或 token bucket 平滑恢復流量？

Psychology

恢復時刻最容易被誤認成可以全速前進

人看到資源恢復，就自然想讓所有等待工作立刻補進度；但系統剛恢復時往往最脆弱。

補償心理放大尖峰

先前被擋住的工作想一次補回來，容易形成比原本更高的瞬時負載。

全量通知看似簡單

Broadcast 實作方便，卻把選擇權交給所有等待者同時競爭。

平均流量遮住同步性

一分鐘平均 QPS 可接受，不代表同一秒內的喚醒尖峰可承受。

處理驚群問題時，請把恢復視為需要節流的事件，而不是單純解除封鎖。

Applications

把同時恢復改成分散恢復

Cache、連線池、檔案事件與裝置重連都可能需要驚群保護。

Cache stampede

用 singleflight 或 stale-while-revalidate 讓熱門 key 只有一個重建者。

連線恢復

大量裝置斷線後重連要加入 jitter 和分批 gate，避免服務剛恢復又被打滿。

Worker 喚醒

工作佇列避免全量 broadcast，改用精準通知或固定數量 worker 消費。

Misreadings

驚群不是單純流量太大

流量總量可能不變，真正危險是等待者在同一時刻做同一件事。

只擴容後端 校正

擴容能提高容量，但同步尖峰仍可能超過新容量。

把重試當韌性 校正

沒有 jitter 的重試會讓失敗者在同一時間再次聚集。

忽略熱門鍵 校正

少數 key 或資源就能引發全局尖峰，平均分布會掩蓋它。

Questions

讓恢復不要同時起跑

下面兩問適合用在 cache、重試與喚醒設計 review。

這個事件會喚醒多少等待者？

估算最壞情況下同時被喚醒的 client、worker 或 request 數量。

同一份工作能否只做一次？

對熱門 key、重連或初始化流程加入合併、排隊或 single owner。

延伸路線

可接著讀 cache stampede，理解熱門快取同時失效時的重建尖峰。
