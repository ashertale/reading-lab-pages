典範轉移

Paradigm Shift

典範轉移描述的是一套共同問題、方法和標準被另一套取代；改變的不只是答案，而是什麼算問題、什麼算好證據。

Read For

理解常態科學、異常累積、不可通約性和社群轉向如何改變知識框架。

Orientation

有時候新理論不是回答舊問題，而是重畫問題的地圖

典範提供研究社群共享的範例、工具、價值和標準；轉移發生時，連評分規則也會跟著改。

Core Tension

舊典範能解很多問題，才會穩固；新典範初期未必完整，卻可能更好地組織那些長期被壓住的異常。

01 辨認常態工作

一個領域平時如何定義好問題和好解法？

02 追蹤異常累積

哪些現象被反覆視為例外、修補或暫時放置？

03 看標準是否改變

轉移不只是新答案勝出，也包含評估標準的改寫。

Setup

微服務不是單純把模組拆小，而是改變了部署、責任和失敗觀

最小場景是一個架構變化同時改寫了團隊如何理解系統。

最小劇情

單體架構長期靠集中 release 管理品質。當組織轉向微服務，問題不再只是如何避免所有故障，而是如何讓局部故障可隔離、可觀測、可獨立部署。

真正難點

新典範帶來新成本，也讓舊指標不再完整描述成功。

Key Point

典範轉移要你看評估規則是否改變，而不是只看工具名稱是否更新。

Lenses

檢查問題、工具和標準是否一起移動

真正的轉移會改變整套實務，不只是新增 buzzword。

問題定義

哪些問題被新框架凸顯，哪些被降級？

方法工具

新的量測、語言和操作方式是什麼？

成功標準

社群如何重新判斷什麼算優雅、正確或有價值？

Psychology

舊典範會保護已經熟練的人，也保護他們的評分方式

框架轉換威脅的不只是理論，還有專家身份、訓練投入和制度位置。

專業沉沒成本

舊框架越熟練，越難承認新標準。

新框架早期粗糙

初期缺陷容易被用來證明它不成熟。

語言斷裂

雙方使用相同詞彙，可能指向不同評估世界。

判斷轉型時，請問它是否改變了我們如何定義錯誤和成功。

Applications

在轉型中同時管理舊標準與新標準

技術架構、AI 工具和研究領域都會經歷框架轉換。

架構轉型

從單體到服務化時，指標要加入部署自治、觀測性和故障隔離。

AI 工作流

導入生成式工具不只是加速寫作，也改變審查、責任和品質標準。

研究策略

異常案例若越來越多，可能需要新分類和新方法，而非更多補丁。

Misreadings

典範轉移不是任何流行詞都能宣稱

真正的轉移要看問題、方法、標準和社群實務是否一起變。

把工具升級當典範轉移 校正

換工具不一定改變框架。

以為舊典範全錯 校正

舊框架通常在某些範圍仍有效。

忽略過渡成本 校正

轉移期間常需要雙軌指標和教育。

Questions

問改變的是答案，還是評分規則

下面兩問適合用在技術轉型和研究方向評估。

新框架讓哪些舊問題變得不重要？

這能顯示標準是否移動。

我們是否仍用舊指標評估新實務？

避免用過時尺衡量新方法。
