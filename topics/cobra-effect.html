Orientation

好意的獎勵，怎麼養出反方向的行為

眼鏡蛇效應不是在說人天生愛鑽漏洞，而是在說制度一旦只獎勵可數的表面成果，理性行為可能會把問題本身變成資產。

Core Tension

設計者想用獎勵消除問題，卻把「產生可被獎勵的問題」變成一條收益路徑。短期報表看似進步，長期系統卻更依賴原本要消滅的東西。

01 先看獎勵單位

獎每一件、每一次、每個數字時，現場會先學會增加可計分單位。

02 再看驗證難度

如果驗證只看回報表面，制度會鼓勵包裝、拆分與重分類。

03 最後看退出機制

取消獎勵時，原本被養大的副作用可能一次釋放回系統。

Setup

把問題縮到最小

最小故事是：你想減少某個壞東西，於是付錢給每一個「被消滅的壞東西」。制度看似直接，卻漏掉了人可以影響供給端。

最小劇情

政府、公司或平台宣布按件給獎，要求現場提交可驗證證明。只要證明比真實改善容易，大家就會先優化證明。

真正難點

問題不在獎勵本身，而在制度沒有分清楚「消除存量」和「創造可申報事件」。當後者更容易，補救措施就會餵大問題。

用獎勵加速解決
→
現場發現可操弄供給
→
補救方案反而製造更多問題
Key Point

眼鏡蛇效應可以視為古德哈特定律的制度版：當被獎勵的 proxy 比真目標更容易生產，系統就會開始生產 proxy。

Lenses

四種檢查誘因反噬的透鏡

判斷一個方案是否危險，不只看目標是否善良，還要看現場能不能低成本偽造進步。

獎勵是否按件計算

按件獎勵會把事件切得越細越有利。若沒有品質門檻，拆分和重複申報會比真正修復更划算。

問題供給是否可被操弄

只要參與者能製造、延後、重新分類或引入更多問題，獎勵就可能改變供給端，而不是消除存量。

驗證是否晚於獎勵

如果錢、分數或聲望先發出去，後續才檢查長期效果，制度就會偏向快速可見的表演。

取消後是否有反彈

制度退場時，被獎勵養大的行為不會自動消失。沒有退場計畫，副作用可能集中爆發。

制度設計 看似改善 可能反噬
按問題件數給獎 回報量快速上升 人會製造或拆分更多可回報事件。
只看短期移除數 存量短暫下降 供給端被刺激，長期總量變高。
只驗收表面證明 證據包很完整 證據品質提升，真實改善未必同步。
有退出與抽查 短期速度較慢 較能避免把壞行為變成穩定生意。

Psychology

為什麼局部理性會把制度整體推歪

越是急著修正明顯痛點，越容易把注意力放在可立即計數的動作，而不是系統行為會如何回應。

行動感壓過因果感

發獎勵、設 KPI、收證明都很像在做事。問題是制度真正改變的是人的選擇空間，不只是人的努力程度。

短期曲線太迷人

一開始數字常會變漂亮，因為低成本案例先被收割。等現場學會反向生產，失真才會浮出來。

責任被推給個人

制度失靈後，管理者容易說有人作弊。但很多時候，是設計把不想要的行為變成合理選擇。

重點是說明制度設計如何把人推向局部理性，而不是把問題簡化成人品故事。

Applications

把誘因副作用帶回產品與制度設計

工程團隊常需要用獎勵或指標改變行為。眼鏡蛇效應提醒你：先檢查制度會不會讓人創造更多可被獎勵的工作。

Bug bounty 與問題回報

若只按件數給獎，回報者可能拆小、重複或提交低品質案例。較好的設計會把嚴重度、可重現性、修復價值和濫用偵測放進規則。

測試通過率

若只獎勵 pass rate，團隊可能避開 flaky 或高風險測項。真正要獎勵的應是風險降低，而不是漂亮的通過曲線。

量產異常處理

若供應商按「處理件數」被評分，最容易被處理的異常會優先被推動，根因複雜但影響大的問題反而被拖延。

  • 設計獎勵前先問：參與者能不能低成本製造更多可領獎事件？
  • 把獎勵延後到長期效果被驗證，通常比只看提交證明更安全。

Misreadings

三個常見誤讀

這個概念容易被拿來嘲笑政策失敗，但它真正有用的地方是替制度設計做壓力測試。

所以不要用獎勵 校正

不是。獎勵仍然有用，但要獎勵難以偽造、接近真目標、且能被長期驗證的行為。

這只是有人作弊 校正

有些情況是作弊，有些只是制度把錯誤行為合理化。修人之前，先修誘因。

只要加更多稽核就好 校正

稽核能降低濫用，但如果獎勵單位本身錯了，稽核只是在替壞指標加成本。

Questions

把它變成你的制度檢查表

下面兩個問題適合在設 KPI、bonus、bug bounty 或外包驗收前先問一次。

如果大家都照規則最大化收益

系統會真的更接近目標，還是只會產生更多可申報、可計分、可包裝的成果？

如果明天取消獎勵

被制度養大的行為會自然消失，還是會把原本壓住的副作用釋放回現場？
