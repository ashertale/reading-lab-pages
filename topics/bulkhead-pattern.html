隔艙模式

Bulkhead Pattern

隔艙模式把資源切成可控區塊，讓一個區塊卡死時，不會拖垮整艘系統。

Orientation

不要讓低價值工作吃掉救命資源

隔艙模式把 thread、connection、queue、memory 或 service instances 分成獨立池，讓故障停在邊界內。

Core Tension

共享資源最省，但也最容易讓一個慢 downstream、buggy feature 或流量尖峰把整個系統拖住。

01 先找共享耗盡點

thread pool、socket、heap、queue depth 都可能是失效傳播通道。

02 再切核心路徑

控制、恢復、安全與管理面不該和低價值背景工作完全共用資源。

03 最後接受利用率代價

隔離會犧牲部分效率，換來故障時的可預測降級。

Setup

Telemetry 卡住，控制命令也失靈

設備把 telemetry upload、remote command、firmware check 都丟進同一個 worker pool；網路慢時，upload 任務佔滿 worker。

最小場景

低優先上傳任務等待 timeout，command handler 排在後面，使用者以為設備死機。

真正難點

功能彼此看似無關，卻透過 thread pool 和 socket pool 共享同一個耗盡點。

低價值任務變慢
→
共用資源被吃滿
→
核心路徑無法取得執行機會
Key Point

隔艙模式把故障傳播路徑改成可設計的邊界。

Lenses

隔離要切在會耗盡的地方

服務拆開不代表隔離完成；若仍共用同一個資料庫連線池或 executor，失效仍會穿過去。

資源池

獨立 thread pool、connection pool 或 queue quota 能防止單一路徑吃滿全部。

優先級

安全與控制路徑需要保留容量，不應與 bulk work 競爭到最後一格。

降級模式

隔艙滿了時要回傳 degraded，而不是讓請求無限等待。

觀測性

每個隔艙都需要水位、拒絕數、延遲與飽和時間。

切法保護風險
獨立 thread pool 慢任務不吃掉控制 worker 池太小會降低吞吐
連線配額 壞 downstream 不佔滿所有 sockets 需要依流量調整配額
記憶體區隔 低優先 buffer 不壓垮核心狀態 浪費未用保留空間

Psychology

共享池在正常時最迷人

早期負載小時，共享 pool 看起來簡潔又高效；直到某條路徑變慢，耦合才浮現。

效率偏誤

大家會看到平均利用率提高，卻低估故障時的競爭。

功能邊界幻覺

程式碼模組分開，不代表 runtime resource 也分開。

降級不敢設計

怕拒絕請求看起來壞，結果讓整個系統一起等死。

隔艙的 review 重點是問：哪一類失敗可以被允許局部發生？

Applications

雲端服務和 RTOS 任務都有隔艙問題

隔艙不是微服務專用；嵌入式系統裡的 task、queue、DMA buffer 與 heap 也需要邊界。

Cloud backend

把報表查詢、批次工作與交易 API 分開資源池，避免慢查詢拖垮交易。

RTOS firmware

讓 safety monitor、control loop 與 telemetry 分別有優先級和 queue 限額。

Connection pools

不同 downstream 設獨立配額，避免單一依賴故障耗盡所有連線。

  • 為每個隔艙設 saturation metric；沒有觀測性，隔離只是一張架構圖。
  • 用故障演練驗證：某隔艙滿載時，核心路徑是否仍在 deadline 內完成。

Misreadings

隔艙不是把所有東西切碎

切太細會降低利用率、增加配置成本，甚至讓排程更難理解。

服務拆開就等於隔艙 校正

如果共用同一個 DB pool、executor 或 rate limit，故障仍可傳播。

隔艙越多越安全 校正

過度切分會造成資源閒置與難以調參。

隔艙可替代 timeout 校正

隔離需要搭配 timeout、circuit breaker 與 fallback 才能完整控制失效。

Questions

哪個耗盡點不該被共享

如果某個 feature 卡住

它會吃掉哪些 thread、connection、memory 或 queue，是否影響安全或控制路徑？

如果隔艙滿了

系統會明確拒絕、降級、排隊，還是讓 caller 無限等待？

延伸路線

可與 circuit breaker、blast radius、backpressure 與 graceful degradation 一起看，形成完整失效控制策略。
