O 環理論

O-Ring Theory

O 環理論說明複雜任務的價值會被最弱環節拉低；多個高品質步驟相乘時，一個小缺陷也可能讓整體歸零。

Orientation

有些流程不是加總分數，而是一個環節破掉就全盤失效

O 環理論來自挑戰者號事故的隱喻：當任務由互補步驟組成，整體成功率等於每一環品質相乘。

Core Tension

在低容錯系統裡，局部便宜或局部普通不一定省錢，因為失敗會拖垮所有高品質投入。

01 分辨加法與乘法工作

有些工作可由平均品質彌補，有些工作被最弱環節決定。

02 找單點低品質

小零件、小流程或小權限錯誤可能造成巨大損失。

03 做品質匹配

高品質人員和流程需要搭配相同水準的相鄰環節。

Setup

整個 release 都準備好了，只差一個 migration 沒被驗證

最小場景是一個小步驟承載整體成功。

最小劇情

團隊完成新功能、測試和監控，但資料庫 migration 腳本未在真實資料量上跑過。上線後 migration 卡住，導致整個 release 回滾。

真正難點

被忽略的步驟看似只佔工作量一小部分，卻決定整體是否能成功交付。

多環節互補
→
一環品質偏低
→
整體成果被拉低
Key Point

O 環理論要求你找出乘法型環節，因為那些地方不能靠其他環節表現優秀來抵消。

Lenses

識別哪些環節沒有補償空間

品質管理要看失敗是否可局部隔離，還是會讓整體歸零。

互補程度

各步驟是否缺一不可，或可由其他步驟補救？

失敗擴散

單一缺陷會局部降級，還是造成整體失敗？

匹配需求

高技能人員是否被低品質工具或流程卡住？

Psychology

小環節容易被視為不值得高規格處理

工作量小不代表風險小；人常用投入比例錯估失敗影響。

比例錯覺

小零件、小腳本和小表單看起來不像大風險。

平均品質直覺

人習慣用平均表現評估團隊。

明星迷思

強者加入低品質流程後，產出仍可能被流程拖住。

對乘法型流程，review 最弱環節比稱讚最強環節更有價值。

Applications

把低容錯環節列成獨立風險

航太、醫療、軟體部署和資料管線都需要看品質乘法。

軟體發布

migration、rollback、權限和監控要和功能本身同級 review。

供應鏈

便宜零件若不可替代，可能讓高價組件一起報廢。

團隊配置

高能力團隊若缺少 QA 或維運支援，整體交付仍受限。

Misreadings

O 環理論不是說每個環節都要最高規格

它要求把高規格放在互補且不可補償的環節上。

到處加品質 校正

非關鍵環節過度工程會浪費。

只找人弱點 校正

工具、流程和介面也可能是 O 環。

忽略降級設計 校正

若能隔離失敗，乘法風險就可降低。

Questions

問哪個小環節能讓整件事失敗

哪些步驟失敗後無法由其他步驟補救？

把它們列成 O 環。

最弱環節是否得到與影響相稱的資源？

用失敗影響而非工作量分配注意力。
