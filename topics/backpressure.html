背壓

Backpressure

當上游產生資料的速度快過下游消化能力時，真正成熟的系統不會假裝一切都沒事，而會讓壓力往回傳，逼整條鏈路一起面對容量現實。

Read For

這一頁要把『先收下再說』和『系統有能力誠實告訴上游我吃不下了』拆開，因為沒有背壓的高吞吐常只是延後爆炸。

主題類型
流量控制 / 系統穩定
核心問題
Slow consumers need a way to signal producers to slow down

讀完產出

Orientation

吞得下不代表消化得了，真正的穩定要會把壓力往回傳

背壓的價值在於承認容量是系統共同事實，不是靠某一層偷偷堆 buffer 就能永遠遮住。

Core Tension

上游想追 throughput，下游受限於 CPU、IO、網路或人力；若沒有壓力回傳，瓶頸只會從一個 queue 移到另一個 queue，最後以更壞方式爆出來。

01 先找真正瓶頸

不是每次都要怪 producer 太快，有時是 consumer 設計、批次策略或外部依賴太慢。

02 再決定壓力如何回傳

可以減速、丟棄、拒絕、批次化，重點是不要假裝容量無限。

03 最後把隊列當緩衝而不是垃圾桶

buffer 能吸收突波，但若被拿來長期掩蓋失衡，遲早會把延遲和記憶體一起吃爆。

Setup

從一條上游很順、下游一直喘的處理鏈開始

背壓常在你以為系統只是『偶爾慢一下』時默默形成，直到某天 queue 長到所有人都看得見。

最小劇情

感測資料每秒進來 1,000 筆，但下游只處理得掉 600 筆；短期靠 buffer 撐住，長期延遲、記憶體與超時一起上升。

真正難點

如果上游永遠不知道下游已經滿了，整條鏈路就會靠最脆弱的一段默默承受全部壓力。

上游輸出快於下游消化
→
中間緩衝越堆越高
→
若無背壓終將以延遲或崩潰顯形
Key Point

背壓不是讓系統變慢，而是避免它靠自欺的速度把自己推向更難收拾的失控。

Lenses

四個看容量現實的角度

不是每條資料流都需要同樣策略，但每條資料流都需要知道滿了以後怎麼辦。

突波還是常態

buffer 是拿來吸短暫尖峰，還是正在掩蓋長期失衡？

可丟棄性

資料是否允許採樣、降精度或淘汰較舊項目？

回傳機制

producer 能否知道 consumer 已滿，並主動放慢或暫停？

觀測能力

隊列深度、處理延遲與丟棄率是否被看見，而不是只看入口吞吐？

策略 好處 代價或前提
加 buffer 短期能吸收尖峰 長期失衡時只是在延後爆炸
回傳減速 讓整條鏈路一起面對容量 需要協定或介面支援
丟棄或採樣 保住系統整體穩定 必須接受不是每筆資料都能完整保留

Psychology

為什麼大家容易先堆 queue 再說

因為 buffer 讓問題暫時從畫面上消失，看起來比當場拒絕或降速更像『系統還有在扛』。

吞吐數字很好看

入口還能吃進資料時，團隊容易誤以為整體都健康。

延遲比錯誤更隱形

慢慢堆積的問題不如立即拒絕那麼刺眼。

拒絕聽起來像退步

其實很多時候可預期地拒絕，比假裝接住最後全部爆掉更成熟。

背壓的價值不在優雅術語，而在於它迫使系統對容量誠實。

Applications

把容量誠實帶回服務、裝置與資料流

對串流處理、事件系統與裝置上報來說，背壓常是穩定性與吞吐討論中最容易被晚點才看見的那層。

日誌與遙測

若上報永不節流，下游儲存或解析跟不上時，最後常在最不想丟資料的時刻一起炸。

事件工作流

worker 數量有限卻沒有節流時，佇列會把故障訊號藏成延遲。

UI 串流

畫面更新頻率若不考慮使用者裝置承受度，再即時也會把互動卡死。

  • 看到 queue 增長時，不只要擴容，也要問上游為何有能力一直不知道下游已滿。
  • 背壓策略本身也是產品決策：要慢、要拒、要丟，哪個傷害比較可接受？

Misreadings

三個常見誤讀

背壓不是讓系統保守，而是讓系統不靠幻覺維持表面吞吐。

多開 queue 就能解決 校正

queue 只能搬移壓力，不能消除長期產消失衡。

背壓代表系統變慢 校正

沒有背壓時，系統常只是先看起來快，後面整體更慢更亂。

只有 reactive framework 才談背壓 校正

任何 producer-consumer 鏈路都會遇到這個容量問題。

Questions
