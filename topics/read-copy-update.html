Read-Copy-Update (RCU)

RCU 讓讀者幾乎不用停下來，但更新者必須把替換、可見性與舊物件回收分成不同階段處理。

Orientation

讀者可以快，舊資料不能早死

RCU 把共享資料的更新拆成發布新版本與延後回收舊版本；讀者在短臨界區內拿到的指標必須保持可用。

Core Tension

傳統鎖讓更新者等讀者，RCU 則讓讀者少付同步成本；省下的讀取成本會轉移到更新者的排序、版本替換與 grace period 管理。

01 先保護讀者窗口

讀者進入臨界區後，所見物件在離開前不得被回收。

02 再發布完整版本

更新者要先準備新資料，再用正確的 pointer publish 讓讀者看見。

03 最後等過渡期

回收舊物件前，要確認替換當時已存在的讀者都結束了。

Setup

查表正在跑，控制面想換表

資料面每個封包都查一次表；控制面偶爾更新規則。讓封包路徑拿重鎖會很貴，但直接釋放舊表會炸到正在讀的人。

最小場景

CPU 0 進入 RCU read-side section 後取得舊 route pointer；CPU 1 建好新表並交換全域 pointer。

真正難點

CPU 1 看不到 CPU 0 還會不會 dereference 舊節點，所以不能只因為全域 pointer 已更新就 free。

Key Point

RCU 的安全感來自延後破壞舊世界，而不是讓所有人同步停下來。

Lenses

四個必須一起看的接口

RCU 不是單一 API 名稱，而是一組時間與空間約束；少看任何一段都可能變成 use-after-free。

Read-side section

讀者用極短窗口宣告自己可能持有 RCU 保護的指標。

Pointer publication

新物件的欄位要先初始化完成，才讓其他 CPU 透過指標看到它。

Grace period

它回答的是替換前已開始的讀者是否都離開，而不是所有未來讀者。

Reclamation pressure

更新太快會堆積待回收物件，記憶體壓力可能轉成另一種可靠性問題。

檢查點要確認什麼常見失誤
讀者範圍 哪些路徑會 dereference 指標 忘記中斷、timer 或 workqueue 也可能讀
發布順序 新資料在 pointer 可見前已完整 普通 assignment 遮掉 memory ordering 問題
回收時機 舊讀者已全部離開 交換 pointer 後立刻 kfree

Psychology

沒有鎖的讀者看起來像沒有責任

RCU 的讀者路徑太輕，容易讓 review 只盯著更新程式碼，而忽略讀者拿到的指標仍需要生命週期保護。

Hot path 崇拜

大家喜歡零等待的讀取，但很少同步檢查待回收佇列是否堆高。

測試不容易撞中

use-after-free 常要在特定 CPU interleaving 下才出現，單元測試可能長期全綠。

名字造成錯覺

Read-copy-update 聽起來像更新策略，實際上還包含讀者宣告與回收協議。

評審 RCU 程式碼時，請把讀者、更新者、回收者當成三個角色分別走一次。

Applications

讀多寫少的共享狀態最常受益

RCU 很適合資料讀取極頻繁、更新相對少、且讀者可以容忍新舊版本短暫共存的地方。

Kernel lookup

routing、dentry、policy table 這類 lookup path 可避免每次讀取都撞全域鎖。

Driver state snapshot

狀態表更新不頻繁，但 IRQ 或 poll path 需要低延遲讀取。

配置快取

韌體可用類似世代替換思路發布新配置，舊配置等所有使用者離開後再回收。

  • 若讀者可能睡眠、持有指標跨臨界區，先檢查 RCU flavor 是否匹配。
  • 把待回收數量、grace period 延遲與更新節流納入壓測，不只看讀取吞吐。

Misreadings

RCU 不是把鎖拿掉就結束

它提供的是一種特定生命週期契約；違反契約時，錯誤通常比普通鎖競爭更難查。

等同 reader-writer lock 校正

reader-writer lock 排斥更新與讀取；RCU 允許讀者看舊版本，語意不同。

會自動回收所有東西 校正

RCU 告訴你何時能回收，不會替你修好外部引用或錯誤 ownership。

只適合 kernel 校正

核心是版本發布與延後回收；使用者態或韌體也可借用概念，但要重建等價保證。

Questions

舊指標何時真正沒有人碰

如果更新者現在釋放舊物件

哪些 CPU、ISR、callback 或讀者還可能持有它？證據來自鎖、臨界區，還是只是時間推測？

如果更新暴增

待回收物件會堆在哪裡，是否有節流、批次回收或記憶體壓力警訊？
