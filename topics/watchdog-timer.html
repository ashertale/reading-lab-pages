Watchdog Timer

看門狗計時器不是定時重開機；它要求軟體持續證明自己仍健康，否則把系統拉回可恢復狀態。

Orientation

餵狗應該代表系統真的還活著

Watchdog timer 在軟體未能於期限內重置計時器時觸發恢復動作，常見是 reset 或進入安全狀態。

Core Tension

如果餵狗動作太便宜，壞掉的系統也能繼續餵；如果 timeout 太激進，正常尖峰又會被誤判成死機。

01 先定健康條件

關鍵 task、queue 水位、sensor 更新與 control loop deadline 都可能要納入。

02 再設定時間窗

timeout 要長過合理 worst-case，又短到能在傷害前恢復。

03 最後保存線索

reset 前若可行，留下 reset reason、PC、task state 或 crash counter。

Setup

主迴圈還在跑，通訊已經死了

裝置遠端失聯，現場無法手動重啟。Log 顯示 main loop 還在固定餵狗，但 network task 已卡在 mutex。

最小場景

餵狗放在 timer callback，無論 app 狀態如何都會執行；watchdog 永遠看不見 partial failure。

真正難點

系統健康不是單一 PC 有在前進，而是關鍵功能仍能在 deadline 內完成。

軟體必須證明健康
→
逾時表示證明失敗
→
硬體或 supervisor 觸發恢復
Key Point

Watchdog 最重要的設計是什麼時候不餵。

Lenses

看門狗設計有四個門檻

把 watchdog 加進專案很容易；讓它正確代表健康比較難。

餵狗資格

只有所有關鍵檢查通過才餵，避免單一 alive loop 掩蓋 partial hang。

Timeout 長度

太短造成誤 reset，太長讓現場失效時間過久。

分級恢復

可先關閉輸出、保存診斷、重啟子系統，再由硬體 reset 接手。

窗口式 watchdog

某些設計要求不能太早也不能太晚餵，能抓住 runaway loop。

壞味道問題改善
ISR 固定餵狗 主程式死掉仍被掩蓋 由健康檢查集中授權餵狗
timeout 靠猜 誤 reset 或恢復太慢 用 deadline 與 worst-case measurement 設定
reset 後無紀錄 field failure 無法分析 保存 reset reason 與最小 crash context

Psychology

Reset 很容易被當成萬靈丹

看門狗能提高恢復機率，也可能讓真正的 bug 長期被 reset 遮住。

症狀消失

系統重開後恢復，讓團隊暫時失去追根究柢的壓力。

誤把存活當健康

某個 loop 還在跑，不代表通訊、控制、儲存都正常。

害怕不餵

工程師常怕誤 reset，於是把餵狗條件放得太寬。

好的 watchdog 會同時提高可恢復性與可診斷性，而不是只讓 bug 更安靜。

Applications

遠端設備尤其需要可診斷的恢復

無法人工觸及的設備需要自救；但自救必須保留足夠資訊，否則 field issue 會反覆重演。

RTOS task monitor

每個關鍵 task 更新 heartbeat，supervisor 確認全部在期限內前進才餵硬體 WDT。

安全輸出

timeout 後先關閉馬達、heater 或高壓輸出，再做 reset 或進入故障模式。

Boot loop 防護

連續 watchdog reset 超過門檻時進入 recovery image 或保守配置。

  • 設計餵狗 API 時，讓呼叫者提交健康證據，而不是任何地方都能直接 kick。
  • 在 release test 裡刻意卡住 task、mutex、ISR 與 queue，確認 watchdog 能抓到預期故障。

Misreadings

它不是定時重開機器

Watchdog 是最後防線，不是替代正確錯誤處理與 root cause 分析。

每隔一段時間 reset 比較安全 校正

週期性 reset 會掩蓋 memory leak 或 race condition，還可能中斷正常任務。

有 watchdog 就不怕 deadlock 校正

如果 deadlock 後仍有某段程式餵狗，watchdog 完全看不見。

timeout 越短越好 校正

太短會把正常高負載或 flash erase 誤判成故障。

Questions

誰有資格餵狗

如果只剩 main loop 在跑

哪些 heartbeat、queue drain、sensor update 或 control deadline 會阻止它餵狗？

如果 watchdog reset 發生

下一次開機能否知道前一刻卡在哪個 task、哪個階段、哪個 reset reason？
