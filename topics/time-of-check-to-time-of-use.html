檢查到使用時間差

Time-of-Check to Time-of-Use

TOCTOU 的危險在於檢查通過的那一刻和真正使用資源的那一刻，中間世界已經可能變了。

Orientation

檢查結果有保存期限，使用時可能已經不是同一個世界

檢查到使用時間差描述的是一種競態：你先驗證條件，再根據驗證結果行動，但條件在行動前被改變。

Core Tension

先檢查再使用是直覺流程，但在可變狀態上，分開的兩步會留下被插入或替換的時間窗口。

01 縮短或消除窗口

讓檢查與使用在同一原子操作、同一鎖或同一 transaction 裡完成。

02 使用穩定引用

用 file descriptor、handle、版本號或 capability 指向已確認對象，而不是重新查路徑或名稱。

03 使用前再驗證

若無法原子化，就在真正使用前重新確認權限、版本與狀態。

Setup

檢查的是安全檔案，打開時路徑已被換成符號連結

最小場景是被檢查的對象和被使用的對象看似同名，實際已在中間被替換。

最小劇情

程式先檢查 /tmp/job.out 屬於目前使用者，稍後再用相同路徑寫入。攻擊者在兩步之間把路徑替換成指向敏感檔案的 symlink。

真正難點

程式做了檢查，但檢查結果綁定的是過去那個狀態，不是使用時的實際對象。

先驗證狀態
→
中間狀態被改變
→
根據過期結果執行操作
Key Point

TOCTOU 的修法，是讓檢查和使用共享同一個不可被替換的對象或狀態。

Lenses

找出檢查和使用之間的可變窗口

任何 check-then-use、authorize-then-act 或 read-then-write 都要問中間能否被改變。

對象身份

使用時是否仍是當初檢查的同一個檔案、row、裝置或 session？

狀態版本

檢查後是否用版本號、etag、sequence 或 lock 保證狀態未變？

攻擊者能力

誰能在窗口內改 path、permission、link、buffer 或硬體狀態？

Psychology

人容易把剛檢查過的事當成仍然成立

順序閱讀程式時，檢查和使用緊貼在一起；實際執行時，中間可能足夠讓另一方插入。

程式碼鄰近造成錯覺

兩行程式相鄰不代表執行上不可打斷。

安全檢查給人完成感

看到 access check 很容易放心，卻沒有追問結果是否綁定到使用對象。

競態被歸成偶發

TOCTOU 常需要特定時序，因此難重現時容易被低估。

檢查安全條件時，要同時設計檢查結果如何一路帶到使用點。

Applications

讓安全判斷和操作共享同一個承諾

檔案系統、權限 API、硬體控制和分散式資料都可能有 TOCTOU。

檔案操作

檢查後用已取得的 descriptor 操作，避免重新解析可被替換的路徑。

權限控制

執行敏感操作時重新驗證 capability 或版本，避免先授權後權限已變。

硬體狀態

讀取 ready flag 後若會等待，使用前再確認狀態或鎖住對應資源。

Misreadings

TOCTOU 不是只屬於檔案 symlink 攻擊

凡是檢查結果和使用行為分離，而且狀態可變，都可能有同類問題。

以為檢查越多越安全 校正

多次檢查若都不綁定使用對象，仍可能過期。

只看惡意攻擊 校正

正常併發更新也能讓檢查結果失效。

忽略硬體和韌體 校正

暫存器狀態、DMA buffer 和 sensor ready bit 也可能在檢查後改變。

Questions

問檢查結果如何活到使用點

下面兩問適合用在安全 review、driver code 和資料更新流程。

檢查和使用中間誰能改變狀態？

列出可插入的執行緒、使用者、服務、硬體事件或中斷。

使用時是否仍指向同一個對象？

用 handle、版本、lock 或 transaction 證明不是重新查到另一個東西。

延伸路線

可接著讀 race condition，理解結果如何依賴事件交錯。
