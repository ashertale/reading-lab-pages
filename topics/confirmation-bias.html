Orientation

先抓住這題真正的壓力點

確認偏誤會讓人以為自己正在客觀查證，其實只是把查證流程變成既有答案的辯護律師。

Core Tension

一個好假設需要證據支持，但如果你只尋找支持它的訊號，越努力 debug 反而越可能把錯誤包裝得更完整。

01 先找會推翻它的訊號

不要只問「哪裡證明我對」，先問「如果我錯，哪個觀測最早會露餡」。

02 分開觀測與解釋

log、scope 波形、測試結果是觀測；把它們歸因到某個 root cause 是下一層解釋。

03 讓替代假設同場競爭

至少保留兩個可被驗證的 competing hypotheses，避免整個流程只服務一個故事。

Setup

把問題縮到最小

最小情境是：你已經猜某個模組有問題，接下來每一筆訊號都開始被你讀成那個猜測的延伸。

最小劇情

某個 intermittent bug 看起來像 I2C timing，你開始只重跑 bus 相關測試，對 power sequence、race condition 和測試治具訊號變得不敏感。

真正難點

問題不是你有假設，而是你讓假設主導了證據搜尋。最後留下的不是完整證據，而是被篩選過的支持材料。

Key Point

確認偏誤的解法不是「不要有直覺」，而是把直覺變成可被反駁、可被替代、可被別人重跑的假設。

Lenses

四種容易偏掉的位置

確認偏誤不是單一動作，而是一串小傾斜：找資料、解讀資料、記住資料、最後回報資料都可能偏。

搜尋偏誤

只查會支持目前假設的 log、case 或文件，讓一開始的猜測取得不公平的樣本優勢。

解讀偏誤

同一筆模糊資料被解釋成支持自己；相反訊號則被當成 noise、測試誤差或例外。

記憶偏誤

事後回想時更容易記得命中假設的 case，忘掉那些其實曾經讓假設動搖的觀測。

回報偏誤

在 issue update 裡整理出一條太順的故事線，讓 reviewer 很難看見還沒被排除的 alternative。

Psychology

為什麼人會自然偏向確認

確認既有想法比較省力，也比較保護自尊；尤其在時間壓力、責任壓力或已有投入時更明顯。

認知省力

重新打開問題空間很耗能，大腦會偏好沿著已經建立的故事繼續走。

責任防衛

當某個假設和自己先前判斷綁在一起，反例就不只是資料，而像是對判斷能力的威脅。

團隊同步壓力

一旦會議裡大家都接受某個方向，提出反例的人會感覺自己在拖慢進度。

這裡的心理連結是教學性整理：重點是幫工程決策設計抗偏誤流程，不是把每個判斷錯誤都歸咎於個人固執。

Applications

把它帶回工程現場

確認偏誤在 firmware debug、code review、規格判讀與量產問題分析都很常見，因為這些工作都需要在不完整訊號中快速建模。

Firmware debug

先寫下會推翻目前 root cause 的觀測條件，例如改 sample rate、換板、換治具、換初始化順序後結果應該怎麼變。

Code review

reviewer 不只看 patch 是否符合作者說法，也要問哪些路徑沒有被測到、哪些假設只靠口頭推論。

量產異常

不要只蒐集支持單一供應商或單一批次的資料；要同時檢查環境、測試站、操作流程與韌體版本交互作用。

  • 把 issue note 分成 Observations、Hypotheses、Disconfirming Tests 三欄，降低證據和故事混在一起的機率。
  • 每次 status update 至少保留一個替代假設，直到有明確測試把它淘汰。

Misreadings

三個常見誤讀

這個概念常被拿來指責別人「你有偏見」，但真正有用的讀法是改善驗證流程。

所以直覺都不可信 校正

不是。直覺很適合產生假設；問題是不能讓產生假設的直覺，同時壟斷驗證假設的流程。

只要資料夠多就不會偏 校正

資料很多仍可能是偏樣本。若搜尋策略偏了，更多資料只是更厚的同溫層。

這只是不夠客觀的人才會有 校正

不是。確認偏誤通常是正常認知捷徑在高壓問題裡的副作用，所以要靠流程補強。

Questions

把它變成你的 debug 檢查

下面兩個問題適合放在 root cause 分析與 code review 前。

哪一個測試最可能讓我現在的假設失敗

如果你想不到，代表假設還太模糊，或你正在保護它而不是檢驗它。

我是否能公平描述另一個 root cause

若替代假設被講得很弱，可能不是它真的弱，而是你沒有替它設計像樣的測試。

延伸路線

接著可以順讀 availability heuristic、anchoring、belief perseverance，或把它接回工程裡的 hypothesis-driven debugging。
