Knowledge Study / teams / knowledge / resilience

巴士因子

Bus Factor / Truck Factor

一個系統若只有極少數人真正知道怎麼維護、部署或修它，那它的最大風險常不是技術本身，而是知識過度集中。

Read For

這一頁要把『有高手扛住』和『系統真的可持續』拆開，因為很多團隊的穩定感，其實只是建立在少數人還沒離開。

主題類型
團隊風險 / 知識集中
核心問題
How many people can disappear before the project stalls
主要判準
知識集中度、可替代性、文件可用性、恢復時間
讀完產出
能把知識風險當成系統風險管理，而不只是人力安排問題

Orientation

專案能不能繼續，不只看 code 在不在，也看知道怎麼動它的人還在不在

巴士因子把一個常被情感化的問題變得可談：當關鍵知識只存在少數人腦中，整個系統的續航力就會被綁在他們身上。

Core Tension

專家集中能讓短期效率很高，但一旦維運、排錯與決策脈絡無法被他人接手，系統表面穩定其實只是在借人撐住。

01 先盤點真正不可替代的知識

不是每段 code 都一樣危險，最危險的是只有一兩個人知道的 build、bring-up、排錯與 release 路徑。

02 再看替代是否真的可用

文件存在不代表能接手；要看別人能否靠它完成真實任務。

03 最後把傳承做成日常機制

知識分享若只在出事後補救，通常已經太晚。

Setup

從一個大家都不想打擾的人開始

巴士因子低的團隊常有一種熟悉畫面：某幾個人永遠被問同一批問題，也永遠是最後的故障出口。

最小劇情

某位資深工程師最懂 boot sequence、燒錄流程與 field diagnostic；只要他休假，release 節奏就顯著變慢。

真正難點

真正集中的是脈絡，不只是知識點；別人也許看得懂 code，卻不知道為什麼當初要這樣設計。

關鍵知識集中在少數人
→
他們成為所有高風險路徑的瓶頸
→
人一離開或不可用就暴露系統脆弱
Key Point

巴士因子不是在問『誰最重要』，而是在問『系統有多少部分其實只靠個人記憶在維持』。

Lenses

四個看知識集中風險的角度

不必把所有知識平均化，但你至少要知道哪些區域一旦失人就會讓專案停擺。

路徑唯一性

某個關鍵流程是否只有一種做法，而且只有一個人知道？

脈絡可轉移性

別人能否不靠口耳相傳就理解設計理由、風險邊界與例外處理？

值班壓力

是否總是同樣幾個人被拉去解 incident、帶新人、做最後 release 判斷？

替代演練

團隊有沒有真的練過讓第二人、第三人完成整條關鍵鏈路？

徵兆 表面看起來像 其實在透露什麼
同一人永遠被 @ 他效率最高 知識分布失衡，系統在向單點故障靠攏
文件存在但沒人敢照做 大家比較謹慎 文件不是可執行知識，只是存放知識殘影
某人請假就延後 release 時程剛好不巧 關鍵判準與操作路徑沒有被團隊化

Psychology

為什麼知識集中常被誤當成效率

因為讓最懂的人一直做，短期最省時間；知識傳承看起來反而像在拖慢進度。

專家捷徑很誘人

讓高手直接處理問題，總比培養第二人當下更快。

教別人有顯性成本

pair、文件與演練會占掉可見工時，所以常被往後排。

穩定假象會麻痺風險感

只要關鍵人還在，團隊就容易低估自己其實站在很細的支柱上。

巴士因子不是責怪專家，而是提醒團隊不要把系統可靠性偷偷綁在人類記憶上。

Applications

把知識分散當成工程韌性設計

對 firmware、製造測試與平台維運來說，低巴士因子往往不是抽象風險，而是下一次 release 卡住的直接原因。

Bring-up 與燒錄流程

若只有少數人知道板子異常時該怎麼救，生產與 debug 都會被放大成排隊事件。

CI/CD 與 toolchain

看似自動化的系統若只有一人知道怎麼修 pipeline，仍是隱性單點。

模組 ownership

如果 component owner 只有名字，沒有可替代的操作與判準，ownership 就只是排程上的標籤。

  • 真正的去風險不是叫專家多寫文件，而是讓第二人能獨立走完高風險任務。
  • 每次 incident 與 release 都是檢查巴士因子的好時機：誰是最後必須出現的人？為什麼？

Misreadings

三個常見誤讀

巴士因子不是要求人人都知道一切，而是避免關鍵知識變成單點故障。

只要文件齊全就沒事 校正

文件若沒被他人實際用過，通常還不能算真正可接手。

專家多就是好事 校正

專家重要，但若只有專家能動，系統就還是脆弱。

這是人資問題不是工程問題 校正

知識分布直接決定維運韌性、排錯速度與變更風險。

Questions

如果這個人明天不在線，系統還能穩定往前嗎

下面兩個問題很適合放在 ownership 與 release 盤點裡。

哪三條關鍵路徑目前只有一人真正能獨立完成

那通常就是最先該做 shadowing 與演練的地方。

我說的『可替代』是看過、做過，還是真的能獨立交付

很多團隊高估的是知道流程，不是能執行流程。
