FLP 不可能性

FLP Impossibility

在完全非同步系統裡，只要允許一個程序可能 crash，確定性共識就無法同時保證安全與終止。

Read For

這頁適合用在 distributed lock、cluster election、firmware fleet coordination 與可靠性設計：它提醒你不可能性常來自模型，而不是工程師不夠努力。

Orientation

共識的困難在等待裡

FLP 不可能性說明，在完全非同步模型中，無法分辨一個節點只是慢，還是真的 crash；這讓確定性共識無法保證一定終止。

Core Tension

系統可以堅持安全而繼續等，也可以為了可用性做出猜測；但在 FLP 的模型裡，兩者不能被確定性演算法完整兼得。

01 先說清楚模型

是否有 bounded delay、failure detector、leader lease、quorum、隨機化或外部時鐘？

02 再分安全與活性

安全是不要決定錯；活性是最後要決定。FLP 逼你承認兩者在非同步故障下的張力。

03 最後檢查工程逃生口

實務系統通常加入 timeout、leader、隨機化或部分同步假設，這些都是設計選擇。

Setup

慢節點和死節點長得一樣

把情境縮到三個節點要選出同一個 firmware rollout decision；其中一個節點突然沒有回應。

最小場景

網路可能只是延遲，也可能該節點已 crash。若其他節點繼續等，系統可能永遠停住；若直接前進，可能和遲到訊息產生衝突。

真正難點

問題不是 timeout 數值設多少，而是完全非同步模型不允許你用時間準確區分慢與死。

Key Point

FLP 的力道在於揭露：你若想要實務可用，就必須明講自己加了哪個額外假設。

Lenses

讀共識保證時要拆的四個詞

很多分散式系統文件會說高可用、強一致、容錯；FLP 讓你把這些詞拆回模型條件。

非同步

訊息可能延遲任意久，演算法不能依賴固定時間上限。

Crash fault

節點可能停止回應；其他節點不能可靠知道它是否永遠停了。

Deterministic

沒有隨機選擇來打破對稱，也不能靠機率式終止保證。

Consensus

所有正確節點要決定同一值，且該值符合有效性條件。

加入的假設 買到什麼 留下什麼風險
timeout / partial synchrony 實務上能判斷超時 網路極端延遲時仍可能誤判。
quorum 用多數交集維持安全 少數分割區犧牲可用性。
隨機化 避開某些確定性僵局 保證通常變成機率式。

Psychology

人會把罕見延遲當成可以忽略

工程直覺常說『網路通常很快』，但定理討論的是保證；保證不能靠通常。

timeout 給人確定感

設定 5 秒 timeout 很像做了判斷，其實只是把故障偵測變成工程假設。

成功案例掩蓋模型

cluster 長期正常運作，容易讓人忘記它依賴的是部分同步與 failover policy。

一致性語言太簡略

強一致、高可用、容錯這些詞若沒有模型條件，很容易變成行銷式承諾。

FLP 的價值不是讓人悲觀，而是讓保證寫得誠實。

Applications

用在 leader election 與 rollout 協調

Firmware fleet、gateway cluster 或測試基礎設施只要需要多節點決定，就會碰到時間、故障與一致性的交換。

分散式鎖

若 lock service 依賴 lease，就要說清楚 clock drift、renewal 與 partition 時的行為。

批次 rollout

多個 controller 是否同時推同一批裝置，取決於 leader election 的故障模型與 fence 機制。

測試資源協調

runner pool 或硬體治具分配若跨節點同步，partition 時要選擇停住、降級或接受重複分配風險。

  • 讀任何共識保證時，先找 timeout、clock、quorum 與 failure detector 的假設。
  • 把 partition 行為寫成產品規格：哪邊繼續服務，哪邊停止，如何恢復。

Misreadings

不可能性不等於不能做系統

FLP 不否定 Paxos、Raft 或其他實務系統；它說的是在特定模型下不能同時保證某些性質。

所以共識演算法都沒用 校正

錯。實務演算法加入額外假設，並在那些假設下提供有用保證。

加 timeout 就推翻 FLP 校正

timeout 改變了模型；它不是推翻定理，而是接受部分同步或故障偵測假設。

只要網路夠好就安全 校正

平均好不代表保證成立；要寫清楚極端延遲與 partition 時的選擇。

Questions

找出共識保證背後的假設

如果一個節點沒有回應

系統如何區分慢、crash、partition 與 GC pause？這個判斷依賴哪些時間假設？

如果網路分割發生

你的設計優先保安全、可用性、還是人工介入？恢復時怎麼處理已做出的決定？
