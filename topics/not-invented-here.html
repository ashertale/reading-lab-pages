Not Invented Here

非我發明症候群讓團隊低估外部方案，只因它不是自己做的；問題不在自製本身，而在身份感壓過技術判斷。

分辨合理自建、整合風險和身份防衛，避免把外部成熟方案排除在評估之外。

Orientation

不是自己做的，不代表就不懂；自己做的，也不代表就最合適

NIH 的核心是偏好來源而非品質：團隊因為所有權、面子或控制感，對外部方案設定更嚴苛標準。

Core Tension

自建可能帶來差異化和控制；外部方案可能更成熟、更便宜。真正要比的是生命週期成本和風險，而不是血統。

01 先定義不可妥協需求

把安全、延遲、授權、可維護性等硬條件寫清。

02 同標準比較

自建和外部方案都要算開發、維運、升級和退出成本。

03 分辨控制幻覺

擁有原始碼不等於有能力長期維護。

Setup

外部 library 已解決 90% 問題，團隊仍想重寫一套

最小場景是一個自建提案沒有被要求承擔完整成本。

最小劇情

團隊需要 feature flag 系統。成熟開源方案支援審計、分群和 rollback；內部有人主張自建，理由是更符合我們需求，但尚未估算值班、UI、權限和 migration 成本。

真正難點

自建理由可能成立，但若只嚴格審查外部方案，評估就已經偏斜。

Key Point

非我發明症候群要你把來源偏好剝掉，讓方案在同一張成本與風險表上比較。

Lenses

把自製衝動拆成需求、風險和情緒

有些拒絕外部方案是專業判斷，有些只是團隊身份在說話。

硬需求

外部方案是否真的無法滿足核心限制？

維運負擔

自建後誰升級、修漏洞、寫文件和支援使用者？

身份訊號

拒絕理由是否含有我們比較懂、外面都不可靠這類泛化語句？

Psychology

自己做會帶來掌控感，也會遮住長期負擔

工程師喜歡理解和掌控系統，這是優點；但掌控感不等於總成本低。

所有權偏誤

自己設計的東西更容易被看見優點。

外部不信任

陌生 codebase 的風險被放大，內部未知數被低估。

能力展示

自建有時承載了證明團隊能力的需求。

build vs buy 文件要強制列出自建五年維護成本，否則 NIH 很難被看見。

Applications

用對稱評估取代來源偏好

套件採用、雲端服務和內部平台都適合用同一套比較框架。

開源採用

檢查社群活躍、授權、漏洞回應和退出計畫。

雲端服務

比較供應商鎖定與自建可靠性成本。

內部平台

平台團隊需證明自建能力帶來差異化，而不只是控制權。

Misreadings

NIH 不是說外部方案一定比較好

有些核心能力確實該自建；問題是拒絕外部方案時必須拿出對稱證據。

把採用等同偷懶 校正

利用成熟方案可讓團隊專注差異化。

忽略供應鏈風險 校正

外部依賴仍需治理，不是盲目引入。

把控制當免費 校正

控制權需要人力、知識和長期承諾支撐。

Questions

問拒絕外部方案的證據是否對稱

如果這是內部方案，我們會用同樣標準挑剔嗎？

檢查評估是否偏向自家。

自建的五年維運帳單是什麼？

把升級、漏洞和支援都算進去。

延伸路線

可接著讀 build vs buy，理解何時自建、採購或採用開源更合理。
