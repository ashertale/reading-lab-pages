困惑代理問題

Confused Deputy Problem

危險不一定來自權限不夠，而可能來自一個有權限的元件被引導去替別人做本來不該做的事；問題是代理被搞混了，不是單點權限看起來太大。

Read For

分清楚『誰有權』和『誰在替誰用權』之間的差別，避免把授權漏洞只看成單點權限大小問題。

Orientation

有權限不一定危險，替錯的人使用權限才危險

這個工程概念會一直被拿出來講，是因為 當高權限元件看不清自己是在替誰做事時，它就可能在合法權限內替不該被服務的對象完成敏感操作。

Core Tension

當高權限元件看不清自己是在替誰做事時，它就可能在合法權限內替不該被服務的對象完成敏感操作。

01 先分清誰是請求者、誰是執行者

只知道執行者有權還不夠，還要知道它此刻在替誰行事。

02 再讓授權跟著上下文走

若代理元件拿自己的權限直接處理外部請求，就很容易混掉真正應該檢查的主體。

03 最後縮小可被代理的能力

把高權限能力切細、顯式授權，通常比事後補黑名單更穩。

Setup

把問題放回一個小型系統

困惑代理問題不是一般的權限過大而已，它關鍵在於高權限者開始替錯的人使用那份權力。

最小劇情

一個有寫入系統檔案權限的工具接收外部給的路徑參數，結果被誘導去覆寫原本不該讓使用者碰的目標檔。

真正難點

難點在於表面上看，每個參與者都像在做自己被允許做的事，漏洞卻出在權限上下文被混掉了。

高權限元件接收外部請求
→
代理上下文沒有被清楚傳遞
→
合法權限被誤用到錯的目標上
Key Point

困惑代理問題的重點不是『權限太大』四個字，而是授權決策若沒把請求來源與目的綁清楚，就會讓代理替錯的人出力。

Lenses

幾條真正有用的工程判準

一旦分不清誰在發請求、誰在動手、權限又是替誰開的，代理漏洞就會冒出來。

代理上下文

系統是否有保留『這個操作是替誰做的』資訊，並用它決定授權。

權限繼承

高權限元件若直接用自己的全權服務外部請求，風險會顯著升高。

目標混淆

真正被操作的資源是否和請求者原本應被允許接觸的資源一致。

判準 當下看起來像什麼 真正代價
只看執行者權限 審查較快 最容易漏掉代理其實在替誰用這個權限
把代理上下文帶進授權 設計較細 卻較能阻止合法元件被借刀誤用
高權限大包服務 開發省事 一旦上下文混淆，爆點通常很深

Psychology

為什麼團隊容易在這裡失手

人很容易只看「這個服務本來就有權限」，卻忘了真正該問的是它現在在替誰用。

角色簡化

人容易把『有權限的服務』直接當成可信任的服務，忽略它也可能被利用。

合法外表安慰

只要操作看起來走的是正式工具鏈，就比較難想到權限其實被借道誤用。

上下文偷懶

明確傳遞代理上下文很麻煩，所以系統常偷懶用『反正這服務本來就能做』帶過。

Applications

放回真實系統後它會改變什麼

後端代辦、檔案工具和自動化流程最常在這裡出事，因為服務自己的高權限太容易蓋過請求者身分。

檔案工具

高權限工具必須把目標資源與請求來源綁清楚，不能只看自己本身有權。

後端服務

代辦操作若不帶使用者上下文，服務就容易拿自己的權限替別人做超出授權的事。

自動化流程

跨服務代辦時要分清誰在簽名、誰在批准、誰在真正承擔這次操作的權限。

  • 後端代辦: 把使用者上下文帶進授權，而不是只看服務自己有沒有權。
  • 檔案工具: 別讓高權限工具直接替外部參數打開任何目標。

Misreadings

別讓這條原則變成口號

若一開始就把 困惑代理問題 聽成「所以高權限服務都不該存在」，後面連「不是。問題是高權限服務不能在代理上下文混亂時直接拿自己全權辦事。」這層校正都很難真正進場。

所以高權限服務都不該存在 校正

不是。問題是高權限服務不能在代理上下文混亂時直接拿自己全權辦事。

只要做路徑白名單就夠了 校正

若沒把代理主體綁清楚，很多變體仍可能借其他路徑進場。

這只是老式系統漏洞 校正

任何會代辦、轉交、代簽或代操作的現代服務都可能踩到。

Questions

把它變成下一次重構前的追問

這個元件此刻是在替誰執行權限，而那個『誰』有沒有被明確帶到授權檢查裡

這通常是漏洞最早的切口。

如果把執行者自己的高權限拿掉，只剩請求者原本該有的能力，這個操作還做得成嗎

若做得成太多，代理設計多半有問題。
