Backward Compatibility

向後相容不是永遠不改，而是讓新版本在舊客戶端、舊資料與舊操作節奏仍存在時能安全演進。

Read For

理解相容層、版本協商、資料遷移和 deprecation 如何讓系統改變不斷線。

Orientation

新版本上線時，舊世界通常還沒下線

向後相容要求新系統仍能服務既有使用者、資料和整合方；它讓改變可逐步發生，而不是一次切斷。

Core Tension

相容保護使用者和生態系，也會留下舊分支、舊欄位和舊行為；沒有退出條件時，相容會變成永久負擔。

01 定義外部契約

先分清哪些行為被使用者或裝置依賴，哪些只是內部實作。

02 做可展開變更

新增欄位、雙讀雙寫、feature flag 和版本協商比直接替換安全。

03 寫清移除條件

Deprecation 要有觀測資料、通知節奏和最後移除日期。

Setup

新 API 改了欄位語意，舊 app 還在用昨天的假設

最小場景是服務端已升級，但客戶端、資料或文件仍停在舊契約。

最小劇情

後端把 status 從字串改成物件以支援更多狀態。最新版 app 正常解析，倉庫裡大量舊版裝置卻把回應視為格式錯誤而停止同步。

真正難點

開發團隊看見的是新 schema 比較好，現場看見的是舊客戶端不能同時升級。

契約被新版本改變
→
舊使用者仍按舊假設呼叫
→
相容缺口變成現場故障
Key Point

向後相容的核心是承認部署不是瞬間事件，而是一段新舊共存期。

Lenses

把版本演進視為時間線

相容策略要回答新增、共存、觀測、通知和移除各階段的責任。

契約面

欄位、錯誤碼、時序、預設值和 side effect 是否被外部依賴？

共存面

新舊版本是否會同時讀寫同一資料或連到同一服務？

退出面

如何知道舊版本已少到可移除，誰負責通知和處理例外？

Psychology

做新功能的人容易高估升級同步性

開發環境常全是最新版；現場卻有舊 app、離線裝置、第三方整合和手動操作流程。

新版視角太乾淨

測試矩陣若只包含最新版，就會看不見舊客戶端的長尾。

內部重構被誤當外部透明

團隊知道語意不變，但外部可能依賴了原本的格式和錯誤行為。

舊版負擔讓人想快切

維護相容很煩，因此容易把 deprecation 寫成願望而不是計畫。

相容設計要用部署現實寫，而不是用 release note 的理想時間寫。

Applications

讓新舊版本能共存一段時間

API、韌體協定和資料格式都需要相容窗口與可觀測退出條件。

API 演化

新增欄位保持 optional，避免改變既有欄位型別和語意，必要時用版本協商。

韌體協定

裝置端不能即時更新時，server 要支援舊 command set 到實際裝置存量下降。

資料遷移

用 expand-migrate-contract 流程先讓新舊程式都能讀，再逐步移除舊欄位。

Misreadings

向後相容不是永遠背著舊包袱

相容需要退出計畫；沒有 deprecation，系統會被舊假設綁死。

把相容當不能改 校正

可以改，但要讓舊使用者在遷移窗口內不被切斷。

只看語法相容 校正

格式沒變不代表語意沒變；預設值、錯誤碼和排序也可能是契約。

用公告取代觀測 校正

發信通知不代表舊版本已消失，移除前要看真實流量和裝置存量。

Questions

找出還活著的舊世界

哪些舊客戶端會和新版本共存？

列出 app、裝置、第三方整合、批次 job 和離線資料。

移除舊契約的觀測條件是什麼？

以流量、錯誤率、版本分布或裝置回報作為退出依據。

延伸路線

可接著讀 versioning，理解版本標記、契約變更和遷移策略。
