Race Condition

Orientation

邏輯正確不保證交錯後仍正確

競態條件發生在多個執行者讀寫同一狀態，而結果依賴誰先誰後；單一路徑測試常看不見它。

Core Tension

併發提高吞吐與反應速度，但只要共享狀態沒有被原子化或序列化，速度就會換來不可預期結果。

01 界定共享狀態

先找出哪些變數、檔案、row、裝置暫存器或外部資源被多方讀寫。

02 保護檢查加更新

Check-then-act、read-modify-write 和初始化流程常需要同一個原子邊界。

03 測試交錯而非路徑

用 stress、fuzz、delay injection 或 model checking 製造不同執行順序。

Setup

兩個執行緒都看見庫存還有一件，於是各自賣出一件

最小場景是檢查和更新分開，中間被另一個執行者插入。

最小劇情

服務 A 和服務 B 同時讀到庫存為 1。兩者都通過檢查，接著各自寫入扣減結果，最後系統接受了兩筆訂單。

真正難點

每個服務的單次邏輯都合理；錯在檢查與扣減沒有形成同一個不可分割動作。

多方讀到同一舊狀態
→
各自做出合法判斷
→
交錯寫入產生錯誤結果
Key Point

競態條件的校正，是把關鍵狀態轉移設計成單一原子承諾。

Lenses

把時間交錯當成輸入條件

若結果只在某種順序下正確，就要問這個順序由誰保證。

原子邊界

哪些讀取、判斷和寫入必須一起完成，不能被插入？

序列化點

系統是否有單一資料庫交易、鎖、compare-and-swap 或 message queue 承諾順序？

可重放性

log 是否記錄足夠時序資訊，讓偶發錯誤能被重建？

Psychology

人會按自己閱讀程式的順序想像執行順序

競態難抓，因為程式碼有固定排列，但執行時間線可能有許多交錯。

線性閱讀造成錯覺

看 code 時自然一行行讀，容易忘記別的執行者可在兩行之間插入。

偶發重現降低優先級

無法穩定重現的錯誤容易被歸類為環境問題。

局部 lock 給人安全感

保護了某個函式不代表保護了完整狀態轉移。

排查競態時，不只問這段 code 做什麼，也問它被誰在何時打斷。

Applications

讓共享狀態只有一個承諾順序

競態可能出現在執行緒、ISR、資料庫交易和分散式服務之間。

ISR/main loop

共享旗標或計數器要用 atomic、critical section 或 lock-free pattern 明確保護。

資料庫更新

庫存、餘額或配額應用交易或 conditional update，而不是先查再寫。

跨服務流程

對同一資源的狀態轉移用 idempotency key、版本號或單一 owner 序列化。

Misreadings

競態不是只有多執行緒才會有

任何多方讀寫同一狀態且結果依賴時序的地方，都可能有競態。

以為加一把鎖就夠 校正

鎖的範圍要覆蓋完整不變式，否則只是讓部分操作安全。

把偶發當硬體怪問題 校正

偶發性正是競態特徵之一，不能因為難重現就排除。

忽略跨程序和跨服務 校正

資料庫 row、檔案、queue 和遠端 API 也都可能形成共享狀態。

Questions

找出誰能插在兩步之間

下面兩問適合用在併發 review 和偶發 bug debug。

哪個狀態被多方讀寫？

先列共享變數、資料列、檔案、裝置暫存器與外部資源。

檢查和更新是否不可分割？

若中間能被另一方插入，就需要交易、鎖、CAS 或序列化隊列。
