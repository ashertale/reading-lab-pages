Priority Ceiling Protocol

優先權天花板協議不是等高優先權任務被擋住才補救，而是預先用資源天花板限制誰能進入臨界區。

Orientation

把最壞阻塞限制在設計裡

Priority ceiling protocol 為每個共享資源指定 ceiling，並在任務進入臨界區時調整或限制排程，避免無界反轉與錯誤巢狀。

Core Tension

互斥鎖保護資料，卻可能破壞 deadline；即時系統不能只問資料有沒有 race，也要問阻塞多久。

01 先列資源使用者

每個 mutex、bus、driver queue 都要知道哪些任務可能進入。

02 再計算 ceiling

資源 ceiling 等於可能鎖住它的最高任務優先權。

03 最後做阻塞分析

schedulability 要納入低優先權臨界區造成的 bounded blocking。

Setup

低優先權任務拿著 I2C，控制迴路在等

Logger 正在讀 EEPROM 持有 I2C lock；control task 醒來要讀 sensor，卻被 lock 擋住，中優先權通訊任務又搶走 CPU。

最小場景

priority inheritance 會在高優先權任務被擋住後提升 logger；ceiling protocol 則更早限制可能造成問題的進入。

真正難點

功能上所有 task 都正確，排程交錯卻讓 deadline miss。

Key Point

即時鎖的正確性包含時間界線，不只是 mutual exclusion。

Lenses

從資源天花板看可排程性

Priority ceiling 的不同變體介入時機不同；共同目標是讓阻塞可分析、避免 deadlock。

Resource ceiling

一個資源的 ceiling 由會使用它的最高優先權任務決定。

Original protocol

任務只有在被高優先權任務阻塞時才提升持鎖者 priority。

Immediate protocol

任務一取得資源就提升到該資源 ceiling，實作較直接。

Nested critical sections

錯誤鎖順序會造成 deadlock；ceiling rule 可提前擋下危險進入。

協議介入時機適合注意
Priority inheritance 高優先權已經被阻塞後 可緩解反轉，但 deadlock/阻塞分析仍要小心
OCPP 鎖請求與阻塞關係形成時 阻塞行為可分析，實作較複雜
ICPP 任務拿鎖的瞬間 較易實作，但 priority movement 更多

Psychology

測不到 deadline miss 不代表排程安全

即時錯誤常需要特定任務釋放時間、持鎖長度與中斷時機；平常跑得順只代表沒碰到最壞交錯。

互斥成功的安慰

資料沒壞不代表 deadline 沒被低優先權臨界區吃掉。

中優先權任務被忽略

priority inversion 的麻煩常是第三方任務延長持鎖者釋放時間。

鎖範圍膨脹

為了方便把 I2C transaction、等待與資料處理包在同一把鎖裡，阻塞時間就失控。

RTOS code review 應把每把鎖的 worst-case hold time 寫在任務時間預算旁。

Applications

共享匯流排和驅動鎖最需要天花板視角

韌體常有多個任務共用同一條低速 bus 或 driver state；這些資源會把資料同步問題變成排程問題。

I2C sensor bus

控制任務、診斷任務與 logger 共享 bus 時，ceiling 可限制低優先權長交易的影響。

SPI flash driver

NVM 寫入或 erase 很長，應拆出非阻塞流程或明確禁止高優先權路徑等待。

Motor control RTOS

控制迴路資源要避免被 telemetry 或 shell command 的鎖持有時間拖累。

  • 把鎖內等待外設完成的時間列為 blocking，不要只算 CPU 指令時間。
  • 若資源會被 ISR、task 與 deferred work 共用，分別標出它們是否參與同一套 ceiling 規則。

Misreadings

天花板不是把系統全部墊高

它要求精準知道資源使用集合；粗暴提升所有鎖，只會製造延遲與假安全感。

等同 priority inheritance 校正

inheritance 多半在阻塞發生後補救；ceiling 在進入臨界區時限制風險。

用了就不用鎖順序 校正

協議能降低 deadlock 風險，但資源模型與鎖範圍仍要被設計清楚。

只跟作業系統有關 校正

若 driver 或 HAL 把長 transaction 包成黑盒，ceiling 分析仍會失真。

Questions

最壞情況下，高優先權任務被誰擋多久

如果高優先權任務醒來

它可能被哪些低優先權臨界區擋住，每段最長 hold time 是多少？

如果鎖內會等硬體

等待時間是否可界定，timeout、DMA completion 或 bus error 會不會放大阻塞？

延伸路線

建議接 priority inversion、rate monotonic analysis、mutex protocol 與 watchdog timing，一起看即時可靠性。
