不注意盲視

Inattentional Blindness

注意力像窄頻通道：當任務把它綁住時，眼前再明顯的訊號也可能沒有進入意識。

Read For

分辨「看過畫面」和「注意到訊號」的差異，別把漏看誤當粗心。

主題類型
注意力 / 知覺限制

Orientation

人不是攝影機；我們看見的是任務要求我們處理的那一部分

不注意盲視說明，資訊出現在視野裡不代表它被處理；注意力的任務設定會決定什麼有機會被看見。

Core Tension

專注能讓人完成困難任務，也會讓任務外的訊號失去入口。系統若只靠人順便注意，就等於沒有防線。

01 任務會設定視野

人被要求數數、比對或追蹤時，注意力會服務那個任務。

02 未預期訊號最容易漏掉

告警、異常或錯字若不在檢查目標裡，就算很明顯也可能被跳過。

03 防線要外部化

把關鍵訊號交給 checklist、告警規則或雙人分工，而不是期待人忙中自動看見。

Setup

一個 reviewer 可以逐行讀過 diff，卻仍漏掉整個風險類型

最小場景是檢查者很專注，只是專注在錯的問題上。

最小劇情

Reviewer 被要求確認命名、格式與測試是否更新。他仔細看完整個 patch，卻沒有注意到新增路徑繞過了既有權限檢查。

真正難點

他不是沒看，也不是不用心；他的注意任務沒有把權限邊界列為目標。

Key Point

不注意盲視的校正不是叫人更努力看，而是改變他被要求看什麼。

Lenses

檢查漏看時，先問注意力被派去做什麼

同一張畫面可以支援很多任務；任務不同，看見的世界也不同。

任務指令

明確寫下檢查目標，避免把安全、隱私或相容性留給順便發現。

訊號形式

關鍵訊號若和背景長得太像，就算在畫面中也容易失去注意入口。

冗餘觀察

讓不同人或不同工具看不同風險，不要讓一個專注任務承擔所有偵測。

Psychology

注意力節省成本，也會製造盲點

人若同時處理所有訊號，很快就會過載；因此大腦會選擇性處理，但選擇本身有代價。

專注會排他

把資源集中在一件事上，其他通道就變暗。這是能力限制，不是單純意志問題。

預期塑造偵測

人比較容易看見自己預期會出現的東西；未預期事件需要更強提示才會進入意識。

事後明顯不代表當下明顯

回放畫面時，知道答案的人會覺得異常很醒目；當下操作者沒有這個優勢。

把不注意盲視放進流程設計，會讓檢查從責怪個人轉向安排注意資源。

Applications

不要讓關鍵風險藏在『順便看一下』裡

工程流程裡，最危險的漏看通常不是畫面空白，而是訊號沒有被分配到責任通道。

Code review

把 reviewer 分工成行為、測試、安全與相容性，而不是所有人都泛泛看完整體。

監控告警

告警要對應明確 action。若只是多一個 dashboard 數字，人在 incident 壓力下很可能看不到。

操作流程

高壓操作要用 checklist 和 pre-flight gate 承接注意力，而不是靠操作者記得所有例外。

Misreadings

漏看不等於眼睛沒看到

這個概念提醒我們檢查注意力分配，而不是用『怎麼會看不到』結案。

把漏看都歸咎於粗心 校正

粗心有時存在，但若多人在同任務下都漏同類訊號，流程才是主要線索。

以為多訓練就能解決 校正

訓練有用，但未預期訊號仍需要工具、提示與分工承接。

把畫面存在當成資訊已傳達 校正

資訊出現在 UI 或文件裡，不代表人在當下任務中會處理它。

Questions

把注意力當作有限資源來設計

這個風險被誰明確負責看？

如果答案是大家都會看到，它很可能其實沒有人負責。

當下任務會讓哪類訊號變暗？

操作者專注在速度、格式或數字時，哪些異常需要額外提示？

延伸路線

可接著讀 change blindness，對照人如何漏掉已發生的變化。
