事前驗屍

Premortem

事前驗屍把團隊帶到未來失敗現場：假設計畫已經慘敗，再回頭找今天不敢說、沒看見或被低估的原因。

Orientation

與其問會不會失敗，不如先假設已經失敗

事前驗屍的技巧在於解除樂觀敘事：當失敗已被設定為事實，成員更容易提出平時怕掃興的風險。

Core Tension

計畫啟動時需要士氣和承諾；但過早一致會讓真實疑慮被壓到太晚才爆出。

01 先設定失敗事實

不是問可能失敗嗎，而是假設一年後確定失敗。

02 讓每個人獨立寫原因

避免第一個發言者框住大家。

03 轉成具體防護

風險清單要接到 owner、trigger 和 mitigation。

Setup

release 前十分鐘的焦慮，應該提前兩週被請上桌

最小場景是一個團隊在上線前做反向演練。

最小劇情

新支付流程準備上線。團隊假設三週後上線失敗、退款暴增、客服爆量；每個人寫下最可能原因，最後發現 rollback 腳本、灰度門檻和客服話術都未完成。

真正難點

若正常問風險，大家可能只列已知項目；假設失敗後，隱性焦慮更容易浮現。

Key Point

事前驗屍把悲觀變成一個短時間、可操作的設計工具。

Lenses

讓沉默的風險取得發言權

好 premortem 不只是列風險，而是降低說出風險的社交成本。

獨立輸入

先匿名或個別收集，再合併討論。

失敗路徑

每個原因要能連到具體事件序列。

觸發條件

什麼訊號出現時要啟動備案？

Psychology

正向計畫會讓異議看起來像不支持

事前驗屍提供一個允許懷疑的劇本，讓成員不用冒著破壞氣氛的風險。

群體和諧

大家不想在啟動會上扮演唱反調的人。

計畫謬誤

團隊低估整合時間和外部依賴。

責任模糊

風險若沒有 owner，就會留在大家都知道的狀態。

premortem 的結尾不能是這些都要注意，而是每個高風險項都有下一步。

Applications

把失敗想像接到工程控制點

專案、上線和策略決策都適合在承諾前做短版 premortem。

Release readiness

把假想事故轉成 rollback、alert、runbook 和客服準備。

專案規劃

把依賴延遲、需求變更和人力缺口轉成 trigger。

策略下注

列出市場、競品和內部執行失敗的早期訊號。

Misreadings

事前驗屍不是開悲觀大會

它把悲觀限制在明確流程內，目標是改計畫，而不是削弱信心。

停在風險清單 校正

沒有 owner 和 mitigation 的清單只是焦慮存檔。

讓主管先說 校正

權威先發言會壓扁獨立判斷。

只找大災難 校正

很多失敗來自中等風險疊加。

Questions

問未來失敗的新聞標題會怎麼寫

下面兩問適合用在專案 kickoff 和上線審查。

假設已經失敗，最可信的三條原因鏈是什麼？

用過去式寫，避免空泛可能性。

哪些行動今天就能降低這些原因？

把風險轉成 owner、trigger 和日期。
