能力式安全

Capability-Based Security

能力式安全把權限附在不可偽造的參照上；能碰什麼，不靠全域身份臨時判斷，而靠你實際持有哪個 capability。

Orientation

權限跟著 handle 走，而不是跟著全域名字走

Capability 通常是一個不可偽造的參照，既指向物件，也攜帶允許的操作集合。

Core Tension

傳路徑或名字很方便，但接收者用誰的權限解析名字，常在系統變複雜後變得不清楚。

01 先移除全域查找

被隔離的元件不應任意從全域 namespace 找資源。

02 再傳遞已授權參照

caller 把自己願意授權的 handle 交給 callee。

03 最後衰減權限

只需要讀取就傳 read-only capability，不給完整控制權。

Setup

Helper 以自己的權限替 plugin 開檔

Plugin 沒有敏感檔案權限，卻能要求 helper 處理一個路徑。若 helper 不檢查授權主體，就成了 confused deputy。

最小場景

plugin 傳入 /secure/calibration.key，helper 用 service account 成功 open，然後把內容送回 plugin。

真正難點

helper 的行為看似只是服務請求，但 authority 其實被 plugin 借走了。

caller 持有 capability
→
委派有限 handle
→
callee 只能在授權範圍內操作
Key Point

能力式設計讓授權從『請你幫我找這個名字』改成『我把這個已授權物件交給你』。

Lenses

能力要不可偽造，也要能縮小

Capability 的安全性不只在 handle 形式，還在系統能否阻止偽造、限制傳遞並支持權限衰減。

Unforgeability

程式不能靠猜數字或組字串取得 capability；必須由系統或持有人授予。

Delegation

持有人可以把 capability 交給另一個元件，形成明確授權鏈。

Attenuation

權限可以被包成較小集合，例如 read-only、限速或一次性。

Revocation

撤銷通常需要額外設計，例如 indirection、lease 或 wrapper object。

設計權限來源主要風險
路徑字串 接收者解析 namespace 時的 ambient authority confused deputy、TOCTOU
檔案描述符 open 時已檢查，之後傳 handle 生命週期與過度授權
Object capability 不可偽造 reference 加 message revocation 與權限衰減設計

Psychology

名字比權限好想，卻也更容易搞錯主體

人習慣說『打開這個檔案』，而不是說『用我授予你的只讀 handle 讀這個物件』。

Ambient authority 太順手

process 一啟動就擁有大量全域權限，短期開發很方便。

字串 API 的誘惑

路徑、URL、device id 容易傳遞，但常把查找和授權綁在一起。

最小權限被延後

先給完整權限最快，等插件或自動化增加後才發現權限面積過大。

安全 API review 應刻意追問：這個操作的 authority 是從哪個參照來的？

Applications

IPC 和沙箱最能看見 capability 的差別

當系統開始讓插件、子程序或遠端元件替彼此工作，權限委派方式就會決定隔離邊界是否清楚。

File descriptor passing

父程序先 open 需要的檔案，再把受限 fd 傳給 sandbox 子程序。

Plugin API

插件只拿到特定 project、device 或 build artifact 的 handle，而不是整個工作目錄權限。

Firmware command token

診斷命令可用短效 token 限定目標設備、操作集合與有效時間。

  • 避免讓低權限元件傳字串要求高權限元件代為解析全域資源。
  • 設計 capability 時同步設計衰減、過期、審計與撤銷路徑。

Misreadings

Capability 不是任何 token 都算

真正重要的是不可偽造、攜帶權限、可受系統保護地傳遞，而不是名稱叫 token 或 handle。

等同 POSIX capabilities 校正

Linux/POSIX capabilities 多是 process privilege bit，不是物件參照權限模型。

有 UUID 就不可偽造 校正

若 token 可猜、可重放或未綁定權限範圍，就不是可靠 capability。

ACL 就完全相反 校正

實務系統可混用；關鍵是哪些操作靠查全域表，哪些靠持有參照。

Questions

這個權限是誰交給誰的

如果收到的是名字

接收者用誰的權限解析它，caller 是否能誘導 deputy 碰到不該碰的物件？

如果收到的是 capability

它能否被複製、轉交、縮小、過期或撤銷，審計能否追到授權鏈？
