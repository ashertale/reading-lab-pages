側信道攻擊

Side-Channel Attack

演算法可以在數學上正確，實作仍可能透過時間、功耗、快取或電磁訊號把秘密漏出去。

Read For

適合看 secure boot、金鑰比較、加密加速器、smart card、TrustZone 邊界與任何秘密資料會碰到實體或微架構效應的地方。

Orientation

秘密不只會從輸出流出，也會從過程流出

Side channel 指攻擊者不直接破解規格，而是觀察實作時附帶產生的訊號，例如耗時、功耗、cache hit 或電磁輻射。

Core Tension

安全設計常證明演算法輸出不洩密；硬體和軟體實作卻會把中間狀態壓到可觀測世界。

01 先找可觀測量

時間、電流、cache、錯誤訊息與 RF/EM 訊號都可能成為線索。

02 再問秘密相關性

若 secret bit 會改變分支或 memory access，訊號就可能可分類。

03 最後估攻擊樣本

單次量測很吵不代表安全；大量樣本可把噪聲平均掉。

Setup

Token 比對早一點返回

函式逐 byte 比對輸入 token 和真值；遇到第一個不相等就 return false。攻擊者量回應時間，就能猜前綴長度。

最小場景

第一個 byte 猜對時，比完全猜錯多跑一次比較；多次量測後，差異可被統計放大。

真正難點

程式的輸出只有 true/false，但 timing side channel 額外提供了接近程度。

Key Point

側信道讓安全邊界從 API 介面擴展到物理與微架構行為。

Lenses

攻擊者讀的是哪一種旁路訊號

不同側信道需要不同防護；把 leakage class 分清楚，才不會用錯藥。

Timing

秘密影響執行時間、重試次數、branch 或記憶體延遲。

Power/EM

資料切換與硬體單元活動會留下功耗或電磁特徵。

Cache

共享微架構狀態讓攻擊者透過 hit/miss 推測 victim access pattern。

Fault-assisted

故意注入 glitch、電壓或時鐘錯誤，觀察錯誤輸出推回內部狀態。

訊號常見場景防護方向
時間 密碼比對、模冪、branch on secret 常數時間、固定路徑
功耗 AES、ECC、smart card 遮罩、隨機化、平衡邏輯
快取 shared CPU、TEE、雲端 co-location 隔離、flush 策略、constant access pattern

Psychology

抽象演算法讓人忘記機器會發熱、耗電、走捷徑

我們習慣把程式想成理想步驟；真正晶片上的資料路徑會留下許多不在規格書輸出欄位裡的痕跡。

單元測試看不見

functional test 只看結果正確，不會告訴你不同 secret 的時間分布是否可分。

效能優化衝突

lookup table、early exit、branch prediction 常讓程式更快，也可能讓 secret 進入可觀測路徑。

噪聲帶來假安全感

示波器上看起來很亂，統計工具仍可能把相關性挖出來。

做安全 review 時，請把攻擊者當成能量測系統行為的人，而不只是能呼叫 API 的人。

Applications

韌體安全要把密鑰操作放到可量測世界裡看

MCU、secure element、boot ROM 與加密加速器都可能在秘密資料運算時留下外部訊號。

Secure boot

簽章驗證、hash 比對與錯誤碼要避免透露哪一步或哪個 byte 接近成功。

Crypto driver

DMA、cache line、table lookup 與 interrupt timing 可能把 key-dependent access 暴露出去。

Production test

治具若能重複觸發加密操作，樣本數可能足以進行功耗分析。

  • 秘密控制流程和秘密控制記憶體位置，都要被標成高風險程式碼。
  • 防護要跟攻擊模型對齊；常數時間不會自動解決功耗或 fault injection。

Misreadings

不是只有密碼學家才需要擔心

任何秘密會影響可觀測行為的地方，都可能形成資訊旁路。

演算法安全就夠 校正

AES、RSA 或 ECC 正確不代表你的實作不會透過 timing 或 power 洩密。

加一點 random delay 就好 校正

隨機延遲可能被平均掉，也可能破壞系統即時性卻沒堵住主要訊號。

攻擊者一定要拆機 校正

有些 timing/cache side channel 可遠端或在同機隔離邊界內利用。
