分散式計算謬誤

Fallacies of Distributed Computing

Orientation

分散式系統最常壞在那些你以為不用寫下來的前提

這組謬誤提醒我們，遠端呼叫不是本地函式；網路、延遲、頻寬、安全、拓撲和管理成本都會進入設計。

Core Tension

分散式架構讓系統可以擴展和解耦，但每切出一個遠端邊界，就多一組不可靠與不可即時知道的狀態。

01 把遠端當不可靠

每個 RPC、message 和同步流程都要有 timeout、重試和失敗語意。

02 把延遲當產品成本

使用者體感、deadline 和資源持有時間都會被遠端邊界改寫。

03 把操作納入設計

版本、拓撲、安全、監控和 ownership 是分散式系統的一部分。

Setup

一個看似簡單的服務切分，讓本地判斷變成多次遠端等待

最小場景是團隊把模組拆成服務，卻仍用本地函式的心智模型使用它。

最小劇情

原本一次本地權限檢查被拆成三個服務呼叫。設計文件只寫流程成功時序，沒有 timeout、partial failure、版本不一致或觀測欄位。

真正難點

服務切分帶來組織和部署彈性，也把網路現實放進每一次判斷。

本地邊界變遠端邊界
→
隱含網路假設未被處理
→
延遲與故障進入使用者路徑
Key Point

分散式謬誤的用途，是把沒有說出口的樂觀前提變成可設計、可測試的條件。

Lenses

逐條審查遠端邊界的假設

每個跨網路互動都要問可靠性、延遲、頻寬、安全、拓撲和操作責任。

可靠與延遲

呼叫失敗、超時、慢回覆和重複回覆時，上游如何處理？

資料與版本

兩端 schema、語意和部署節奏不一致時，是否有兼容策略？

操作與安全

誰監控、誰 rotate secret、誰處理拓撲變更和 incident ownership？

Psychology

人喜歡把遠端依賴想成比較慢的本地函式

這個錯覺讓設計圖保持簡潔，卻把大部分事故條件藏在箭頭裡。

成功路徑太順

白板上畫箭頭時，失敗、延遲和重試通常不會自己出現。

平台抽象太漂亮

SDK 讓遠端呼叫像函式，容易淡化網路和版本差異。

責任邊界模糊

跨團隊服務讓功能容易組合，也讓故障定位和變更協調更難。

分散式設計 review 最值得問的，是每條箭頭在壞掉、變慢、重複或版本不同時怎麼辦。

Applications

把遠端邊界當成故障邊界

微服務、IoT 裝置和跨區資料同步都會觸碰這組謬誤。

微服務

每個同步呼叫都設定 timeout、fallback 和 tracing，不讓一個依賴拖垮整條 request path。

IoT 裝置

裝置要能處理雲端不可達、時鐘偏移和批次補傳，而不是假設網路隨時存在。

跨區同步

把延遲、費用和拓撲變化納入容量與一致性設計。

Misreadings

這組謬誤不是反對分散式系統

它反對的是用單機心智模型設計跨網路系統。

把它當成老派警語 校正

雲平台降低很多成本，但沒有消除延遲、分區、版本和 ownership。

只在大型系統才考慮 校正

兩個服務、一個裝置加雲端，也已經是分散式系統。

用重試處理所有問題 校正

重試會增加流量和重複操作，必須搭配 idempotency、backoff 和 budget。

Questions

把每條遠端箭頭拆開

這條遠端呼叫失敗、變慢或重複時怎麼辦？

分別回答 timeout、retry、fallback、idempotency 和錯誤回傳。

哪個假設沒有被觀測或測試？

網路、延遲、頻寬、拓撲、安全和操作責任都要能被驗證。
