語意化版本

Semantic Versioning

版本號真正承載的不是排隊順序，而是變更承諾；一旦版本號不再對應相容性，整個生態就會開始用猜的。

Orientation

版本號一旦不再對應承諾，生態就會開始靠運氣存活

把 語意化版本 放進真實系統後，最先跳出來的不是定義，而是這個張力：當升版不再清楚表示哪些變更安全、哪些需要遷移時，使用者的真正成本會被推到整合現場。

Core Tension

當升版不再清楚表示哪些變更安全、哪些需要遷移時，使用者的真正成本會被推到整合現場。

01 先把公開契約說清楚

如果連哪些東西算 API 都說不清，版本號自然也不可能誠實。

02 再讓版本號對應真變更

破壞性就是破壞性，不應因為想降低阻力就假裝它只是小修。

03 最後設計遷移節奏

版本號能告訴人風險，但真正的相容性治理仍需要 deprecation 與遷移期。

Setup

先看現場最早卡住的地方

語意化版本要處理的不是數字整齊，而是下游能不能提前讀懂這次升版到底會不會咬人。

最小劇情

某 SDK 將回傳欄位語義改掉卻只升 patch 版，結果下游裝置雖然編譯成功，監控邏輯卻在正式環境悄悄失準。

真正難點

難點不是不會編版本號，而是很多團隊想保留『看起來沒那麼嚇人』的升版表面，於是把變更誠實度犧牲掉。

對外介面發生改變
→
版本號沒有如實表態
→
整合成本被推給下游現場
Key Point

語意化版本最重要的不是格式，而是讓變更成本能在發布時被提前看見，而不是等到整合現場才爆。

Lenses

別讓同一個名詞包掉全部設計判斷

真正出事的常不是程式碼改多少，而是版本號有沒有誠實說出契約究竟動了哪一層。

承諾邊界

版本號承諾的是哪一層：封包格式、API 回傳、CLI 行為，還是整體產品體驗。

破壞性判定

不只編譯錯才算 break；語義改變、預設值改變與時序改變都可能是破壞。

生態信任

一旦使用者學會版本號不可信，他們就會回到 pin 版本、人工測爆與口耳相傳。

判準 當下看起來像什麼 真正代價
誠實升 major 短期看起來比較嚇人 卻能讓下游提早安排遷移與測試
把 break 假裝成 minor 升版阻力表面較小 真正代價被推到現場與事故後面
沒定義公開 API 看起來比較自由 版本號最後很難對外形成可信承諾

Psychology

人和團隊為什麼會往錯的簡化走

大家容易把版本號當行政欄位填完就好，因為真正的遷移成本通常是下游在現場替你付。

害怕嚇跑使用者

很多團隊不想升 major，是因為它象徵麻煩，但問題只會被延後到別人身上。

模糊比較省事

版本號模糊時，內部協調比較少，但外部整合成本會快速升高。

局部正確幻覺

只要自己這邊測試過了，團隊就容易低估語義微調對下游造成的實際傷害。

Applications

放回真實系統後它會改變什麼

回到 SDK、韌體和內部服務發布，semver 的價值其實是降低跨團隊猜測。

SDK 發布

把回傳欄位、錯誤語義與預設值變動都納入版本契約，而不是只盯函式簽名。

韌體相容

若 bootloader 或 config schema 有 break，就該明確暴露遷移需求，不要讓現場猜。

內部服務

即便只服務組織內部，誠實版本號仍能減少跨團隊升版時的隱性摩擦。

  • SDK 發布: 把語義改動視為契約變動，而不只是程式碼差異。
  • 韌體升版: 讓現場能從版本號提早判斷遷移風險。

Misreadings

這個概念最常被怎樣讀歪

若一開始就把 語意化版本 聽成「語意化版本只是一套命名格式」，後面連「若沒有公開契約和遷移紀律，格式本身不會自動產生信任。」這層校正都很難真正進場。

語意化版本只是一套命名格式 校正

若沒有公開契約和遷移紀律，格式本身不會自動產生信任。

只要程式還能跑就不算 break 校正

語義與時序的變動，同樣可能讓下游系統默默壞掉。

內部系統不用在意版本誠實 校正

越是跨團隊依賴多的內部服務，越需要靠版本號降低整合猜測。

Questions

下次做設計前先問這兩句

我們這次變更到底改動了哪一層承諾，版本號有沒有如實說出來

若說不清，升版策略大多半已經開始靠運氣。

如果下游只看 release note 和版本號，不進原始碼，能不能判斷這次升版風險

不能的話，契約通常還不夠清楚。
