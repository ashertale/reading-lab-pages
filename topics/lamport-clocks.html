Lamport Clocks

在沒有共享時鐘的系統裡，事件順序不該先問幾點發生，而要先問誰可能影響了誰。

Orientation

沒有全局現在，也能談先後

Lamport 邏輯時鐘用遞增計數與訊息傳遞規則，捕捉事件之間的 happened-before 關係。

Core Tension

邏輯時鐘能保證因果先後被排序，但反過來不成立：時間戳較小不一定代表物理上較早，也不能分辨所有並發細節。

01 先定義 happened-before

同一 process 內的順序與訊息送收關係，構成分散式事件的因果骨架。

02 再傳遞邏輯時間

發送訊息時帶上 clock，接收方把本地 clock 推到更大，讓因果關係反映在數字上。

03 最後承認並發

沒有因果路徑的事件可能只是並發；不要硬把牆上時間當成真正順序。

Setup

兩個 log 說自己先發生

想像 gateway 和 device 都有本地 clock，兩邊 log 顯示彼此都比對方早送出狀態更新。

最小場景

NTP 漂移、RTC 校準與低功耗喚醒造成 timestamp 不一致；若只用 wall-clock 排序，debug 會把結果讀成不可能的倒因為果。

真正難點

真正需要的是知道哪個事件可能影響另一個事件，而不是誰的本地時間數字比較小。

Key Point

Lamport clocks 讓你在不信任全局時間時，仍能保留足夠的因果排序。

Lenses

三種時間不要混在一起

分散式除錯常失敗在把物理時間、邏輯時間與因果關係混成一種順序。

物理時間

牆上時鐘用來觀察延遲與使用者時間，但會受 drift、sync interval 與 suspend 影響。

邏輯時間

Lamport clock 只保證如果 A happened-before B，clock(A) 會小於 clock(B)。

並發事件

沒有訊息或因果路徑的事件不能靠 Lamport clock 判定真實先後。

更強表示

若要辨識並發與因果細節，可以看 vector clock 等更豐富的表示。

需求 Lamport clock 適合嗎 理由
保持因果順序 適合 訊息送收會推進接收方 clock。
量測真實延遲 不適合單獨使用 邏輯數字不是秒數。
分辨兩事件是否並發 不足 需要 vector clock 或額外資訊。

Psychology

人太相信 timestamp

Timestamp 看起來客觀，卻常只是某台機器對時間的局部看法；分散式系統裡尤其危險。

數字排序很方便

把 log 依 timestamp 排列很自然，但 clock drift 會把故事排錯。

物理直覺太強

人習慣世界有共同現在；系統節點卻可能只共享訊息，不共享精準時間。

因果線索被忽略

request id、sequence、ack 與 message payload 往往比 wall-clock 更能說明事件關係。

Lamport clocks 的心智轉換是：先找訊息路徑，再談事件故事。

Applications

用在跨節點 log 與協定排序

Firmware 與邊緣系統越來越常跨 MCU、gateway、cloud 與測試基礎設施；邏輯時間能讓除錯資料更可讀。

Distributed logging

在跨節點訊息裡帶 sequence 或 logical clock，可以重建 happened-before，而不是只靠本地 timestamp。

狀態同步

多控制器交換狀態時，邏輯時間可避免舊訊息覆蓋新狀態的部分情境。

測試框架

runner、fixture 與 DUT 之間的事件若有邏輯序號，failure triage 會少很多時間排序爭議。

  • 跨節點訊息至少帶 request id、sequence 與本地事件序號，讓因果關係能被重建。
  • 不要用 Lamport clock 量延遲；它能排序因果，不是替代精準時鐘。

Misreadings

邏輯時鐘不是比較準的時鐘

Lamport clocks 的價值是因果排序，不是校準物理時間。

clock 數字大代表真實較晚 校正

不一定。它代表邏輯上吸收了更多事件或訊息關係。

可以判斷所有事件先後 校正

不能。並發事件沒有因果路徑時，Lamport clock 只能給出某種總排序輔助。

不需要 wall-clock 校正

仍需要。物理時間對 timeout、latency 與使用者時間很重要，只是它不該獨自承擔因果分析。
