最小驚訝原則

Principle of Least Astonishment

當介面、命名或行為讓使用者總得停下來猜『它為什麼這樣』，成本不只是在困惑那一刻，而是整體信任與學習曲線一起變差。

Read For

這一頁要把『有創意的設計』和『讓人一直被意外打斷』拆開，因為好介面常不是最花俏，而是最不需要重建心智模型。

Orientation

當每一步都得重新猜規則，再聰明的功能也會變得很重

最小驚訝原則的核心不是消滅一切意外，而是避免系統在不必要的地方打破使用者合理預期。

Core Tension

特立獨行的設計有時能很精巧，但若它違反主流慣例或前後語意不一致，就會把每一次互動都變成額外思考成本。

01 先找使用者已帶進來的預期

命名、按鈕位置與 API 行為最好尊重既有語境，不要每次都自創語法。

02 再追一致性而不只追新奇

局部最漂亮的設計若和整體不合，最終仍會讓人困惑。

03 最後把驚訝留給值得的地方

若真要打破慣例，必須明確換來更高價值，而不是只是作者偏好。

Setup

從一個名字看似合理、實際卻常讓人用錯的介面開始

最小驚訝原則在 API、CLI、設定頁與任何需要預測系統行為的地方都很重要。

最小劇情

某個 `reset()` 函式不是回到預設值，而是清空所有狀態並重啟連線；名稱雖不算錯，卻遠比使用者預期的破壞性更高。

真正難點

真正昂貴的不是第一次驚訝，而是你從此開始不敢再自然相信系統行為。

介面違反既有預期
→
使用者被迫停下來重猜規則
→
信任與流暢度一起下降
Key Point

最小驚訝原則在保護的，不只是易用性，也包括使用者敢不敢把更多操作交給系統。

Lenses

四個看預期對齊的角度

不是所有驚訝都錯，但你至少要知道自己現在打破的是哪一種慣例。

語言慣例

這個名字在同類工具、語言或產品裡通常代表什麼？

區域一致性

同產品內相似元件是否用相同詞彙與相同行為規則？

破壞半徑

如果使用者理解錯了，代價只是多點一次，還是可能造成資料損失或流程中斷？

學習補償

這個非典型設計是否帶來足夠大好處，值得使用者重建預期？

設計選擇 短期看起來像 長期影響
沿用通用慣例 少了些新鮮感 學習成本低、可預測性高
自創語意但未明說 介面很有個性 使用者需要反覆踩雷才能學會
打破慣例但清楚標示 需要額外說明 至少能把驚訝成本控制在入口

Psychology

為什麼讓人驚訝的設計會特別耗能

因為每一次意外都會迫使人停止自動化操作，重新確認系統到底是不是照自己理解在運作。

預測失敗會打斷流

人一旦得停下來驗證，原本流暢的任務就被切斷。

驚訝會傷信任

幾次意外後，使用者會更保守、更常 double check，整體效率跟著下降。

作者視角容易自洽

設計者太熟自己系統時，很容易看不見別人帶進來的既有心智模型。

最小驚訝原則不是迎合所有直覺，而是尊重合理預期，避免不必要的心智摩擦。

Applications

把預期一致性帶回 CLI、API 與產品流程

對工程工具與裝置介面來說，可預測性常比巧妙更值錢，因為它直接影響操作信心。

CLI 指令

若 `--force` 有時只是跳過提示、有時卻連驗證都略過，使用者會很快對整個工具失去直覺。

API 命名

名稱若暗示是純函式，實際卻會產生副作用，就會讓整個 SDK 變得難以安全推理。

設定介面

存檔、套用與重置若每頁語意不一致，使用者很容易在高風險操作上做錯判斷。

  • 當設計選項拉扯不下時，先問哪個版本最符合目標使用者既有預期。
  • 若不得不違反慣例，就要在命名、文件或互動提示上明確補回那份驚訝成本。

Misreadings

三個常見誤讀

最小驚訝原則不是叫你永遠保守，而是要求你對打破預期的代價有自覺。

只要熟了就不算驚訝 校正

熟悉後能適應，不代表初次與間隔使用的成本不大。

創新就不能用這原則 校正

可以創新，但要知道你在換走哪份可預測性。

驚訝只是 UX 問題 校正

在 API 與工具鏈裡，驚訝常直接變成 bug 與事故。

Questions

這個行為對使用者來說，是自然延伸還是突然翻臉

如果我第一次看到這個名稱，會合理預期它做什麼

若與真實副作用差很多，就值得重想。

同一類操作在系統其他地方是否也遵守同樣規則

一致性通常比局部巧妙更重要。

延伸路線
