兩階段提交

Two-Phase Commit

想讓多個參與者一起做『要嘛都做、要嘛都不做』的決定，代價通常不是零；兩階段提交替你換來原子性，卻也同時把等待、阻塞和協調單點帶進系統。

分清楚『想要一次一致』和『系統願意為此付出多少等待與阻塞成本』之間的交易。

Orientation

一致性不是免費午餐，尤其當你想讓很多人同時說 yes 的時候

兩階段提交 真正麻煩的地方，不在名詞，而在它逼你承認：只要你要求多方一起達成單一原子決定，就得把等待、協調、阻塞與恢復難題一起買進來。

Core Tension

只要你要求多方一起達成單一原子決定，就得把等待、協調、阻塞與恢復難題一起買進來。

01 先確認你真的需要原子一致

不是所有跨服務更新都值得用最重的協調換來全有全無。

02 再把阻塞成本算進來

一旦有人卡住，整體等待與資源占用會成為系統成本，而不是邊角例外。

03 最後設計失敗替代方案

有些場景用補償、saga 或明確狀態機，比硬追單次原子性更合適。

Setup

把問題放回一個小型系統

兩階段提交最難的地方不在多寫一次確認，而在你從此得面對誰卡住、誰等待、誰來恢復的整串問題。

最小劇情

訂單、庫存和付款三個服務想保證『要嘛都成功、要嘛都不算』，結果真正的複雜度不是寫三次 update，而是其中一方卡住時大家怎麼一起等、一起回退。

真正難點

難點在於原子語意看起來很乾淨，讓人容易低估協調層本身的脆弱與延遲成本。

→

Key Point

兩階段提交最值得學的不是流程口訣，而是它逼你正視：想要跨多方的一致，通常得拿可用性與簡單度去換。

Lenses

幾條真正有用的工程判準

你要的不只是一起成功，而是一起等待、一起卡住、一起在故障後找回同一個決定。

原子性收益

兩階段提交真正買來的是全有全無的一致保證。

阻塞風險

參與者在等待最終決定時，會帶著鎖、資源或不確定狀態一起卡住。

恢復複雜度

協調者或參與者在中途失敗時，誰知道最終該 commit 還是 abort，本身就是系統問題。

判準 當下看起來像什麼 真正代價
強追單次原子一致 語意最乾淨 阻塞與恢復成本最高
接受補償式一致 語意較沒那麼爽快 卻能降低協調與可用性壓力
忽略等待狀態 設計看起來簡單 實際上最容易在故障時整包卡住

Psychology

為什麼團隊容易在這裡失手

人很容易先被「要嘛都成功、要嘛都不算」的整齊感吸引，晚一點才發現自己買進的是協調成本。

語意潔癖

只要『全有全無』看起來比較乾淨，人就會低估為此付出的分散式成本。

單機類比

很多人會用本機交易直覺去想像跨服務交易，忽略網路與失敗恢復帶來的額外代價。

失敗想像不足

在一切順利時，兩階段提交流程很漂亮；真正難的是中途不順時誰能知道真相。

Applications

把它帶回架構、流程與介面

跨服務交易、工作流和資料系統選型都得先回答：這份原子一致，值不值得把阻塞成本一起買進來。

跨服務交易

在採用兩階段提交前，先問自己是否真的比補償流程更需要單次原子性。

工作流設計

若步驟間已可被清楚重試與補償，硬綁在同一交易裡未必划算。

資料系統選型

對分散式資料庫的一致性承諾，要連阻塞與恢復代價一起理解，而不只看 brochure。

  • 跨服務交易: 先分清楚真需要的原子性等級。
  • 工作流設計: 對阻塞與恢復成本做顯性評估。

Misreadings

別讓這條原則變成口號

若一開始就把 兩階段提交 聽成「所以兩階段提交不好」，後面連「不是。它在需要原子性的場景非常有價值，只是代價不能被浪漫化。」這層校正都很難真正進場。

所以兩階段提交不好 校正

不是。它在需要原子性的場景非常有價值，只是代價不能被浪漫化。

只要網路夠快就沒事 校正

速度能改善一些成本，但阻塞與故障恢復問題不會因此消失。

補償流程一定比較差 校正

很多業務場景本來就更適合明確補償，而不是追求一次性完美原子。

Questions

把它變成下一次重構前的追問

這個場景真的需要單次原子一致，還是其實可以接受可追蹤的補償與中間狀態

先答這題，架構才不會過重。

如果協調者或其中一方在最尷尬的時刻掛掉，其他人要靠什麼知道該 commit 還是 abort

這是理解代價的關鍵問題。
