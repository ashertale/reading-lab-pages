錯誤預算

Error Budget

可靠性目標不是越接近 100% 越好；錯誤預算把允許的失敗量變成產品速度與穩定性的共同語言。

Read For

這頁適合用在 release gate、field reliability、服務 SLO 與 firmware rollout：它會把可靠性討論從情緒拉回可消耗風險。

Orientation

可靠性也有可花用的邊界

錯誤預算把 SLO 留下的不可用空間視為可管理風險：服務可以失敗一點，但不能超過使用者可接受的範圍。

Core Tension

過度追求可靠會犧牲功能速度與成本；過度消耗預算則代表使用者已承受太多失敗，接下來應該優先穩定性。

01 先定 SLO

沒有明確服務目標，錯誤預算就只是事後找藉口。

02 再追 burn rate

預算消耗速度比月底剩多少更早揭露風險。

03 最後連到決策

預算充足時可承擔變更；預算耗盡時應調整發布、修復或降風險。

Setup

是否停止本週 rollout

想像 firmware 分階段推送後，crash rate 和 reconnect error 開始上升，但產品團隊仍希望擴到更多裝置。

最小場景

若已有明確 SLO 與錯誤預算，討論可以變成：目前 burn rate 是否會在觀測窗口內耗盡預算？繼續 rollout 會不會越界？

真正難點

沒有預算語言時，可靠性團隊只會說太危險，產品團隊只會說不能停；雙方缺少共同判準。

Key Point

錯誤預算的用途不是允許亂壞，而是讓速度與穩定性在同一份約束下談判。

Lenses

把 reliability 變成可操作治理

Error budget 只有在接上 SLO、觀測窗口、burn-rate alert 與發布政策後，才會真正改變團隊行為。

SLO 選擇

目標要反映使用者體驗與業務成本；不是每個系統都需要 99.999%。

預算消耗

短時間快速消耗比長期緩慢消耗更需要 immediate response。

發布政策

預算低時可以暫停高風險變更、縮小 rollout 或提高驗證門檻。

投資回饋

長期預算不足表示 reliability work 不是額外項，而是交付能力的一部分。

預算狀態 合理動作 避免
預算充足 允許正常發布與實驗 把可靠性目標當成永遠不能碰的牆。
快速燃燒 縮小 rollout、啟動 incident review 等到月底才結算。
預算耗盡 暫停高風險變更、優先修復 繼續把所有失敗視為個別例外。

Psychology

100% 可靠聽起來道德正確

沒有人想公開支持失敗；錯誤預算的價值是承認零失敗通常昂貴且不一定對使用者最有利。

穩定性被當成否決權

沒有預算時，可靠性常變成誰能喊停；有預算後，喊停需要對照消耗率。

速度被當成冒進

產品變更會消耗風險，但只要仍在預算內，速度本身不是錯。

指標讓責任共享

同一份 budget 讓 product、dev、SRE 與 support 共同面對 tradeoff。

錯誤預算把『該不該冒風險』換成『我們還有多少可承擔風險』。

Applications

用在 firmware rollout 與 field reliability

Firmware 雖不一定是雲服務，也能用 error budget 管理分批推送、crash-free rate、連線成功率與恢復時間。

分階段發布

每個 rollout stage 可綁定 crash rate、rollback rate 與 support ticket 的預算門檻。

Field diagnostics

若某版本快速燃燒錯誤預算，自動縮小曝光並保留更多診斷資料。

Release gate

預算耗盡期間可以只允許修復與低風險變更，避免把不穩定疊上新功能。

  • 替產品化指標選一個使用者感知的 SLO，例如成功連線率、crash-free sessions 或更新成功率。
  • 把 burn rate 接到 rollout 控制，而不是只在週報裡回顧。

Misreadings

錯誤預算不是出事額度

它不是讓團隊故意花掉失敗，而是讓可靠性風險被提前量化與治理。

有預算就可以亂發 校正

錯。預算是風險邊界，發布仍要有監控、回滾與分階段控制。

SLO 越高越好 校正

過高 SLO 可能浪費成本與速度；目標要對齊使用者實際感知。

月底看一次就夠 校正

burn rate 的斜率才是早期警訊；等結算常太晚。
