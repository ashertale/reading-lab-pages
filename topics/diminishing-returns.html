報酬遞減

Diminishing Returns

報酬遞減描述的是額外投入仍有收益，但每多投入一單位帶來的增益越來越小。

Read For

分辨總收益仍上升和邊際收益下降，避免把更多投入誤認成等比例改善。

Orientation

更多不是沒用，而是下一份更多可能不值它的成本

報酬遞減讓我們從總量思考轉向邊際思考：第十個人、第十次優化、第十個功能不一定和第一個一樣有價值。

Core Tension

投入增加通常會改善結果，但系統中有瓶頸、協調成本和飽和點，讓後續投入逐漸變貴。

01 看邊際而非總量

問下一單位投入帶來多少新收益，而不是已投入總共帶來多少。

02 找出限制因素

土地、注意力、架構瓶頸、用戶時間或協調成本可能限制增益。

03 設定停止規則

在投入前先定義邊際收益低於什麼門檻就轉向。

Setup

第一个工程師修掉最大瓶頸，第五個工程師開始彼此等待 review

最小場景是同一目標下的追加投入越來越難產生同等效果。

最小劇情

團隊把更多人投入同一個效能專案。前兩人解掉明顯慢查詢，後續人力開始爭用測試環境、互相同步設計，改善幅度變小。

真正難點

問題不是後來的人沒能力，而是可並行的高價值工作被先拿走，剩下工作受到協調和瓶頸限制。

Key Point

報酬遞減要問下一份投入的回報，而不是靠過去回報外推未來。

Lenses

找到下一單位投入會撞上的限制

邊際收益下降通常不是神祕現象，而是某個限制因素開始主導。

瓶頸位置

目前限制結果的是人力、架構、資料、使用者注意力還是流程？

可並行性

新增投入能獨立產出，還是需要和既有人員高度同步？

替代用途

同一單位資源投入其他方向是否有更高回報？

Psychology

早期成功會讓人高估繼續加碼的效果

人容易把前幾次投入的高回報外推，忘了那些回報常來自先摘低垂果實。

線性直覺

多一倍人力應該多一倍產出，這種想法很自然但常失真。

沉沒投入牽引

已經投入很多後，人更想再加一點把成果拉高。

停止點難說出口

承認邊際變低像是否定前面投入，其實只是承認階段改變。

每次追加資源前，都要重新估下一單位，不要用第一單位的收益說服自己。

Applications

用邊際收益管理加碼

效能、人力、功能和學習都會遇到邊際收益變小。

效能優化

前幾個瓶頸修完後，下一毫秒可能成本很高，應重新比較使用者價值。

人力配置

把更多人塞進同一專案前，先看工作是否可切分。

產品功能

第十個設定選項可能增加複雜度多於新增價值。

Misreadings

報酬遞減不是報酬變負

收益可能仍在增加，只是增加速度下降；是否停止要看邊際成本和替代用途。

以為一遞減就該停 校正

只要邊際收益仍高於替代方案，繼續投入可能合理。

只看平均回報 校正

平均回報可能很好，下一單位卻已不值得。

忽略階段差異 校正

早期探索、成長和成熟期的邊際曲線不同。

Questions

問下一單位還值不值

下一單位投入的預期收益是多少？

不要用累積收益或過去高回報代替邊際估算。

同一資源放到別處會怎樣？

邊際比較必須包含替代用途。

延伸路線

可接著讀 marginal utility，理解額外一單位帶來的主觀或經濟價值如何變化。
