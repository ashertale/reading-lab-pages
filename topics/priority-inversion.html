優先權反轉

Priority Inversion

優先權反轉時，高優先工作不是輸給更高優先工作，而是被低優先工作握住的資源間接擋住。

Read For

理解 priority inheritance、鎖持有時間與中等優先工作如何讓即時系統錯過 deadline。

Orientation

最高優先權也不能越過它需要的鎖

優先權反轉揭露一個現實：排程器看到的是任務優先序，但真正阻塞高優先任務的可能是低優先任務持有的共享資源。

Core Tension

優先權讓重要工作先跑，但共享資源會把任務綁成依賴鏈；如果排程器不知道這條鏈，就會讓中等任務放大阻塞時間。

01 縮短低優先持鎖

低優先任務持有高優先任務需要的鎖時，任何長操作都可能破壞 deadline。

02 使用繼承或天花板協定

讓持有關鍵資源的低優先任務暫時提升，避免被中等任務插隊。

03 把鎖列入 deadline 分析

即時分析不能只看 task runtime，還要看資源等待與最大阻塞時間。

Setup

高優先控制迴圈在等低優先 logger 釋放 mutex

最小場景是低優先任務持有資源，高優先任務等待，而中等優先任務持續搶 CPU。

最小劇情

低優先 logger 持有 SPI mutex 寫入外部 flash；高優先控制任務需要同一條 SPI 讀感測器而被阻塞。此時中等優先通訊任務反覆執行，logger 無法釋放 mutex。

真正難點

排程器照優先權運作，結果卻讓高優先任務間接輸給中等任務。

Key Point

優先權反轉要從資源依賴看，不是只從任務優先級表看。

Lenses

把阻塞時間納入即時性分析

高優先任務的 worst case 不只包含自己的執行時間，也包含等待低優先持有者釋放資源。

持鎖區間

低優先任務在 critical section 裡是否做 I/O、配置記憶體或等待事件？

中等任務干擾

當高優先任務被阻塞時，中等優先任務是否能讓持鎖者長時間跑不到？

協定支援

RTOS mutex 是否啟用 priority inheritance 或 priority ceiling？

Psychology

任務表看起來正確，讓人低估共享資源的暗線

工程師很容易相信優先級設定已表達重要性，但 mutex、bus 和 buffer 會重寫實際等待關係。

設定表給人安全感

看到 control loop 是最高優先，容易忽略它仍可能卡在低優先資源上。

偶發 deadline miss 難定位

優先權反轉依賴特定交錯，log 可能只顯示高優先任務晚了。

中等任務被忽略

真正延長阻塞的常不是持鎖者，而是把持鎖者擠下 CPU 的中等任務。

即時系統 review 要把共享資源畫成依賴圖，而不是只看 task priority。

Applications

讓高優先任務少等不可控資源

RTOS、driver 和服務調度都可能因共享資源出現優先序失真。

RTOS mutex

關鍵 mutex 啟用 priority inheritance，並限制 critical section 最大長度。

Driver 設計

把慢速外設操作拆出高優先路徑，避免控制任務等待 flash 或 console。

服務排程

高優先 request 若依賴低優先背景 job 釋放資源，需設計資源隔離或提升策略。

Misreadings

優先權反轉不是優先級設錯而已

任務優先級可能完全合理；錯的是共享資源讓低優先任務掌握了高優先任務的進展。

把高優先再調高 校正

再高也不能使用被鎖住的資源，除非解決持鎖者跑不到或持鎖太久。

只責怪低優先任務 校正

低優先任務可能只是持有資源，真正放大阻塞的是中等優先任務。

忽略非 mutex 資源 校正

Bus、DMA、buffer 和連線池也可能形成同樣依賴。

Questions

找出高優先任務正在等誰

下面兩問適合用在 RTOS timing review 和 deadline miss debug。

高優先任務會等哪些低優先資源？

列出 mutex、bus、buffer、driver queue 和外部 I/O。

等待時誰能讓持有者跑不到？

檢查中等優先任務是否會延長低優先持鎖時間。
