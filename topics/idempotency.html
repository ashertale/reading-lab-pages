冪等性

Idempotency

當操作可能重送、重試或被使用者重複觸發時，真正寶貴的不是『一定只會發生一次』，而是『多來幾次也不會把世界越改越歪』。

Read For

這一頁要把『送一次成功』和『重送也不會出事』拆開，因為真實系統常活在逾時、重試與不知道對方到底做沒做的世界裡。

Orientation

重試不可怕，可怕的是每重試一次都讓狀態往不同方向多滑一步

冪等性之所以重要，不是因為大家愛重複做事，而是因為真實系統裡你常無法確定前一次到底有沒有成功。

Core Tension

如果失敗時不敢重送，可靠性會很差；如果重送會重複扣款、重複下單或重複寫壞狀態，系統同樣危險。

01 先承認重試一定會發生

timeout、網路中斷與使用者連點都是常態，不是例外。

02 再區分動作與結果

不是所有操作都能天生冪等，但很多可透過 request id、狀態檢查與去重做到結果冪等。

03 最後把不確定收斂成可重入

當你不知道前一次做沒做完，系統最好能安全地再做一次同樣請求。

Setup

從一個回應逾時但你不知道對方有沒有真的做完的請求開始

冪等性最常在你最不想冒險的時候出現：你不敢不重送，也不敢重送。

最小劇情

支付 API 回應 timeout，你不知道訂單是否已建立；若直接重送又沒有去重，使用者可能被扣兩次。

真正難點

問題不在操作本身，而在外界看到的狀態與請求生命周期常不完全同步。

請求結果不明
→
發起重試或重送
→
若無冪等保護就重複產生副作用
Key Point

冪等性是在替不確定網路環境買安全重試能力，而不是追求數學上的漂亮性質。

Lenses

四個看安全重試的角度

不是每個操作都該被強迫冪等，但任何會被重送的操作都應該先想後果。

副作用型態

這個操作是在設定目標狀態，還是在每次都累加或創造新事件？

請求識別

系統能否辨認『這其實是同一件事重來一次』？

狀態查詢

執行前或執行後，呼叫方是否有辦法查到當前真實狀態？

失敗窗口

在回應遺失、部分完成或重複提交時，系統會怎麼收斂？

操作類型 冪等設計較自然 若沒做會怎樣
設定某值 把狀態設成目標值 重送通常問題較小
新增事件或資源 需要 request id 或去重鍵 重送可能創造重複紀錄
累加扣減 需轉成帶識別的交易或對帳流程 最容易在重試時造成財務或數量錯亂

Psychology

為什麼大家常低估冪等性的重要

因為在理想流程裡，每個操作看起來都只做一次；直到真實網路與人類行為把它打碎。

成功路徑太乾淨

只看 happy path 時，很難感受到 timeout 與重送的複雜度。

重複觸發常被當作使用者問題

其實很多重送是系統為了可靠性自己必須做的。

副作用延後才爆

冪等缺失常不是當下壞，而是在之後對帳、退款與事件追查時才痛。

冪等性不是每次都免費，但對需要可靠重試的系統來說，它通常比想像中便宜得多。

Applications

把安全重試帶回 API、任務與設備控制

對後端服務、工作佇列與裝置控制來說，冪等性是把不可靠通道變得可操作的核心設計之一。

REST API

建立訂單、發送任務與更新狀態若有 idempotency key，就比較不怕 client 超時後重送。

工作佇列

consumer crash 後訊息重投是常態，handler 若不冪等，重跑就可能越處理越亂。

裝置命令

某些控制命令若重發會切換兩次，代表它更像 toggle，不適合活在不可靠鏈路裡。

  • 不知道請求結果時，最好的系統不是要求人別重試，而是允許安全重試。
  • 凡是帶副作用的操作，都值得先問『如果它被重送三次，世界會怎麼變』。

Misreadings

三個常見誤讀

冪等性常被講得很抽象，其實它核心只是在處理真實世界的重複與不確定。

只有 HTTP method 才有冪等性 校正

HTTP 只是其中一種表達；更重要的是業務操作本身能否安全重複。

冪等就代表沒有副作用 校正

不是。它可以有副作用，只是重複執行後不應持續累加新結果。

做了 retry 就自然需要冪等 校正

還差一步：你得真的設計識別、去重與狀態收斂。

Questions

如果這個操作被重送三次，我還能接受結果嗎

下面兩個問題很適合放在 API 與工作流設計時。

當我不知道前一次有沒有成功時，系統允許我安全重送嗎

若不行，呼叫方就會被迫在盲區裡賭運氣。

這個操作是設定狀態，還是在創造新事件

答案通常會直接決定冪等策略該怎麼做。
