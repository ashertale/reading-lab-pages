可觀測性

Observability

可觀測性不是收集很多圖表，而是讓你能從系統外部輸出推回內部狀態與失敗原因。

Orientation

看得到數字，不代表推得回原因

可觀測性關心的是從輸出推斷內部狀態的能力；儀表板很多但無法回答為什麼，仍然不可觀測。

Core Tension

系統需要訊號才能診斷，但訊號太少會盲，太多又會淹沒；重點是讓關鍵路徑、上下文和因果鏈可被查詢。

01 從問題倒推訊號

先列事故時要回答的問題，再決定 log、metric、trace 和 event。

02 保留上下文

單點數字不足以診斷，request id、版本、裝置型號、區域和依賴狀態常是關鍵。

03 支援未知問題

可觀測性要能讓人提出新查詢，而不是只能看預先定義好的告警。

Setup

告警說錯誤率上升，卻無法回答是哪個版本、哪條路徑、哪個依賴

最小場景是監控知道壞了，但缺少把壞事切開的維度。

最小劇情

服務 500 error 增加，dashboard 只有全域錯誤率。團隊不知道是新版、特定 region、某個 device model，還是第三方 API 造成，只能逐一猜測。

真正難點

監控提供狀態燈，可觀測性需要提供可追問的診斷路徑。

外部症狀出現
→
訊號缺少上下文
→
診斷只能靠猜測與重試
Key Point

可觀測性的價值，是在事故中縮短從症狀到原因的距離。

Lenses

用事故問題檢查訊號設計

好的訊號要能回答誰受影響、哪裡改變、哪條依賴失效，以及為什麼現在發生。

切分維度

訊號是否能依版本、租戶、裝置、region、feature flag 和依賴切開？

因果鏈

Trace 或 correlation id 是否能串起入口、內部服務、資料庫和外部 API？

資料品質

Log 是否結構化、時間同步、採樣合理，且不洩漏敏感資料？

Psychology

團隊容易把更多資料當成更多理解

資料量增加很容易帶來安全感，但如果沒有查詢路徑和脈絡，事故時仍然只能翻牆式搜尋。

儀表板帶來錯覺

滿屏圖表像控制室，卻可能都只回答已知問題。

告警和診斷被混淆

告警負責叫醒人，可觀測資料負責讓人知道接下來查哪裡。

事後才想欄位

事故發生後才發現缺 version 或 request id，代表訊號設計沒有從 debug 問題倒推。

可觀測性不是儲存所有東西，而是保存足夠回答下一個未知問題的上下文。

Applications

替下一次事故預留提問能力

服務、韌體和資料管線都需要能切分、串接和追問的訊號。

微服務

用 trace id 串起入口和依賴呼叫，並讓 error metric 可依版本和 feature flag 切分。

韌體裝置

上報錯誤時包含 firmware version、硬體 revision、電源狀態和最近操作序列。

資料管線

記錄批次 id、來源 offset、schema version 和轉換階段，讓錯誤資料可追溯。

Misreadings

可觀測性不是 log 越多越好

沒有結構和問題導向的 log 只會增加成本與噪音。

把 monitoring 當 observability 校正

Monitoring 看已知狀態，可觀測性要支援未知問題的探索。

只收集不演練 校正

如果 on-call 不知道怎麼用訊號排查，資料存在也沒用。

忽略隱私和成本 校正

可觀測資料也需要 retention、採樣和敏感資訊控制。

Questions

問訊號能否回答事故問題

下面兩問適合用在服務設計、韌體 telemetry 與 incident review。

如果明天錯誤率上升，我能按哪些維度切開？

版本、區域、租戶、裝置、依賴和 feature flag 是否都在訊號裡？

症狀到根因的鏈路是否能串起來？

Request、job 或 device event 是否有可追蹤的 correlation id？

延伸路線

可接著讀 monitoring，對照已知狀態監控和未知問題診斷的差異。
