集體行動困境

Collective Action Problem

集體行動困境出現於大家都受益於某個共同結果，卻各自有理由少出力、慢出力或等別人先出力。

理解公共利益、個人誘因、搭便車和協調成本如何讓明明值得的事做不起來。

Orientation

每個人都希望結果存在，但不一定希望自己付出成本

集體行動困境不是大家不懂共同利益，而是個人理性和群體理性之間出現落差。

Core Tension

共同成果需要多人投入；但若每個人都能享受成果而不必貢獻，貢獻者就會被懲罰，成果也可能不足。

01 找出受益和付費者

誰享受結果，誰承擔成本，兩者是否分離？

02 降低貢獻摩擦

讓貢獻更小、更明確、更可見，搭便車就比較不會成為默認。

03 設計執行和回饋

只有道德呼籲通常不夠，需要規則、互惠、聲譽或強制機制。

Setup

大家都想要乾淨的 CI，卻都把 flaky test 留給下一個人

最小場景是共同品質由多人共享，但修復成本落在當下某一個人身上。

最小劇情

團隊都抱怨 CI 不可靠，因為 flaky test 讓每次 merge 都變慢。可是每個人手上的 feature 都很急，於是看到偶發失敗就 rerun，沒人停下來修。

真正難點

修好 flaky test 的收益由全隊共享，成本卻由某個人立即承擔。

Key Point

集體行動問題要靠制度把共同利益轉回個人可承擔、可回收的行動。

Lenses

把共同成果拆成貢獻單位

只說大家應該投入太模糊；需要讓貢獻變得具體、可分配、可追蹤。

貢獻粒度

是否能把大公共問題拆成小任務、輪值或明確 owner？

搭便車成本

不貢獻者是否仍享受同樣收益，且沒有任何可見成本？

執行機制

靠自願、互惠、聲譽、預算、排班還是正式規則來維持？

Psychology

人會等別人先證明這件事值得付出

當貢獻成本立即而收益分散，人很容易把延後行動合理化。

責任被稀釋

受益者越多，每個人越容易覺得自己不是關鍵。

公平感影響投入

如果只有少數人一直貢獻，他們很快會降低投入。

短期壓力打敗公共品

公共維護的收益長期才顯現，feature deadline 卻在眼前。

公共品要能活下來，必須把『對大家好』翻譯成『這週誰做哪一步』。

Applications

把公共維護變成可分配工作

團隊品質、平台治理和安全維護都容易卡在集體行動問題。

工程衛生

把 flaky test、dependency update 和文件維護排進輪值或容量預算。

平台治理

讓受益方承擔審核、回報或 moderation 成本，而不是只享受平台流量。

安全維護

共用元件漏洞修復要有明確 owner 和期限，避免每個產品線等別人先升級。

Misreadings

不是大家自私就完了

即使每個人都善意，誘因和成本分配仍可能讓共同成果不足。

只靠道德呼籲 校正

喊大家重視公共品通常不如改變成本和回饋。

把搭便車者妖魔化 校正

很多人只是被眼前任務壓住，制度要讓貢獻可行。

忽略規模變化 校正

小團隊靠默契可行，大團隊需要更明確機制。

Questions

問公共利益如何回到個人行動

誰享受成果，誰付出成本？

如果兩者分離，就需要補償、輪值或規則。

下一個最小貢獻單位是什麼？

把公共問題拆成本週可完成、可追蹤的一步。
