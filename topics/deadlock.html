死結

Deadlock

死結不是系統安靜無事，而是每個參與者都握著別人需要的資源，然後等待別人先放手。

Read For

掌握相互等待、鎖順序、資源持有與超時策略如何決定系統是否能繼續前進。

主題類型
併發問題 / 相互等待
核心問題
Everyone waits, no one can move
主要判準
互斥、持有等待、循環等待、破局策略
讀完產出

Orientation

每個人都在等，代表問題已經從速度變成進展性

死結的核心不是慢，而是沒有任何一方能在現有條件下完成並釋放資源。

Core Tension

鎖和交易保護一致性，但資源取得順序若沒有規則，保護機制本身會讓所有參與者停在原地。

01 固定取得順序

多把鎖或多個資源必須有全域順序，避免 A 等 B、B 等 A。

02 縮短持有時間

不要在持有鎖時做 I/O、等待使用者或呼叫未知延遲的外部服務。

03 提供破局機制

timeout、deadlock detection、交易重試與可回滾操作要預先設計。

Setup

兩個交易各自拿到一半資源，然後都在等另一半

最小場景是每個參與者都持有資源，同時等待對方持有的資源。

最小劇情

交易 A 先鎖住帳戶 X，再等帳戶 Y；交易 B 先鎖住帳戶 Y，再等帳戶 X。兩邊都不會自然完成，也不會自動釋放。

真正難點

單看任一交易都合理；問題出在多個交易交錯後形成循環等待。

各自取得部分資源
→
等待另一方釋放
→
循環等待阻斷進展
Key Point

死結要靠設計破壞必要條件，而不是期待執行緒自己想通。

Lenses

檢查死結四條件是否同時成立

死結通常需要互斥、持有等待、不可搶占與循環等待一起出現。

資源圖

把執行緒、交易、鎖和資源畫成等待圖，循環通常比 log 更誠實。

取得順序

確認所有路徑是否以同樣順序拿鎖或開交易。

釋放策略

等待時是否能超時、回滾、釋放部分資源或讓 scheduler 介入？

Psychology

人容易審查單一路徑，卻漏掉路徑之間的交錯

併發錯誤常藏在組合狀態裡；單看每段 code 都有道理，合在一起才卡死。

局部正確感很強

每段程式都有保護資料的理由，讓人低估鎖之間的全域關係。

重現困難降低警覺

死結可能只在特定時序出現，無法重現時容易被當成偶發。

timeout 被當成補丁

只加 timeout 可能讓系統恢復，但沒有移除循環等待的設計原因。

debug 死結時，先畫等待圖；修死結時，先定取得順序和釋放規則。

Applications

把資源順序寫成接口契約

併發程式、資料庫交易和跨服務 workflow 都可能在不同層形成死結。

Mutex 設計

對多鎖路徑定義全域 lock ordering，並用 assert 或 wrapper 防止反序取得。

資料庫交易

讓交易以穩定順序更新 rows，並對 deadlock retry 設計 idempotent 操作。

跨服務流程

避免服務在等待遠端回覆時持有本地不可釋放資源。

Misreadings

死結不是普通慢查詢

慢查詢終究會完成；死結在沒有破局機制時不會自然前進。

只加 timeout 就算修好 校正

Timeout 是破局，不一定是修設計；仍要找循環等待來源。

只看 CPU 使用率 校正

死結時 CPU 可能很低，低使用率不代表系統健康。

把責任推給資料庫 校正

資料庫能偵測交易死結，但應用層仍要設計重試和資源順序。

Questions

畫出誰握著誰需要的東西

下面兩問適合用在 deadlock debug 和併發設計 review。

是否存在等待循環？

把執行緒和資源畫成圖，確認是不是 A 等 B、B 又等 A。

哪個死結條件可以被破壞？

從鎖順序、持有時間、搶占或 timeout 中選一個真正落地的破局策略。

延伸路線

可接著讀 concurrency，理解多執行單元共享狀態時的基本風險。
