Orientation

非我發明症候群 不像一句立場聲明，它更像討論時的排序器：先把 來源偏見 放回桌面，再決定 控制慾 和 採納阻力 要怎麼分配。

Core Tension

有些團隊不是因為外部方案真的比較差才拒絕，而是因為它不是自己做的、自己控制的、自己命名的。

01 第一步先把 來源偏見 寫清楚

讀 非我發明症候群 的第一步，不是贊成或反對，而是確認誰把 來源偏見 先當真。

02 再把 控制慾 的成本攤開

控制慾 一旦只留在抽象層，這條原則就很容易被拿去替某一方省成本。

03 最後問 採納阻力 怎麼落地

採納阻力 不是附註；它決定這條原則是可操作的判準，還是說完就散的口號。

Setup

哪個前提一旦不寫，非我發明症候群 就會被講歪

這類原則一離開場景就會變口號；放回 協作與採納 裡的實際判斷後，誰要先說明、誰要先承擔、誰要先驗收會立刻分開。

最小劇情

外部函式庫、現成流程或別隊的方案看起來可行，但只因來源不在自己掌控下，就被默默降級或排斥。

Key Point

非我發明症候群 真正幫你做的，不是替某個立場加分，而是把 來源偏見、控制慾 和 採納阻力 排成可檢查的順序。

Lenses

先把 來源影響價值感、控制比效率重要 與 重做帶來身份安全感 分開看

來源影響價值感、控制比效率重要 與 重做帶來身份安全感 看起來都在談 非我發明症候群，其實各自是在回答不同的追問：誰先說明、誰先承擔、誰來收尾。

來源影響價值感

來源影響價值感 這條線在乎的，不是誰比較有氣勢，而是 來源偏見 一旦沒被交代，現場會怎麼開始失真。

控制比效率重要

順著 控制比效率重要 讀，你會發現爭點不在口號，而在 控制慾 是否真的被變成判準。

重做帶來身份安全感

把焦點放在 重做帶來身份安全感 時，你其實是在說：採納阻力 應該先被寫清楚，否則後面的結論都站不穩。

Psychology

現場通常怎麼把 非我發明症候群 讀成順手口號

真正的困難通常不在反對者，而在支持者也常不願意把 控制慾 的代價和 採納阻力 的驗收一併寫出來。

自己做比較安心

自己做比較安心 常不是純粹錯誤，而是一種討論捷徑：先把自己放在合理的一邊，再晚點處理 來源偏見。

外部依賴像風險

碰到 外部依賴像風險 時，最危險的是大家以為原則已經被引用，實際上 控制慾 還沒被具體說明。

採納別人的功勞感太低

採納別人的功勞感太低 會讓人覺得只要把話說得夠完整，就不必再補 採納阻力 的證據或代價。

真正要防的不是有人沒背過 非我發明症候群，而是整個討論都在引用它，卻沒人把 來源偏見 和 控制慾 寫成可檢查條件。

Applications

回到 第三方元件採用、跨團隊共用 與 平台整合

這類原則離開 第三方元件採用 或 跨團隊共用 很容易變格言；一回到實際流程，你就得回答誰先說明、誰先買單、誰來驗證。

第三方元件採用

只要到了 第三方元件採用，你就能看出 來源偏見 究竟是被真的承擔，還是只是被口頭提過。

跨團隊共用

在 跨團隊共用 這種場景，非我發明症候群 幫你的不是贏辯論，而是把 控制慾 先釘在桌面上，避免事後才補。

平台整合

平台整合 最需要 非我發明症候群 的時候，通常就是大家都急著往前走，卻還沒把 採納阻力 寫成共識。

Misreadings

三個常見誤讀

最麻煩的不是有人反對 非我發明症候群，而是很多人自以為同意，卻同意的是不同版本。

非我發明症候群 只是好聽口號 校正

真正難的是把 非我發明症候群 翻成一組可檢查的判準，而不是把它當成結論本身。

只要背了 非我發明症候群 就能直接套用 校正

原則若不落回 來源偏見、控制慾 和代價分配，通常會在現場變成誤用。

非我發明症候群 只適合高層決策 校正

其實從 code review 到 協作與採納 的日常判斷，都會碰到同樣的壓力。

Questions

把 非我發明症候群 追到真的要回應的人

拿這兩個問題去問，你會很快看出討論是在處理 非我發明症候群，還是在借它壓人。

如果 第三方元件採用 的會議只剩十分鐘

如果三件事只能先問一件，你會先逼誰交代 來源偏見、控制慾 或 採納阻力 裡最容易被跳過的那一段？

如果有人直接引用 非我發明症候群

你會要求對方補哪一段：來源偏見 的前提、控制慾 的代價，還是 採納阻力 的驗收方式？
