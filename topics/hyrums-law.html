Hyrum's Law

只要系統有足夠多使用者，任何可被觀察到的行為都可能被某人當成依賴，因此真正的介面常比文件寫的還大。

Orientation

公開 API 之外，還長著一層看不見的相容性表面

Hyrum's Law 抓到的是大型系統演化的現實：只要使用者夠多，行為本身就會變成規格。

Core Tension

工程師以為自己在改內部實作，使用者依賴的卻常是排序、時序、錯誤訊息、邊界值乃至 bug 的具體樣子。

01 先盤點可觀察行為

所有會被下游看見的欄位、順序、時間與例外路徑，都可能已被拿去自動化。

02 再估隱性消費者

真正危險的常不是文件使用者，而是那些沒說話、卻默默依賴現狀的工具與腳本。

03 最後設計退場

想改行為時，要靠 telemetry、版本漸進、fallback 與清楚 deprecation，不要只靠勇氣。

Setup

從一個很普通的重構開始看

海勒姆定律之所以麻煩，是因為它常在你覺得『這根本不算介面』的地方出現。

最小劇情

你把設定檔輸出的 key 順序改成更漂亮的排序，結果下游某支老腳本其實靠原本順序做比對，部署後整批工具失靈。

真正難點

你沒有正式承諾過那個順序，但觀察到的穩定行為已經被現場當成契約的一部分。

行為長期穩定存在
→
使用者默默圍著它建流程
→
看似內部的改動變成破壞性變更
Key Point

Hyrum's Law 不是在反對重構，而是在提醒你真正的介面是『使用者可觀察且可依賴的總和』。

Lenses

四個看隱性契約的角度

當你準備動一個成熟系統時，先看清楚依賴面到底長到哪裡。

文件契約

這是你公開聲明要維持的部分，但它往往只是冰山露出水面的那一角。

行為契約

輸出排列、預設值、延遲曲線與容錯邏輯，經常是使用者真正綁上的東西。

規模效應

使用者越多、接法越多，越不可能只剩你想像中的正規用法。

回收成本

你不是不能改，而是要把變更成本視為產品成本的一部分。

行為 文件怎麼寫 下游可能怎麼依賴
欄位順序 未明說 腳本拿來做 diff、hash 或正則解析
錯誤碼內容 只保證失敗 monitoring 與客服流程靠字串判斷分類
回應時間型態 沒有 SLA 排程器假設某個 timeout 下必定完成
歷史 bug 理論上是錯的 某些客戶端早就把它當 workaround 的基礎

Psychology

為什麼團隊老覺得『這不可能有人在用』

因為作者腦中想的是設計意圖，使用者手上拿的是可觀察結果，兩者本來就不會自然對齊。

設計者偏好意圖視角

你知道什麼是 implementation detail，就容易低估別人根本只能看到黑盒行為。

沉默使用者不會舉手

最危險的依賴者常不是常開會的人，而是多年沒動過的腳本和流程。

重構成就感會遮住相容成本

乾淨實作讓人很有掌控感，但下游承受的是被動遷移的現實。

海勒姆定律提醒你：API 設計不是只寫文件，而是要對『實際被依賴的表面』保持警覺。

Applications

把相容性表面帶回介面治理

從 firmware 狀態機到 CLI 工具輸出，任何可被觀察的穩定行為都值得被當成風險面積來管理。

CLI 與工具鏈

人類看得懂的輸出常被機器默默解析，因此格式、空格與排序都可能成為契約。

韌體狀態與 timing

host 端可能依賴某個 retry 時序、boot wait 或 status code，即使你從未正式承諾。

資料格式演化

新增欄位通常比改語意安全，但就算是新增，也可能讓嚴格 parser 當場崩掉。

  • 在高使用量介面上做變更時，把 telemetry、灰度 rollout 與 compatibility tests 當成設計的一部分。
  • 如果你打算說『這只是內部細節』，先問一句：外部真的完全觀察不到嗎？

Misreadings

三種容易用錯的方式

這條定律不是讓人放棄進化，而是讓演化更像工程，不像賭運氣。

所以什麼都不能改 校正

不是。可以改，但要把兼容策略與遷移成本一起設計。

只要沒寫進文件就不算 breaking change 校正

現場不會因為文件沒寫就停止依賴可觀察行為。

多補文件就能解決 校正

文件有幫助，但對既有依賴者來說，行為驗證比宣告更重要。

Questions

改內部之前，先問外部到底看到了什麼

下面兩個問題非常適合放進 release review 與 API 變更評估。

這個改動真的只在內部嗎

若把輸入、輸出、時序、錯誤與順序都算進來，它還只是內部細節嗎？

誰可能依賴我沒承諾的東西

有沒有老腳本、客戶端、測試治具或 human workflow 是靠現況存活的？
