責任分散

Diffusion of Responsibility

當一件事同時屬於很多人時，它很容易變成實際上誰都沒真的擁有；參與者越多，個人越可能把第一步想成『應該會有人處理』。

Read For

這一頁要把『大家都有責任』和『因此誰都不會先動』拆開，因為很多空窗不是缺能力，而是缺明確的第一責任人。

Orientation

責任一旦被攤太薄，主動性常不是平均分配，而是一起蒸發

責任分散提醒你：共享責任若沒有結構化 ownership，很容易退化成集體旁觀。

Core Tension

多人參與本來能帶來支援與備援，但若誰該先動、誰該收尾不清楚，群體反而更可能出現行動空窗。

01 先把第一步綁到人

越重要的事，越不能只說『大家留意』。

02 再讓行動可見

若沒人知道別人是否已接手，觀望就會自然蔓延。

03 最後分清共享與模糊

共享責任需要清楚交接，不是把責任平均稀釋。

Setup

從一個所有人都在現場、事情卻還是拖著沒動的時刻開始

責任分散在 incident、值班、群組協作與大型會議裡尤其常見。

最小劇情

bug 通知發到大群組，大家都看到了，但因為沒指名 owner，最初十分鐘反而沒人真正接住。

真正難點

人不是故意懶，而是群體情境改寫了每個人對『是不是我該先動』的感受。

多人同時知道一件事
→
個人責任感被稀釋
→
行動空窗出現或拖延放大
Key Point

責任分散最常偽裝成『大家都在關心』，但真正缺的是誰在負責。

Lenses

四個看 ownership 稀釋的角度

只要你看到一件事人人有份、人人模糊，就值得懷疑責任分散正在發生。

指派明確度

是否有清楚 owner，而不是一個模糊的『各位請處理』？

狀態可見性

大家看得出來誰已接手、處理到哪裡嗎？

行動門檻

要開始處理是否需要太多請示或確認？

群體規模

參與者越多、角色越重疊，責任越容易被平均到看不見。

場景 責任分散看起來像 後果
大群組通知 大家都看到了 最初無人真正接手
會議待辦 共識是要做 會後沒有明確 owner 便自然蒸發
值班升級 有很多人在線 每個人都以為別人已經在處理

Psychology

為什麼人一多，主動性反而可能下降

因為當責任不再明確落在自己身上時，觀望就會被合理化成正常等待。

他人存在會降低緊迫感

只要還有很多人也知道，個人就更容易先等等看。

不確定誰該動

角色模糊時，出手本身反而像冒進。

共享關注會被誤讀成共享處理

大家都 aware，不代表大家都在 act。

責任分散不是人變冷漠，而是群體結構改變了責任感的落點。

Applications

把 ownership 補回事件處理與協作流程

對工程團隊來說，責任分散最常讓 incident、待辦和跨團隊依賴在『大家都知道』中悄悄失速。

Incident command

重大事件若沒有明確指揮與分工，很容易一堆人在線卻沒有有效推進。

PR review

指派 reviewers 很多不代表 review 會更快，反而可能人人等別人先看。

跨部門跟進

只把議題留在共同 Slack 頻道，常比指定 owner 更容易掉。

  • 『大家留意』幾乎等於沒指派；關鍵事項最好永遠落到名字而不是群體。
  • 越多人參與的流程，越需要狀態可見與 ownership 轉移規則。

Misreadings

三個常見誤讀

責任分散不是說團隊合作不好，而是說合作需要結構化責任。

人多就一定沒效率 校正

不是。問題在於責任有沒有被清楚配置。

這和偷懶一樣 校正

很多時候是 ownership 設計失敗，不是動機低。

只在旁觀者效應才有 校正

現代團隊協作與線上溝通一樣常見。

Questions

這件事現在是真的有人接住，還是只是很多人都知道

下面兩個問題很適合放在會後與 incident 中。

如果現在要追進度，我能直接說出負責的名字嗎

答不出來時，多半就是責任已被稀釋。

答案會直接影響行動速度。
