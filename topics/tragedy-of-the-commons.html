公地悲劇

Tragedy of the Commons

當每個人都能從共享資源多拿一點，成本卻由所有人分攤，理性的個體行動可能耗盡整個系統。

Orientation

先抓住這題真正的壓力點

公地悲劇的重點不是人壞，而是制度讓每個人拿資源時得到完整好處，付成本時只承擔一小部分。

Core Tension

個體多用一點看起來合理，因為額外收益落在自己身上；但當每個人都這樣做，共享資源就被耗損到大家都不能好好使用。

01 先定義公地

找出哪些資源是多人共用、可被耗損、又缺乏清楚邊界。

02 再看誘因

誰獲得使用收益？誰承擔維護成本？這兩者是否分離？

03 最後設治理

用邊界、配額、監測、共同決策和衝突處理來替代模糊自律。

Setup

把問題縮到最小

最小故事可以是一個共享測試平台：每個團隊都想多占一點時間，最後大家都排不到穩定窗口。

最小劇情

測試設備有限，每個 team 都把自己的 job 排滿，還常常不釋放環境。單一 team 這樣做很合理，因為能提高自己的交付機率。

真正難點

環境被占滿、資料被污染、維護沒人負責後，所有 team 的交付速度都下降，但沒有單一 team 覺得自己是主因。

個體多拿一點共享資源
→
成本被所有使用者分攤
→
資源品質或可用性下降
Key Point

公地悲劇不是靠呼籲大家善良解決，而是靠把使用權、維護責任、監測和制裁設計清楚。

Lenses

四個治理公地的判準

不是所有共享資源都會悲劇化；關鍵在於有沒有能讓使用與維護重新對齊的制度。

邊界清楚

誰可以使用資源、何時使用、使用多少，需要被明確定義。

成本回饋

過度使用的人要看到成本，不然每個人都會把維護當成別人的問題。

可觀測性

若使用狀態不可見，團隊無法分辨正常需求與資源濫用。

共同修規則

被規則影響的人需要能參與調整，否則制度容易不符合現場條件。

共享資源 悲劇化訊號 可能治理方式
測試設備 長期排滿、環境常髒 配額、預約、使用後清理檢查
code review 時間 少數人長期被塞爆 review rotation、owner 分流、PR 大小限制
共用模組 人人改、沒人維護 清楚 owner、變更規則、相容性測試
technical debt budget 每個 feature 都借一點 債務上限、還款 milestone、公開 debt register

Psychology

為什麼大家會一起把公地用壞

在公地裡，個人的局部理性很容易和集體長期利益分離。

成本被稀釋

我多用一點造成的損害很小，但收益很明確，所以很容易合理化。

責任被稀釋

資源壞掉時，每個人都只覺得自己只是眾多使用者之一。

短期壓力壓過長期維護

交付 deadline 很明確，資源退化則比較慢、比較難歸因。

這裡的心理連結是教學性整理：重點是說明制度如何塑造行為，不是把共享資源問題歸咎於個人品格。

Applications

把它帶回工程現場

工程團隊裡的公地常不是草地，而是測試環境、shared code、平台團隊注意力與 release 穩定性。

Shared test lab

若沒有預約、清理與責任回報，測試平台會變成每個人都想用、但越用越不可靠的資源。

Shared ownership code

共用底層模組若沒有 owner 和 contract，feature team 會各自做局部修改，長期破壞整體一致性。

Release risk budget

每個 team 都想塞一個小風險進 release，總體風險最後超過整個系統能承受的範圍。

  • 治理共享資源時，先讓使用量、排隊時間、失敗率與維護成本可見。
  • 不要只設禁令；要讓受影響的人能參與規則調整，規則才會貼近現場。

Misreadings

三個常見誤讀

公地悲劇常被講成「共享一定失敗」，但這會錯過後續治理研究的重要修正。

所以共享資源一定要私有化 校正

不是。Ostrom 的研究顯示，許多社群能用清楚規則和共同治理長期管理共享資源。

只要大家自律就好 校正

自律有用，但高壓和誘因不對齊時，仍需要邊界、監測與可執行規則。

這只適用自然資源 校正

工程裡的測試環境、review 能量、共用模組和穩定性 budget 也有公地特徵。

Questions

把它變成你的治理檢查

下面兩個問題適合用在 shared resource、platform ownership 或 release governance。

這個資源的使用者和維護者是不是同一群人

如果不是，誰在替過度使用付成本？誰有權調整規則？

資源耗損是否被看見

如果使用量、失敗率、清理時間和 queue time 不可見，團隊很難自我校正。
