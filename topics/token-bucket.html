令牌桶

Token Bucket

令牌桶讓系統允許短暫爆發，但把長期平均速率鎖住；它管的不是每一秒都一樣，而是爆發不能無限借未來。

Orientation

允許爆發，但不允許無限透支

令牌桶用固定速率補充 token，請求到來時消耗 token；桶容量決定可累積的短期 burst。

Core Tension

完全禁止 burst 會犧牲延遲與使用體驗；完全放任 burst 會讓下游資源被尖峰打穿。

01 先定補充速率

補充速率代表長期平均允許量，應對齊下游可承受容量。

02 再定桶容量

容量代表可容忍的瞬間爆發，太小會卡正常尖峰，太大會讓下游吃到過大衝擊。

03 最後決定超額行為

token 不足時要 drop、delay、降級或標記，這是產品與可靠性決策。

Setup

錯誤重試把服務打更壞

想像裝置連不上伺服器後立刻密集重試；單台裝置看似合理，整個 fleet 同時重試就會放大故障。

最小場景

每台裝置在失敗後快速 retry，上游剛恢復就被尖峰打滿；若每台裝置有 token bucket，短期可重試，但長期速率受控。

真正難點

單次請求是否合理，不等於群體負載可承受；rate limiter 要管的是總體時間分布。

Key Point

令牌桶把『偶爾快一點』和『長期太多』拆成兩個參數。

Lenses

設計令牌桶的四個旋鈕

一個 token bucket 的行為主要由補充速率、桶容量、消耗單位與超額策略決定。

補充速率 r

代表平均允許工作量；要根據下游吞吐與公平性設定。

桶容量 b

代表可累積信用；越大越能吸收突發，也越可能把尖峰傳給下游。

消耗權重

不同請求可消耗不同 token，例如大封包、昂貴 log 或高成本操作。

超額處理

delay 適合 shaping，drop 適合 protection，mark 適合後續分級處理。

需求 參數傾向 可能副作用
保護下游 低 r、小 b、超額 drop 使用者尖峰體驗較差。
吸收自然 burst 合理 r、中等 b、超額 queue 排隊延遲需受控。
高成本操作 依成本消耗多 token 需要估算權重與公平性。

Psychology

平均值讓尖峰看起來無害

團隊常拿平均 QPS 或平均 log rate 判斷安全；令牌桶逼你同時看 burst 能量。

尖峰短所以被忽略

幾秒鐘的 burst 可能足以耗盡 thread、buffer 或上游 quota。

公平性被單機視角遮住

每台裝置都覺得自己不多，但 fleet 同步後會形成群體尖峰。

排隊看似溫和

把超額請求全部排隊可能只是把 overload 延後，並增加 latency 與記憶體壓力。

Rate limiting 不是只為了拒絕請求，而是讓負載形狀可被下游承受。

Applications

用在 firmware log、重試與 IO 節流

Embedded 系統常有有限 buffer、flash endurance、radio bandwidth 與 cloud quota；令牌桶能把這些限制轉成可調策略。

Log throttling

高頻錯誤可消耗 token，token 不足時只保留摘要，避免 log storm 吃掉 flash 或 uplink。

Retry control

失敗重試使用 token bucket 加 jitter，可避免大規模裝置同時敲上游。

IO scheduling

低優先工作可用較小補充速率，避免背景寫入長期壓縮前景 latency。

  • 為每個 rate limiter 同時寫出平均速率、最大 burst 與超額行為。
  • 若請求成本差異大，讓 token 消耗反映成本，而不是每次都消耗 1。

Misreadings

它不是單純每秒最多 N 次

Token bucket 和固定窗口限制不同；它保留短期信用，也因此需要小心 burst 容量。

桶越大越友善 校正

桶太大會把累積信用一次釋放給下游，造成尖峰。

平均速率足夠就安全 校正

平均安全不代表瞬間安全；buffer、thread 與上游 quota 都會被尖峰影響。

超額一定要 drop 校正

可以 delay、降級、合併或標記，取決於請求是否仍有時效價值。
