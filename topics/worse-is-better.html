更差反而更好

Worse Is Better

一個設計如果夠簡單、夠容易實作、夠容易被帶出去活下來，它有時會比內部更一致、理論更漂亮的設計更快勝出；不是因為它真更好，而是因為它更容易活。

Read For

Orientation

有些系統不是因為最漂亮才贏，而是因為最容易先活下來

這個工程概念會一直被拿出來講，是因為 當簡單性、實作速度與一致性拉扯時，較粗糙但更容易普及的方案常會先取得生態優勢。

Core Tension

當簡單性、實作速度與一致性拉扯時，較粗糙但更容易普及的方案常會先取得生態優勢。

01 先分開設計美感和擴散條件

一個內部更漂亮的系統，不一定比較容易被安裝、理解與接上現場。

02 再看哪些摩擦最決定存活

部署難度、學習成本和相容性，常比理論一致性更早決定誰先活下來。

03 最後決定你要贏哪種戰場

如果目標是普及，就得正視存活條件；如果目標是核心品質，也要承認擴散可能慢。

Setup

從最小可運作場景開始

更差反而更好之所以常讓人不舒服，是因為它逼你承認設計優雅和生態存活不一定走同一條路。

最小劇情

團隊有一個架構更一致的內部工具，但另一個功能比較少、接口比較醜的版本因為更好裝、更好接，反而先被整個組織用起來。

真正難點

難點在於工程師常直覺地把內部美感和外部存活能力綁在一起，現實卻未必如此。

兩種設計各有優劣
→
較簡單可活的版本先被採用
→
生態優勢反過來鞏固原本較粗糙的選擇
Key Point

更差反而更好不是在歌頌粗糙，而是在提醒你：生態採用和內部優雅是兩種不同勝利條件。

Lenses

幾條真正有用的工程判準

有時先活下來的不是最漂亮的設計，而是那個讓別人最少卡關的版本。

簡單性

簡單不只是功能少，也包含實作、理解與接入門檻較低。

一致性

理論上更整齊的設計，未必在現場更容易被使用者完整接住。

生態滾動

一旦某方案先被廣泛採用，周邊工具、文件與習慣也會一起替它加分。

判準 當下看起來像什麼 真正代價
內部設計較優雅 原則很整齊 若導入摩擦高，可能很難先活進生態
功能較少但容易用 看起來沒那麼漂亮 卻可能因採用速度快而先取得地盤
忽略擴散條件 工程上比較舒服 產品與生態上可能輸得很早

Psychology

為什麼團隊容易在這裡失手

人很容易替自己喜歡的設計加上道德分數，於是忽略了別人其實只是在選更容易用的工具。

優雅偏愛

做得漂亮會讓工程師很有成就感，也更容易低估導入端的摩擦。

先活下來效應

一旦某方案先到位，人就會自然替它補文件、補教學、補依賴。

粗糙羞恥

承認『這個比較醜但比較能活』對很多團隊來說情感上很難。

Applications

工程現場最常用到它的地方

工具推廣、API 設計和產品策略都會被這個現象提醒：採用摩擦本身就是競爭條件。

內部工具推廣

先讓最少阻力版本活進日常，再逐步把一致性和完整性往回補。

API 設計

一個容易接上的窄接口，常比大而全但進入成本高的接口更容易長出生態。

產品策略

別只問哪個設計更完美，也要問哪個版本比較可能先被真正用起來。

  • 工具推廣: 別忽視安裝、學習與接入門檻對存活的影響。
  • API 設計: 有時窄而可用比大而全更容易長出生態。

Misreadings

這個概念最常被怎樣讀歪

若一開始就把 更差反而更好 聽成「所以粗糙永遠會贏」，後面連「不是。它只是在提醒擴散條件很重要，不是說品質可以永久被忽略。」這層校正都很難真正進場。

所以粗糙永遠會贏 校正

不是。它只是在提醒擴散條件很重要，不是說品質可以永久被忽略。

只要先活就不用再修 校正

很多粗糙勝出的系統若不持續補課，最終也會被自己的債反咬。

這代表優雅沒有價值 校正

優雅依然重要，只是它未必是最先決定生死的因子。

Questions

把它變成下一次重構前的追問

我們現在比較在乎的是內部一致性，還是讓方案先活進更多使用場景

先說清楚，取捨才不會假裝沒有代價。

若這個設計真的比較漂亮，它卡在現場的最大摩擦是什麼，能不能先拆掉那個摩擦

很多好設計輸的其實不是原理。
