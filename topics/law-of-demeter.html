迪米特法則

Law of Demeter

一個物件或模組若一路穿透別人的內部結構做事，表面上很方便，實際上常是在偷偷擴大耦合面。

Read For

這一頁會把『好像很順手的鏈式存取』和『其實在跨越太多內部邊界』拆開，因為很多脆弱設計都是從知道太多開始。

Orientation

知道太多內部細節的呼叫，看起來方便，改起來最痛

迪米特法則常被簡化成不要寫長鏈，但它更深的意思是：把知識限制在你真正應該知道的近鄰範圍。

Core Tension

直接深入別人的內部結構能少寫一些轉接碼，但也會讓每次內部調整都長出額外的相依與破壞半徑。

01 先問我為何需要知道這麼深

如果必須穿好多層才能完成工作，通常代表責任切分有問題。

02 再把操作拉回擁有者

與其外部穿透內部細節，不如讓真正擁有資料與規則的模組自己提供操作。

03 最後控制變更半徑

邊界越清楚，內部重構時越不會連動傷到整串呼叫者。

Setup

從一段看似普通的鏈式呼叫開始

真正麻煩的不是點很多下，而是這串點擊代表你已經知道了太多不屬於你的結構細節。

最小劇情

某段程式用 `device.config.transport.port.timeout` 直接改值；只要中間任一層重構，外部邏輯就會跟著裂開。

真正難點

短期這樣很快，長期卻讓外部模組對內部樹狀結構產生依賴，封裝形同虛設。

外部知道太多內部結構
→
內部重構就波及外部
→
耦合面越滾越大
Key Point

迪米特法則不是形式主義，而是在提醒你：模組邊界失守時，重構成本會被悄悄外溢。

Lenses

四個看耦合擴張的角度

不必把每個 getter 都視為原罪，但你要能分辨哪些互動正在讓邊界變薄。

知識穿透

呼叫端是否必須知道對方內部的資料結構、排序或子物件關係？

責任錯位

這個操作原本更應該由誰來做：資料擁有者，還是外部協調者？

變更半徑

一個內部命名或結構調整，會不會連動改掉很多不相干的上層邏輯？

抽象品質

若對外介面不足，大家就會自然繞進內部補洞。

做法 當下看起來的好處 之後常見代價
外部一路取到最深欄位 開發快、少寫 wrapper 封裝失效，任何內部改動都容易破外部
把操作收回模組內 需要先想清楚介面 重構自由度更高，邊界更穩
新增中介 service 協調 層次更清楚 若拆太碎也可能增加跳轉成本

Psychology

為什麼工程師很容易一路點進去

因為內部資料就在眼前，直接拿來用會比退一步重想責任分工更省力。

區域最短路徑誘惑

改當前功能時，直接抓到值比設計新介面更快。

封裝成本被延後

邊界問題不會立刻爆炸，所以大家容易把它當未來再說。

熟悉內部會降低警覺

自己很懂那段 code 時，更容易忘記別人之後得一起承擔這層知識穿透。

這個法則真正保護的是改動自由，而不只是程式碼看起來乾不乾淨。

Applications

把邊界意識帶回系統與韌體模組設計

對 firmware 與平台程式來說，很多日後難拆的相依，都是從『先直接穿進去拿』開始。

驅動與平台層

若上層邏輯直接碰到底層暫存器布局，之後換晶片或抽象層就會非常痛。

設定物件

一串深層欄位常意味著設定責任散落各處，誰能保證一致性開始變模糊。

API 設計

若客戶端必須知道 server 內部資源結構才用得順，就代表介面還沒站穩。

  • 看到長鏈式存取時，不只問風格，還要問它暴露了哪些本不該外露的內部知識。
  • 若某個外部模組老是需要拆開別人的物件做事，通常是介面責任還沒切好。

Misreadings

三個常見誤會

迪米特法則很容易被縮成語法禁令，但它真正是邊界管理原則。

只要有兩個點就是錯 校正

不是。重點在於是否跨越不該知道的內部結構。

多包一層一定比較好 校正

如果 wrapper 只是轉述而沒有收攏責任，也可能只是多一層噪音。

這只適用 OOP 校正

任何模組化系統都會遇到『誰知道太多』的問題。

Questions

這段程式是在合作，還是在偷看別人的內部

如果對方內部結構改名或重排，這段邏輯會壞掉嗎

若會，它很可能已經知道了太多。

真正擁有這個規則的人，是否應該自己提供完成操作的介面

若答案是是，就代表責任該往內收。
