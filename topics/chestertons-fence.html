Chesterton's Fence

在拆掉一條規則、一道流程或一段舊設計之前，先弄懂它當初為什麼會被放在那裡。

Orientation

看似礙事的設計，到底在替誰擋風險

切斯特頓圍欄不是要你崇拜傳統，而是提醒你：眼前看似多餘的東西，可能正默默吸收你還沒看見的風險。

Core Tension

當一個規則、流程或程式碼片段看起來很笨重時，我們很自然想把它移除；但如果我們不知道它原本在防什麼，簡化就可能只是把保護機制換成更昂貴的事故。

01 先問它在防什麼

每一道圍欄背後通常都有一個失敗案例、約束條件，或曾經很真實的成本。

02 再問今天還成不成立

理解存在理由，不等於永遠保留。重點是先理解，再判斷它是否已經過時。

03 最後設計替代保護

如果真的要拆，最好同步補上更清楚、更便宜、可驗證的新護欄。

Setup

把問題縮到最小

這個原則可以用一個很小的故事來理解：你看到路中間有一道看似礙事的圍欄，直覺想先拆掉它。

最小劇情

某個 reviewer gate、double-check、延遲等待、互鎖條件或例外處理，讓流程顯得慢又醜。新人看了覺得明顯可以省掉。

真正難點

你眼前看到的是成本，但建立它的人當初看到的可能是事故、誤用、法規、硬體限制，或一種今天已經不明顯的失敗模式。

Key Point

切斯特頓圍欄不是「先別動」，而是「如果連它為何存在都說不出來，你還沒有足夠資訊去安全地動它」。

Lenses

四種看圍欄的透鏡

同一個舊機制可能是歷史包袱，也可能是沒被文件化的保護。先分清楚它屬於哪一類，後續動作才不會混亂。

歷史理由

它可能是對某次真實事故的回應。不了解事故背景時，今天看起來多餘的限制，當年可能是救命措施。

隱形約束

有些東西存在不是因為理論漂亮，而是因為供應商、硬體時序、製程誤差、法規或外部接口真的那樣要求。

局部醜陋 vs. 全局穩定

某段 workaround 也許局部難看，但它可能在整體上壓住了更大的不穩定來源。局部簡潔不一定等於系統更好。

可替代性

真正成熟的拆法不是直接移除，而是先找到更便宜、更清楚、可驗證的新機制來承接舊功能。

Psychology

為什麼人總先看到圍欄的成本

這個原則抓到的是一種很常見的心理傾向：我們比較容易看見眼前成本，不容易看見被它阻止的未發生事故。

新鮮視角的自信

剛接手系統時，很多東西會顯得不合理。我們容易把「我現在看不懂」誤認成「它本來就沒理由」。

結果偏差

如果事故現在沒有發生，我們會低估預防措施的價值，忘記它的成功常常表現為「什麼都沒發生」。

簡化的誘惑

刪掉東西通常比理解脈絡快，也比較有成就感。但系統真正困難的地方，常在那些看似多餘的邊界條件裡。

重點不是指責保守，而是說明人為什麼會系統性低估預防性設計的價值。

Applications

回到 refactor、review 與保護機制

對 firmware、driver、測試流程與跨團隊協作來說，切斯特頓圍欄幾乎天天都在發生。很多看似醜的設計，背後都是某種成本交換。

硬體互鎖與保護條件

某個上電順序、延遲等待或 fallback path 可能很醜，但它可能正在吸收元件公差、供電穩定性或量產條件的不一致。

Legacy code path

一段看似多餘的 compatibility code 可能不是因為沒人整理，而是因為現場仍有老版本裝置、工具鏈或客戶環境依賴它。

流程與 release gate

多一道 sign-off、額外 validation 或 checklist 往往不是為了形式，而是為了把跨部門資訊落差與責任斷點補起來。

  • 當你想刪一個東西時，先追三件事：它原本在防什麼、今天還成不成立、如果移除用什麼替代。
  • 真正好的重構不是把圍欄拆掉，而是把隱藏理由顯性化，然後把保護機制換成更乾淨的版本。

Misreadings

三個常見誤讀

這個概念最容易被誤用成保守口號，但它真正要求的其實是更高品質的理解與變更設計。

所以舊東西都不能動 校正

不是。理解原因之後，很多圍欄其實就可以拆；只是你要先知道自己正在接手哪個功能。

只要是 legacy 就一定有深意 校正

也不是。有些確實只是債。但切斯特頓圍欄要求的是先驗證，而不是先浪漫化 legacy。

理解了理由就必須保留原樣 校正

不需要。理解的價值在於讓你能設計更好的替代方案，而不是永遠綁在舊形式上。

Questions

把它變成你的變更前檢查表

下面兩個問題很適合在 code review、流程重構、或 release 改版前快速過一遍。

如果你今天拆掉它

你能不能具體說出：它原本在防什麼失敗模式？哪些使用者、裝置、情境或跨部門接口可能最先受影響？

如果你認為它已經過時

你有沒有證據證明當年的約束已經消失？你準備用什麼更清楚、成本更低、可驗證的新機制來接手？

延伸路線

延伸閱讀可看 technical debt、normal accidents、safety margins；也可以回到工程管理裡的 change management 與 postmortem culture。
