p-Hacking

當人一邊看資料、一邊改切法、停測點、子群體或指標，只要有足夠多次嘗試，原本普通的雜訊也可能被磨成看似有意義的結果。

Read For

看懂為什麼某些看起來『有數據證明』的結論，其實只是被反覆試到剛好顯著。

主題類型
研究偏差 / 結果篩選

Repeatedly slicing data or stopping tests until significance appears

Orientation

不是每個顯著結果都在說話，有些只是被試到剛好浮上來

p-hacking 的力道，不在口號，而在它逼你看見：只要檢定、切法和停點足夠靈活，資料裡總有機會長出一個看似漂亮的顯著結果，即使背後沒有真效應。

Core Tension

01 先分開探索與驗證

探索可以很自由，但若要宣稱結論成立，就應有事前設計或額外驗證。

02 再讓停測與指標規則前置

很多偽發現就長在『看一眼再改一下』的彈性裡。

03 最後把嘗試次數透明化

看過多少切法、換過多少指標，直接影響結果該被信到什麼程度。

Setup

別先談大詞，先看最小互動怎麼失真

p-hacking 最好先從一個看似普通的分析流程讀起，因為危險往往不是單次造假，而是自由度累加。

最小劇情

團隊做 A/B test，先看整體沒差，又切新客、切老客、改觀察天數、換指標，最後終於找到一個剛好過門檻的子群體結論。

真正難點

難點在於每一步單看都像合理探索，但多步疊起來後，顯著性就不再代表原本以為的那種證據強度。

Key Point

p-hacking 的重點不是禁止探索，而是把探索和驗證分開，別讓雜訊穿上『被證明了』的外衣。

Lenses

幾個會讓結果分岔的判準

一個結果為什麼會顯著，往往不是只看最後那個數字，而是看它背後到底走過多少條路。

多重比較

測越多子群、越多指標，越有機會在純雜訊裡挖到顯著。

停測彈性

邊看邊停會讓結果門檻失去原本假設的意義。

敘事包裝

找到結果後再回頭組故事，會讓偶然顯得像原本就該如此。

制度線索 表面上像什麼 真正風險
事前固定設計 探索空間較少 但結論的證據解讀比較穩
邊看邊改 更容易找到『有趣結果』 卻很可能只是提高假陽性率
探索後再驗證 流程較長 能同時保留發現和可信度

Psychology

為什麼人和制度會往這裡滑

人遇到沒有故事的結果時，很自然就會想再多看一眼、多切一次，直到資料終於給出想要的訊號。

結果飢渴

只要很想看到有差異，人就會自然偏好繼續翻資料直到找到什麼。

局部合理化

每一次切群或延長觀察都看起來不離譜，因而更難感受到總體偏差累積。

顯著崇拜

只要出現顯著星號，很多人就會暫時停止再問設計過程。

這一段要看的，是 結果飢渴、局部合理化 這類反應如何把 p-hacking 指向的制度壓力在現場放大。

Applications

把它帶回管理、政策與組織設計

A/B test、產品分析和研究治理都得防這件事，因為探索一旦假裝成驗證，整個結論就會變形。

A/B test

在看結果前就先寫下主要指標、觀察窗口與停止規則。

產品分析

探索性切群可以做，但要標清楚是探索而不是已確認結論。

研究報告

把試過多少版本和最後挑哪個版本交代清楚，遠比只秀最好看那張圖誠實。

  • A/B test: 先寫停測與主要指標規則。
  • 產品分析: 把探索性發現和驗證性結論清楚分欄。

Misreadings

這個制度概念最常被怎樣誤用

若一開始就把 p-hacking 聽成「所以不能探索資料」，後面連「不是。問題是探索不能直接偽裝成已驗證結論。」這層校正都很難真正進場。

所以不能探索資料 校正

不是。問題是探索不能直接偽裝成已驗證結論。

只要結果顯著就夠了 校正

若設計過程高度彈性，顯著性本身的解讀就會大幅鬆動。

這只是學術界問題 校正

商業分析、產品實驗和內部 dashboard 一樣很容易踩到。
