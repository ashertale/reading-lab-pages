故障半徑

Blast Radius

同樣的錯誤，差別在於它能炸到多遠；故障半徑設計就是讓壞事停在可承受的邊界內。

Orientation

可靠性不只是在防錯，也是在限制錯

故障半徑描述一個錯誤能影響的範圍；好的系統承認錯誤會發生，並預先讓它難以跨越邊界。

Core Tension

追求效率常會共享資源、共用權限與一次性大規模部署；這些選擇會降低成本，也會放大單一錯誤的影響範圍。

01 先畫失效域

按使用者、地區、SKU、版本、資料、權限或硬體資源切出錯誤可停住的邊界。

02 再限制擴散路徑

限流、隔離、fallback、circuit breaker 與 feature flag 都是在縮小錯誤傳播。

03 最後演練恢復

邊界若沒有回滾、觀測與操作手冊，事故時很容易被人工跨過。

Setup

一個壞設定推到所有裝置

想像雲端下發新的 power policy，設定本身有 bug；若沒有分批、版本門檻與快速回滾，所有在線裝置會同時受影響。

最小場景

同一個配置服務、同一把權限與同一個 rollout 開關讓錯誤跨越全部 SKU；support、伺服器與使用者體驗一起被拖進事故。

真正難點

錯誤本身也許很小，真正昂貴的是它沒有被限制在 canary 或單一失效域內。

單點錯誤
→
共享通道放大
→
缺少隔離導致全面影響
Key Point

故障半徑的設計目標是：即使判斷錯了，也只讓一小群系統付學費。

Lenses

從哪些邊界縮小半徑

故障擴散通常沿著共享資源、共用設定、過大權限與無分階段部署流動。

部署半徑

canary、staged rollout 與自動暫停能避免新版本一次碰到所有使用者。

資料半徑

多租戶資料、共享 cache 與全域 migration 需要隔離與回復計畫。

權限半徑

一把過大的 token 或 admin path 會讓單一憑證失誤影響整個系統。

資源半徑

共享 queue、thread pool 或 rate limit 若未隔離，局部 overload 會拖垮其他流量。

半徑來源 縮小方式 代價
一次性 rollout canary、分批、health gate 發布較慢、操作較複雜。
共享資源池 tenant/class isolation 資源利用率可能下降。
全域權限 least privilege、scoped token 權限管理成本增加。

Psychology

小錯讓人忽略大範圍

人常按錯誤本身大小評估風險，卻忘記傳播路徑決定事故規模。

一次推完比較省事

大批次部署讓流程乾淨，卻把所有未知風險集中在同一刻。

共享資源看起來有效率

高利用率常以降低隔離為代價；平常省下的資源可能在事故時放大成本。

手動回滾被高估

事故時資訊不足、壓力高，若回滾不是預先演練的路徑，很容易慢半拍。

故障半徑思考會把『出錯機率』補上一句：出錯後最遠會到哪裡？

Applications

用在 rollout、配置與權限

Firmware 與 device fleet 管理特別需要 blast radius：裝置數量大、環境分散、回收成本高。

Phased rollout

先用 canary 裝置、內部群組、低風險 SKU 或單一地區承接未知風險。

Remote config

設定下發要有 schema validation、scope、kill switch 與版本相容檢查。

Feature flag

flag 不只是開關，也要有目標群組、依賴條件與自動停用規則。

  • 每個高風險變更都寫出最大影響範圍與最短回滾路徑。
  • 把共享資源按重要性或租戶隔離，避免低優先流量吃掉高優先服務。

Misreadings

縮小半徑不是降低品質要求

故障隔離不代表可以少測；它承認測試不能窮盡，讓未知風險有邊界。

有 canary 就不用驗證 校正

錯。Canary 是最後一道現場觀測，不是取代設計與測試。

半徑越小越好 校正

半徑太小會讓發布很慢；要按風險、回滾能力與觀測品質設定。

隔離只屬於雲端 校正

Device fleet、工廠測試、bootloader 更新與配置下發都需要失效域。

Questions

每個變更都問最遠會炸到哪

如果這個版本有致命 bug

最多會影響哪些 SKU、地區、使用者與相依服務？多久能停止擴散？

如果共享資源被打滿

哪些重要工作會一起受影響？是否有隔離 queue、quota 或降級路徑？
