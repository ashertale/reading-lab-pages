機率校準

Calibration

一個模型說 80% 時，長期是否真的約有八成會發生？校準關心的不是排序漂亮，而是信心能不能被拿來決策。

Read For

適合用在 failure prediction、defect risk scoring、測試優先級與任何會把機率交給人或自動流程的模型。

Orientation

分數會排序，不等於機率可信

校準問的是預測機率和長期發生比例是否對得上；它比單次猜中更關心信心刻度是否誠實。

Core Tension

一個模型可以很會把高風險排前面，卻把 60% 說成 90%。若 downstream 拿這個數字做成本決策，過度自信會比單純錯排更麻煩。

01 先分清任務

排序任務看前後順序；機率任務要求數字能和實際比例對齊。

02 再看信心分桶

把 0.7 到 0.8 的樣本放一起，觀察實際事件比例。

03 最後連到決策

若決策使用期望成本，未校準機率會把成本估錯。

Setup

一個太有把握的 defect predictor

模型把某批裝置標成 80% fail risk，產線決定加測；幾週後回看，該區間實際 fail rate 明顯低很多。

最小場景

分數高的確比低分危險，但 80% 桶只有 45% 事件率；排序有用，機率刻度不可信。

真正難點

團隊可能以為模型只是保守，實際上 downstream 已把這個數字轉成加測成本、排程與庫存決策。

模型輸出機率
→
實際比例不匹配
→
成本決策被信心誤差帶偏
Key Point

校準把「模型覺得很可能」改成可審計的長期比例問題。

Lenses

四個檢查信心刻度的方法

校準不只是畫一張圖，而是確認模型輸出能否進入決策算式。

Reliability diagram

把預測機率分桶，對照每桶實際事件率，看線是否貼近對角線。

Proper scoring rule

Brier score 或 log loss 會懲罰錯誤又過度自信的預測。

資料漂移

原本校準的模型，遇到新產品線、新測試條件或供應商變化後可能失準。

分群校準

整體看起來準，某個機種、地區或 firmware branch 仍可能系統性偏高或偏低。

現象可能代表先查
高分桶實際比例偏低 模型過度自信 訓練分布、label delay、後處理校準
低分桶仍常出事 低估尾端風險 漏掉的特徵或分群偏差
排序好但成本差 AUC 與機率校準分離 決策是否使用預測機率本身

Psychology

小數點會讓人以為你懂了

0.83 看起來比「高風險」更科學，但精密呈現不等於可靠校準。

信心外觀

帶小數的機率容易被管理流程當成客觀事實，而不是模型輸出的估計。

單次結果迷思

80% 事件沒發生不代表模型錯；要看同類預測的長期比例。

成功排序的遮蔽

高風險排前面很有價值，但它可能遮住機率刻度偏移。

校準訓練的是團隊對不確定性的語感：不是問它有沒有猜中，而是問信心是否可兌現。

Applications

當機率會進入流程，校準就不是裝飾

只要預測分數會觸發加測、停線、升級處理或自動拒絕，它就需要校準檢查。

Predictive maintenance

若 30% failure risk 會決定是否提前換件，就要確認同分數桶的真實故障比例。

Test prioritization

排序可以只看相對分數；若要估算省下多少風險，機率刻度必須可信。

Risk dashboard

高層看到 90% 風險會做資源決策，模型過度自信會造成反覆救火。

  • 同時報告 discrimination 指標與 calibration 指標，避免一個漂亮排序掩蓋錯誤機率。
  • 每次產品線或資料來源改變後，重新畫分群 reliability diagram。

Misreadings

校準不是把分數調到好看

校準要服務未來決策，而不是事後讓歷史圖表更平滑。

準確率高就校準好 校正

分類正確率只看門檻後結果，校準看門檻前的機率刻度。

校準會讓模型更會排序 校正

校準通常調整機率解讀，不保證 ranking 變好。

整體校準就夠 校正

某些 subgroup 可能嚴重偏差，尤其是不同機種、地區或測試站。
