誘發需求

Induced Demand

誘發需求說的是供給增加會改變使用行為；容量不是只把壅塞清掉，也可能邀請新的需求進場。

Read For

理解道路、服務容量、雲端資源和流程加速如何讓使用者重新安排行為。

主題類型
交通 / 容量與使用

Capacity creates demand

Orientation

把瓶頸變寬，使用者不一定維持原本用法

當道路、API 配額或服務速度改善，原本被壅塞、價格或等待壓住的需求可能被釋放，最後新容量又被消耗掉。

Core Tension

擴容能短期舒緩壓力，卻也降低使用成本；需求曲線移動後，系統可能回到新的壅塞點。

01 看使用成本

時間、金錢、等待和摩擦降低時，需求通常會增加。

02 估計潛在需求

當下排隊只是可見需求，還有被抑制的需求。

03 搭配需求管理

價格、配額、優先級和替代方案可避免容量被無差別填滿。

Setup

API 配額提高後，客戶開始把原本離線批次改成即時呼叫

最小場景是一個技術團隊擴容後，使用者立刻改變工作流。

最小劇情

平台把每分鐘 API 上限提高十倍，原本抱怨限流的客戶先是順暢許多；幾週後，他們把更多內部流程改成即時查詢，尖峰流量再次接近上限。

真正難點

擴容前的需求不是固定數字，而是由成本和限制塑造的行為結果。

容量提高
→
使用成本下降
→
新需求填入
Key Point

誘發需求要求容量規劃同時建模人會怎麼改用系統。

Lenses

把擴容視為改變價格訊號

即使沒有收費，等待時間、失敗率和操作麻煩也是價格。

被壓抑需求

有哪些需求因舊容量限制而沒有表現出來？

替代行為

使用者是否會從其他流程轉移到新容量？

管理手段

是否需要優先級、配額或差別定價來保留關鍵用途？

Psychology

舒適一點後，人很快把新便利當成基本配備

需求不是只來自原本的排隊者，也來自人對新便利的重新適應。

摩擦消失會創造習慣

原本嫌麻煩的行為會變成日常。

短期改善掩蓋長期回填

擴容後前幾週的順暢可能不是新均衡。

免費感降低節制

若使用者看不到容量成本，就不會自主管理。

擴容 PRD 要加一欄：使用者會新增哪些以前不做的行為。

Applications

容量規劃需要需求治理

道路、雲端成本和內部工具都可能在變快後被重新填滿。

雲端資源

提高自動擴縮上限前，先設定成本警示和優先工作負載。

資料平台

查詢變快後，使用者可能增加探索性查詢，需要資源隔離。

城市交通

拓寬道路若不搭配公共運輸與定價，通勤模式會重新調整。

Misreadings

誘發需求不是說永遠不要擴容

擴容可能必要，但不能把需求當成固定不變的外生數字。

把擴容視為無效 校正

短期舒緩和關鍵容量仍有價值。

忽略服務品質 校正

有些需求被誘發，是因為之前服務本來就不合理地差。

只看供給側 校正

需求管理和替代方案同樣重要。

Questions

問新容量會邀請誰改變行為

下面兩問適合用在容量 review 和成本治理。

哪些需求目前被等待、限流或麻煩壓住？

估計擴容後可能被釋放的用量。

我們如何保護關鍵用途不被新需求擠掉？

設計配額、優先級或成本可見性。
