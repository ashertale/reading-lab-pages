路燈效應

Streetlight Effect

人會傾向在最好找、最好量、最好觀察的地方努力，即使真正的問題根本不在那裡。

Read For

這頁要幫你練一個判斷：當分析工作進展很順時，先懷疑自己是不是只在有光的地方搜尋，而不是在問題真正發生的地方搜尋。

Orientation

會發光的地方，不一定就是答案所在

路燈效應的危險不在於偷懶，而在於努力方向被量測便利性悄悄接管，讓錯誤看起來像進度。

Core Tension

你可以非常勤奮地分析資料、調整儀表板、寫更多腳本，但如果搜尋空間從一開始就選錯，勤奮只會更快地遠離答案。

01 先問問題在哪

不要因為某處容易記錄就假設問題也最可能出現在那裡。

02 再問光從哪來

量測方便、工具成熟、日誌完整，這些都可能只是路燈，不是真因所在。

03 最後補黑暗區

真正有價值的下一步，常是設法照亮盲區，而不是在亮處再多找一次。

Setup

把問題縮到最小

這個偏誤很常混進專業工作，因為它常偽裝成『先從有資料的地方開始』這種很合理的流程。

最小劇情

某次系統延遲異常，團隊持續檢查應用層 log 與 dashboard，因為那些資料最好拿；結果真正瓶頸其實在底層 I/O 與網路抖動。

真正難點

有光的地方通常代表工具、資料與熟悉度都在那裡，所以人很容易把可觀測性誤當成因果優先順序。

只在好量的地方找
→
盲區沒有被照亮
→
錯把可見性當成真相
Key Point

路燈效應不是叫你別從容易處下手，而是提醒你不要把『容易找』直接升格成『最值得找』。

Lenses

三個檢查搜尋方向的角度

遇到難題時，可以從問題位置、資料生成方式與工具偏好三面看自己的搜尋是否被路燈綁架。

問題位置

真實故障點可能在你沒 log、沒感測器、沒權限，或跨團隊邊界的地方。

資料生成方式

現在手上的資料反映的是被記錄的世界，不一定是完整發生的世界。

工具舒適區

人會自然偏向自己最熟的觀測面，例如只看 code、只看報表、或只看 UI。

看法 看起來像 真正風險
只看現成資料 分析很快開始 真因留在沒有儀表板的地方。
建立新觀測點 前期變慢 需要成本，但能把黑暗區變成可驗證空間。
跨層確認 溝通更麻煩 多一點摩擦，但能防止局部觀測冒充全貌。

Psychology

為什麼可見性這件事會偷換成重要性

這個偏誤之所以黏人，是因為它會把舒適感、熟悉度與進度感打包成看似專業的忙碌。

有資料就像有掌控感

可視化與 log 讓人覺得自己正在接近答案，即使那些資料只照亮了局部。

陌生區域會被自然迴避

跨 layer、跨裝置或跨團隊的調查成本高，因此人會不自覺把搜尋停在熟悉地帶。

偏差常不是因為懶，而是因為可觀測性會偽裝成合理的工作順序。

Applications

把搜尋成本偏差帶回 debug 與分析

在 debug、研究設計、KPI 制定與測試覆蓋策略裡，路燈效應都很常把有限注意力導到錯地方。

Firmware debug

如果你只看 MCU log，卻從不量電源、時序或匯流排實際波形，就可能一直在亮處打轉。

產品指標設計

若團隊只追蹤容易上報的互動指標，卻沒有能力看到真正重要但難量的失敗體驗，KPI 就會偏向路燈。

  • 每次分析卡住時，先列一張『我們看不到什麼』清單，比再看一次 dashboard 更有價值。
  • 若某個問題總是在熟悉資料裡找不到答案，下一步通常是補觀測，而不是加碼同一種分析。

Misreadings

路燈效應不是叫你別用方便資料

知道有路燈效應後，也不要把所有現成資料都當成可疑；問題在於把它們當成唯一入口。

所以現成資料沒價值 校正

不是。它們常是起點，但不應該自動升格成完整世界。

只要多收資料就能解決 校正

若收集方向仍受路燈支配，你只是在更大規模地記錄同一塊亮區。

Questions

查問題時，先問哪裡只是比較好找

當你覺得分析很努力卻沒進展時，下面兩個問題特別值得先問。

我們是在找真因，還是在找可量到的因

目前所有證據是否都來自同一個容易觀測的面向，以致真正的故障空間根本沒有被碰到？

如果真問題在黑暗區

你需要新增什麼感測、權限、實驗或跨團隊資訊，才能讓盲區真正可檢查？
