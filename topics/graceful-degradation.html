優雅降級

Graceful Degradation

優雅降級的目標不是假裝沒壞，而是在部分能力失效時保留最重要的服務與清楚的使用者語意。

Read For

理解核心功能、非核心功能、降級觸發、使用者提示與恢復路徑如何一起設計。

Orientation

壞掉時仍能提供較小但誠實的服務

優雅降級承認完整功能可能不可用，於是先定義哪些能力可以暫停、替換或變慢，哪些核心承諾必須保住。

Core Tension

完整體驗需要許多依賴，但可用性需要在依賴失效時切小；降級設計就是把全有全無改成分層承諾。

01 排出核心功能

先定義沒有它就不能稱為服務的最小能力。

02 明確降級觸發

依賴錯誤率、延遲、資源水位或安全狀態達到門檻時自動降級。

03 說清使用者語意

降級不是沉默失敗，要讓使用者知道哪些功能暫停、資料可能延遲。

Setup

推薦服務掛了，結帳流程不應一起掛掉

最小場景是非核心依賴失效，如果沒有隔離，核心流程跟著失效。

最小劇情

電商頁面載入時同步等待個人化推薦。推薦服務延遲暴增後，商品頁和結帳按鈕也被卡住，使用者無法完成原本可完成的購買。

真正難點

推薦提升體驗，但在故障時不該和核心交易共享同一個存活條件。

非核心依賴變慢
→
核心路徑同步等待
→
完整頁面一起失效
Key Point

優雅降級要先把功能分層，讓非核心能力失效時不拖走核心承諾。

Lenses

把完整體驗拆成可失效層

每個功能都要有故障時的替代行為，而不是只寫正常情況的整合路徑。

核心與附加

哪個功能必須保留，哪個可隱藏、延後、用靜態值替代？

觸發與退出

什麼指標觸發降級，什麼條件恢復完整功能？

狀態告知

使用者、操作員和監控是否知道目前處於降級模式？

Psychology

團隊容易在正常體驗上精雕細琢，卻晚到事故才設計壞掉畫面

降級流程不像新功能那麼吸引人，但它決定故障時使用者看到的是小範圍限制還是整體崩潰。

完整體驗偏見

設計和測試常以所有依賴可用為前提。

降級被當成失敗承認

團隊可能不願在 UI 或 API 中明示部分不可用，結果讓錯誤更混亂。

恢復路徑被忽略

只知道怎麼進入降級，不知道何時安全退出，會造成長時間功能缺失。

優雅降級是一份故障時的產品規格，不只是 try-catch。

Applications

讓非核心依賴失效時核心仍能走完

服務 UI、韌體控制和資料產品都能透過分層承諾降低故障範圍。

服務頁面

推薦、評論或個人化失敗時，保留商品資訊和購買流程。

韌體控制

非必要感測器失效時切到保守模式，保留安全控制和告警。

資料產品

即時分析不可用時顯示上次更新時間和延遲資料，不讓整頁空白。

Misreadings

優雅降級不是把錯誤藏起來

它要誠實呈現較小能力；隱藏錯誤只會讓使用者和操作員失去判斷。

把降級當靜默 fallback 校正

使用者若需要知道資料延遲或功能停用，就不能假裝正常。

所有功能都保留一點 校正

有些功能在不完整時反而危險，應該明確關閉。

只處理前端畫面 校正

後端依賴、資料一致性和操作 runbook 也要支援降級。

Questions

問故障時保留哪個承諾

下面兩問適合用在 resilience review 和產品故障規格。

完整功能失效時，最小可用服務是什麼？

列出必須保留、可替代、可暫停和必須關閉的功能。

使用者如何知道現在是降級狀態？

定義文案、狀態碼、時間戳和操作限制。

延伸路線

可接著讀 fault tolerance，理解局部故障如何被系統吸收。
