費茲定律

Fitts's Law

越遠、越小的目標越難快速命中；介面不是只看功能位置，也要看手、眼與游標抵達它的成本。

Read For

適合檢查 touch panel、factory tool、debug console 與任何需要頻繁點選的 UI：它把「不好按」翻成可以討論的幾何條件。

Orientation

好按不是感覺，是距離與目標寬度的合約

費茲定律把指向動作看成速度與精度的交換：目標越小、離起點越遠，抵達時間與錯誤機率都會上升。

Core Tension

介面設計常把所有命令排成整齊格子，但使用者的手不會平均使用每個格子；高頻、低風險的動作若放在難命中的位置，整潔版面會變成日常摩擦。

01 先看操作頻率

每天點上百次的命令，應該比季度設定更容易抵達。

02 再看目標容錯

寬一點的熱區、清楚的邊界與安全間距，能降低點錯成本。

03 最後處理危險動作

破壞性命令不一定要難找，但不該與高頻安全動作共享同一條手勢路徑。

Setup

從一顆太小的 Reflash 按鈕開始

想像一個內部燒錄工具，主要流程需要反覆點 Reflash、Read log、Reset device；最常用的按鈕卻小、遠、靠近危險命令。

最小場景

游標每次從 log 區移到右上角小按鈕，旁邊就是清除設定。長時間測試後，手一偏就觸發錯誤命令。

真正難點

這不是單純 UI 醜，而是把高頻操作設計成高精度任務，讓疲勞與趕時間直接變成操作風險。

常用動作離手太遠
→
目標又小又靠近危險項
→
速度一上來就產生錯點
Key Point

費茲定律提醒你：可用性不只在功能有沒有做，而在使用者能否以合理成本穩定命中。

Lenses

四個幾何判準

同一顆按鈕在規格書裡只是命令，在人手裡會變成移動距離、目標寬度與鄰近風險。

移動距離

游標、手指或視線從目前工作區移到目標的距離越長，動作越慢也越容易被中斷。

有效寬度

真正可點的不只是視覺按鈕，而是使用者敢快速命中的安全區域。

操作頻率

高頻命令的幾何成本會被乘上次數，低頻命令則可以保留更多確認步驟。

鄰近後果

小目標旁如果是 destructive action，錯點代價會比單純多花一秒嚴重很多。

介面選擇會改善什麼要小心什麼
放大高頻按鈕 降低瞄準時間與失誤 不要讓危險命令也一起變順手
把常用命令貼近工作區 縮短滑鼠或手指路徑 避免遮住需要觀察的狀態資訊
增加危險命令間距 降低誤觸與連點事故 間距不是替代權限控管

Psychology

手會疲勞，注意力也會省力

人不是每次都慢慢瞄準；高壓、重複與時間壓力會讓操作越來越依賴肌肉記憶。

熟練帶來快，也帶來偏移

熟手會加速，於是小熱區和相鄰危險區的問題更容易露出來。

視覺整齊會遮住手感

平均排列看起來公平，卻忽略不同命令的使用密度。

錯誤常被怪到人身上

若多數人都在同一個角落點錯，問題多半不是訓練不足，而是目標設計不誠實。

費茲定律不是替使用者找藉口，而是要求設計者把人體限制放進規格。

Applications

用在觸控、工具列與硬體旁的軟 UI

韌體工程常會做內部工具、量產面板、touch HMI 或測試 fixture UI；這些地方的操作錯誤很容易被誤認成流程紀律問題。

產線工具

把燒錄、讀狀態、收 log 等高頻安全動作放進大熱區，並讓清除資料與 reset 分離。

觸控產品

戴手套、震動、濕手與小螢幕都會縮小實際可命中的有效寬度。

Debug dashboard

高頻 filter、pause、copy log 應貼近觀察區，不要每次都要求工程師跨越整個畫面。

  • 把操作頻率、錯點代價與起始位置一起標在 UI review 裡，比只討論顏色更有用。
  • 危險命令可以用間距、二次確認與不同手勢處理，而不是只靠紅色文字提醒。

Misreadings

大按鈕不是萬用解

把所有元素都放大，只會把畫面擠壞；費茲定律要求的是按使用情境分配操作成本。

所有東西越大越好 校正

高頻安全動作需要大熱區；低頻設定與危險命令需要清楚但不必搶手。

只適用滑鼠，不適用觸控 校正

手指、觸控筆、旋鈕游標與遙控器焦點移動都會遇到距離與寬度問題。

套公式就能設計 UI 校正

公式能提醒方向，實際仍要測量錯誤率、操作路徑與使用環境。
