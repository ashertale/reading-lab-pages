漏桶演算法

Leaky Bucket

漏桶把突發輸入轉成受控輸出；它不是讓流量消失，而是決定何時排隊、何時丟棄、何時平滑。

Orientation

平均能處理，不代表尖峰能吞下

漏桶用固定或受控速率排出工作，將短時間湧入的 burst 存進有限容量的桶裡。

Core Tension

如果桶夠大，burst 可以被延後處理；如果桶滿了，系統必須明確選擇丟棄、降級或把壓力推回上游。

01 先設 leak rate

漏出速率應對齊 downstream 真正可持續處理能力。

02 再設 bucket size

容量越大越能吸收 burst，但也會增加 worst-case latency。

03 最後定 overflow

滿桶時的行為比正常路徑更重要，因為事故通常從那裡開始。

Setup

事件突然湧入，處理緒平均夠快仍然爆 queue

一個 sensor hub 平常每秒幾十筆 event，干擾時瞬間跳到數千筆；平均處理能力看起來足夠，queue 卻先被尖峰塞滿。

最小場景

上游 burst 在 200 ms 內灌入，consumer 每秒只能穩定處理固定數量。

真正難點

若不限制進桶或出桶速率，短暫尖峰會變成記憶體壓力、延遲尾巴與 watchdog 風險。

Key Point

漏桶的價值在於把不可預測尖峰轉成可規劃的速率與明確的 overflow 決策。

Lenses

桶容量和漏速會共同塑造延遲

漏桶不是一個單純 limiter；它同時是 queue、meter 與 policy boundary。

Leak rate

控制下游每單位時間看到多少工作，保護 CPU、網路或 API。

Bucket depth

吸收 burst 的緩衝量，直接影響最大排隊延遲。

Drop policy

tail drop、priority drop 或 sampling 會保留不同類型的工作。

Token bucket 對照

Token bucket 常允許累積 token 後短暫 burst；leaky bucket queue 更強調平滑輸出。

設計選擇好處代價
大桶 吸收較大 burst 延遲尾巴變長
小桶 快速暴露過載 容易丟掉短暫尖峰
低 leak rate 保護 downstream 使用者等待或資料堆積

Psychology

平均值讓尖峰看起來不存在

很多容量規劃只看平均 QPS 或平均 event rate，卻忽略所有壞事都集中在短時間內。

平均吞吐幻覺

一分鐘 600 筆不等於每秒穩定 10 筆；它可能是第一秒 600 筆。

Buffer 安慰感

加大 queue 會讓丟包變少，但也可能把延遲推到使用者無法接受。

丟棄羞恥

工程師常不想承認需要 drop policy，結果讓系統在滿載時隨機壞。

漏桶設計應該把 overflow 當正常規格，而不是等事故時才補 if statement。

Applications

從 API 到 interrupt storm 都能削峰

只要上游可能比下游快，漏桶就能把 burst 變成可觀察、可控制的壓力。

API rate limit

限制客戶端請求進入服務的速率，必要時回傳 429 或排隊。

Log upload

設備重連後不要一次上傳所有 log，可用漏桶平滑對 server 的壓力。

Firmware event queue

對非關鍵事件做 sampling 或 drop，保留 critical fault path 的資源。

  • 把 bucket occupancy 當作健康訊號；長期高水位代表 leak rate 或上游行為不匹配。
  • 重要事件和低價值 telemetry 不要共用同一個桶，否則低價值 burst 會吃掉關鍵容量。

Misreadings

桶不是無限水庫

漏桶能塑造流量，但不能創造處理能力；滿桶時的選擇會暴露真實優先順序。

有 bucket 就不會掉資料 校正

容量有限；超過時必須 drop、delay 或 backpressure。

漏速越低越安全 校正

過低會累積延遲，甚至讓使用者看到過期資料。

和 token bucket 沒差 校正

兩者常可對照，但 burst allowance 與輸出平滑語意不同。
