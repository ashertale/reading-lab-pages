斷路器模式

Circuit Breaker Pattern

當依賴已經在失敗時，繼續照常重試只會把局部故障放大成系統性拖垮；斷路器的價值就在於承認該暫停了。

Orientation

真正昂貴的，常不是第一次失敗，而是還假裝它沒失敗

斷路器模式 真正麻煩的地方，不在名詞，而在它逼你承認：當依賴已經處於高錯誤或高延遲狀態時，持續灌流量只會把上游資源也一起拖進故障裡。

Core Tension

當依賴已經處於高錯誤或高延遲狀態時，持續灌流量只會把上游資源也一起拖進故障裡。

01 先定義什麼叫依賴不健康

沒有清楚門檻時，團隊通常會把每次失敗都當成再試一次就好。

02 再把隔離當成保護而不是放棄

切斷流量不是承認失敗，而是替其他還能活的能力保留資源。

03 最後設計可控恢復

半開探測的價值，在於讓系統不是永久封鎖，而是有節奏地確認依賴是否回來。

Setup

先看現場最早卡住的地方

斷路器模式最好先從資源被拖死的那一刻看起，因為它要切斷的不是單次錯誤，而是故障蔓延。

最小劇情

裝置管理服務依賴遠端憑證 API；當 API 逾時後，上游還持續同步重試，每個 request 都卡住執行緒池，結果連本來可以處理的本地工作也一起停擺。

真正難點

難點不在切斷呼叫本身，而在團隊往往怕『明明還有一點機會成功』，於是不敢承認該先保全整體。

依賴開始高延遲或高錯誤
→
上游照常灌請求與重試
→
局部故障擴散成全局塞車
Key Point

斷路器模式的關鍵不是悲觀，而是承認：暫時拒絕比持續拖垮整體更負責任。

Lenses

別讓同一個名詞包掉全部設計判斷

這一段會把失敗門檻、半開恢復和降級路徑分開看，免得把斷路器誤當重試參數微調。

擴散成本

這個依賴失敗時，會吃掉多少執行緒、連線、記憶體與等待時間。

恢復節奏

恢復不該靠全量流量猛灌，而要有受控探測與回復條件。

降級策略

被切斷後，系統是明確失敗、回退快取，還是提供較小但可接受的服務。

判準 當下看起來像什麼 真正代價
持續照常重試 看起來沒有放棄任何請求 常把依賴故障擴散成上游全面塞車
及時斷路 部分功能會暫時拒絕 卻能保住其他能力與恢復空間
沒有半開探測 實作比較單純 要嘛永遠封鎖，要嘛一恢復就又被全量流量撞壞

Psychology

人和團隊為什麼會往錯的簡化走

人在依賴失靈時很容易加碼重試，因為「再試一次」看起來總比承認現在該停更積極。

僥倖重試

只要偶爾有成功案例，人就容易高估『再試一次』的價值。

不願顯式失敗

很多團隊寧可拖長等待，也不想讓使用者立刻看到錯誤，結果反而把更大範圍一起拖壞。

局部樂觀

負責單一請求的人會想救這一單，卻較難同時看到整體資源正在被抽乾。

Applications

把它帶回架構、流程與介面

放進 API 依賴、裝置連線和內部服務，斷路器其實是在替整體韌性留生路。

外部 API 依賴

當上游服務持續超時時，斷路器能保住本地執行緒池與快取服務。

裝置側網路資源

對慢或不穩定的遠端端點先限流與斷路，比讓裝置一直卡在等待更健康。

內部微服務

把單點依賴的失敗明確隔離，才能避免全站級 cascade。

  • 外部 API 依賴: 先保住本地資源，不讓遠端故障擴散。
  • 裝置網路行為: 對不穩定端點做斷路與降級，而不是無限等待。

Misreadings

這個概念最常被怎樣讀歪

若一開始就把 斷路器模式 聽成「有重試就夠了」，後面連「沒有斷路條件的重試，常只是更快地把自己也拖進去。」這層校正都很難真正進場。

有重試就夠了 校正

沒有斷路條件的重試，常只是更快地把自己也拖進去。

斷路器代表系統不可靠 校正

相反，它是承認依賴不可靠後，仍努力保住整體可靠性的一種設計。

只要切掉就行 校正

沒有恢復探測與降級策略，斷路器只完成了一半。

Questions

下次做設計前先問這兩句

這個依賴如果連續失敗三十秒，最先被它拖死的是哪種資源

先答出來，才知道斷路門檻該怎麼設。

依賴被切斷後，系統還打算怎麼提供較小但可接受的行為

若沒有答案，斷路只會從拖垮變成失明。
