Raft 共識演算法

Raft Consensus

Raft 用 leader、term 與 replicated log 把共識拆得較容易理解，但它仍要求多數派、穩定日誌與非拜占庭假設。

Orientation

共識先交給一個被多數派承認的 leader

Raft 把伺服器分成 follower、candidate 與 leader；正常情況由 leader 接收命令、複製日誌並在多數派保存後提交。

Core Tension

Leader 讓系統比較好懂，卻也讓正確性集中在選主、term 單調性與日誌匹配規則上。

01 先用 term 分世代

每次選舉進入新 term，舊 leader 的權威會被較新 term 擋下。

02 再用多數派授權

leader 必須拿到 majority vote，提交也依賴 majority replication。

03 最後維持日誌前綴

已提交的 entry 不能被未來 leader 覆蓋，這是 replicated state machine 的核心。

Setup

Leader 寫到一半就消失

三節點叢集要更新 active channel。Leader 把 entry 發給一個 follower 後當機；client 到底能不能把這次寫入視為成功？

最小場景

若 entry 已在多數派保存並被 leader commit，後續 leader 必須保留它；若只在少數節點，可能被覆寫。

真正難點

節點重啟、網路延遲與重複 RPC 都會讓同一段 log 看起來有多個版本。

Follower timeout 發起選舉
→
新 leader 對齊 log
→
多數派保存後提交命令
Key Point

Raft 的安全性不靠大家同時在線，而靠 majority 與 log matching 限制未來能承認什麼。

Lenses

從五個欄位追日誌是否站得住

看 Raft 問題時，先別跳到整個叢集；term、index、commit index、last applied 與 vote record 往往已經說出大半真相。

Term

每個 RPC 都帶 term，讓節點能辨認過期權威。

Election timeout

隨機 timeout 降低 split vote；太短會被網路 jitter 擾亂。

Log matching

若兩份 log 在同 index/term 相同，之前的 prefix 應相同。

Snapshot

日誌不能無限長；快照要保留足夠資訊讓落後節點追上。

情境Raft 依靠工程風險
Leader crash 新 term election 與 majority vote timeout 設太敏感造成頻繁改選
Follower log 落後 AppendEntries 對齊 prefix 大量補 log 影響恢復時間
Cluster membership change joint consensus 或等價過渡 一次改太多節點破壞多數派交集

Psychology

易懂的演算法會讓人想自己寫一版

Raft 的貢獻之一是可理解性；但 production 正確性藏在磁碟、時鐘、網路與 membership change 的邊角。

把 demo 當規格

動畫能說明正常路徑，卻不會替你處理 fsync failure 或 torn write。

過度相信 leader

Leader 是便利的協調點，不是絕對真理；term 與 majority 才是權威來源。

忽略慢節點

一個長期落後的 follower 會牽動 snapshot、catch-up 與維運策略。

Raft review 最怕只討論 election happy path，卻沒有逐項列出持久化與重啟後的不變量。

Applications

控制面的狀態最適合用 replicated log 思考

當系統需要所有節點按同一順序套用命令，Raft 的 log 模型比共享資料庫直覺更清楚。

Metadata store

服務發現、cluster config、lease 狀態可透過 replicated log 保持一致。

Firmware channel

多個管理節點用同一串命令決定哪些 device group 進入哪個 release channel。

Feature flag control

旗標變更若需要強順序，可把開關操作納入共識 log。

  • Raft 處理的是 crash fault，不是節點惡意撒謊；安全通訊與身份驗證仍要另外設計。
  • 先決定哪些狀態真的需要強一致，否則 leader bottleneck 會把所有更新都拉進同一條隊伍。

Misreadings

Raft 好懂，不代表沒有坑

它降低了心智負擔，但不會消除分散式系統的基本成本。

等同拜占庭容錯 校正

Raft 假設節點不惡意偽造訊息；壞節點模型不同時要換工具。

只要三台就永遠可用 校正

三台只能容忍一台失效；網路分區後少數派必須停止提交。

Log replication 就是資料備份 校正

它複製的是命令順序；snapshot、compaction 與 state machine determinism 仍要處理。
