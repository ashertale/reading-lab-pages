Orientation

簡單為什麼有用，但又不是越少越好

奧卡姆剃刀不是「最簡單的一定對」。它只在幾個解釋都能承擔目前證據時，要求你不要額外增加沒有必要的東西。

Core Tension

人想要完整解釋，很容易補上看似聰明的假設；但每多一個假設，就多一個可能錯、難測、難維護的負擔。剃刀的作用是把證據還沒要求的複雜度先削掉。

01 先確認解釋力相近

只有在幾個候選解釋都能說明證據時，簡潔才開始有判準價值。

02 再數必要假設

少假設不是少細節，而是少那些沒有被證據要求的實體、例外與補丁。

03 最後設計測試

好的簡化會讓下一個測試更清楚，而不是把未知藏起來。

Setup

把問題縮到最小

你面前有兩個解釋：一個需要單一失效點，另一個需要三個剛好同時發生的條件。兩者目前都說得通，你先追哪一個？

最小劇情

一台板子偶發重啟。A 假設是某條 power rail 在高負載下 droop；B 假設是 compiler bug、sensor noise 與排程競態剛好疊在一起。

真正難點

A 不一定對，B 也不一定錯。奧卡姆剃刀只說：在證據還沒逼你接受複合故事前，先驗證較少假設、較容易推翻的模型。

多個解釋都能涵蓋現象

→

Key Point

剃刀不是替你下判決，而是替你排序下一步調查：先處理不需要額外假設的路徑，再讓資料決定是否需要更複雜的模型。

Lenses

四種判斷簡潔是否有用的透鏡

真正的簡潔不是短句子，而是用較少的未驗證承諾解釋同樣多的現象。

解釋力

如果簡單模型漏掉關鍵現象，它不是剃刀，而是忽略證據。先確認候選模型都能承擔已知資料。

假設負債

每個額外假設都是一筆未驗證負債。負債越多，模型越容易變成為了保住結論而臨時加補丁。

可測性

好的簡化通常會讓測試更直接。若一個說法簡單但不可測，它比較像故事，不像工程假設。

錯誤代價

在安全、醫療或高風險系統中，較簡單的假設也要配合失敗代價。剃刀不能取代風險控制。

情境 剃刀建議 避免誤用
兩個模型都能解釋資料 先測假設較少者 不要把它當成最終真理。
簡單模型漏掉關鍵例外 保留例外並重新建模 不要為了漂亮而刪證據。
複雜模型不可避免 把每個新增假設拆成可測項 不要讓複雜變成不可審查。
高風險決策 用剃刀排序調查 仍要保留安全邊界與 fallback。

Psychology

為什麼人會把俐落感誤當成真相感

複雜解釋有時比較像專業，尤其在壓力下，人會用更大的故事替不確定感找出口。

複雜感帶來掌控感

一個有很多細節的故事聽起來比較像深入分析，但細節不等於證據，可能只是更多未測假設。

不想承認普通錯誤

真正原因常是接線、版本、時序或配置錯誤。人會偏好稀有原因，因為它比較不像自己的基本檢查失敗。

討論被漂亮架構吸走

設計評審中，複雜架構常比較有舞台感。剃刀提醒你問：這些抽象層真的由需求逼出來嗎？

重點是說明人為什麼會把多假設誤認成深度，而不是把簡潔當成品味問題。

Applications

回到假設數量、debug 與方案比較

對 firmware debug、系統設計與 root cause analysis 來說，奧卡姆剃刀是排序工具：先追最少假設、最容易驗證的路徑。

Debug 假設排序

先檢查版本、配置、電源、時鐘、reset reason、邊界條件。這些原因不酷，但常比罕見 compiler bug 更少假設。

架構設計

抽象層、plugin system、通用框架都可能有價值；但如果需求還沒要求，先用較少 moving parts 的設計保留學習速度。

Root cause 報告

避免把多個未驗證原因堆成一個看似完整的故事。每個原因都要能被證據支持，或被列為待驗證假設。

  • 把「目前證據支持什麼」和「我想像可能發生什麼」分成兩欄。
  • 當複雜模型真的必要，將它拆成可逐一推翻的小假設。

Misreadings

三個常見誤讀

奧卡姆剃刀很常被講成一句漂亮話，真正使用時反而需要很多條件。

最簡單的答案一定對 校正

不是。它只在證據同等時給偏好；新資料出現後，較複雜的模型可能才是正確模型。

越少程式碼越好 校正

不一定。少程式碼若靠隱含前提、魔術常數或不可見耦合達成，假設負債反而更高。

複雜就是錯 校正

複雜有時是現實要求。問題不是複雜，而是沒有證據要求的複雜。
