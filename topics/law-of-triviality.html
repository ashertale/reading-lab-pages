瑣事定律

Law of Triviality

團隊常把最多討論時間花在最容易理解、最容易表態的小事上，反而讓真正高風險的大題目被快速帶過。

Read For

這一頁要幫你看見：會議時間不一定跟議題重要性成比例，往往更跟大家能不能輕鬆發言成比例。

主題類型
組織現象 / 注意力錯配
核心問題
Groups spend more time on easier, smaller issues

Orientation

越容易說話的題目，越容易吃掉整場會議

瑣事定律抓到的是組織注意力的偏斜，不是因為大家愚蠢，而是因為參與門檻不同。

Core Tension

高風險議題往往需要專業、準備與承擔，而油漆顏色、命名與小流程則人人都能出聲；結果時間分配反而顛倒。

01 先看誰容易開口

能被所有人快速理解的議題，最容易長成長討論。

02 再看誰真正承擔風險

真正大的決策常只剩少數人有足夠背景與責任感能深入說。

03 最後用流程校正

若不主動設計時間與決策權重，注意力自然會往瑣事聚集。

Setup

從一場被小事拖走的會議開始

這個定律經典得近乎日常：真正昂貴的問題常被十分鐘帶過，瑣碎細節卻聊半小時。

最小劇情

架構風險、法規依賴與資料遷移只花了十幾分鐘，反而 button 文案與 naming 聊到會議超時。

真正難點

不是小事不重要，而是它們因為人人都懂、人人都能評價，天然比較會搶時間。

議題理解門檻低
→
更多人樂於發言
→
時間被從重要但難懂的題目抽走
Key Point

瑣事定律最實際的用法，是把會議時間當作稀缺資源來保護，而不是任它被可發言性牽引。

Lenses

四個看注意力錯配的角度

想判斷一場討論是不是落進這個模式，可以先看這四個訊號。

低門檻議題過度膨脹

越容易形成個人品味意見，越容易聊不停。

高門檻議題被快速帶過

真正複雜的問題因為少數人懂，反而沒被集體追問。

參與感替代了風險感

大家對能參與的題目更有存在感，未必代表那是更大的風險。

缺乏主持校正

若主持不主動控時與切題，討論自然滑向大家都能講的地方。

議題類型 討論時看起來像 真正風險
命名與樣式 意見很多、很熱烈 容易吃掉核心設計風險的時間
架構與法規依賴 很快點頭帶過 若判斷錯，代價遠高於瑣碎細節
流程小優化 每個人都有感 容易把會議變成零碎偏好交換
高成本決策 只有少數人發言 沉默不代表理解，只代表門檻高

Psychology

為什麼大家都比較想聊油漆，不想聊發電機

因為能懂、能講、能表現存在感的題目，本來就更容易聚集注意力。

可參與感很誘人

對人人看得懂的題目發表意見，能快速產生自己有貢獻的感覺。

複雜題目有挫敗感

若議題需要很多背景，人就更容易沉默或假裝已經理解。

主持若不切題，熱度就會勝出

熱鬧不代表重要，只代表表態成本低。

這條定律不是貶低小事，而是提醒你：時間分配若不被設計，就不會自然接近風險分配。

Applications

把注意力管理帶回會議設計

工程組織裡最常見的版本，是 architecture review 被 bikeshedding，真正的邊界條件反而沒被說清楚。

Design review

UI 細節與命名往往比 failure mode 更容易引發集體討論。

Roadmap 規劃

功能排序聊很久，實際依賴、法規與 migration cost 卻沒有同等時間。

PR 討論

格式與小重構最容易收到大量 comment，關鍵假設與風險可能反而被滑過。

  • 對高風險議題預留保護時段與明確 owner，避免它被會議熱點吃掉。
  • 把可非同步處理的瑣碎題目移出主會議，讓同步時間留給難而重要的判斷。

Misreadings

別把它當成『小事都不重要』

真正的問題不是小事存在，而是小事和大事搶了錯的比例。

所以細節不用管 校正

不是。細節也重要，只是未必要占用最稀缺的同步決策時間。

只要熱烈討論就是壞事 校正

熱烈沒問題，前提是它沒有吃掉對高風險問題的必要注意力。

解法是禁止大家發言 校正

更好的做法是調整議程結構，而不是壓低參與。

Questions

這場會議在解真正昂貴的問題嗎

這兩個問題很適合主持人在中途拉回焦點。

我們現在聊最久的，是最重要的，還是只是最好聊的

若把議題成本與影響排一遍，時間分配還合理嗎？

真正高風險的問題，有沒有被同等程度追問

是不是因為太難、太少人懂，就被草率點頭帶過了？
