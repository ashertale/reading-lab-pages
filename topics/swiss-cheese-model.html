瑞士起司模型

Swiss Cheese Model

事故往往不是單一道防線失效，而是多層防護各自的小漏洞剛好排成一條通道，讓風險一路穿透。

Orientation

真正危險的不是一個洞，而是很多洞剛好排成一直線

瑞士起司模型之所以常被引用，是因為它用很直觀的方式描述多層保護如何一起失守。

Core Tension

單層防線幾乎永遠不完美；真正的安全來自缺陷不會同時對齊，而不是幻想每層都零漏洞。

01 先列防線

先問系統有哪些層次在擋風險，而不是只看最後失敗點。

02 再辨洞的性質

有些漏洞是長期 latent condition，有些是當下 active failure。

03 最後看穿透路徑

事故不是洞存在就會發生，而是多個洞在同一時刻被串起來。

Setup

從一場看似很多人都失手的事故開始

這個模型最有用的地方，是把『連環失誤』翻成『多層保護沒有互相補位』。

最小劇情

需求澄清沒做好、測試治具剛好漏一項、review 沒觸及邊界條件、發布門檻又放寬，結果問題一路通過每一道門。

真正難點

每一層單看都像偶發小缺口，但事故發生在這些缺口剛好排成可穿透的時候。

多層防線各有漏洞
→
漏洞在特定時機對齊
→
風險一路穿透成事故
Key Point

瑞士起司模型最有價值的提醒是：別問哪片起司最爛，先問為什麼洞能連成一線。

Lenses

四個看防線失守的角度

若想用這個模型分析事故，通常要同時看這四件事。

潛在條件

長期存在但不顯眼的制度、資源或設計缺口，平常就把洞挖好了。

活躍失誤

當下的操作失誤、誤判或違規，常是最後被看見的那一層。

防線相依

如果多層保護彼此過度依賴，同一種錯就可能一起穿透。

恢復能力

即使前面漏了，後面是否還有機會檢出、隔離、回滾或降級。

防線層 看起來像 洞可能長在哪
需求與設計 規則寫得有 邊界條件沒被明講或被誤讀
測試與驗證 有 test 就安全 場景覆蓋不夠、治具模型太理想
Review 與流程 gate 有人簽名就算過 注意力被瑣事帶走，關鍵假設沒被追問
發布與監控 可以上就先上 缺少及時偵測與回滾通道

Psychology

為什麼事故後人總想只抓最後那個洞

因為最後一個可見錯誤最容易被指出來，但它通常不是整條穿透路徑的全部。

最後觸發者最顯眼

人會自然把注意力放在事故前最後一個動作或最後一個人身上。

潛在條件較無戲劇性

制度與資源缺口平常安靜存在，事後比較難被直覺當成主角。

單點責任故事較簡單

把事故濃縮成一個人的失誤，比承認防線群整體設計不足更省力。

這個模型不否認個別失誤，而是要求你把個別失誤放回更完整的防線脈絡裡看。

Applications

把多層防線思維帶回 review 與發布

對 firmware、infra 與安全關鍵流程來說，真正的防禦不是一關做滿，而是多關互補。

發布流程

requirements、unit test、integration test、staging 與 monitoring 都是不同層次的起司，不能假設其中一層萬能。

硬體安全互鎖

sensor、firmware、driver、UI 提示與人工檢查若有共同盲區，就可能在極端情境下一起漏。

資安防禦

ACL、驗證、監控與回應程序若共享同一錯誤假設，攻擊就能一次穿透多層。

  • 做事故檢討時，至少列出四層以上防線，逐層找出洞如何連線，而不是只找最後責任人。
  • 最值得修的常不是『把最後一層再拉高』，而是讓不同防線不要共享同一盲點。

Misreadings

別把模型縮成『每層都有洞所以算了』

瑞士起司模型不是宿命論，而是防線設計語言。

既然每層都有洞，事故無可避免 校正

不是。真正目標是避免洞在同一時刻對齊，並提高穿透成本。

只要多加幾層就安全 校正

層數不是重點；若層與層共享同樣盲點，厚度再多也沒用。

這模型只適合飛安 校正

任何多層防禦系統都可以借這種視角檢查自身。

Questions

這場事故到底穿過了哪些起司

如果最後那個人沒失手，前面哪一層其實仍然在漏

這能幫你避免把責任全部壓在最後一個可見動作上。

哪些防線的洞其實來自同一個根因

若多層共享同樣的假設錯誤，補一層通常不夠。
